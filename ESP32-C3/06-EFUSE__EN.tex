\chapter{eFuse Controller (EFUSE)}
\label{mod:efuse}


\section{Overview}

\chipname{} contains a 4096-bit eFuse controller to store parameters. Once an eFuse bit is programmed to 1, it can never be reverted to 0. The eFuse controller programs individual bits of parameters in eFuse according to user configurations. From outside the chip, eFuse data can only be read via the eFuse Controller. If read-protection for some data is not enabled, that data is readable from outside the chip. If read-protection is enabled, that data can not be read from outside the chip. In all cases, however, some keys stored in eFuse can still be used internally by hardware cryptography modules such as RSA\_DS, HMAC, etc., without exposing this data to the outside world.

 \section{Features}

\begin{itemize}
    \item 4096-bit One-time programmable storage
    \item Configurable write protection
    \item Configurable read protection
    \item Various hardware encoding schemes against data corruption
\end{itemize}

\section{Functional Description}

\subsection{Structure}

eFuse data is organized in 11 blocks (BLOCK0 \verb+~+ BLOCK10).

BLOCK0, which holds most parameters, has 9 bits that are readable but useless to users, and 60 further bits are reserved for future use.

Table \ref{tab:efuse-block0-para} lists all the parameters accessible (readable and usable) to users in BLOCK0 and their offsets, bit widths, as well as information on whether their configuration is directly accessible by hardware, and whether they are protected from programming.

The {\bfseries \hyperref[fielddesc:EFUSEWRDIS]{EFUSE\_WR\_DIS}} parameter is used to disable the writing of other parameters, while {\bfseries \hyperref[fielddesc:EFUSERDDIS]{EFUSE\_RD\_DIS}} is used to disable users from reading BLOCK4 \verb+~+ BLOCK10. For more information on these two parameters, please see Section \ref{sec:efuse-wr-dis} and Section \ref{sec:efuse-rd-dis}.

Among these blocks, BLOCK4 \verb+~+ 9 stores KEY0 \verb+~+ 5, respectively. Up to six 256-bit keys can be written into eFuse. Whenever a key is written, its purpose value should also be written (see table \ref{tab:efuse-key-purpose}). For example, when a key for the JTAG function in HMAC Downstream mode is written to KEY\textcolor{myred}{3} (i.e., BLOCK7), its key purpose value 6 should also be written to EFUSE\_KEY\_PURPOSE\_\textcolor{myred}{3}.

\begin{tiplisting}
Do not program the XTS-AES key into the KEY5 block, i.e., BLOCK9. Otherwise, the key may be unreadable. Instead, program it into the preceding blocks, i.e., BLOCK4 \verb+~+ BLOCK8. The last block, BLOCK9, is used to program other keys.
\end{tiplisting}

BLOCK1 \verb+~+ BLOCK10 use the RS coding scheme, so there are some restrictions on writing to these parameters. For more detailed information, please refer to Section \ref{sec:efuse-data-storage} and Section \ref{sec:efuse-software-programming-para}.

\subsubsection{\hyperref[fielddesc:EFUSEWRDIS]{EFUSE\_WR\_DIS}}
\label{sec:efuse-wr-dis}

Parameter \hyperref[fielddesc:EFUSEWRDIS]{EFUSE\_WR\_DIS} determines whether individual eFuse parameters are write-protected. After \hyperref[fielddesc:EFUSEWRDIS]{EFUSE\_WR\_DIS} has been programmed, execute an eFuse read operation so the new values would take effect.

Column ``Write Protection by \hyperref[fielddesc:EFUSEWRDIS]{EFUSE\_WR\_DIS} Bit Number" in Table  \ref{tab:efuse-block0-para} and Table \ref{tab:efuse-block-1-10-para} list the specific bits in \hyperref[fielddesc:EFUSEWRDIS]{EFUSE\_WR\_DIS} that disable writing.

When the write protection bit of a parameter is set to 0, it means that this parameter is not write-protected and can be programmed, unless it has been programmed before.

When the write protection bit of a parameter is set to 1, it means that this parameter is write-protected and none of its bits can be modified, with non-programmed bits always remaining 0 while programmed bits always remain 1.

\subsubsection{\hyperref[fielddesc:EFUSERDDIS]{EFUSE\_RD\_DIS}}
\label{sec:efuse-rd-dis}

Only the eFuse blocks in BLOCK4 \verb+~+ BLOCK10 can be individually read protected to prevent any access from outside the chip, as shown in column ``Read Protection by \hyperref[fielddesc:EFUSERDDIS]{EFUSE\_RD\_DIS} Bit Number" of Table \ref{tab:efuse-block-1-10-para}. After \hyperref[fielddesc:EFUSERDDIS]{EFUSE\_RD\_DIS} has been programmed, execute an eFuse read operation so the new values would take effect.

If the corresponding \hyperref[fielddesc:EFUSERDDIS]{EFUSE\_RD\_DIS} bit is 0, then the eFuse block can be read by users; if the corresponding \hyperref[fielddesc:EFUSERDDIS]{EFUSE\_RD\_DIS} bit is 1, then the parameter controlled by this bit is user protected.

Other parameters that are not in BLOCK4 \verb+~+ BLOCK10 can always be read by users.

When BLOCK4 \verb+~+ BLOCK10 are set to be read-protected, the data in these blocks are not readable by users, but they can still be read by hardware cryptography modules, if the EFUSE\_KEY\_PURPOSE\_\textcolor{myred}{\it{n}} bit is set accordingly.

\subsubsection{Data Storage}
\label{sec:efuse-data-storage}

Internally, eFuses use hardware encoding schemes to protect data from corruption, which are invisible for users.

All BLOCK0 parameters except for \hyperref[fielddesc:EFUSEWRDIS]{EFUSE\_WR\_DIS} are stored with four backups, meaning each bit is stored four times. This backup scheme is not visible to users.

BLOCK1 \verb+~+ BLOCK10 use RS (44, 32) coding scheme that supports up to 6 bytes of automatic error correction. The primitive polynomial of RS (44, 32) is \(p(x)=x^8+x^4+x^3+x^2+1\).

    \centering
    \fbox{\includegraphics[width=1.0\textwidth]{\modulefiles/figures/06-eFuse-shift-reg-cir-1.eps}}
    \caption{Shift Register Circuit (first 32 output)}
    \label{fig:efuse-lfsr-circuit-1}
\end{figure}

\begin{figure}[ht!]
    \centering
    \fbox{\includegraphics[width=1.0\textwidth]{\modulefiles/figures/06-eFuse-shift-reg-cir-2.eps}}
    \caption{Shift Register Circuit (last 12 output)}
    \label{fig:efuse-lfsr-circuit-2}
\end{figure}

The shift register circuit shown in Figure \ref{fig:efuse-lfsr-circuit-1} and \ref{fig:efuse-lfsr-circuit-2} processes 32 data bytes using RS (44, 32). This coding scheme encodes 32 bytes of  data into 44 bytes:

\begin{itemize}
    \item Bytes [0:31] are the data bytes itself
    \item Bytes [32:43] are the encoded parity bytes stored in 8-bit flip-flops DFF1, DFF2, ..., DFF12 (gf\_mul\_n, where n is an integer, is the result of multiplying a byte of data ...)
\end{itemize}

After that, the hardware burns into eFuse the 44-byte codeword consisting of the data bytes followed by the parity bytes.

When the eFuse block is read back, the eFuse controller automatically decodes the codeword  and applies error correction if needed.

Because the RS check codes are generated on the entire 256-bit eFuse block, each block can only be written once.

\subsection{Programming of Parameters}
\label{sec:efuse-software-programming-para}

The eFuse controller can only program eFuse parameters in one block at a time. BLOCK0 \verb+~+ BLOCK10 share the same address range to store the parameters to be programmed. Configure parameter \hyperref[fielddesc:EFUSEBLKNUM]{EFUSE\_BLK\_NUM} to indicate which block should be programmed.

{\bfseries Programming BLOCK0}

When \hyperref[fielddesc:EFUSEBLKNUM]{EFUSE\_BLK\_NUM} is set to 0, BLOCK0 will be programmed. Register \hyperref[regdesc:EFUSEPGMDATA0REG]{EFUSE\_PGM\_DATA0\_REG} stores \hyperref[fielddesc:EFUSEWRDIS]{EFUSE\_WR\_DIS}. Registers \hyperref[regdesc:EFUSEPGMDATA1REG]{EFUSE\_PGM\_DATA1\_REG} \verb+~+ \hyperref[regdesc:EFUSEPGMDATA5REG]{EFUSE\_PGM\_DATA5\_REG} store the information of parameters to be programmed. Note that 9 BLOCK0 bits are readable but useless to users and must always be set to 0 in the programming registers. The specific bits are:

\begin{itemize}
    \item \hyperref[regdesc:EFUSEPGMDATA1REG]{EFUSE\_PGM\_DATA1\_REG}[24:21]
    \item \hyperref[regdesc:EFUSEPGMDATA1REG]{EFUSE\_PGM\_DATA1\_REG}[31:27]
\end{itemize}

Data in registers \hyperref[regdesc:EFUSEPGMDATA6REG]{EFUSE\_PGM\_DATA6\_REG} \verb+~+ \hyperref[regdesc:EFUSEPGMDATA7REG]{EFUSE\_PGM\_DATA7\_REG} and \hyperref[regdesc:EFUSEPGMCHECKVALUE0REG]{EFUSE\_PGM\_CHECK\_VALUE0\_REG} \verb+~+ \hyperref[regdesc:EFUSEPGMCHECKVALUE2REG]{EFUSE\_PGM\_CHECK\_VALUE2\_REG} are ignored when programming BLOCK0.

{\bfseries Programming BLOCK1}

When \hyperref[fielddesc:EFUSEBLKNUM]{EFUSE\_BLK\_NUM} is set to 1, registers \hyperref[regdesc:EFUSEPGMDATA0REG]{EFUSE\_PGM\_DATA0\_REG} \verb+~+ \hyperref[regdesc:EFUSEPGMDATA5REG]{EFUSE\_PGM\_DATA5\_REG} store the BLOCK1 parameters to be programmed. Registers \hyperref[regdesc:EFUSEPGMCHECKVALUE0REG]{EFUSE\_PGM\_CHECK\_VALUE0\_REG} \verb+~+ \hyperref[regdesc:EFUSEPGMDATA2REG]{EFUSE\_PGM\_DATA2\_REG} store the corresponding RS check codes. Data in registers \hyperref[regdesc:EFUSEPGMDATA6REG]{EFUSE\_PGM\_DATA6\_REG} \verb+~+ \hyperref[regdesc:EFUSEPGMDATA7REG]{EFUSE\_PGM\_DATA7\_REG} are ignored when programming BLOCK1, and the RS check codes will be calculated with these bits all treated as 0.

{\bfseries Programming BLOCK2 \verb+~+ 10}

When \hyperref[fielddesc:EFUSEBLKNUM]{EFUSE\_BLK\_NUM} is set to 2 \verb+~+ 10, registers \\ \hyperref[regdesc:EFUSEPGMDATA0REG]{EFUSE\_PGM\_DATA0\_REG} \verb+~+ \hyperref[regdesc:EFUSEPGMDATA7REG]{EFUSE\_PGM\_DATA7\_REG} store the parameters to be programmed to this block. Registers \hyperref[regdesc:EFUSEPGMCHECKVALUE0REG]{EFUSE\_PGM\_CHECK\_VALUE0\_REG} \verb+~+ \hyperref[regdesc:EFUSEPGMCHECKVALUE2REG]{EFUSE\_PGM\_CHECK\_VALUE2\_REG} store the corresponding RS check codes.

{\bfseries Programming process}

The process of programming parameters is as follows:

\begin{enumerate}
\item Configure the value of parameter \hyperref[fielddesc:EFUSEBLKNUM]{EFUSE\_BLK\_NUM} to determine the block to be programmed.

\item Write parameters to be programmed to registers \hyperref[regdesc:EFUSEPGMDATA0REG]{EFUSE\_PGM\_DATA0\_REG} \verb+~+ \hyperref[regdesc:EFUSEPGMDATA7REG]{EFUSE\_PGM\_DATA7\_REG} and  \hyperref[regdesc:EFUSEPGMCHECKVALUE0REG]{EFUSE\_PGM\_CHECK\_VALUE0\_REG} \verb+~+ \hyperref[regdesc:EFUSEPGMCHECKVALUE2REG]{EFUSE\_PGM\_CHECK\_VALUE2\_REG}.

\item Make sure the eFuse programming voltage VDDQ is configured correctly as described in Section \ref{sec:efuse-vddq-timing}.

\item Configure the field \hyperref[fielddesc:EFUSEOPCODE]{EFUSE\_OP\_CODE} of register \hyperref[regdesc:EFUSECONFREG]{EFUSE\_CONF\_REG} to 0x5A5A.

\item Configure the field \hyperref[fielddesc:EFUSEPGMCMD]{EFUSE\_PGM\_CMD} of register \hyperref[regdesc:EFUSECMDREG]{EFUSE\_CMD\_REG} to 1.

\item Poll register \hyperref[regdesc:EFUSECMDREG]{EFUSE\_CMD\_REG} until it is 0x0, or wait for a PGM\_DONE interrupt. For more information on how to identify a PGM/READ\_DONE interrupt, please see the end of Section \ref{sec:efuse-software-read-para}.

\item Clear the parameters in \hyperref[regdesc:EFUSEPGMDATA0REG]{EFUSE\_PGM\_DATA0\_REG} \verb+~+ \hyperref[regdesc:EFUSEPGMDATA7REG]{EFUSE\_PGM\_DATA7\_REG} and \\ \hyperref[regdesc:EFUSEPGMCHECKVALUE0REG]{EFUSE\_PGM\_CHECK\_VALUE0\_REG} \verb+~+ \hyperref[regdesc:EFUSEPGMCHECKVALUE2REG]{EFUSE\_PGM\_CHECK\_VALUE2\_REG}.

\item Trigger an eFuse read operation (see Section \ref{sec:efuse-software-read-para}) to update eFuse registers with the new values.
\item Check error record registers. If the values read in error record registers are not 0, the programming process should be performed again following above steps 1 \verb+~+ 7. Please check the following error record registers for different eFuse blocks:
\begin{itemize}
    \item BLOCK0: \hyperref[regdesc:EFUSERDREPEATERR0REG]{EFUSE\_RD\_REPEAT\_ERR0\_REG} \verb+~+ \hyperref[regdesc:EFUSERDREPEATERR4REG]{EFUSE\_RD\_REPEAT\_ERR4\_REG}
    \item BLOCK1: \hyperref[regdesc:EFUSERDRSERR0REG]{EFUSE\_RD\_RS\_ERR0\_REG}[2:0], \hyperref[regdesc:EFUSERDRSERR0REG]{EFUSE\_RD\_RS\_ERR0\_REG}[7]
    \item BLOCK2: \hyperref[regdesc:EFUSERDRSERR0REG]{EFUSE\_RD\_RS\_ERR0\_REG}[6:4], \hyperref[regdesc:EFUSERDRSERR0REG]{EFUSE\_RD\_RS\_ERR0\_REG}[11]
    \item BLOCK3: \hyperref[regdesc:EFUSERDRSERR0REG]{EFUSE\_RD\_RS\_ERR0\_REG}[10:8], \hyperref[regdesc:EFUSERDRSERR0REG]{EFUSE\_RD\_RS\_ERR0\_REG}[15]
    \item BLOCK4: \hyperref[regdesc:EFUSERDRSERR0REG]{EFUSE\_RD\_RS\_ERR0\_REG}[14:12], \hyperref[regdesc:EFUSERDRSERR0REG]{EFUSE\_RD\_RS\_ERR0\_REG}[19]
    \item BLOCK5: \hyperref[regdesc:EFUSERDRSERR0REG]{EFUSE\_RD\_RS\_ERR0\_REG}[18:16], \hyperref[regdesc:EFUSERDRSERR0REG]{EFUSE\_RD\_RS\_ERR0\_REG}[23]
    \item BLOCK6: \hyperref[regdesc:EFUSERDRSERR0REG]{EFUSE\_RD\_RS\_ERR0\_REG}[22:20], \hyperref[regdesc:EFUSERDRSERR0REG]{EFUSE\_RD\_RS\_ERR0\_REG}[27]
    \item BLOCK7: \hyperref[regdesc:EFUSERDRSERR0REG]{EFUSE\_RD\_RS\_ERR0\_REG}[26:24], \hyperref[regdesc:EFUSERDRSERR0REG]{EFUSE\_RD\_RS\_ERR0\_REG}[31]
    \item BLOCK8: \hyperref[regdesc:EFUSERDRSERR0REG]{EFUSE\_RD\_RS\_ERR0\_REG}[30:28], \hyperref[regdesc:EFUSERDRSERR1REG]{EFUSE\_RD\_RS\_ERR1\_REG}[3]
    \item BLOCK9: \hyperref[regdesc:EFUSERDRSERR1REG]{EFUSE\_RD\_RS\_ERR1\_REG}[2:0], \hyperref[regdesc:EFUSERDRSERR1REG]{EFUSE\_RD\_RS\_ERR1\_REG}[2:0][7]
    \item BLOCK10: \hyperref[regdesc:EFUSERDRSERR1REG]{EFUSE\_RD\_RS\_ERR1\_REG}[2:0][6:4]
\end{itemize}
\end{enumerate}

{\bfseries Limitations}

In BLOCK0, each bit can be programmed separately. However, we recommend to minimize programming cycles and program all the bits of a parameter in one programming action. In addition, after all parameters controlled by a certain bit of \hyperref[fielddesc:EFUSEWRDIS]{EFUSE\_WR\_DIS} are programmed, that bit should be immediately programmed. The programming of parameters controlled by a certain bit of \hyperref[fielddesc:EFUSEWRDIS]{EFUSE\_WR\_DIS},
and the programming of the bit itself can even be completed at the same time. Repeated programming of already programmed bits is strictly forbidden, otherwise, programming errors will occur.

BLOCK1 cannot be programmed by users as it has been programmed at manufacturing.

BLOCK2 \verb+~+ 10 can only be programmed once. Repeated programming is not allowed.

\subsection{User Read of Parameters}
\label{sec:efuse-software-read-para}

Users cannot read eFuse bits directly. The eFuse Controller hardware reads all eFuse bits and stores the results to their corresponding registers in its memory space. Then, users can read eFuse bits by reading the registers that start with EFUSE\_RD\_. Details are provided in Table \ref{tab:efuse-softw-read-para}.

{\bfseries Updating eFuse read registers}
\label{sec:efuse-update-reg}

The eFuse Controller reads internal eFuses to update corresponding registers. This read operation happens on system reset and can also be triggered manually by users as needed (e.g., if new eFuse values have been programmed). The process of triggering a read operation by users is as follows:

\begin{enumerate}
    \item Configure the field \hyperref[fielddesc:EFUSEOPCODE]{EFUSE\_OP\_CODE} in register \hyperref[regdesc:EFUSECONFREG]{EFUSE\_CONF\_REG} to 0x5AA5.
    \item Configure the field \hyperref[fielddesc:EFUSEREADCMD]{EFUSE\_READ\_CMD} in register \hyperref[regdesc:EFUSECMDREG]{EFUSE\_CMD\_REG} to 1.
    \item Poll register \hyperref[regdesc:EFUSECMDREG]{EFUSE\_CMD\_REG} until it is 0x0, or wait for a READ\_DONE interrupt. Information on how to identify a PGM/READ\_DONE interrupt is provided below in this section.
    \item Read the values of each parameter from memory.
\end{enumerate}
The eFuse read registers will hold all values until the next read operation.

{\bfseries Error detection}

Error record registers allow users to detect if there are any inconsistencies in the stored backup eFuse parameters.

Registers EFUSE\_RD\_REPEAT\_ERR0 \verb+~+ 3\_REG indicate if there are any errors of programmed parameters (except for \hyperref[fielddesc:EFUSEWRDIS]{EFUSE\_WR\_DIS}) in BLOCK0 (value 1 indicates an error is detected, and the bit becomes invalid; value 0 indicates no error).

Registers EFUSE\_RD\_RS\_ERR0 \verb+~+ 1\_REG store the number of corrected bytes as well as the result of RS decoding during eFuse reading BLOCK1 \verb+~+ BLOCK10.

The values of above registers will be updated every time after the eFuse read registers have been updated.

{\bfseries Identifying program/read operation}

    The methods to identify the completion of a program/read operation are described below. Please note that bit 1 corresponds to a program operation, and bit 0 corresponds to a read operation.
    \begin{itemize}
    \item Method one:
        \begin{enumerate}
            \item Poll bit 1/0 in register \hyperref[regdesc:EFUSEINTRAWREG]{EFUSE\_INT\_RAW\_REG} until it becomes 1, which represents the completion of a program/read operation.
        \end{enumerate}
    \item Method two:
        \begin{enumerate}
            \item Set bit 1/0 in register \hyperref[regdesc:EFUSEINTENAREG]{EFUSE\_INT\_ENA\_REG} to 1 to enable the eFuse Controller to post a PGM/READ\_DONE interrupt.
            \item Configure the Interrupt Matrix to enable the CPU to respond to eFuse interrupt signals, see Chapter \ref{mod:intmtrx} \textit{\nameref{mod:intmtrx}}.
            \item Wait for the PGM/READ\_DONE interrupt.
            \item Set bit 1/0 in register \hyperref[regdesc:EFUSEINTCLRREG]{EFUSE\_INT\_CLR\_REG} to 1 to clear the PGM/READ\_DONE interrupt.
    \end{enumerate}
\end{itemize}


{\bfseries Note}


When eFuse controller updating its registers, it will use \hyperref[regdesc:EFUSEPGMDATA0REG]{EFUSE\_PGM\_DATA\textcolor{red}{\it{n}}\_REG} (n=0, 1, .., 7) again to store data. So please do not write important data into these registers before this updating process initiated. \\
During the chip boot process, eFuse controller will update eFuse data into registers which can be accessed by users automatically. Users can get programmed eFuse data by reading corresponding registers. Thus, it is no need to update eFuse read registers in such case.

\subsection{eFuse VDDQ Timing}
\label{sec:efuse-vddq-timing}

The eFuse Controller operates with 20 MHz of clock frequency, and its programming voltage VDDQ should be configured as follows:

\begin{itemize}
    \item \hyperref[fielddesc:EFUSEDACNUM]{EFUSE\_DAC\_NUM} (the rising period of VDDQ): The default value of VDDQ is 2.5 V and the voltage increases by 0.01 V in each clock cycle. Thus, the default value of this parameter is 255;
    \item \hyperref[fielddesc:EFUSEDACCLKDIV]{EFUSE\_DAC\_CLK\_DIV} (the clock divisor of VDDQ): The clock period to program VDDQ should be larger than 1 $\mu$s;
    \item \hyperref[fielddesc:EFUSEPWRONNUM]{EFUSE\_PWR\_ON\_NUM} (the power-up time for VDDQ): The programming voltage should be stabilized after this time, which means the value of this parameter should be configured to exceed the result of \hyperref[fielddesc:EFUSEDACCLKDIV]{EFUSE\_DAC\_CLK\_DIV} times \hyperref[fielddesc:EFUSEDACNUM]{EFUSE\_DAC\_NUM};
    \item \hyperref[fielddesc:EFUSEPWROFFNUM]{EFUSE\_PWR\_OFF\_NUM} (the power-out time for VDDQ): The value of this parameter should be larger than 10 $\mu$s.
\end{itemize}

\subsection{The Use of Parameters by Hardware Modules}

Some hardware modules are directly connected to the eFuse peripheral in order to use the parameters listed in Table \ref{tab:efuse-block0-para} and Table \ref{tab:efuse-block-1-10-para}, specifically those marked with ``Y" in columns ``Accessible by Hardware". Users cannot intervene in this process.

\subsection{Interrupts}
\begin{itemize}
\item PGM\_DONE interrupt: Triggered when eFuse programming has finished. To enable this interrupt, set the \hyperref[fielddesc:EFUSEPGMDONEINTENA]{EFUSE\_PGM\_DONE\_INT\_ENA} field of register \hyperref[regdesc:EFUSEINTENAREG]{EFUSE\_INT\_ENA\_REG} to 1;
\item READ\_DONE interrupt: Triggered when eFuse reading has finished. To enable this interrupt, set the \\\hyperref[fielddesc:EFUSEREADDONEINTENA]{EFUSE\_READ\_DONE\_INT\_ENA} field of register \hyperref[regdesc:EFUSEINTENAREG]{EFUSE\_INT\_ENA\_REG} to 1.
\end{itemize}
