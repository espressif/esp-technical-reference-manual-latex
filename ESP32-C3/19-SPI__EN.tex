\chapter{SPI Controller (SPI)}
\label{mod:spi}


\section{Overview}

The Serial Peripheral Interface (SPI) is a synchronous serial interface useful for communication with external peripherals.
The \chipname{} chip integrates three SPI controllers:
\begin{itemize}
  \item SPI0,
  \item SPI1,
  \item General Purpose SPI2 (GP-SPI2).
\end{itemize}

SPI0 and SPI1 controllers are primarily reserved for internal use. This chapter mainly focuses on the GP-SPI2 controller.

\subfile{\modulefiles/tables/terms.tex}

\section{Features}



Some of the key features of GP-SPI2 are:
\begin{itemize}
    \item Master and slave modes
    \item Half- and full-duplex communications
    \item CPU- and DMA-controlled transfers
    \item Various data modes:
        \begin{itemize}
            \item 1-bit SPI mode
            \item 2-bit Dual SPI mode
            \item 4-bit Quad SPI mode
            \item QPI mode
        \end{itemize}
    \item Configurable module clock frequency:
        \begin{itemize}
            \item Master: up to 80 MHz
            \item Slave: up to 60 MHz
        \end{itemize}
    \item Configurable data length:
    \begin{itemize}
        \item CPU-controlled transfer in master mode or in slave mode: 1 \verb+~+ 64 B
        \item DMA-controlled single transfer in master mode: 1 \verb+~+ 32 KB
        \item DMA-controlled configurable segmented transfer in master mode: data length is unlimited
        \item DMA-controlled single transfer or segmented transfer in slave mode: data length is unlimited
    \end{itemize}
    \item Configurable bit read/write order
    \item Independent interrupts for CPU-controlled transfer and DMA-controlled transfer
    \item Configurable clock polarity and phase
    \item Four SPI clock modes: mode 0 \verb+~+ mode 3
    \item Six CS lines in master mode: CS0 \verb+~+ CS5
    \item Able to communicate with SPI devices, such as a sensor, a screen controller, as well as a flash or RAM chip
\end{itemize}

\subsection{CPU-Controlled Data Transfer}\label{sec:spi-func-descr-cpu-ctrl-data-trans}
GP-SPI2 provides 16 x 32-bit data buffers, i.e., \hyperref[regdesc:SPIW0REG]{SPI\_W0\_REG} \verb+~+ \hyperref[regdesc:SPIW15REG]{SPI\_W15\_REG}, see Figure \ref{fig:spi-data-buffer-cpu-ctrl-data-trans}.
CPU-controlled transfer indicates the transfer, in which the data to send is from GP-SPI2 data buffer and the received data is stored to GP-SPI2 data buffer.
In such transfer, every single transaction needs to be triggered by the CPU, after its related registers are configured. For such reason, the CPU-controlled transfer is always single transfers (consisting of only one transaction). CPU-controlled mode supports full-duplex communication and half-duplex communication.

\subsubsection{CPU-Controlled Master Mode} \label{sec:spi-cpu-ctrl-master}

In a CPU-controlled master full-duplex or half-duplex transfer, the RX or TX data is saved to or sent from \hyperref[regdesc:SPIW0REG]{SPI\_W0\_REG} \verb+~+ \hyperref[regdesc:SPIW15REG]{SPI\_W15\_REG}. The bits \hyperref[fielddesc:SPIUSRMOSIHIGHPART]{SPI\_USR\_MOSI\_HIGHPART} and \hyperref[fielddesc:SPIUSRMISOHIGHPART]{SPI\_USR\_MISO\_HIGHPART} control which buffers are used, see the list below.

\begin{itemize}
    \item TX data
    \begin{itemize}
        \item When \hyperref[fielddesc:SPIUSRMOSIHIGHPART]{SPI\_USR\_MOSI\_HIGHPART} is cleared, i.e., high part mode is disabled, TX data is from \hyperref[regdesc:SPIW0REG]{SPI\_W0\_REG} \verb+~+ \hyperref[regdesc:SPIW15REG]{SPI\_W15\_REG} and the data address is incremented by 1 on each byte transferred. If the data byte length is larger than 64, the data in \hyperref[regdesc:SPIW0REG]{SPI\_W0\_REG}[7:0] \verb+~+ \hyperref[regdesc:SPIW15REG]{SPI\_W15\_REG}[31:24] may be sent more than once. For instance, if 66 bytes (byte0 \verb+~+ byte65) need to be sent, then the address of byte65 is the result of (65 \% 64 = 1), i.e., byte65 is from \hyperref[regdesc:SPIW0REG]{SPI\_W0\_REG}[15:8], and byte64 is from \hyperref[regdesc:SPIW0REG]{SPI\_W0\_REG}[7:0]. In this case, the content of \hyperref[regdesc:SPIW0REG]{SPI\_W0\_REG}[15:0] may be sent more than once.
        \item When \hyperref[fielddesc:SPIUSRMOSIHIGHPART]{SPI\_USR\_MOSI\_HIGHPART} is set, i.e., high part mode is enabled, TX data is from \hyperref[regdesc:SPIW8REG]{SPI\_W8\_REG} \verb+~+ \hyperref[regdesc:SPIW15REG]{SPI\_W15\_REG} and the data address is incremented by 1 on each byte transferred. If the data byte length is larger than 32, the data in \hyperref[regdesc:SPIW8REG]{SPI\_W8\_REG}[7:0] \verb+~+ \hyperref[regdesc:SPIW15REG]{SPI\_W15\_REG}[31:24] may be sent more than once.
    \end{itemize}
    \item RX data
    \begin{itemize}
        \item When \hyperref[fielddesc:SPIUSRMISOHIGHPART]{SPI\_USR\_MISO\_HIGHPART} is cleared, i.e., high part mode is disabled, RX data is saved to \hyperref[regdesc:SPIW0REG]{SPI\_W0\_REG} \verb+~+ \hyperref[regdesc:SPIW15REG]{SPI\_W15\_REG}, and the data address is incremented by 1 on each byte transferred. If the data byte length is larger than 64, the data in \hyperref[regdesc:SPIW0REG]{SPI\_W0\_REG}[7:0] \verb+~+ \hyperref[regdesc:SPIW15REG]{SPI\_W15\_REG}[31:24] may be overwritten. For instance, when 66 bytes (byte0 \verb+~+ byte65) are received, byte65 and byte64 are stored to the addresses of (65 \% 64 = 1) and (64 \% 64 = 0), i.e., \hyperref[regdesc:SPIW0REG]{SPI\_W0\_REG}[15:8] and \hyperref[regdesc:SPIW0REG]{SPI\_W0\_REG}[7:0]. In this case, the content of \hyperref[regdesc:SPIW0REG]{SPI\_W0\_REG}[15:0] may be overwritten.

        \item When \hyperref[fielddesc:SPIUSRMISOHIGHPART]{SPI\_USR\_MISO\_HIGHPART} is set, i.e., high part mode is enabled, the RX data is saved to \hyperref[regdesc:SPIW8REG]{SPI\_W8\_REG} \verb+~+ \hyperref[regdesc:SPIW15REG]{SPI\_W15\_REG}, and the data address is incremented by 1 on each byte transferred. If the data byte length is larger than 32, the content of \hyperref[regdesc:SPIW8REG]{SPI\_W8\_REG} \verb+~+ \hyperref[regdesc:SPIW15REG]{SPI\_W15\_REG} may be overwritten.
    \end{itemize}
\end{itemize}

\begin{tiplisting}
\vspace{-2em}
\begin{itemize}
    \item TX/RX data address mentioned above both are byte-addressable. Address 0 stands for \hyperref[regdesc:SPIW0REG]{SPI\_W0\_REG}[7:0], and Address 1 for \hyperref[regdesc:SPIW0REG]{SPI\_W0\_REG}[15:8], and so on. The largest address is \hyperref[regdesc:SPIW15REG]{SPI\_W15\_REG}[31:24].
    \item To avoid any possible error in TX/RX data, such as TX data being sent more than once or RX data being overwritten, please make sure the registers are configured correctly.
\end{itemize}
\end{tiplisting}

\subsubsection{CPU-Controlled Slave Mode} \label{sec:spi-cpu-ctrl-slave}

In a CPU-controlled slave full-duplex or half-duplex transfer, the RX data or TX data is saved to or sent from \hyperref[regdesc:SPIW0REG]{SPI\_W0\_REG} \verb+~+ \hyperref[regdesc:SPIW15REG]{SPI\_W15\_REG}, which are byte-addressable.

\begin{itemize}
    \item In full-duplex communication, the address of \hyperref[regdesc:SPIW0REG]{SPI\_W0\_REG} \verb+~+ \hyperref[regdesc:SPIW15REG]{SPI\_W15\_REG} starts from 0 and is incremented by 1 on each byte transferred. If the data address is larger than 63, the content of \hyperref[regdesc:SPIW15REG]{SPI\_W15\_REG}[31:24] is overwritten.
    \item In half-duplex communication, the ADDR value in transmission format is the start address of the RX or TX data, corresponding to the registers \hyperref[regdesc:SPIW0REG]{SPI\_W0\_REG} \verb+~+ \hyperref[regdesc:SPIW15REG]{SPI\_W15\_REG}. The RX or TX address is incremented by 1 on each byte transferred. If the address is larger than 63 (the highest byte address, i.e. \hyperref[regdesc:SPIW15REG]{SPI\_W15\_REG}[31:24]), the address of overflowing data is always 63 and only the content of \hyperref[regdesc:SPIW15REG]{SPI\_W15\_REG}[31:24] is overwritten.
\end{itemize}

According to your applications, the registers \hyperref[regdesc:SPIW0REG]{SPI\_W0\_REG} \verb+~+ \hyperref[regdesc:SPIW15REG]{SPI\_W15\_REG} can be used as:
\begin{itemize}
    \item data buffers only
    \item data buffers and status buffers
    \item status buffers only
\end{itemize}


\subsection{DMA-Controlled Data Transfer}\label{sec:spi-func-descr-dma-ctrl-data-transfer}

DMA-controlled transfer refers to the transfer, in which GDMA RX module receives data and GDMA TX module sends data. This transfer is supported both in master mode and in slave mode.

A DMA-controlled transfer can be

\begin{itemize}
    \item a single transfer, consisting of only one transaction. GP-SPI2 supports this transfer both in master and slave modes.
    \item a configurable segmented transfer, consisting of several transactions (segments). GP-SPI2 supports this transfer only in master mode. For more information, see Section \ref{sec:spi-dma-ctrl-config-seg-transfer}.
    \item a slave segmented transfer, consisting of several transactions (segments). GP-SPI2 supports this transfer only in slave mode. For more information, see Section \ref{sec:spi-slave-single-seg-trans}.
\end{itemize}

A DMA-controlled transfer only needs to be triggered once by CPU. When such transfer is triggered, data is transferred by the GDMA engine from or to the DMA-linked memory, without CPU operation.

DMA-controlled mode supports full-duplex communication, half-duplex communication and functions described in Section \ref{sec:spi-func-descr-gp-spi2-master} and Section \ref{sec:spi-func-descr-gp-spi2-slave}. Meanwhile, the GDMA RX module is independent from the GDMA TX module, which means that there are four kinds of full-duplex communications:

\begin{itemize}
    \item Data is received in DMA-controlled mode and sent in DMA-controlled mode.

    \item Data is received in DMA-controlled mode but sent in CPU-controlled mode.

    \item Data is received in CPU-controlled mode but sent in DMA-controlled mode.

    \item Data is received in CPU-controlled mode and sent in CPU-controlled mode.

\end{itemize}


\subsubsection{GDMA Configuration}\label{sec:spi-dma-config}


\begin{itemize}
    \item Select a GDMA channel\textcolor{red}{\it{n}}, and configure a GDMA TX/RX descriptor, see Chapter \ref{mod:dma} \textit{\nameref{mod:dma}}.

    \item Set the bit \hyperref[fielddesc:GDMAINLINKSTARTCHN]{GDMA\_INLINK\_START\_CH\textcolor{red}{\it{n}}} or \hyperref[fielddesc:GDMAOUTLINKSTARTCHN]{GDMA\_OUTLINK\_START\_CH\textcolor{red}{\it{n}}} to start GDMA RX/TX engine.

    \item Before all the GDMA TX buffer is used or the GDMA TX engine is reset, if  \hyperref[fielddesc:GDMAOUTLINKRESTARTCHN]{GDMA\_OUTLINK\_RESTART\_CH\textcolor{red}{\it{n}}} is set, a new TX buffer will be added to the end of the last TX buffer in use.

    \item GDMA RX buffer is linked in the same way as the GDMA TX buffer, by setting \hyperref[fielddesc:GDMAINLINKSTARTCHN]{GDMA\_INLINK\_START\_CH\textcolor{red}{\it{n}}} or \hyperref[fielddesc:GDMAINLINKRESTARTCHN]{GDMA\_INLINK\_RESTART\_CH\textcolor{red}{\it{n}}}.

    \item The TX and RX data lengths are determined by the configured GDMA TX and RX buffer respectively, both of which are 0 \verb+~+ 32 KB.

    \item Initialize GDMA inlink and outlink before GDMA starts. The bits \hyperref[fielddesc:SPIDMARXENA]{SPI\_DMA\_RX\_ENA} and \hyperref[fielddesc:SPIDMATXENA]{SPI\_DMA\_TX\_ENA} in register \hyperref[regdesc:SPIDMACONFREG]{SPI\_DMA\_CONF\_REG} should be set, otherwise the read/write data will be stored to/sent from the registers \hyperref[regdesc:SPIW0REG]{SPI\_W0\_REG} \verb+~+ \hyperref[regdesc:SPIW15REG]{SPI\_W15\_REG}.

\end{itemize}

In master mode, if GDMA\_IN\_SUC\_EOF\_CH\textcolor{red}{\it{n}}\_INT\_ENA is set, then the interrupt GDMA\_IN\_SUC\_EOF\_CH\textcolor{red}{\it{n}}\_INT will be triggered when one single transfer or one configurable segmented transfer is finished.

The only difference between DMA-controlled transfers in master mode and in slave mode is on the GDMA RX control:

\begin{itemize}
    \item When the bit \hyperref[fielddesc:SPIRXEOFEN]{SPI\_RX\_EOF\_EN} is cleared, a GDMA\_IN\_SUC\_EOF\_CH\textcolor{red}{\it{n}}\_INT interrupt may be generated after the CS is pulled high once:
        \begin{itemize}
            \item In a slave single transfer, if \hyperref[fielddesc:SPIDMASLVSEGTRANSEN]{SPI\_DMA\_SLV\_SEG\_TRANS\_EN} is cleared and GDMA\_IN\_SUC\_EOF\_CH\textcolor{red}{\it{n}}\_INT\\\_ENA is set, a GDMA\_IN\_SUC\_EOF\_CH\textcolor{red}{\it{n}}\_INT interrupt will be triggered once the single transfer is done.
            \item In a slave segmented transfer, if both \hyperref[fielddesc:SPIDMASLVSEGTRANSEN]{SPI\_DMA\_SLV\_SEG\_TRANS\_EN} and GDMA\_IN\_SUC\_EOF\_CH\textcolor{red}{\it{n}}\_\\INT\_ENA are set, a GDMA\_IN\_SUC\_EOF\_CH\textcolor{red}{\it{n}}\_INT interrupt also is triggered once the command (\hyperref[other:spi-item-cmd7]{CMD7} or \hyperref[other:spi-item-end-seg-trans]{End\_SEG\_TRANS}) is received correctly.
        \end{itemize}

    \item When the bit \hyperref[fielddesc:SPIRXEOFEN]{SPI\_RX\_EOF\_EN} is set, the generation of GDMA\_IN\_SUC\_EOF\_CH\textcolor{red}{\it{n}}\_INT also depends on the length of transferred data.
        \begin{itemize}
            \item In a slave single transfer, if \hyperref[fielddesc:SPIDMASLVSEGTRANSEN]{SPI\_DMA\_SLV\_SEG\_TRANS\_EN} is cleared and GDMA\_IN\_SUC\_EOF\_CH\textcolor{red}{\it{n}}\_\\INT\_ENA is set, a GDMA\_IN\_SUC\_EOF\_CH\textcolor{red}{\it{n}}\_INT interrupt will be generated once the single transfer is done or the length of GDMA RX received data is equal to (\hyperref[fielddesc:SPIMSDATABITLEN]{SPI\_MS\_DATA\_BITLEN} + 1).
            \item In a slave segmented transfer, if \hyperref[fielddesc:SPIDMASLVSEGTRANSEN]{SPI\_DMA\_SLV\_SEG\_TRANS\_EN} is set, a GDMA\_IN\_SUC\_EOF\_CH\textcolor{red}{\it{n}}\_INT interrupt will be generated once the command (\hyperref[other:spi-item-cmd7]{CMD7} or \hyperref[other:spi-item-end-seg-trans]{End\_SEG\_TRANS}) is received correctly or the length of GDMA RX received data is equal to (\hyperref[fielddesc:SPIMSDATABITLEN]{SPI\_MS\_DATA\_BITLEN} + 1).
            \end{itemize}
\end{itemize}


\subsubsection{GDMA TX/RX Buffer Length Control}\label{sec:spi-dma-tx-rx-buffer-length-ctrl}

It is recommended that the length of configured GDMA TX/RX buffer is equal to the length of real transferred data.

\begin{itemize}
    \item If the length of configured GDMA TX buffer is shorter than that of real transferred data, the extra data will be the same as the last transferred data. \hyperref[int:SPIOUTFIFOEMPTYERRINT]{SPI\_OUTFIFO\_EMPTY\_ERR\_INT} and \hyperref[int:GDMAOUTEOFINT]{GDMA\_OUT\_EOF\_CH\textcolor{red}{\it{n}}\_INT} are triggered.

    \item If the length of configured GDMA TX buffer is longer than the that of real transferred data, the TX buffer is not fully used, and the remaining buffer is available for following transaction even if a new TX buffer is linked later. Please keep it in mind. Or save the unused data and reset DMA.

    \item If the length of configured GDMA RX buffer is shorter than that of real transferred data, the extra data will be lost. The interrupts \hyperref[int:SPIINFIFOFULLERRINT]{SPI\_INFIFO\_FULL\_ERR\_INT} and \hyperref[int:SPITRANSDONEINT]{SPI\_TRANS\_DONE\_INT} are triggered. But \hyperref[int:GDMAINSUCEOFINT]{GDMA\_IN\_SUC\_EOF\_CH\textcolor{red}{\it{n}}\_INT} interrupt is not generated.

    \item If the length of configured GDMA RX buffer is longer than that of real transferred data, the RX buffer is not fully used, and the remaining buffer is discarded. In the following transaction, a new linked buffer will be used directly.

\end{itemize}

\subsection{Data Flow Control in GP-SPI2 Master and Slave Modes}\label{sec:spi-func-descr-data-flow-ctrl-gp-spi2-master-slave}

CPU-controlled and DMA-controlled transfers are supported in GP-SPI2 master and slave modes.
CPU-controlled transfer means that data transfers between registers \hyperref[regdesc:SPIW0REG]{SPI\_W0\_REG} \verb+~+ \hyperref[regdesc:SPIW15REG]{SPI\_W15\_REG} and the SPI device. DMA-controlled transfer means that data transfers between the configured GDMA TX/RX buffer and the SPI device. To select between the two transfer modes, configure \hyperref[fielddesc:SPIDMARXENA]{SPI\_DMA\_RX\_ENA} and \hyperref[fielddesc:SPIDMATXENA]{SPI\_DMA\_TX\_ENA} before the transfer starts.

\subsubsection{GP-SPI2 Functional Blocks}

\begin{figure}[H]
    \centering
    \fbox{\includegraphics[width=0.9\textwidth]{19-SPI/figures/ESP726_GPSPI_Block_Diagram.eps}}
    \caption{GP-SPI2 Block Diagram}
    \label{fig:spi-gp-spi2-block-diag}
\end{figure}

Figure \ref{fig:spi-gp-spi2-block-diag} shows main functional blocks in GP-SPI2, including:
\begin{itemize}
    \item \hyperref[sec:spi-state-machine]{Master FSM}: all the features, supported in GP-SPI2 master mode, are controlled by this state machine together with register configuration.
    \item SPI Buffer: \hyperref[regdesc:SPIW0REG]{SPI\_W0\_REG} \verb+~+ \hyperref[regdesc:SPIW15REG]{SPI\_W15\_REG}, see Figure \ref{fig:spi-data-buffer-cpu-ctrl-data-trans}. The data transferred in CPU-controlled mode is prepared in this buffer.
    \item Timing Module: capture data on FSPI bus.
    \item spi\_mst/slv\_din/dout\_ctrl: convert the TX/RX data into bytes.
    \item spi\_rx\_afifo: store the received data.
    \item buf\_tx\_afifo: store the data to send.
    \item dma\_tx\_afifo: store the data from GDMA.
    \item clk\_spi\_mst: this clock is the module clock of GP-SPI2 and derived from PLL\_CLK. It is used in GP-SPI2 master mode, to generate SPI\_CLK signal for data transfer and for slaves.
    \item SPI\_CLK Generator: generate SPI\_CLK by dividing clk\_spi\_mst. The divider is determined by SPI\_CLKCNT\_N and SPI\_CLKDIV\_PRE.
    \item SPI\_CLK\_out Mode Control: output the SPI\_CLK signal for data transfer and for slaves.
    \item SPI\_CLK\_in Mode Control: capture the SPI\_CLK signal from SPI master when GP-SPI2 works as a slave.

\end{itemize}

\subsubsection{Data Flow Control in Master Mode}\label{sec:spi-data-flow-ctrl-master}

\begin{figure}[H]
    \centering
    \fbox{\includegraphics[width=0.9\textwidth]{19-SPI/figures/ESP726_GPSPI_Mst_Data_Flow.eps}}
    \caption{Data Flow Control in GP-SPI2 Master Mode}
    \label{fig:spi-data-flow-ctrl-gp-spi2-master}
\end{figure}

Figure \ref{fig:spi-data-flow-ctrl-gp-spi2-master} shows the data flow of GP-SPI2 in master mode. Its control logic is as follows:
\begin{itemize}
    \item RX data: data in FSPI bus is captured by Timing Module, converted in units of bytes by spi\_mst\_din\_ctrl module, and then stored in corresponding addresses according to the transfer modes.
    \begin{itemize}
        \item CPU-controlled transfer: the data is stored to registers \hyperref[regdesc:SPIW0REG]{SPI\_W0\_REG}  \verb+~+ \hyperref[regdesc:SPIW15REG]{SPI\_W15\_REG}.
        \item DMA-controlled transfer: the data is stored to GDMA RX buffer.
    \end{itemize}
    \item TX data: the TX data is from corresponding addresses according to transfer modes and is saved to buf\_tx\_afifo.
    \begin{itemize}
        \item CPU-controlled transfer: TX data is from \hyperref[regdesc:SPIW0REG]{SPI\_W0\_REG} \verb+~+ \hyperref[regdesc:SPIW15REG]{SPI\_W15\_REG}.
        \item DMA-controlled transfer: TX data is from GDMA TX buffer.
    \end{itemize}
\end{itemize}

The data in buf\_tx\_afifo is sent out to Timing Module in 1/2/4-bit modes, controlled by GP-SPI2 state machine. The Timing Module can be used for timing compensation. For more information, see Section \ref{sec:spi-gp-spi2-timing-compensation}.

\subsubsection{Data Flow Control in Slave Mode}\label{sec:spi-data-flow-ctrl-slave}

\begin{figure}[H]
    \centering
    \fbox{\includegraphics[width=0.9\textwidth]{19-SPI/figures/ESP726_GPSPI_Slv_Data_Flow.eps}}
    \caption{Data Flow Control in GP-SPI2 Slave Mode}
    \label{fig:spi-data-flow-ctrl-gp-spi2-slave}
\end{figure}

Figure \ref{fig:spi-data-flow-ctrl-gp-spi2-slave} shows the data flow in GP-SPI2 slave mode. Its control logic is as follows:
\begin{itemize}
    \item In CPU/DMA-controlled full-duplex/half-duplex modes, when an external SPI master starts the SPI transfer, data on the FSPI bus is captured, converted into unit of bytes by spi\_slv\_din\_ctrl module, and then is stored in spi\_rx\_afifo.
    \begin{itemize}
        \item In CPU-controlled full-duplex transfer, the received data in spi\_rx\_afifo will be later stored into registers \hyperref[regdesc:SPIW0REG]{SPI\_W0\_REG}  \verb+~+ \hyperref[regdesc:SPIW15REG]{SPI\_W15\_REG}, successively.

        \item In half-duplex \hyperref[other:spi-item-wrbuf]{Wr\_BUF} transfer, when the value of address (SLV\_ADDR[7:0]) is received, the received data in spi\_rx\_afifo will be stored in the related address of registers \hyperref[regdesc:SPIW0REG]{SPI\_W0\_REG}  \verb+~+ \hyperref[regdesc:SPIW15REG]{SPI\_W15\_REG}


        \item In DMA-controlled full-duplex transfer or in half-duplex \hyperref[other:spi-item-wrdma]{Wr\_DMA} transfer, the received data in spi\_rx\_afifo will be stored in the configured GDMA RX buffer.

    \end{itemize}

    \item In CPU-controlled full-/half-duplex transfer, the data to send is stored in buf\_tx\_afifo. In DMA-controlled full-/half-duplex transfer, the data to send is stored in dma\_tx\_afifo. Therefore, Rd\_BUF transaction controlled by CPU and Rd\_DMA transaction controlled by DMA can be done in one slave segmented transfer. TX data comes from corresponding addresses according the transfer modes.
    \begin{itemize}
        \item In CPU-controlled full-duplex transfer, when \hyperref[fielddesc:SPISLAVEMODE]{SPI\_SLAVE\_MODE} and \hyperref[fielddesc:SPIDOUTDIN]{SPI\_DOUTDIN} are set and SPI\_DMA\\\_TX\_ENA is cleared, the data in \hyperref[regdesc:SPIW0REG]{SPI\_W0\_REG} \verb+~+ \hyperref[regdesc:SPIW15REG]{SPI\_W15\_REG} will be stored into buf\_tx\_afifo;

        \item In CPU-controlled half-duplex transfer, when \hyperref[fielddesc:SPISLAVEMODE]{SPI\_SLAVE\_MODE} is set, \hyperref[fielddesc:SPIDOUTDIN]{SPI\_DOUTDIN} is cleared, Rd\_BUF command and SLV\_ADDR[7:0] are received, the data started from the related address of \hyperref[regdesc:SPIW0REG]{SPI\_W0\_REG} \verb+~+ \hyperref[regdesc:SPIW15REG]{SPI\_W15\_REG} will be stored into buf\_tx\_afifo;

        \item In DMA-controlled full-duplex transfer, when \hyperref[fielddesc:SPISLAVEMODE]{SPI\_SLAVE\_MODE}, \hyperref[fielddesc:SPIDOUTDIN]{SPI\_DOUTDIN} and SPI\_DMA\_TX\_\\ENA are set, the data in the configured GDMA TX buffer will be stored into dma\_tx\_afifo;

        \item In DMA-controlled half-duplex transfer, when \hyperref[fielddesc:SPISLAVEMODE]{SPI\_SLAVE\_MODE} is set, \hyperref[fielddesc:SPIDOUTDIN]{SPI\_DOUTDIN} is cleared, and Rd\_DMA command is received, the data in the configured GDMA TX buffer will be stored into dma\_tx\_afifo.

    \end{itemize}

The data in buf\_tx\_afifo or dma\_tx\_afifo is sent out by spi\_slv\_dout\_ctrl module in 1/2/4-bit modes.
\end{itemize}

\subsection{GP-SPI2 Works as a Master}\label{sec:spi-func-descr-gp-spi2-master}

GP-SPI2 can be configured as a SPI master by clearing the bit \hyperref[fielddesc:SPISLAVEMODE]{SPI\_SLAVE\_MODE} in \hyperref[regdesc:SPISLAVEREG]{SPI\_SLAVE\_REG}. In this operation mode, GP-SPI2 provides clock signal (the divided clock from GP-SPI2 module clock) and six CS lines (CS0 \verb+~+ CS5).

\begin{tiplisting}
\vspace{-2em}
\begin{itemize}
    \item The length of transferred data must be in unit of bytes, otherwise the extra bits will be lost. The extra bits here means the result of total data bits \% 8.
    \item To transfer bits not in unit of bytes, consider implementing it in CMD state or ADDR state.
\end{itemize}
\end{tiplisting}

\subsubsection{State Machine}
\label{sec:spi-state-machine}

When GP-SPI2 works as a master, the state machine controls its various states during data transfer, including configuration (CONF), preparation (PREP), command (CMD), address (ADDR), dummy (DUMMY), data out (DOUT), and data in (DIN) states. GP-SPI2 is mainly used to access 1/2/4-bit SPI devices, such as flash and external RAM, thus the naming of GP-SPI2 states keeps consistent with the sequence naming of flash and external RAM. The meaning of each state is described as follows and Figure \ref{fig:spi-gp-spi2-state-machine-master} shows the workflow of GP-SPI2 state machine.

\begin{enumerate}
    \item IDLE: GP-SPI2 is not active or is in slave mode.
    \item CONF: only used in DMA-controlled \hyperref[sec:spi-dma-ctrl-config-seg-transfer]{configurable segmented transfer}. Set \hyperref[fielddesc:SPIUSR]{SPI\_USR} and \hyperref[fielddesc:SPIUSRCONF]{SPI\_USR\_CONF} to enable this state. If this state is not enabled, it means the current transfer is a single transfer.
    \item PREP: prepare an SPI transaction and control SPI CS setup time. Set \hyperref[fielddesc:SPIUSR]{SPI\_USR} and \hyperref[fielddesc:SPICSSETUP]{SPI\_CS\_SETUP} to enable this state.
    \item\label{other:spi-item-cmd} CMD: send command sequence. Set \hyperref[fielddesc:SPIUSR]{SPI\_USR} and \hyperref[fielddesc:SPIUSRCOMMAND]{SPI\_USR\_COMMAND} to enable this state.
    \item\label{other:spi-item-addr} ADDR: send address sequence. Set \hyperref[fielddesc:SPIUSR]{SPI\_USR} and \hyperref[fielddesc:SPIUSRADDR]{SPI\_USR\_ADDR} to enable this state.
    \item DUMMY (wait cycle): send dummy sequence. Set \hyperref[fielddesc:SPIUSR]{SPI\_USR} and \hyperref[fielddesc:SPIUSRDUMMY]{SPI\_USR\_DUMMY} to enable this state.
    \item\label{other:spi-item-data} DATA: transfer data.
    \begin{itemize}
    \item DOUT: send data sequence. Set \hyperref[fielddesc:SPIUSR]{SPI\_USR} and \hyperref[fielddesc:SPIUSRMOSI]{SPI\_USR\_MOSI} to enable this state.
    \item DIN: receive data sequence. Set \hyperref[fielddesc:SPIUSR]{SPI\_USR} and \hyperref[fielddesc:SPIUSRMISO]{SPI\_USR\_MISO} to enable this state.
    \end{itemize}
    \item DONE: control SPI CS hold time. Set \hyperref[fielddesc:SPIUSR]{SPI\_USR} to enable this state.
\end{enumerate}

\begin{figure}[H]

    \caption{GP-SPI2 State Machine in Master Mode}\label{fig:spi-gp-spi2-state-machine-master}

Legend to state flow:
\begin{itemize}
    \item \textcolor{orange}{\textbf{—}}:
indicates corresponding state condition is not satisfied; repeats current state.
    \item \textcolor{cyan}{\textbf{—}}: corresponding registers are set and conditions are satisfied; goes to next state.
    \item \textcolor{red}{\textbf{—}}: state registers are not set; skips one or more following states, depending on the registers of the following states are set or not.
\end{itemize}

Explanation to the conditions listed in the figure above:
\begin{itemize}
    \item CONF condition: gpc[17:0] >= \hyperref[fielddesc:SPICONFBITLEN]{SPI\_CONF\_BITLEN}[17:0]
    \item PREP condition: gpc[4:0] >= \hyperref[fielddesc:SPICSSETUPTIME]{SPI\_CS\_SETUP\_TIME}[4:0]
    \item CMD condition: gpc[3:0] >= \hyperref[fielddesc:SPIUSRCOMMANDBITLEN]{SPI\_USR\_COMMAND\_BITLEN}[3:0]
    \item ADDR condition: gpc[4:0] >= \hyperref[fielddesc:SPIUSRADDRBITLEN]{SPI\_USR\_ADDR\_BITLEN}[4:0]
    \item DUMMY condition: gpc[7:0] >= \hyperref[fielddesc:SPIUSRDUMMYCYCLELEN]{SPI\_USR\_DUMMY\_CYCLELEN}[7:0]
    \item DOUT condition: gpc[17:0] >= \hyperref[fielddesc:SPIMSDATABITLEN]{SPI\_MS\_DATA\_BITLEN}[17:0]
    \item DIN condition: gpc[17:0] >= \hyperref[fielddesc:SPIMSDATABITLEN]{SPI\_MS\_DATA\_BITLEN}[17:0]

    \item DONE condition: (gpc[4:0] >= \hyperref[fielddesc:SPICSHOLDTIME]{SPI\_CS\_HOLD\_TIME}[4:0] || \hyperref[fielddesc:SPICSHOLD]{SPI\_CS\_HOLD} == 1'b0)

\end{itemize}





A counter (gpc[17:0]) is used in the state machine to control the cycle length of each state. The states CONF, PREP, CMD, ADDR, DUMMY, DOUT, and DIN can be enabled or disabled independently. The cycle length of each state can also be configured independently.

    \item Configure DIN state related registers.
        \begin{itemize}
            \item Configure read data bit length in \hyperref[fielddesc:SPIMSDATABITLEN]{SPI\_MS\_DATA\_BITLEN}. \hyperref[fielddesc:SPIMSDATABITLEN]{SPI\_MS\_DATA\_BITLEN} = bit length expected - 1.
            \item Set \hyperref[fielddesc:SPIFREADQUAD]{SPI\_FREAD\_QUAD} and \hyperref[fielddesc:SPIUSRMISO]{SPI\_USR\_MISO}.
            \item Clear \hyperref[fielddesc:SPIFREADDUAL]{SPI\_FREAD\_DUAL}.
            \item Configure GDMA in DMA-controlled mode. In CPU controlled mode, no action is needed.
        \end{itemize}

    \item Clear \hyperref[fielddesc:SPIUSRMOSI]{SPI\_USR\_MOSI}.
    \item Set \hyperref[fielddesc:SPIDMAAFIFORST]{SPI\_DMA\_AFIFO\_RST}, \hyperref[fielddesc:SPIBUFAFIFORST]{SPI\_BUF\_AFIFO\_RST}, and \hyperref[fielddesc:SPIRXAFIFORST]{SPI\_RX\_AFIFO\_RST} to reset these buffers.

    \item Set \hyperref[fielddesc:SPIUSR]{SPI\_USR} to start GP-SPI2 transfer.
\end{enumerate}

When writing data (DOUT state), \hyperref[fielddesc:SPIUSRMOSI]{SPI\_USR\_MOSI} should be configured instead, while \hyperref[fielddesc:SPIUSRMISO]{SPI\_USR\_MISO} should be cleared. The output data bit length is the value of \hyperref[fielddesc:SPIMSDATABITLEN]{SPI\_MS\_DATA\_BITLEN} + 1. Output data should be configured in GP-SPI2 data buffer (\hyperref[regdesc:SPIW0REG]{SPI\_W0\_REG} \verb+~+ \hyperref[regdesc:SPIW15REG]{SPI\_W15\_REG}) in CPU-controlled mode, or GDMA TX buffer in DMA-controlled mode. The data byte order is incremented from LSB (byte 0) to MSB.

Pay special attention to the command value in \hyperref[fielddesc:SPIUSRCOMMANDVALUE]{SPI\_USR\_COMMAND\_VALUE} and to address value in \hyperref[fielddesc:SPIUSRADDRVALUE]{SPI\_USR\_\\ADDR\_VALUE}.

The configuration of command value is as follows:

\begin{itemize}

\item If \hyperref[fielddesc:SPIUSRCOMMANDBITLEN]{SPI\_USR\_COMMAND\_BITLEN} < 8, the command value is written to \hyperref[fielddesc:SPIUSRCOMMANDVALUE]{SPI\_USR\_COMMAND\_VALUE}[7:0]. Command value is sent as follows.

\end{itemize}

\item If 7 < \hyperref[fielddesc:SPIUSRCOMMANDBITLEN]{SPI\_USR\_COMMAND\_BITLEN} < 16, the command value is written to \hyperref[fielddesc:SPIUSRCOMMANDVALUE]{SPI\_USR\_COMMAND\_VALUE}[15:\\0]. Command value is sent as follows.

\end{itemize}


The configuration of address value is as follows:

\subsubsection{Full-Duplex Communication (1-bit Mode Only)}\label{sec:spi-full-duplex-1-bit}

\textbf{Introduction}

GP-SPI2 supports SPI full-duplex communication. In this mode, SPI master provides CLK and CS signals, exchanging data with SPI slave in 1-bit mode via MOSI (FSPID, sending) and MISO (FSPIQ, receiving) at the same time. To enable this communication mode, set the bit \hyperref[fielddesc:SPIDOUTDIN]{SPI\_DOUTDIN} in register \hyperref[regdesc:SPIUSERREG]{SPI\_USER\_REG}. Figure \ref{fig:spi-full-duplex-gp-spi2-master-comm-slave} illustrates the connection of GP-SPI2 with its slave in full-duplex communication.

\begin{figure}[H]
    \centering
    \includegraphics[width=0.4\textwidth]{19-SPI/figures/19-spi-gp-spi2-fd}
    \caption{Full-Duplex Communication Between GP-SPI2 Master and a Slave}
    \label{fig:spi-full-duplex-gp-spi2-master-comm-slave}
\end{figure}

In full-duplex communication, the behavior of states CMD, ADDR, DUMMY, DOUT and DIN are configurable. Usually, the states CMD, ADDR and DUMMY are not used in this communication. The bit length of transferred data is configured in \hyperref[fielddesc:SPIMSDATABITLEN]{SPI\_MS\_DATA\_BITLEN}. The actual bit length used in communication equals to (\hyperref[fielddesc:SPIMSDATABITLEN]{SPI\_MS\_DATA\_BITLEN} + 1).

\textbf{Configuration}

To start a data transfer, follow the steps below:
\begin{itemize}
    \item Configure the IO path via IO MUX or GPIO matrix between GP-SPI2 and an external SPI device.
    \item Configure APB clock (APB\_CLK, see Chapter \ref{mod:resclk} \textit{\nameref{mod:resclk}}) and module clock (clk\_spi\_mst) for the GP-SPI2 module.
    \item Set \hyperref[fielddesc:SPIDOUTDIN]{SPI\_DOUTDIN} and clear \hyperref[fielddesc:SPISLAVEMODE]{SPI\_SLAVE\_MODE}, to enable full-duplex communication in master mode.

    \item Configure GP-SPI2 registers listed in Table \ref{tab:spi-reg-state-ctrl-1-2-4-bit}.

    \item Configure SPI CS setup time and hold time according to Section \ref{sec:spi-cs-setup-hold-time-ctrl}.

    \item Set the property of FSPICLK according to Section \ref{sec:spi-gp-spi2-clk-ctrl}.

    \item Prepare data according to the selected transfer mode:
    \begin{itemize}
        \item In CPU-controlled MOSI mode, prepare data in registers \hyperref[regdesc:SPIW0REG]{SPI\_W0\_REG} \verb+~+ \hyperref[regdesc:SPIW15REG]{SPI\_W15\_REG}.

        \item In DMA-controlled mode,
        \begin{itemize}
            \item configure \hyperref[fielddesc:SPIDMATXENA]{SPI\_DMA\_TX\_ENA}/\hyperref[fielddesc:SPIDMARXENA]{SPI\_DMA\_RX\_ENA}
            \item configure GDMA TX/RX link
            \item start GDMA TX/RX engine, as described in Section \ref{sec:spi-func-descr-dma-ctrl-data-transfer} and Section \ref{sec:spi-func-descr-data-flow-ctrl-gp-spi2-master-slave}.
        \end{itemize}
    \end{itemize}

    \item Configure interrupts \label{other:spi-item-config-interrupts-spi-slave} and wait for SPI slave to get ready for transfer.

    \item Set \hyperref[fielddesc:SPIDMAAFIFORST]{SPI\_DMA\_AFIFO\_RST}, \hyperref[fielddesc:SPIBUFAFIFORST]{SPI\_BUF\_AFIFO\_RST}, and \hyperref[fielddesc:SPIRXAFIFORST]{SPI\_RX\_AFIFO\_RST} to reset these buffers.

    \item Set \hyperref[fielddesc:SPIUSR]{SPI\_USR} in register \hyperref[regdesc:SPICMDREG]{SPI\_CMD\_REG} to start the transfer and wait for the configured interrupts.

\end{itemize}
\subsubsection{Half-Duplex Communication (1/2/4-bit Mode)}
\label{sec:spi-half-duplex-1-2-4-bit}

\textbf{Introduction}

In this mode, GP-SPI2 provides CLK and CS signals. Only one side (SPI master or slave) can send data at a time, while the other side receives the data. To enable this communication mode, clear the bit \hyperref[fielddesc:SPIDOUTDIN]{SPI\_DOUTDIN} in register \hyperref[regdesc:SPIUSERREG]{SPI\_USER\_REG}. The standard format of SPI half-duplex communication is CMD + [ADDR +] [DUMMY +] [DOUT or DIN]. The states ADDR, DUMMY, DOUT, and DIN are optional, and can be disabled or enabled independently.

As described in Section \ref{sec:spi-reg-config-state-bit-mode-ctrl}, the properties of GP-SPI2 states: CMD, ADDR, DUMMY, DOUT and DIN, such as cycle length, value, and parallel bus bit mode, can be set independently. For the register configuration, see Table \ref{tab:spi-reg-state-ctrl-1-2-4-bit}.

The detailed properties of half-duplex GP-SPI2 are as follows:

\begin{enumerate}
    \item CMD: 0 \verb+~+ 16 bits, master output, slave input.
    \item ADDR: 0 \verb+~+ 32 bits, master output, slave input.
    \item DUMMY: 0 \verb+~+ 256 FSPICLK cycles, master output, slave input.
    \item DOUT: 0 \verb+~+ 512 bits (64 B) in CPU-controlled mode and 0 \verb+~+ 256 Kbits (32 KB) in DMA-controlled mode, master output, slave input.

    \item DIN: 0 \verb+~+ 512 bits (64 B) in CPU-controlled mode and 0 \verb+~+ 256 Kbits (32 KB) in DMA-controlled mode, master input, slave output.
\end{enumerate}

\textbf{Configuration}

The register configuration is as follows:
\begin{enumerate}
    \item Configure the IO path via IO MUX or GPIO matrix between GP-SPI2 and an external SPI device.
    \item Configure APB clock (APB\_CLK) and module clock (clk\_spi\_mst) for the GP-SPI2 module.

    \item Clear \hyperref[fielddesc:SPIDOUTDIN]{SPI\_DOUTDIN} and \hyperref[fielddesc:SPISLAVEMODE]{SPI\_SLAVE\_MODE}, to enable half-duplex communication in master mode.

    \item Configure GP-SPI2 registers listed in Table \ref{tab:spi-reg-state-ctrl-1-2-4-bit}.

    \item Configure SPI CS setup time and hold time according to Section \ref{sec:spi-cs-setup-hold-time-ctrl}.

    \item Set the property of FSPICLK according to Section \ref{sec:spi-gp-spi2-clk-ctrl}.

    \item Prepare data according to the selected transfer mode:
    \begin{itemize}
        \item In CPU-controlled MOSI mode, prepare data in registers \hyperref[regdesc:SPIW0REG]{SPI\_W0\_REG} \verb+~+ \hyperref[regdesc:SPIW15REG]{SPI\_W15\_REG}.

        \item In DMA-controlled mode,
        \begin{itemize}
            \item configure \hyperref[fielddesc:SPIDMATXENA]{SPI\_DMA\_TX\_ENA}/\hyperref[fielddesc:SPIDMARXENA]{SPI\_DMA\_RX\_ENA}
            \item configure GDMA TX/RX link
            \item start GDMA TX/RX engine, as described in Section \ref{sec:spi-func-descr-dma-ctrl-data-transfer} and Section \ref{sec:spi-func-descr-data-flow-ctrl-gp-spi2-master-slave}.
        \end{itemize}
    \end{itemize}

    \item Configure interrupts and wait for SPI slave to get ready for transfer.

    \item Set \hyperref[fielddesc:SPIDMAAFIFORST]{SPI\_DMA\_AFIFO\_RST}, \hyperref[fielddesc:SPIBUFAFIFORST]{SPI\_BUF\_AFIFO\_RST}, and \hyperref[fielddesc:SPIRXAFIFORST]{SPI\_RX\_AFIFO\_RST} to reset these buffers.

    \item Set \hyperref[fielddesc:SPIUSR]{SPI\_USR} in register \hyperref[regdesc:SPICMDREG]{SPI\_CMD\_REG} to start the transfer and wait for the configured interrupts.
\end{enumerate}

\subsubsection{DMA-Controlled Configurable Segmented Transfer} \label{sec:spi-dma-ctrl-config-seg-transfer}
\begin{tiplisting}
Note that there is no separate section on how to configure a single transfer in master mode, since the CONF state of a configurable segmented transfer can be skipped to implement a single transfer.
\end{tiplisting}

\textbf{Introduction}

When GP-SPI2 works as a master, it provides a feature named: configurable segmented transfer controlled by DMA.

A DMA-controlled transfer in master mode can be
\begin{itemize}
    \item a single transfer, consisting of only one transaction;
    \item or a configurable segmented transfer, consisting of several transactions (segments).
\end{itemize}

In a configurable segmented transfer, the registers of its each single transaction (segment) are configurable. This feature enables GP-SPI2 to do as many as transactions (segments) as configured when such transfer is triggered once by CPU. Figure \ref{fig:spi-config-seg-trans-dma-ctrl-master} shows how this feature works.



\begin{figure}[H]
    \centering
    \fbox{\includegraphics[width=0.9\textwidth]{19-SPI/figures/ESP726_GPSPI_SCT_20210819.eps}}
    \caption{Configurable Segmented Transfer in DMA-Controlled Master Mode}
    \label{fig:spi-config-seg-trans-dma-ctrl-master}
\end{figure}


As shown in Figure \ref{fig:spi-config-seg-trans-dma-ctrl-master}, the registers for one transaction (segment \regindex{n}) can be reconfigured by GP-SPI2 hardware according to the content in its Conf\_buf\regindex{n} during a CONF state, before this segment starts.

It's recommended to provide separate GDMA CONF links and CONF buffers (Conf\_buf\regindex{i} in Figure \ref{fig:spi-config-seg-trans-dma-ctrl-master}) for each CONF state. A GDMA TX link is used to connect all the CONF buffers and TX data buffers (Tx\_buf\regindex{i} in Figure \ref{fig:spi-config-seg-trans-dma-ctrl-master}) into a chain. Hence, the behavior of the FSPI bus in each segment can be controlled independently.

For example, in a configurable segmentent transfer, its segment\textcolor{red}{\it i}, segment\textcolor{red}{\it j}, and segment\textcolor{red}{\it k} can be configured to full-duplex, half-duplex MISO, and half-duplex MOSI, respectively. \textcolor{red}{\it i}, \textcolor{red}{\it j}, and \textcolor{red}{\it k} are integer variables, which can be any segment number.

Meanwhile, the state of GP-SPI2, the data length and cycle length of the FSPI bus, and the behavior of the GDMA, can be configured independently for each segment. When this whole DMA-controlled transfer (consisting of several segments) has finished, a GP-SPI2 interrupt, \hyperref[int:SPIDMASEGTRANSDONEINT]{SPI\_DMA\_SEG\_TRANS\_DONE\_INT}, is triggered.

\textbf{Configuration}

 \begin{enumerate}
    \item Configure the IO path via IO MUX or GPIO matrix between GP-SPI2 and an external SPI device.
    \item Configure APB clock (APB\_CLK) and module clock (clk\_spi\_mst) for GP-SPI2 module.

    \item Clear \hyperref[fielddesc:SPIDOUTDIN]{SPI\_DOUTDIN} and \hyperref[fielddesc:SPISLAVEMODE]{SPI\_SLAVE\_MODE}, to enable half-duplex communication in master mode.

    \item Configure GP-SPI2 registers listed in Table \ref{tab:spi-reg-state-ctrl-1-2-4-bit}.

    \item Configure SPI CS setup time and hold time according to Section \ref{sec:spi-cs-setup-hold-time-ctrl}.

    \item Set the property of FSPICLK according to Section \ref{sec:spi-gp-spi2-clk-ctrl}.

   \item Prepare descriptors for GDMA CONF buffer and TX data (optional) for each segment. Chain the descriptors of CONF buffer and TX buffers of several segments into one linked list.

   \item Similarly, prepare descriptors for RX buffers for each segment and chain them into one linked list.
   \item Configure all the needed CONF buffers, TX buffers and RX buffers, respectively for each segment before this DMA-controlled transfer begins.

   \item Point \hyperref[fielddesc:GDMAOUTLINKADDRCHN]{GDMA\_OUTLINK\_ADDR\_CH\textcolor{red}{\it{n}}} to the head address of the CONF and TX buffer descriptor linked list, and then set \hyperref[fielddesc:GDMAOUTLINKSTARTCHN]{GDMA\_OUTLINK\_START\_CH\textcolor{red}{\it{n}}} to start the TX GDMA.

   \item Clear the bit \hyperref[fielddesc:SPIRXEOFEN]{SPI\_RX\_EOF\_EN} in register \hyperref[regdesc:SPIDMACONFREG]{SPI\_DMA\_CONF\_REG}. Point \hyperref[fielddesc:GDMAINLINKADDRCHN]{GDMA\_INLINK\_ADDR\_CH\textcolor{red}{\it{n}}} to the head address of the RX buffer descriptor linked list, and then set \hyperref[fielddesc:GDMAINLINKSTARTCHN]{GDMA\_INLINK\_START\_CH\textcolor{red}{\it{n}}} to start the RX GDMA.

   \item Set \hyperref[fielddesc:SPIUSRCONF]{SPI\_USR\_CONF} to enable CONF state.
   \item Set \hyperref[fielddesc:SPIDMASEGTRANSDONEINTENA]{SPI\_DMA\_SEG\_TRANS\_DONE\_INT\_ENA} to enable the \hyperref[int:SPIDMASEGTRANSDONEINT]{SPI\_DMA\_SEG\_TRANS\_DONE\_INT} interrupt. Configure other interrupts if needed according to Section \ref{sec:spi-interrupts}.

   \item Wait for all the slaves to get ready for transfer.

    \item Set \hyperref[fielddesc:SPIDMAAFIFORST]{SPI\_DMA\_AFIFO\_RST}, \hyperref[fielddesc:SPIBUFAFIFORST]{SPI\_BUF\_AFIFO\_RST} and \hyperref[fielddesc:SPIRXAFIFORST]{SPI\_RX\_AFIFO\_RST}, to reset these buffers.

   \item Set \hyperref[fielddesc:SPIUSR]{SPI\_USR} to start this DMA-controlled transfer.

   \item Wait for \hyperref[int:SPIDMASEGTRANSDONEINT]{SPI\_DMA\_SEG\_TRANS\_DONE\_INT} interrupt, which means this transfer has finished and the data has been stored into corresponding memory.
 \end{enumerate}

\textbf{Configuration of CONF Buffer and Magic Value}\label{other:spi-para-config-conf-buffer-magic-value}

On GP-SPI2, only registers which will change from the last transaction (segment) need to be re-configured to new values in CONF state. The configuration of other registers can be skipped (i.e. kept the same) to save time and chip resources.

The first word in GDMA CONF buffer\textcolor{red}{\it i}, called SPI\_BIT\_MAP\_WORD, defines whether each GP-SPI2 register is to be updated or not in segment\textcolor{red}{\it i}. The relation of SPI\_BIT\_MAP\_WORD and GP-SPI2 registers to update can be seen in Table \ref{tab:spi-master-bm-table-conf-state} Bitmap (BM) Table. If a bit in the BM table is set to 1, its corresponding register value will be updated in this segment. Otherwise, if some registers should be kept from being changed, the related bits should be set to 0.

Then new values of all the registers to be modified should be placed right after SPI\_BIT\_MAP\_WORD, in consecutive words in the CONF buffer.

To ensure the correctness of the content in each CONF buffer, the value in SPI\_BIT\_MAP\_WORD[31:28] is used as ``magic value", and will be compared with \hyperref[fielddesc:SPIDMASEGMAGICVALUE]{SPI\_DMA\_SEG\_MAGIC\_VALUE} in the register \hyperref[regdesc:SPISLAVEREG]{SPI\_SLAVE\_REG}. The value of \hyperref[fielddesc:SPIDMASEGMAGICVALUE]{SPI\_DMA\_SEG\_MAGIC\_VALUE} should be configured before this DMA-controlled transfer starts, and can not be changed during these segments.

    \begin{itemize}
        \item If SPI\_BIT\_MAP\_WORD[31:28] == \hyperref[fielddesc:SPIDMASEGMAGICVALUE]{SPI\_DMA\_SEG\_MAGIC\_VALUE}, this DMA-controlled transfer continues normally; the interrupt \hyperref[int:SPIDMASEGTRANSDONEINT]{SPI\_DMA\_SEG\_TRANS\_DONE\_INT} is triggered at the end of this DMA-controlled transfer.
        \item If SPI\_BIT\_MAP\_WORD[31:28] != \hyperref[fielddesc:SPIDMASEGMAGICVALUE]{SPI\_DMA\_SEG\_MAGIC\_VALUE}, GP-SPI2 state (spi\_st) goes back to IDLE and the transfer is ended immediately. The interrupt \hyperref[int:SPIDMASEGTRANSDONEINT]{SPI\_DMA\_SEG\_TRANS\_DONE\_INT} is still triggered, with \hyperref[fielddesc:SPISEGMAGICERRINTRAW]{SPI\_SEG\_MAGIC\_ERR\_INT\_RAW} bit set to 1.
    \end{itemize}

\subsection{GP-SPI2 Works as a Slave}\label{sec:spi-func-descr-gp-spi2-slave}

GP-SPI2 can be used as a slave to communicate with an SPI master. As a slave, GP-SPI2 supports 1-bit SPI, 2-bit dual SPI, 4-bit quad SPI, and QPI modes, with specific communication formats. To enable this mode, set \hyperref[fielddesc:SPISLAVEMODE]{SPI\_SLAVE\_MODE} in register \hyperref[regdesc:SPISLAVEREG]{SPI\_SLAVE\_REG}.

The CS signal must be held low during the transmission, and its falling/rising edges indicate the start/end of a single or segmented transmission. The length of transferred data must be in unit of bytes, otherwise the extra bits will be lost. The extra bits here means the result of total bits \% 8.

\subsubsection{Communication Formats}\label{sec:spi-slave-communication-format}

In GP-SPI2 slave mode, SPI full-duplex and half-duplex communications are available. To select from the two communications, configure \hyperref[fielddesc:SPIDOUTDIN]{SPI\_DOUTDIN} in register \hyperref[regdesc:SPIUSERREG]{SPI\_USER\_REG}.

Full-duplex communication means that input data and output data are transmitted simultaneously throughout the entire transaction. All bits are treated as input or output data, which means no command, address or dummy states are expected. The interrupt \hyperref[int:SPITRANSDONEINT]{SPI\_TRANS\_DONE\_INT} is triggered once the transaction ends.

In half-duplex communication, the format is CMD+ADDR+DUMMY+DATA (DIN or DOUT).
\begin{itemize}
    \item ``DIN" means that an SPI master reads data from GP-SPI2.
    \item ``DOUT" means that an SPI master writes data to GP-SPI2.
\end{itemize}

The detailed properties of each state are as follows:

\begin{enumerate}
   \item CMD:
        \begin{itemize}
            \item Indicate the function of SPI slave;
            \item One byte from master to slave;
            \item Only the values in Table \ref{tab:spi-cmd-values-spi-mode} and Table \ref{tab:spi-cmd-values-qpi-mode} are valid;
            \item Can be sent in 1-bit SPI mode or 4-bit QPI mode.
        \end{itemize}

   \item ADDR:
        \begin{itemize}
            \item The address for \hyperlink{other:spi-item-wrbuf}{Wr\_BUF} and \hyperlink{other:spi-item-rdbuf}{Rd\_BUF} commands in CPU-controlled transfer, or placeholder bits in other transfers and can be defined by application;
            \item One byte from master to slave;
            \item Can be sent in 1-bit, 2-bit or 4-bit modes (according to the command).
        \end{itemize}

   \item DUMMY:
        \begin{itemize}
            \item It's value is meaningless. SPI slave prepares data in this state;
            \item Bit mode of FSPI bus is also meaningless here;
            \item Last for eight SPI\_CLK cycles.
        \end{itemize}

   \item DIN or DOUT:

        \begin{itemize}
            \item Data length can be 0 \verb+~+ 64 B in CPU-controlled mode and unlimited in DMA-controlled mode;
            \item Can be sent in 1-bit, 2-bit or 4-bit modes according to the CMD value.
        \end{itemize}

   \end{enumerate}

\begin{tiplisting}
The states of ADDR and DUMMY can never be omitted in any half-duplex communications.
\end{tiplisting}
\vspace{-1em}
When a half-duplex transaction is complete, the transferred CMD and ADDR values are latched into \hyperref[fielddesc:SPISLVLASTCOMMAND]{SPI\_SLV\_\\LAST\_COMMAND} and \hyperref[fielddesc:SPISLVLASTADDR]{SPI\_SLV\_LAST\_ADDR} respectively. The \hyperref[fielddesc:SPISLVCMDERRINTRAW]{SPI\_SLV\_CMD\_ERR\_INT\_RAW} will be set if the transferred CMD value is not supported by GP-SPI2 slave mode. The \hyperref[fielddesc:SPISLVCMDERRINTRAW]{SPI\_SLV\_CMD\_ERR\_INT\_RAW} can only be cleared by software.



\subsubsection{Supported CMD Values in Half-Duplex Communication} \label{sec:spi-cmd-values-half-duplex}

In half-duplex communication, the defined values of CMD determine the transfer types. Unsupported CMD values are disregarded, meanwhile the related transfer is ignored and \hyperref[fielddesc:SPISLVCMDERRINTRAW]{SPI\_SLV\_CMD\_ERR\_INT\_RAW} is set. The transfer format is CMD (8 bits) + ADDR (8 bits) + DUMMY (8 SPI\_CLK cycles) + DATA (unit in bytes). The detailed description of CMD[3:0] is as follows:

\begin{enumerate}
    \item \label{other:spi-item-wrbuf} 0x1 (Wr\_BUF): CPU-controlled write mode. Master sends data and GP-SPI2 receives data. The data is stored in the related address of \hyperref[regdesc:SPIW0REG]{SPI\_W0\_REG} \verb+~+ \hyperref[regdesc:SPIW15REG]{SPI\_W15\_REG}.

    \item \label{other:spi-item-rdbuf} 0x2 (Rd\_BUF): CPU-controlled read mode. Master receives the data sent by GP-SPI2. The data comes from the related address of \hyperref[regdesc:SPIW0REG]{SPI\_W0\_REG} \verb+~+ \hyperref[regdesc:SPIW15REG]{SPI\_W15\_REG}.

    \item \label{other:spi-item-wrdma} 0x3 (Wr\_DMA): DMA-controlled write mode. Master sends data and GP-SPI2 receives data. The data is stored in GP-SPI2 GDMA RX buffer.

    \item 0x4 (Rd\_DMA): DMA-controlled read mode. Master receives the data sent by GP-SPI2. The data comes from GP-SPI2 GDMA TX buffer.

    \item \label{other:spi-item-cmd7} 0x7 (CMD7): used to generate an \hyperref[int:SPISLVCMD7INT]{SPI\_SLV\_CMD7\_INT} interrupt. It can also generate a GDMA\_IN\_SUC\_EOF\\\_CH\textcolor{red}{\it{n}}\_INT interrupt in a slave segmented transfer when GDMA RX link is used. But it will not end GP-SPI2's slave segmented transfer.

    \item 0x8 (CMD8): only used to generate an \hyperref[int:SPISLVCMD8INT]{SPI\_SLV\_CMD8\_INT} interrupt, which will not end GP-SPI2's slave segmented transfer.

    \item 0x9 (CMD9): only used to generate an \hyperref[int:SPISLVCMD9INT]{SPI\_SLV\_CMD9\_INT} interrupt, which will not end GP-SPI2's slave segmented transfer.

    \item 0xA (CMDA): only used to generate an \hyperref[int:SPISLVCMDAINT]{SPI\_SLV\_CMDA\_INT} interrupt, which will not end GP-SPI2's slave segmented transfer.

\end{enumerate}

The detail function of CMD7, CMD8, CMD9, and CMDA commands is reserved for user definition. These commands can be used as handshake signals, the passwords of some specific functions, the triggers of some user defined actions, and so on.

1/2/4-bit modes in states of CMD, ADDR, DATA are supported, which are determined by value of CMD[7:4]. The DUMMY state is always in 1-bit mode and lasts for eight SPI\_CLK cycles. The definition of CMD[7:4] is as follows:

\begin{enumerate}
    \item 0x0: CMD, ADDR, and DATA states all are in 1-bit mode.
    \item 0x1: CMD and ADDR are in 1-bit mode. DATA is in 2-bit mode.
    \item 0x2: CMD and ADDR are in 1-bit mode. DATA is in 4-bit mode.
    \item 0x5: CMD is in 1-bit mode. ADDR and DATA are in 2-bit mode.
    \item 0xA: CMD is in 1-bit mode, ADDR and DATA are in 4-bit mode. Or in QPI mode.
\end{enumerate}

In addition, if the value of CMD[7:0] is 0x05, 0xA5, 0x06, or 0xDD, DUMMY and DATA states are omitted. The definition of CMD[7:0] is as follows:

\begin{enumerate}
    \item \label{other:spi-item-end-seg-trans}0x05 (End\_SEG\_TRANS): master sends 0x05 command to end slave segmented transfer in SPI mode.
    \item 0xA5 (End\_SEG\_TRANS): master sends 0xA5 command to end slave segmented transfer in QPI mode.
    \item 0x06 (En\_QPI): GP-SPI2 enters QPI mode when receiving the 0x06 command and the bit \hyperref[fielddesc:SPIQPIMODE]{SPI\_QPI\_MODE} in register \hyperref[regdesc:SPIUSERREG]{SPI\_USER\_REG} is set.
    \item 0xDD (Ex\_QPI): GP-SPI2 exits QPI mode when receiving the 0xDD command and the bit \hyperref[fielddesc:SPIQPIMODE]{SPI\_QPI\_MODE} is cleared.
\end{enumerate}

\subsubsection{Slave Single Transfer and Slave Segmented Transfer}\label{sec:spi-slave-single-seg-trans}

When GP-SPI2 works as a slave, it supports full-duplex and half-duplex communications controlled by DMA and by CPU. DMA-controlled transfer can be a single transfer, or a slave segmented transfer consisting of several transactions (segments). The CPU-controlled transfer can only be one single transfer, since each CPU-controlled transaction needs to be triggered by CPU.

In a slave segmented transfer, all transfer types listed in Table \ref{tab:spi-cmd-values-spi-mode} and Table \ref{tab:spi-cmd-values-qpi-mode} are supported in a single transaction (segment). It means that CPU-controlled transaction and DMA-controlled transaction can be mixed in one slave segmented transfer.

It is recommended that in a slave segmented transfer:

\begin{itemize}
    \item CPU-controlled transaction is used for handshake communication and short data transfers.

    \item  DMA-controlled transaction is used for large data transfers.
\end{itemize}


\subsubsection{Configuration of Slave Single Transfer}\label{sec:spi-config-slave-single-trans}

In slave mode, GP-SPI2 supports CPU/DMA-controlled full-duplex/half-duplex single transfers. The register configuration procedure is as follows:
\begin{enumerate}
    \item Configure the IO path via IO MUX or GPIO matrix between GP-SPI2 and an external SPI device.

    \item Configure APB clock (APB\_CLK).

    \item Set the bit \hyperref[fielddesc:SPISLAVEMODE]{SPI\_SLAVE\_MODE}, to enable slave mode.

    \item Configure \hyperref[fielddesc:SPIDOUTDIN]{SPI\_DOUTDIN}:
    \begin{itemize}
        \item 1: enable full-duplex communication.
        \item 0: enable half-duplex communication.
    \end{itemize}

    \item Prepare data:
    \begin{itemize}
        \item if CPU-controlled transfer mode is selected and GP-SPI2 is used to send data, then prepare data in registers \hyperref[regdesc:SPIW0REG]{SPI\_W0\_REG} \verb+~+ \hyperref[regdesc:SPIW15REG]{SPI\_W15\_REG}.
        \item if DMA-controlled transfer mode is selected,
        \begin{itemize}
            \item configure \hyperref[fielddesc:SPIDMATXENA]{SPI\_DMA\_TX\_ENA}/\hyperref[fielddesc:SPIDMARXENA]{SPI\_DMA\_RX\_ENA} and \hyperref[fielddesc:SPIRXEOFEN]{SPI\_RX\_EOF\_EN}.
            \item configure GDMA TX/RX link.
            \item start GDMA TX/RX engine, as described in Section \ref{sec:spi-func-descr-dma-ctrl-data-transfer} and Section \ref{sec:spi-func-descr-data-flow-ctrl-gp-spi2-master-slave}.
        \end{itemize}
    \end{itemize}

    \item Set \hyperref[fielddesc:SPIDMAAFIFORST]{SPI\_DMA\_AFIFO\_RST}, \hyperref[fielddesc:SPIBUFAFIFORST]{SPI\_BUF\_AFIFO\_RST}, and \hyperref[fielddesc:SPIRXAFIFORST]{SPI\_RX\_AFIFO\_RST} to reset these buffers.

    \item Clear \hyperref[fielddesc:SPIDMASLVSEGTRANSEN]{SPI\_DMA\_SLV\_SEG\_TRANS\_EN} in register \hyperref[regdesc:SPIDMACONFREG]{SPI\_DMA\_CONF\_REG} to enable slave single transfer mode.

    \item Set \hyperref[fielddesc:SPITRANSDONEINTENA]{SPI\_TRANS\_DONE\_INT\_ENA} in \hyperref[regdesc:SPIDMAINTENAREG]{SPI\_DMA\_INT\_ENA\_REG} and wait for the interrupt \hyperref[int:SPITRANSDONEINT]{SPI\_TRANS\_DONE\_INT}. In DMA-controlled mode, it is recommended to wait for the interrupt GDMA\_IN\_SUC\_EOF\_CH\textcolor{red}{\it{n}}\_INT when GDMA RX buffer is used, which means that data has been stored in the related memory. Other interrupts described in Section \ref{sec:spi-interrupts} are optional.
\end{enumerate}


\subsubsection{Configuration of Slave Segmented Transfer in Half-Duplex}\label{sec:spi-config-slave-seg-trans-half-duplex}

GDMA must be used in this mode. The register configuration procedure is as follows:

\begin{enumerate}
   \item  Configure the IO path via IO MUX or GPIO matrix between GP-SPI2 and an external SPI device.

   \item Configure APB clock (APB\_CLK).

    \item Set \hyperref[fielddesc:SPISLAVEMODE]{SPI\_SLAVE\_MODE} to enable slave mode.

    \item Clear \hyperref[fielddesc:SPIDOUTDIN]{SPI\_DOUTDIN} to enable half-duplex communication.

    \item Prepare data in registers \hyperref[regdesc:SPIW0REG]{SPI\_W0\_REG} \verb+~+ \hyperref[regdesc:SPIW15REG]{SPI\_W15\_REG}, if needed.

    \item Set \hyperref[fielddesc:SPIDMAAFIFORST]{SPI\_DMA\_AFIFO\_RST}, \hyperref[fielddesc:SPIBUFAFIFORST]{SPI\_BUF\_AFIFO\_RST} and \hyperref[fielddesc:SPIRXAFIFORST]{SPI\_RX\_AFIFO\_RST} to reset these buffers.

    \item Set bits \hyperref[fielddesc:SPIDMARXENA]{SPI\_DMA\_RX\_ENA} and \hyperref[fielddesc:SPIDMATXENA]{SPI\_DMA\_TX\_ENA}. Clear the bit \hyperref[fielddesc:SPIRXEOFEN]{SPI\_RX\_EOF\_EN}. Configure GDMA TX/RX link and start GDMA TX/RX engine, as shown in Section \ref{sec:spi-func-descr-dma-ctrl-data-transfer} and Section \ref{sec:spi-func-descr-data-flow-ctrl-gp-spi2-master-slave}.

    \item Set \hyperref[fielddesc:SPIDMASLVSEGTRANSEN]{SPI\_DMA\_SLV\_SEG\_TRANS\_EN} in \hyperref[regdesc:SPIDMACONFREG]{SPI\_DMA\_CONF\_REG} to enable slave segmented transfer.

    \item Set \hyperref[fielddesc:SPIDMASEGTRANSDONEINTENA]{SPI\_DMA\_SEG\_TRANS\_DONE\_INT\_ENA} in \hyperref[regdesc:SPIDMAINTENAREG]{SPI\_DMA\_INT\_ENA\_REG} and wait for the interrupt \hyperref[int:SPIDMASEGTRANSDONEINT]{SPI\_\\DMA\_SEG\_TRANS\_DONE\_INT}, which means that the segmented transfer has finished and data has been put into the related memory. Other interrupts described in Section \ref{sec:spi-interrupts} are optional.
\end{enumerate}

When End\_SEG\_TRANS (0x05 in SPI mode, 0xA5 in QPI mode) is received by GP-SPI2, this slave segmented transfer is ended and the interrupt \hyperref[int:SPIDMASEGTRANSDONEINT]{SPI\_DMA\_SEG\_TRANS\_DONE\_INT} is triggered.

\subsubsection{Configuration of Slave Segmented Transfer in Full-Duplex}\label{sec:spi-config-slave-seg-trans-full-duplex}
GDMA must be used in this mode. In such transfer, the data is transferred from and to the GDMA buffer. The interrupt GDMA\_IN\_SUC\_EOF\_CH\textcolor{red}{\it{n}}\\\_INT is triggered when the transfer ends. The configuration procedure is as follows:

\begin{enumerate}
   \item Configure the IO path via IO MUX or GPIO matrix between GP-SPI2 and an external SPI device.
   \item Configure APB clock (APB\_CLK).

    \item Set \hyperref[fielddesc:SPISLAVEMODE]{SPI\_SLAVE\_MODE} and \hyperref[fielddesc:SPIDOUTDIN]{SPI\_DOUTDIN}, to enable full-duplex communication in slave mode.

    \item Set \hyperref[fielddesc:SPIDMAAFIFORST]{SPI\_DMA\_AFIFO\_RST}, \hyperref[fielddesc:SPIBUFAFIFORST]{SPI\_BUF\_AFIFO\_RST}, and \hyperref[fielddesc:SPIRXAFIFORST]{SPI\_RX\_AFIFO\_RST} bit, to reset these buffers.

    \item Set \hyperref[fielddesc:SPIDMATXENA]{SPI\_DMA\_TX\_ENA}/\hyperref[fielddesc:SPIDMARXENA]{SPI\_DMA\_RX\_ENA}. Configure GDMA TX/RX link and start GDMA TX/RX engine, as shown in Section \ref{sec:spi-func-descr-dma-ctrl-data-transfer} and Section \ref{sec:spi-func-descr-data-flow-ctrl-gp-spi2-master-slave}.

    \item Set the bit \hyperref[fielddesc:SPIRXEOFEN]{SPI\_RX\_EOF\_EN} in register \hyperref[regdesc:SPIDMACONFREG]{SPI\_DMA\_CONF\_REG}. Configure \hyperref[fielddesc:SPIMSDATABITLEN]{SPI\_MS\_DATA\_BITLEN}[17:0] in register \hyperref[regdesc:SPIMSDLENREG]{SPI\_MS\_DLEN\_REG} to the byte length of the received DMA data.

    \item Set \hyperref[fielddesc:SPIDMASLVSEGTRANSEN]{SPI\_DMA\_SLV\_SEG\_TRANS\_EN} in \hyperref[regdesc:SPIDMACONFREG]{SPI\_DMA\_CONF\_REG} to enable slave segmented transfer mode.

    \item Set \hyperref[fielddesc:GDMAINSUCEOFCHNINTENA]{GDMA\_IN\_SUC\_EOF\_CH\textcolor{red}{\it{n}}\_INT\_ENA} and wait for the interrupt \hyperref[int:GDMAINSUCEOFINT]{GDMA\_IN\_SUC\_EOF\_CH\textcolor{red}{\it{n}}\_INT}.
\end{enumerate}


\section{CS Setup Time and Hold Time Control}\label{sec:spi-cs-setup-hold-time-ctrl}

SPI bus CS (SPI\_CS) setup time and hold time are very important to meet the timing requirements of various SPI devices (e.g. flash or PSRAM).

CS setup time is the time between the CS falling edge and the first latch edge of SPI bus CLK (SPI\_CLK). The first latch edge for mode 0 and mode 3 is rising edge, and falling edge for mode 1 and mode 2.

CS hold time is the time between the last latch edge of SPI\_CLK and the CS rising edge.

In slave mode, the CS setup time and hold time should be longer than 0.5 x T\_SPI\_CLK, otherwise the SPI transfer may be incorrect. T\_SPI\_CLK: one cycle of SPI\_CLK.

In master mode, set the CS setup time by specifying \hyperref[fielddesc:SPICSSETUP]{SPI\_CS\_SETUP} in \hyperref[regdesc:SPIUSERREG]{SPI\_USER\_REG} and  \hyperref[fielddesc:SPICSSETUPTIME]{SPI\_CS\_SETUP\_TIME} in \hyperref[regdesc:SPIUSER1REG]{SPI\_USER1\_REG}:
    \begin{itemize}
        \item If \hyperref[fielddesc:SPICSSETUP]{SPI\_CS\_SETUP} is cleared, the SPI CS setup time is 0.5 x T\_SPI\_CLK.
        \item If \hyperref[fielddesc:SPICSSETUP]{SPI\_CS\_SETUP} is set, the SPI CS setup time is (\hyperref[fielddesc:SPICSSETUPTIME]{SPI\_CS\_SETUP\_TIME} + 1.5) x T\_SPI\_CLK.
    \end{itemize}

Set the CS hold time by specifying \hyperref[fielddesc:SPICSHOLD]{SPI\_CS\_HOLD} in \hyperref[regdesc:SPIUSERREG]{SPI\_USER\_REG} and  \hyperref[fielddesc:SPICSHOLDTIME]{SPI\_CS\_HOLD\_TIME} in \hyperref[regdesc:SPIUSER1REG]{SPI\_USER1\_REG}:
    \begin{itemize}
        \item If \hyperref[fielddesc:SPICSHOLD]{SPI\_CS\_HOLD} is cleared, the SPI CS hold time is 0.5 x T\_SPI\_CLK;
        \item If \hyperref[fielddesc:SPICSHOLD]{SPI\_CS\_HOLD} is set, the SPI CS hold time is (\hyperref[fielddesc:SPICSHOLDTIME]{SPI\_CS\_HOLD\_TIME} + 1.5) x T\_SPI\_CLK.
    \end{itemize}

\section{GP-SPI2 Clock Control}\label{sec:spi-gp-spi2-clk-ctrl}

GP-SPI2 has the following clocks:
\begin{itemize}
    \item clk\_spi\_mst: module clock of GP-SPI2, derived from PLL\_CLK. Used in GP-SPI2 master mode, to generate SPI\_CLK signal for data transfer and for slaves.
    \item SPI\_CLK: output clock in master mode.
    \item APB\_CLK: clock for register configuration.
    \item clk\_hclk\label{clk:clkhclk}: module timing compensation clock of GP-SPI2.
\end{itemize}

In master mode, the maximum output clock frequency of GP-SPI2 is $f_{\textrm{clk\_spi\_mst}}$. To have slower frequencies, the output clock frequency can be divided as follows:

    \[
    f_{\textrm{SPI\_CLK}}=\frac{f_{\textrm{clk\_spi\_mst}}} {\textrm{(\hyperref[fielddesc:SPICLKCNTN]{SPI\_CLKCNT\_N} + 1)(\hyperref[fielddesc:SPICLKDIVPRE]{SPI\_CLKDIV\_PRE} + 1)}}
    \] \label{other:spi-formula-gp-spi2-output-clk-frequency}

The divider is configured by \hyperref[fielddesc:SPICLKCNTN]{SPI\_CLKCNT\_N} and \hyperref[fielddesc:SPICLKDIVPRE]{SPI\_CLKDIV\_PRE} in register \hyperref[regdesc:SPICLOCKREG]{SPI\_CLOCK\_REG}. When the bit \hyperref[fielddesc:SPICLKEQUSYSCLK]{SPI\_CLK\_EQU\_SYSCLK} in register \hyperref[regdesc:SPICLOCKREG]{SPI\_CLOCK\_REG} is set to 1, the output clock frequency of GP-SPI2 will be $f_{\textrm{clk\_spi\_mst}}$. And for other integral clock divisions, \hyperref[fielddesc:SPICLKEQUSYSCLK]{SPI\_CLK\_EQU\_SYSCLK} should be set to 0.

In slave mode, the supported input clock frequency ($f_{\textrm{SPI\_CLK}}$) of GP-SPI2 is:
\begin{itemize}
    \item If $f_{\textrm{APB\_CLK}}$ >= 60 MHz, $f_{\textrm{SPI\_CLK}}$ <= 60 MHz;

    \item If $f_{\textrm{APB\_CLK}}$ < 60 MHz, $f_{\textrm{SPI\_CLK}}$ <= $f_{\textrm{APB\_CLK}}$.
\end{itemize}

    \item Mode 0: CPOL = 0, CPHA = 0; SCK is 0 when the SPI is in idle state; data is changed on the negative edge of SCK and sampled on the positive edge. The first data is shifted out before the first negative edge of SCK.

\end{enumerate}

\section{GP-SPI2 Timing Compensation} \label{sec:spi-gp-spi2-timing-compensation}

\textbf{Introduction}

The I/O lines are mapped via GPIO Matrix or IO MUX. But there is no timing adjustment in IO MUX. The input data and output data can be delayed for 1 or 2 APB\_CLK cycles at the rising or falling edge in GPIO matrix. For detailed register configuration, see Chapter \ref{mod:iomuxgpio} \textit{\nameref{mod:iomuxgpio}}.

Figure \ref{fig:spi-gp-spi2-timing-compensation-ctrl-diag-master} shows the timing compensation control for GP-SPI2 master mode, including the following paths:
\begin{itemize}
    \item ``CLK": the output path of GP-SPI2 bus clock. The clock is sent out by SPI\_CLK out control module, passes through GPIO Matrix or IO MUX and then goes to an external SPI device.

    \item ``IN": data input path of GP-SPI2. The input data from an external SPI device passes through GPIO Matrix or IO MUX, then is adjusted by the Timing Module and finally is stored into spi\_rx\_afifo.

    \item ``OUT": data output path of GP-SPI2. The output data is sent out to the Timing Module, passes through GPIO Matrix or IO MUX and is then captured by an external SPI device.
\end{itemize}

\begin{figure}[H]
    \centering
    \fbox{\includegraphics[width=0.9\textwidth]{19-SPI/figures/ESP726_GPSPI_Timing_Block.eps}}
    \caption{Timing Compensation Control Diagram in GP-SPI2 Master Mode}
    \label{fig:spi-gp-spi2-timing-compensation-ctrl-diag-master}
\end{figure}


Every input and output data is passing through the Timing Module and the module can be used to apply delay in units of $T_{\textrm{clk\_spi\_mst}}$ (one cycle of clk\_spi\_mst) on rising or falling edge.

\textbf{Key Registers}

\begin{itemize}
    \item \hyperref[regdesc:SPIDINMODEREG]{SPI\_DIN\_MODE\_REG}: select the latch edge of input data
    \item \hyperref[regdesc:SPIDINNUMREG]{SPI\_DIN\_NUM\_REG}: select the delay cycles of input data
    \item \hyperref[regdesc:SPIDOUTMODEREG]{SPI\_DOUT\_MODE\_REG}: select the latch edge of output data
\end{itemize}


\textbf{Timing Compensation Example}

Figure \ref{fig:spi-gp-spi2-timing-compensation-example-master} shows a timing compensation example in GP-SPI2 master mode. Note that DUMMY cycle length is configurable to compensate the delay in I/O lines, so as to enhance the performance of GP-SPI2.

\begin{figure}[H]
    \centering
    \fbox{\includegraphics[width=0.7\textwidth]{19-SPI/figures/ESP726_GPSPI_Timing_Seq.eps}}
    \caption{Timing Compensation Example in GP-SPI2 Master Mode}
    \label{fig:spi-gp-spi2-timing-compensation-example-master}
\end{figure}

In Figure \ref{fig:spi-gp-spi2-timing-compensation-example-master}, "p1" is the point of input data of Timing Module, "p2" is the point of output data of Timing Module. Since the input data FSPIQ is unaligned to FSPID, the read data of GP-SPI2 will be wrong without the timing compensation.

To get correct read data, follow the the settings below, assuing $f_{clk\_spi\_mst}$ equals to $f_{SPI\_CLK}$:

\begin{itemize}
    \item Delay FSPID for two cycles at the falling edge of clk\_spi\_mst.
    \item Delay FSPIQ for one cycle at the falling edge of clk\_spi\_mst.
    \item Add one extra dummy cycle.
\end{itemize}


In GP-SPI2 slave mode, if the bit \hyperref[fielddesc:SPIRSCKDATAOUT]{SPI\_RSCK\_DATA\_OUT} in register \hyperref[regdesc:SPISLAVEREG]{SPI\_SLAVE\_REG} is set to 1, the output data is sent at latch edge, which is half an SPI clock cycle earlier. This can be used for slave mode timing compensation.


\section{Interrupts} \label{sec:spi-interrupts}

\textbf{Interrupt Summary}

GP-SPI2 provides an SPI interface interrupt SPI\_INT. When an SPI transfer ends, an interrupt is generated in GP-SPI2. The interrupt may be one or more of the following ones:

\begin{itemize}
\item \label{int:SPIINFIFOFULLERRINT}SPI\_DMA\_INFIFO\_FULL\_ERR\_INT: triggered when GDMA RX FIFO length is shorter than the real transferred data length.
\item \label{int:SPIOUTFIFOEMPTYERRINT}SPI\_DMA\_OUTFIFO\_EMPTY\_ERR\_INT: triggered when GDMA TX FIFO length is shorter than the real transferred data length.

\item \label{int:SPISLVEXQPIINT} SPI\_SLV\_EX\_QPI\_INT: triggered when Ex\_QPI is received correctly in GP-SPI2 slave mode and the SPI transfer ends.
\item \label{int:SPISLVENQPIINT} SPI\_SLV\_EN\_QPI\_INT: triggered when En\_QPI is received correctly in GP-SPI2 slave mode and the SPI transfer ends.
\item \label{int:SPISLVCMD7INT} SPI\_SLV\_CMD7\_INT: triggered when CMD7 is received correctly in GP-SPI2 slave mode and the SPI transfer ends.
\item \label{int:SPISLVCMD8INT} SPI\_SLV\_CMD8\_INT: triggered when CMD8 is received correctly in GP-SPI2 slave mode and the SPI transfer ends.
\item \label{int:SPISLVCMD9INT} SPI\_SLV\_CMD9\_INT: triggered when CMD9 is received correctly in GP-SPI2 slave mode and the SPI transfer ends.
\item \label{int:SPISLVCMDAINT} SPI\_SLV\_CMDA\_INT: triggered when CMDA is received correctly in GP-SPI2 slave mode and the SPI transfer ends.

\item \label{int:SPISLVRDDMADONEINT}SPI\_SLV\_RD\_DMA\_DONE\_INT: triggered at the end of Rd\_DMA transfer in slave mode.
\item \label{int:SPISLVWRDMADONEINT}SPI\_SLV\_WR\_DMA\_DONE\_INT: triggered at the end of Wr\_DMA transfer in slave mode.
\item \label{int:SPISLVRDBUFDONEINT}SPI\_SLV\_RD\_BUF\_DONE\_INT: triggered at the end of Rd\_BUF transfer in slave mode.
\item \label{int:SPISLVWRBUFDONEINT}SPI\_SLV\_WR\_BUF\_DONE\_INT: triggered at the end of Wr\_BUF transfer in slave mode.

\item \label{int:SPITRANSDONEINT}SPI\_TRANS\_DONE\_INT: triggered at the end of SPI bus transfer in both master and slave modes.
\item \label{int:SPIDMASEGTRANSDONEINT}SPI\_DMA\_SEG\_TRANS\_DONE\_INT: triggered at the end of End\_SEG\_TRANS transfer in GP-SPI2 slave segmented transfer mode or at the end of configurable segmented transfer in master mode.
\item \label{int:SPISEGMAGICERRINT} SPI\_SEG\_MAGIC\_ERR\_INT: triggered when a Magic error occurs in CONF buffer during configurable segmented transfer in master mode.


\item \label{int:SPIMSTRXAFIFOWFULLERRINT}
SPI\_MST\_RX\_AFIFO\_WFULL\_ERR\_INT: triggered by RX AFIFO write-full error in GP-SPI2 master mode.

\item \label{int:SPIMSTTXAFIFOREMPTYERRINT}
SPI\_MST\_TX\_AFIFO\_REMPTY\_ERR\_INT: triggered by TX AFIFO read-empty error in GP-SPI2 master mode.

\item \label{int:SPISLVCMDERRINT} SPI\_SLV\_CMD\_ERR\_INT: triggered when a received command value is not supported in GP-SPI2 slave mode.

\item \label{int:SPIAPP2INT}SPI\_APP2\_INT: used and triggered by software. It is only used for user defined function.
\item \label{int:SPIAPP1INT}SPI\_APP1\_INT: used and triggered by software. It is only used for user defined function.

\end{itemize}

\begin{tiplisting}
\vspace{-2em}
\begin{enumerate}
\item If \hyperref[int:GDMAINSUCEOFINT]{GDMA\_IN\_SUC\_EOF\_CH\textcolor{red}{\it{n}}\_INT} is triggered, it means all the RX data has been stored in the RX buffer, and the TX data has been sent to the slave. \label{other:spi-item-gp-spi2-slave-interrupts-note-1}
\item \hyperref[int:SPITRANSDONEINT]{SPI\_TRANS\_DONE\_INT} is triggered when CS is high, which indicates that master has completed the data exchange in \hyperref[regdesc:SPIW0REG]{SPI\_W0\_REG}  \verb+~+ \hyperref[regdesc:SPIW15REG]{SPI\_W15\_REG} with slave in this mode.\label{other:spi-item-gp-spi2-slave-interrupts-note-2}
\item \hyperref[int:SPISLVWRDMADONEINT]{SPI\_SLV\_WR\_DMA\_DONE\_INT} just means that the transmission on the SPI bus is done, but can not ensure that all the push data has been stored in the RX buffer. For this reason, GDMA\_IN\_SUC\_EOF\_CH\textcolor{red}{\it{n}}\_INT is recommended. \label{other:spi-item-gp-spi2-slave-interrupts-note-3}
\vspace{-1em}
\item Or wait for \hyperref[int:SPISLVWRBUFDONEINT]{SPI\_SLV\_WR\_BUF\_DONE\_INT}.\label{other:spi-item-gp-spi2-slave-interrupts-note-4}
\item Or wait for \hyperref[int:SPISLVRDDMADONEINT]{SPI\_SLV\_RD\_DMA\_DONE\_INT}.\label{other:spi-item-gp-spi2-slave-interrupts-note-5}
\item Or wait for \hyperref[int:SPISLVRDBUFDONEINT]{SPI\_SLV\_RD\_BUF\_DONE\_INT}.\label{other:spi-item-gp-spi2-slave-interrupts-note-6}
\item Slave should set the total read data byte length in \hyperref[fielddesc:SPIMSDATABITLEN]{SPI\_MS\_DATA\_BITLEN} before the transfer begins. And set \hyperref[fielddesc:SPIRXEOFEN]{SPI\_RX\_EOF\_EN} 0->1 before the end of the interrupt program.\label{other:spi-item-gp-spi2-slave-interrupts-note-7}
\item Master and slave should define a method to end the segmented transfer, such as via GPIO interrupt and so on.\label{other:spi-item-gp-spi2-slave-interrupts-note-8}
\item Master sends End\_SEG\_TRAN to end the segmented transfer or slave sets the total read data byte length in \hyperref[fielddesc:SPIMSDATABITLEN]{SPI\_MS\_DATA\_BITLEN} and waits for GDMA\_IN\_SUC\_EOF\_CH\textcolor{red}{\it{n}}\_INT.\label{other:spi-item-gp-spi2-slave-interrupts-note-9}
\item Half-duplex Wr\_BUF single transfer can be used in a DMA-controlled segmented transfer.\label{other:spi-item-gp-spi2-slave-interrupts-note-10}
\item Master sends End\_SEG\_TRAN to end the segmented transfer.\label{other:spi-item-gp-spi2-slave-interrupts-note-11}
\item Half-duplex Rd\_BUF single transfer can be used in  a DMA-controlled segmented transfer.\label{other:spi-item-gp-spi2-slave-interrupts-note-12}
\end{enumerate}
\end{tiplisting}
