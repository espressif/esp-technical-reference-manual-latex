\chapter{Low-power Management}\label{mod:lowpowm}

\section{Introduction}

\chipname{} has an advanced Power Management Unit (PMU), which can flexibly power up different power domains of the chip, to achieve the best balance among chip performance, power consumption, and wakeup latency. To simplify power management for typical scenarios, \chipname{} has predefined four power modes, which are preset configurations that power up different combinations of power domains. On top of that, the chip also allows the users to independently power up any particular power domain to meet more complex requirements.

\section{Features}
\chipname{}'s low-power management supports the following features:
\begin{itemize}
    \item Four predefined power modes to simplify power management for typical scenarios
    \item Up to 8 KB of retention memory
    \item 8 x 32-bit retention registers
    \item RTC Boot supported for reduced wakeup latency
\end{itemize}

In this chapter, we first introduce the working process of \chipname{}'s low-power management, then introduce the predefined power modes of the chip, and at last, introduce the RTC boot of the chip.

\section{Functional Description}

\chipname{}'s low-power management involves the following components:

\begin{itemize}
    \item Power management unit: controls the power supply to Analog, RTC and Digital power domains.
    \item Power isolation unit: isolates different power domains, so any powered down power domain does not affect the powered up ones.
    \item Low-power clocks: provide clocks to power domains working in low-power modes. 
    \item RTC timer: logs the status of the RTC main state machine in dedicated registers.
    \item 8 x 32-bit ``always-on" retention registers: These registers are always powered up and are not affected by any low-power modes, thus can be used for storing data that cannot be lost.
    \item 6 x ``always-on" pins: These pins are always powered up and are not affected by any low-power modes, which makes them suitable for working as wakeup sources when the chip is working in the low-power modes (for details, please refer to Section \ref{sec:lpm-wakeup-source}), or can be used as regular GPIOs (for details, please refer to Chapter \ref{mod:iomuxgpio} \textit{\nameref{mod:iomuxgpio}}).
    \item RTC fast memory: 8 KB SRAM that works under CPU clock (CPU\_CLK), which can be used as extended memory.
    \item Voltage regulators: regulate the power supply to different power domains.
\end{itemize}

\subsection{Power Management Unit (PMU)}

\chipname{}'s power management unit controls the power supply to different power domains. The main components of the power management unit include:

\begin{itemize}
    \item RTC main state machine: generates power gating, clock gating, and reset signals.
    \item Power controllers: power up and power down different power domains, according to the power gating signals from the main state machine.
    \item Sleep / wakeup controllers: send sleep or wakeup requests to the RTC main state machine.
    \item Clock controller: selects and powers up/down clock sources.
    \item Protection Timer: controls the transition interval between main state machine states.
\end{itemize}
In \chipname{}'s power management unit, the sleep / wakeup controllers send sleep or wakeup requests to the RTC main state machine, which then generates power gating, clock gating, and reset signals. Then, the power controller and clock controller power up and power down different power domains and clock sources, according to the signals generated by the RTC main state machine, so that the chip enters or exits the low-power modes. The main workflow is shown in Figure \ref{fig:lpm-RTC-structure}.

The RTC timer updates two groups of registers upon any new trigger. The first group logs the time of the current trigger, and the other logs the previous trigger. Detailed information about these two register groups is shown below:
\begin{itemize}
    \item Register group 0: logs the status of RTC timer at the current trigger.
    \begin{itemize}
        \item \hyperref[regdesc:RTCCNTLTIMEHIGH0REG]{RTC\_CNTL\_TIME\_HIGH0\_REG}
        \item \hyperref[regdesc:RTCCNTLTIMELOW0REG]{RTC\_CNTL\_TIME\_LOW0\_REG}
    \end{itemize}
    \item Register group 1: logs the status of RTC timer at the previous trigger.
    \begin{itemize}
    \item \hyperref[regdesc:RTCCNTLTIMEHIGH1REG]{RTC\_CNTL\_TIME\_HIGH1\_REG}
    \item \hyperref[regdesc:RTCCNTLTIMELOW1REG]{RTC\_CNTL\_TIME\_LOW1\_REG}
    \end{itemize}
\end{itemize}

On a new trigger, information on previous trigger is moved from register group 0 to register group 1 (and the original trigger logged in register group 1 is overwritten), and this new trigger is logged in register group 0. Therefore, only the last two triggers can be logged at any time.

It should be noted that any reset / sleep other than power-up reset will not stop or reset the RTC timer.

Also, the RTC timer can be used as a wakeup source. For details, see Section \ref{sec:lpm-wakeup-source}.

\subsubsection{Brownout Detector}

The brownout detector checks the voltage of pins VDD3P3\_RTC, VDD3P3\_CPU, VDDA1, and VDDA2. If the voltage of these pins drops below the predefined threshold (2.7 V by default), the detector would trigger a signal to shut down some power-consuming blocks (such as LNA, PA, etc.) to allow extra time for the digital system to save and transfer important data. 

\hyperref[fielddesc:RTCCNTLBROWNOUTDET]{RTC\_CNTL\_BROWN\_OUT\_DET} indicates the output level of brown-out detector. This register is low level by default, and outputs high level when the voltage of the detected pin drops below the predefined threshold.

\hyperref[fielddesc:RTCCNTLBROWNOUTRSTSEL]{RTC\_CNTL\_BROWN\_OUT\_RST\_SEL} configures the reset type. For more information regarding chip reset and system reset, please refer to \ref{mod:resclk} \textit{\nameref{mod:resclk}}.
    \begin{itemize}
        \item 0: resets the chip       
        \item 1: resets the system               
    \end{itemize}
    
    

The brownout detector has ultra-low power consumption and remains enabled whenever the chip is powered up.

\section{Power Modes Management}

\subsection{Power Domain}\label{sec:lpm-power-domain}

\chipname{} has 9 power domains in three power domain categories:
\begin{itemize}
    \item RTC
        \begin{itemize}
            \item Power management unit (PMU), including RTC timer, fast memory, Always-on registers
        \end{itemize}
    \item Digital
        \begin{itemize}
            \item PD peripherals, including SPI2, GDMA, SHA, RSA, AES, HMAC, DS, Secure Boot
            \item Digital system
            \item Wireless digital circuits
            \item CPU
        \end{itemize}
    \item Analog
    \begin{itemize}
        \item RC\_FAST\_CLK
        \item XTAL\_CLK
        \item PLL
        \item RF circuits
    \end{itemize}
\end{itemize}

\subsection{Reject Sleep}\label{sec:lpm-reject-source}

\chipname{} implements a hardware mechanism that equips the chip with the ability to reject to sleep, which prevents the chip from going to sleep unexpectedly when some peripherals are still working but not detected by the CPU, thus guaranteeing the proper functioning of the peripherals.

All the wakeup sources specified in Table \ref{tab:lpm-wakeup-source} (except UART) can also be configured as the causes to reject sleep.

Users can configure the reject to sleep option via the following registers.
\begin{itemize}
    \item Configure the \hyperref[fielddesc:RTCCNTLSLEEPREJECTENA]{RTC\_CNTL\_SLEEP\_REJECT\_ENA} field to enable or disable the option to reject to sleep:
        \begin{itemize}
            \item Set \hyperref[fielddesc:RTCCNTLLIGHTSLPREJECTEN]{RTC\_CNTL\_LIGHT\_SLP\_REJECT\_EN} to enable reject-to-light-sleep.
            \item Set \hyperref[fielddesc:RTCCNTLDEEPSLPREJECTEN]{RTC\_CNTL\_DEEP\_SLP\_REJECT\_EN} to enable reject-to-deep-sleep.
        \end{itemize}
    \item Read \hyperref[fielddesc:RTCCNTLREJECTCAUSE]{RTC\_CNTL\_SLP\_REJECT\_CAUSE\_REG} to check the reason for rejecting to sleep.
\end{itemize}

\section{Retention DMA}

\chipname{} can power off the CPU in Light\_sleep mode to further reduce the power consumption. To facilitate the CPU to wake up from light\_sleep and resume execution from the previous breakpoint, \chipname{} introduced a retention module.

\chipname{}'s retention module stores CPU information to the Internal SRAM Block9 to Block12 before CPU enters into sleep, and restore such information from Internal SRAM to CPU after CPU wakes up from sleep, thus enabling the CPU to resume execution from the previous breakpoint.

\chipname{}'s Retention DMA:

\begin{itemize}
    \item Retention DMA operates 128-bit wide data, and only supports address alignment of four words.
    \item Retention DMA's link list is specifically designed that it can be used to execute both write and read transactions. The configuration of Retention DMA is similar to that of GDMA: 
    \begin{enumerate}
        \item First allocate enough memory in SRAM before CPU enters sleep to store 432 words\hyperlink{dma-note1}{*}: CPU registers (428 words) and configuration information (4 words).
        \item Then configure the link list according to the memory allocated in the first step. See details in Chapter \ref{mod:dma} \textit{\nameref{mod:dma}}.
    \end{enumerate}


\begin{tiplisting}\label{dma-note1}
\textbf{*} Note that if the memory allocated is smaller than 432 words, then chip can only enter the Light\_sleep mode and cannot further power down CPU.
\end{tiplisting}
\end{itemize}

After configuration, users can enable the Retention function by configuring the  \hyperref[fielddesc:RTCCNTLRETENTIONEN]{RTC\_CNTL\_RETENTION\_EN} field in Register \hyperref[regdesc:RTCCNTLRETENTIONCTRLREG]{RTC\_CNTL\_RETENTION\_CTRL\_REG} to:
\begin{itemize}
    \item Use Retention DMA to store CPU information before the chip enters sleep
    \item Restore information from Retention DMA to CPU after CPU wakes up.
\end{itemize}

\section{RTC Boot}
The wakeup time from Deep-sleep is much longer, compared to the Light-sleep and Modem-sleep, because the ROMs and RAMs are both powered down in this case, and the CPU needs more time for SPI booting. However, it's worth noting that the RTC fast memory remains powered up in the Deep-sleep mode. Therefore, users can store code (so called ``deep sleep wake stub" of up to 8 KB) in the RTC fast memory to avoid the above-mentioned SPI booting, thus speeding up the wakeup process. To use this function, see steps described below:

\begin{enumerate}
    \item Set \hyperref[fielddesc:RTCCNTLSTATVECTORSELPROCPU]{RTC\_CNTL\_STAT\_VECTOR\_SEL\_PROCPU} to 1.
    \item Calculate CRC for the RTC fast memory, and save the result in \hyperref[regdesc:RTCCNTLSTORE7REG]{RTC\_CNTL\_STORE7\_REG[31:0]}.
    \item Set \hyperref[regdesc:RTCCNTLSTORE6REG]{RTC\_CNTL\_STORE6\_REG[31:0]} to the entry address of RTC fast memory.
    \item Send the chip into sleep.
    \item SPI boot and some of the initialization starts after the CPU is powered up. After that, calculate the CRC for the RTC fast memory again. If the result matches with register \hyperref[regdesc:RTCCNTLSTORE7REG]{RTC\_CNTL\_STORE7\_REG[31:0]}, the CPU jumps to the entry address.
\end{enumerate}
