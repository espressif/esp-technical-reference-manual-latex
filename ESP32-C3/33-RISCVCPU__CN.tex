\chapter{ESP-RISC-V CPU}
\label{mod:riscvcpu}



\section{概述}

ESP-RISC-V CPU 是基于 RISC-V ISA 的 32 位内核，包括基本整数 (I)，乘法/除法 (M) 和压缩 (C) 标准扩展。ESP-RISC-V CPU 内核具有 4 级有序标量流水线，针对面积、功耗、性能等进行了优化。CPU 内核架构包含中断控制器 (INTC)、调试模块 (DM) 和用于访问存储器和外设的系统总线 (SYS BUS) 接口。
\begin{figure}[h]
\centering
\includegraphics[scale=1.0,width=0.8\textwidth]{\modulefiles/figures/CPU_C3_BLK_DIAG_v0.1}
\caption{CPU 框图}
\end{figure}


\section{特性}
\label{sec:riscvcpu-features}

\begin{itemize}

    \item 时钟工作频率高达 160 MHz
    \item 通过 IRAM/DRAM 接口零等待周期访问片上 SRAM 和缓存中的程序和数据
    \item 中断控制器 (INTC) 具有多达 31 个向量中断，可配置优先级和阈值级别
    \item 调试模块 (DM) 符合 RISC-V 调试规范 v0.13，支持通过行业标准的 JTAG/USB 端口连接外部调试器
    \item 调试器通过系统总线 (SBA) 直接访问存储器和外设
    \item 硬件触发器符合 RISC-V 调试规范 v0.13，具有多达 8 个断点/观察点
    \item 物理存储器保护 (PMP)，最多可配置 16 个区域
    \item 32 位 AHB 系统总线，用于访问外设
    \item 可配置的核心性能指标事件
    
\end{itemize}

\section{中断控制器}
\label{sec:riscvcpu-int-ctrl}

\subsection{特性}
\label{sec:riscvcpu-int-features}
中断控制器能够捕获、屏蔽来自 RISC-V CPU 外部的中断源，并对中断源的优先级进行动态仲裁。中断控制器具有以下特性：

\begin{itemize}
    \item 多达 31 个具有唯一 ID (1-31) 的异步中断
    \item 支持通过读写存储器匹配寄存器进行配置
    \item 15 个优先级级别，可以分配给不同的中断
    \item 支持电平触发或边沿触发的中断源
    \item 可配置的全局阈值，用于屏蔽优先级较低的中断
    \item 与异常向量地址偏移量匹配的中断 ID
\end{itemize}

中断寄存器的完整列表及详细描述请见章节 \ref{mod:intmtrx} \textit{\nameref{mod:intmtrx}}，\ref{sec:intmtrx-reg-summ} 小节中的“CPU 中断寄存器”。

\subsection{功能描述}
\label{sec:riscvcpu-int-func-descr}

每个中断 ID 都有 5 个属性：

\begin{enumerate}
    \item 使能状态 (0-1)：
    \begin{itemize}
        \item 决定是否允许由 CPU 捕获和处理中断。
        \item 通过写入 \hyperref[regdesc:INTERRUPTCORE0CPUINTENABLEREG]{INTERRUPT\_CORE0\_CPU\_INT\_ENABLE\_REG} 相应的域进行配置。
    \end{itemize}
    \item 类型 (0-1)：
    \begin{itemize}
        \item 在中断信号的上升沿使能闩锁状态。
        \item 通过写入 \hyperref[regdesc:INTERRUPTCORE0CPUINTTYPEREG]{INTERRUPT\_CORE0\_CPU\_INT\_TYPE\_REG} 相应的域进行配置。
        \item 类型保持为 0 的中断称为“电平”类型中断。
        \item 类型保持为 1 的中断称为“边沿”类型中断。
    \end{itemize}
    \item 优先级 (1-15)：
    \begin{itemize}
        \item 当有多个中断在等待时，决定 CPU 先处理哪一个中断。
        \item 通过写入中断 ID \regindex{n} (1-31) 的 \hyperref[regdesc:INTERRUPTCORE0CPUINTPRI0REG]{INTERRUPT\_CORE0\_CPU\_INT\_PRI\_\regindex{n}\_REG} 进行配置。
        \item 优先级为零或小于 \hyperref[regdesc:INTERRUPTCORE0CPUINTTHRESHREG]{INTERRUPT\_CORE0\_CPU\_INT\_THRESH\_REG} 指定阈值的中断将被屏蔽。
        \item 优先级最低为 1，最高为 15。
        \item 具有相同优先级的中断通过其 ID 静态确定优先级，ID 越小，优先级越高。
    \end{itemize}
    \item 等待状态 (0-1)：
    \begin{itemize}
        \item 反映已使能且未被屏蔽的中断信号被捕获时的状态。
        \item 通过读取 \hyperref[regdesc:INTERRUPTCORE0CPUINTEIPSTATUSREG]{INTERRUPT\_CORE0\_CPU\_INT\_EIP\_STATUS\_REG} 中的相应位获得每个中断 ID 的等待状态。
        \item 如果没有更高优先级的中断在等待，则当前在等待的中断将导致 CPU 进入异常。
        \item 如果在等待的中断抢占 CPU 并导致其跳转到相应的异常向量地址，则称该中断为“已声明”。
        \item 所有在等待的中断都为“未声明”。
    \end{itemize}
    \item 清除状态 (0-1)：
    \begin{itemize}
        \item 切换此属性将仅清除已声明的边沿类型中断的等待状态。
        \item 通过先置位然后清零 \hyperref[regdesc:INTERRUPTCORE0CPUINTCLEARREG]{INTERRUPT\_CORE0\_CPU\_INT\_CLEAR\_REG} 中的相应位进行切换。
        \item 电平类型的中断的等待状态不受切换操作的影响，而必须从中断源中清除。
        \item 未声明的边沿类型中断的等待状态可以被清空，方法是先清零 \hyperref[regdesc:INTERRUPTCORE0CPUINTENABLEREG]{INTERRUPT\_CORE0\_CPU\_INT\_ENABLE\_REG} 中的相应位再置位 \hyperref[regdesc:INTERRUPTCORE0CPUINTCLEARREG]{INTERRUPT\_CORE0\_CPU\_INT\_CLEAR\_REG} 中的相同位。
    \end{itemize}
\end{enumerate}

当 CPU 处理在等待的中断时，会进行以下操作：
\begin{itemize}
    \item 将当前未执行指令的地址保存在 \hyperref[regdesc:MEPC]{mepc} 中，以便之后恢复执行。
    \item 将 \hyperref[regdesc:MCAUSE]{mcause} 的值更新为正在处理的中断 ID。
    \item 将 \hyperref[fielddesc:MSTATUSMIE]{MIE} 的状态复制到 \hyperref[fielddesc:MSTATUSMPIE]{MPIE}，然后清零 \hyperref[fielddesc:MSTATUSMIE]{MIE}，从而全局禁用中断。
    \item 通过跳转到 \hyperref[regdesc:MTVEC]{mtvec} 中存储的地址的字对齐偏移量进入异常。
\end{itemize}

表 \ref{tab:riscvcpu-int-isr-addr-map} 列出了每个中断 ID 及其对应的异常向量地址。简而言之，中断 $ID=i$ 的字对齐的异常地址 = $(\hyperref[fielddesc:MTVECBASE]{mtvec} + 4i)$.

 \textit{说明：$ID=0$ 不可用，不能用于捕获中断，这是因为相应的异常向量地址 (\hyperref[fielddesc:MTVECBASE]{mtvec} + 0x00) 已经预留给异常。}

\begin{longtable}{ | r | l || r | l || r | l || r | l | }
\caption{中断 ID 与异常向量地址} \label{tab:riscvcpu-int-isr-addr-map}
\\ \hline
\rowcolor{lightgray}
ID & 地址 & ID & 地址 & ID & 地址 & ID & 地址 \\ \hline
 0 & NA           &  8 & mtvec + 0x20 & 16 & mtvec + 0x40 & 24 & mtvec + 0x60   \\ \hline
 1 & mtvec + 0x04 &  9 & mtvec + 0x24 & 17 & mtvec + 0x44 & 25 & mtvec + 0x64   \\ \hline
 2 & mtvec + 0x08 & 10 & mtvec + 0x28 & 18 & mtvec + 0x48 & 26 & mtvec + 0x68   \\ \hline
 3 & mtvec + 0x0c & 11 & mtvec + 0x2c & 19 & mtvec + 0x4c & 27 & mtvec + 0x6c   \\ \hline

在跳转到异常向量之后，执行流程取决于软件实现，但一般来说该中断将在某个中断服务程序 (ISR) 中被处理（并清除），然后在 CPU 遇到 MRET 指令后恢复正常程序流。

执行 MRET 指令后，CPU 将进行以下操作：
\begin{itemize}
    \item 将 \hyperref[fielddesc:MSTATUSMPIE]{MPIE} 的状态复制回 \hyperref[fielddesc:MSTATUSMIE]{MIE}，然后清零 \hyperref[fielddesc:MSTATUSMPIE]{MPIE}。这意味着，如果之前置位了 \hyperref[fielddesc:MSTATUSMPIE]{MPIE}，则执行 MRET 后 \hyperref[fielddesc:MSTATUSMIE]{MIE} 将被置位，进而全局使能中断。
    \item 跳转到 \hyperref[regdesc:MEPC]{mepc} 中存储的地址，然后恢复执行。
\end{itemize}

软件可以在 ISR 内部实现中断嵌套，具体请参考章节 \ref{sec:riscvcpu-int-suggest-op}。

中断控制器具有以下行为特点：

\begin{itemize}
    \item 仅当中断具有非零优先级、大于或等于阈值寄存器中的值时，它才会反映在 \hyperref[regdesc:INTERRUPTCORE0CPUINTEIPSTATUSREG]{INTERRUPT\_CORE0\_CPU\_INT\_EIP\_STATUS\_REG} 中。
    \item 如果一个中断反映在 \hyperref[regdesc:INTERRUPTCORE0CPUINTEIPSTATUSREG]{INTERRUPT\_CORE0\_CPU\_INT\_EIP\_STATUS\_REG} 中但是还未被处理，则可以通过降低它的优先级或提高全局阈值将其屏蔽（进而防止 CPU 对其进行处理）。
    \item 如果一个中断反映在\\ \hyperref[regdesc:INTERRUPTCORE0CPUINTEIPSTATUSREG]{INTERRUPT\_CORE0\_CPU\_INT\_EIP\_STATUS\_REG} 中，要清除它（防止被处理），则必须将其禁用（如果是边沿属性的中断则需要清除）。
\end{itemize}

\subsection{建议操作}
\label{sec:riscvcpu-int-suggest-op}

\subsubsection{延迟}
\label{sec:riscvcpu-int-latency}

配置中断控制器时应考虑延迟问题。

在稳态操作中，中断控制器的等待时间固定为 4 个周期。稳态操作的意思是最近没有对中断控制器寄存器作任何更改。这意味着一个中断从被中断控制器断言到被 CPU 开始处理刚好消耗 4 个周期。这也意味着，在抢占发生之前，CPU 最多可以执行 5 条指令。

当寄存器被修改时，中断控制器会进入临时状态，然后需要最多 4 个周期才能再次进入稳态。在临时状态期间，中断的顺序可能无法预测，因此，需要软件采取一些安全措施以避免任何同步问题。
 
还须注意的是，中断控制器的配置寄存器位于 APB 地址范围内，因此对这些寄存器的读写访问可能需要消耗几个周期。
 
考虑到上述特征，建议用户在修改中断控制器寄存器时遵循以下操作顺序：

\begin{enumerate}
    \item 保存 \hyperref[fielddesc:MSTATUSMIE]{MIE} 的状态，然后将其清零
    \item 通过“读-修改-写”的方式写中断控制器寄存器
    \item 执行 FENCE 指令以等待所有未完成的写操作完成
    \item 最后，恢复 \hyperref[fielddesc:MSTATUSMIE]{MIE} 的状态
\end{enumerate}

如上述步骤显示，建议用户在配置中断控制器寄存器之前先全局禁用中断 (\hyperref[fielddesc:MSTATUSMIE]{MIE}=0)，然后立即恢复 \hyperref[fielddesc:MSTATUSMIE]{MIE}。

执行完上述操作后，中断控制器将恢复稳态操作。

\subsubsection{配置流程}
\label{sec:riscvcpu-int-config}

默认情况下，\hyperref[regdesc:MSTATUS]{mstatus} 里的 \hyperref[fielddesc:MSTATUSMIE]{MIE} 为 0，即全局禁用中断。在中断堆栈初始化（包括将 \hyperref[regdesc:MTVEC]{mtvec} 设置为中断向量地址）完成之后，软件必须将 \hyperref[fielddesc:MSTATUSMIE]{MIE} 置为 1。

在正常情况下，如果要使能某个中断 \regindex{n}，可以遵循以下步骤：
\begin{enumerate}
    \item 保存 \hyperref[fielddesc:MSTATUSMIE]{MIE} 的状态，然后将其清零
    \item 根据中断的类型（边沿/电平），置位或取消置位 \hyperref[regdesc:INTERRUPTCORE0CPUINTTYPEREG]{INTERRUPT\_CORE0\_CPU\_INT\_TYPE\_REG} 中的第 \regindex{n} 个位
    \item 通过写入 \hyperref[regdesc:INTERRUPTCORE0CPUINTPRI0REG]{INTERRUPT\_CORE0\_CPU\_INT\_PRI\_\regindex{n}\_REG} 指定优先级（最低为 1，最高为 15）
    \item 置位 \hyperref[regdesc:INTERRUPTCORE0CPUINTENABLEREG]{INTERRUPT\_CORE0\_CPU\_INT\_ENABLE\_REG} 中的第 \regindex{n} 个位
    \item 执行 FENCE 指令
    \item 恢复 \hyperref[fielddesc:MSTATUSMIE]{MIE} 的状态
\end{enumerate}

当一个或多个中断在等待时，CPU 将确认（声明）最高优先级的中断，然后跳转到与该中断 ID 相对应的异常向量地址。软件可以通过读取 \hyperref[regdesc:MCAUSE]{mcause} 来推断异常类型（\hyperref[fielddesc:MCAUSEINT]{mcause(31)} 为 1 代表中断，为 0 代表异常）和中断 ID（\hyperref[fielddesc:MCAUSECODE]{mcause(4-0)} 提供中断或异常的 ID）。如果异常向量中的每个表项都是指向不同异常处理程序的跳转指令，则软件无需做此推断。最后，异常处理程序会将程序指引到该中断相应的 ISR。

进入 ISR 后，如果中断为边沿类型，则软件必须切换 \hyperref[regdesc:INTERRUPTCORE0CPUINTCLEARREG]{INTERRUPT\_CORE0\_CPU\_INT\_CLEAR\_REG} 中的第 \regindex{n} 个位，如果是电平类型中断，则必须清除相应的中断源。

软件还可以更新 \hyperref[regdesc:INTERRUPTCORE0CPUINTTHRESHREG]{INTERRUPT\_CORE0\_CPU\_INT\_THRESH\_REG} 的值并置位 \hyperref[fielddesc:MSTATUSMIE]{MIE} 来让更高优先级的中断抢占当前 ISR（即嵌套），但是，在此之前，必须先保存所有状态 CSR（\hyperref[regdesc:MEPC]{mepc}、\hyperref[regdesc:MSTATUS]{mstatus}、\hyperref[regdesc:MCAUSE]{mcause} 等），这是由于发生嵌套时状态 CSR 的值会被覆盖。之后，在退出 ISR 时，再恢复这些 CSR 的值。

最后，程序从 ISR 返回到异常处理程序之后，可以执行 MRET 指令以恢复正常程序流。

如果不再需要中断 \regindex{n} 并且需要将其禁用，则可以遵循以下操作步骤：
\begin{enumerate}
    \item 保存 \hyperref[fielddesc:MSTATUSMIE]{MIE} 的状态，然后将其清零
    \item 读取 \hyperref[regdesc:INTERRUPTCORE0CPUINTEIPSTATUSREG]{INTERRUPT\_CORE0\_CPU\_INT\_EIP\_STATUS\_REG} 检查中断是否在等待
    \item 置位/取消置位 \hyperref[regdesc:INTERRUPTCORE0CPUINTENABLEREG]{INTERRUPT\_CORE0\_CPU\_INT\_ENABLE\_REG} 中的第 \regindex{n} 个位
    \item 如果中断属于边沿类型并且在等待，则必须切换 \hyperref[regdesc:INTERRUPTCORE0CPUINTCLEARREG]{INTERRUPT\_CORE0\_CPU\_INT\_CLEAR\_REG} 中的第 \regindex{n} 个位以清空它的等待状态
    \item 执行 FENCE 指令
    \item 恢复 \hyperref[fielddesc:MSTATUSMIE]{MIE} 的状态
\end{enumerate}

以上只是建议的操作方案，实际操作由软件实现决定。

\section{调试}
\label{sec:riscvcpu-dbg}


\subsection{概述}
\label{sec:riscvcpu-dbg-overview}
本节介绍如何调试和测试在 CPU 内核上运行的软件。调试功能由标准 JTAG 管脚提供，并符合 RISC-V 外部调试支持规范版本 0.13 (RISC-V External Debug Support Specification version 0.13)。

图 \ref{fig:riscvcpu-dbg-sys-overview} 为外部调试系统架构图。

\begin{figure}[h]
\centering
\includegraphics[width=\textwidth/2,keepaspectratio]{\modulefiles/figures/debug_bigfonts.png}
\caption{调试系统架构}\label{fig:riscvcpu-dbg-sys-overview}
\end{figure}

用户与运行调试器（Debugger，例如 GDB）的调试主机（DEBUG HOST，例如笔记本电脑）进行交互。调试器通过调试转换器（Debug Translator，可能包含硬件驱动，例如 OPENOCD）与调试传输硬件（DEBUG TRANSPORT，例如 Olimex USB-JTAG 适配器）进行通信。调试传输硬件通过标准 JTAG 接口将调试主机连接到 ESP-RV 内核的调试传输模块 (JTAG DTM)。JTAG DTM 使用调试模块接口 (DMI) 提供对调试模块 (DM) 的访问。

DM 允许调试器暂停内核。抽象命令提供对 GPR（通用寄存器）的访问。程序缓冲区允许调试器在内核上执行任意代码，从而读取 CPU 内核的其他运行状态。CPU 内核的其他运行状态也可以由其他抽象命令读取。ESP-RV 内核带有一个支持 8 个触发器的触发器模块。当满足触发条件时，内核将自发暂停并通知调试模块。
 
系统总线访问的 block 无需使用 RISC-V 内核即可访问存储器和外设寄存器。


\subsection{特性}
\label{sec:riscvcpu-dbg-features}
基础调试功能具有以下特性：

\begin{itemize}
    \item {向调试器提供有关实现的必要信息}
    \item {支持暂停和恢复 CPU 内核}
    \item {CPU 内核寄存器（包括 CSR）可以由调试器读取/写入}
    \item {CPU 可以从复位后执行的第一条指令开始就被调试}
    \item {可以通过调试器复位 CPU 内核}
    \item {可以在软件断点（植入的断点指令）上暂停 CPU}
    \item {硬件单步调试}
    \item {通过程序缓冲区在暂停的 CPU 中执行任意指令。支持 16 字的程序缓冲区。}
    \item {支持系统总线的字对齐地址访问}
    \item {支持八个硬件触发器（可用作断点/观察点），具体见章节 \ref{sec:riscvcpu-hwt}}
\end{itemize}

\subsection{功能描述}
\label{sec:riscvcpu-dbg-func-descr}

调试机制遵守 RISC-V 外部调试支持规范版本 0.13。有关调试功能的详细介绍，请参考 RISC-V 外部调试支持规范。

\section{硬件触发器}
\label{sec:riscvcpu-hwt}

\subsection{特性}
\label{sec:riscvcpu-hwt-features}

硬件触发器模块提供了断点和观察点功能，供调试使用。硬件触发器具有以下特性：

\begin{itemize}
    \item 8 个独立触发单元
    \item 每个单元都可以配置为匹配程序计数器的地址或存储器访问地址
    \item 可以通过引起断点异常来抢占执行
    \item 可以暂停执行并将控制权转移给调试器
    \item 支持 NAPOT（2 的幂次方对齐）地址编码
\end{itemize}

\subsection{功能描述}
\label{sec:riscvcpu-hwt-func-descr}

硬件触发器模块提供了 4 个 CSR，见\hyperref[sec:riscvcpu-hwt-reg-summ]{寄存器列表}。其中，\hyperref[regdesc:TDATA1]{tdata1} 和 \hyperref[regdesc:TDATA2]{tdata2} 是抽象 CSR，也就是说它们是用于访问 某个触发单元中的内部寄存器的影子寄存器，一次访问一个触发单元。

要选择特定的触发单元，需要将相应的编号 (0-7) 写入 \hyperref[regdesc:TSELECT]{tselect} CSR。当写入有效数值时，抽象 CSR tdata1 和 tdata2 将自动匹配该触发单元的内部寄存器。每个触发单元都有两个内部寄存器，即 \hyperref[regdesc:MCONTROL]{mcontrol} 和 \hyperref[regdesc:MADDRESS]{maddress}，它们分别与 tdata1 和 tdata2 匹配。

向 tselect 写入超过最大编号的数值时会导致该数值被裁剪为最大的编号，此编号可以被读回。这个特性可用于枚举初始化期间或使用调试器时可用的触发器。

由于软件或调试器可能需要知道所选触发器的类型以便正确解读 tdata1 和 tdata2，因此 tdata1 的 4 个位 (31-28) 对所选触发器的类型进行了编码。此域为只读访问属性，并且值始终为 0x2，代表匹配类型触发器，因此，可以推断 tdata1 和 tdata2 会通过 mcontrol 和 maddress 被解读。RISC-V 调试规范 v0.13 提供了其他可能值的信息，但是该触发模块仅支持 0x2 类型。

一旦选定了触发单元，就可以通过置位 mcontrol CSR (tdata1) 中相应的域并将目标地址写入 maddress CSR (tdata2) 来对该触发单元进行配置。

通过写入 mcontrol 的 action 域，可以将每个触发单元配置为引起断点异常或进入调试模式。该域只能从调试器写入，因此默认情况下，触发器（如果启用）将引起断点异常。

每个触发单元的 mcontrol 都有一个 \hyperref[fielddesc:MCONTROLHIT]{hit} 域。在 CPU 暂停或进入异常后，通过读取该域可以查明是否是触发单元触发了。触发器触发后该域会立即被置位，但在恢复操作之前必须被手动清零，虽然不清零不会影响正常执行。

每个触发单元仅支持地址匹配，该地址可以是存储器访问地址，也可以是指令的虚拟地址。通过写入所选触发单元的 maddress (tdata2) CSR，可以指定区域的地址和大小。大于 1 个字节的区域大小通过 NAPOT 编码（见表 \ref{tab:riscvcpu-hwt-table}）指定，并通过置位 mcontrol 中 match 域来使能。注意，根据定义，NAPOT 编码地址的起始地址与区域大小对齐（即，是区域大小的整数倍）。

\begin{longtable}{ | c | c | r | }
\caption{NAPOT 编码的 maddress} \label{tab:riscvcpu-hwt-table}
\\ \hline
\rowcolor{lightgray}
maddress(31-0) & 起始地址 & 大小（字节） \\ \hline
aaa...aaaaaaaaa0 & aaa...aaaaaaaaa0 & $2$ \\ \hline
aaa...aaaaaaaa01 & aaa...aaaaaaaa00 & $4$ \\ \hline
aaa...aaaaaaa011 & aaa...aaaaaaa000 & $8$ \\ \hline

\hyperref[regdesc:TCONTROL]{tcontrol} CSR 对所有触发单元都是通用的。在机器模式下，当程序在异常处理程序中执行时，该寄存器可用于阻止触发器重复引起异常。默认情况下 ISR 内部的断点异常也被禁用，但是，出于调试目的，可以在进入 ISR 之前手动使能断点异常。如果将触发器配置为进入调试模式，则此 CSR 不相关。

\subsection{触发执行流程}
\label{sec:riscvcpu-hwt-suggest-op}

当触发器触发引起硬件线程暂停并进入调试模式时 (\hyperref[fielddesc:MCONTROLACTION]{action} = 1)：
\begin{itemize}
    \item \hyperref[regdesc:DPC]{dpc} 被设置为当前 PC（在解码阶段）
    \item \hyperref[regdesc:DCSR]{dcsr} 的 cause 域被设置为 2，表示暂停是由于触发器触发引起
    \item 与触发的触发器对应的 \hyperref[fielddesc:MCONTROLHIT]{hit} 域被置位
\end{itemize}

当触发器触发引起硬件线程进入异常时 (\hyperref[fielddesc:MCONTROLACTION]{action} = 0)：
\begin{itemize}
    \item \hyperref[regdesc:MEPC]{mepc} 被设置为当前 PC（在解码阶段）
    \item \hyperref[regdesc:MCAUSE]{mcause} 被设置为 3，即断点异常
    \item \hyperref[fielddesc:TCONTROLMPTE]{mpte} 被设置为异常发生之前的 \hyperref[fielddesc:TCONTROLMTE]{mte} 的值
    \item \hyperref[fielddesc:TCONTROLMTE]{mte} 被设置为 0
    \item 与触发的触发器对应的 \hyperref[fielddesc:MCONTROLHIT]{hit} 域被置位
\end{itemize}

\textit{说明：如果两个触发器同时触发，一个 action = 0，action = 1，则硬件线程会暂停并进入调试模式。}

\section{存储器保护}
\label{sec:riscvcpu-pmp}

\subsection{概述}
\label{sec:riscvcpu-pmp-overview}
CPU 内核包含一个物理存储器保护单元，可以供软件设置存储器访问特权（读、写、执行权限）。该物理存储器保护单元不完全等同于 RISC\-/V 指令集手册 V1.10 第二卷“特权架构”中描述的 PMP，下文会详细描述不同之处。

如需了解 RISC-V PMP 的更多信息，请参考 RISC\-/V 指令集手册 V1.10 第二卷“特权架构”。

\subsection{特性}
\label{sec:riscvcpu-pmp-features}
PMP 单元用于控制对物理存储器的访问。它支持 16 个区域，可以对最小 4 个字节大小的区域进行权限设定。ESP32-C3 芯片的物理存储器保护单元与 RISC-V 规范中定义的 PMP 的不同之处在于： 
\begin{itemize}
    \item {不支持静态优先级，即不支持重叠区域}
    \item {支持的最大 NAPOT 范围是 1 GB}
\end{itemize}

根据 RISC-V 特权架构规范，PMP 表项是静态优先级排序的，并且，与存储器访问地址的任意字节匹配的最小编号的 PMP 表项将决定该访问是否成功。这意味着，当任意地址匹配多个 PMP 表项，即多个 PMP 表项的重叠区域时，编号最小的 PMP 表项将决定该访问是否成功。

但是，ESP32-C3 的 RISC-V CPU PMP 单元并未实现静态优先级。因此，软件应确保所有使能的 PMP 表项都有唯一的区域，即它们之间没有重叠区域。如果软件仍试图对具有重叠区域且权限相互矛盾的多个 PMP 表项进行配置，只要访问与其中一个 PMP 表项匹配，则访问成功。如果访问与所有使能的 PMP 表项都不匹配，则会产生一个异常。

\subsection{功能描述}
\label{sec:riscvcpu-pmp-func-descr}

软件可以设置 PMP 单元的配置和地址寄存器，以保存错误并确保安全执行。PMP CSR 只能在机器模式下进行配置。写入、读取和执行权限检测一旦被使能，则将根据 16 个 pmpcfgX 和 pmpaddrX 寄存器（见\hyperref[sec:riscvcpu-pmp-reg-summ]{寄存器列表}）中的配置值作用于用户模式下的所有存储器访问。

默认情况下，PMP 允许机器模式下的所有存储器访问，而撤销用户模式下的所有访问。这意味着必须通过 pmpcfg 和 pmpaddr 寄存器（见\hyperref[sec:riscvcpu-pmp-reg-summ]{寄存器列表}）设置用户模式可以访问的地址范围和有效权限，以确保访问成功。但是在机器模式下没有此要求，因为机器模式下默认 PMP 允许所有访问。如果在机器模式下也需要 PMP 检测，则软件可以将所需 PMP 表项的锁定位置位来使能权限检测。锁定位一旦置位，就只能通过 CPU 复位被清零。

如果从存储器区域提取指令而没有执行权限，则会在处理器级别生成异常，并且在 \hyperref[regdesc:MCAUSE]{mcause} CSR 中异常原因被设置为指令访问错误。同样，任何没有有效读/写权限的读写访问都将生成异常，并且 \hyperref[regdesc:MCAUSE]{mcause} 会更新为读取存储器访问错误或写入存储器访问错误。如果发生存储器读写异常，则存储器访问地址会更新到 \hyperref[regdesc:MTVAL]{mtval} CSR 中。
