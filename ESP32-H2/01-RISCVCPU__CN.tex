\chapter{ESP-RISC-V CPU}
\label{mod:riscvcpu}

\section{概述}



ESP-RISC-V CPU 是一个基于 RISC-V 指令集架构 (ISA) 的 32 位内核，包括基本整数 (I)、乘法/除法 (M)、原子 (A) 和压缩 (C) 标准扩展。ESP-RISC-V CPU 内核具有 4 级有序标量流水线，针对面积、功耗、性能等进行了优化。CPU 内核架构包含调试模块 (DM)、中断控制器 (INTC)、核心本地中断 (CLINT) 和用于访问存储器和外设的系统总线 (SYS BUS) 接口。
\begin{figure}[h]
\centering
\includegraphics[scale=1.0,width=0.8\textwidth]{\modulefiles/figures/CPU_BLK_DIAG_v0.1.pdf}
\caption{CPU 框图}
\end{figure}

\section{特性}
\label{sec:riscvcpu-features}

\begin{itemize}
    \item RISC-V RV32IMAC ISA，四级流水线，时钟工作频率高达 96 MHz
    \item 符合 RISC-V 指令集手册 v2.2 第一卷“非特权架构”(RISC-V Instruction Set Manual Volume I: Unprivileged ISA, Version 2.2) 和 RISC-V 指令集手册 v1.10 第二卷“特权架构” (RISC\-/V Instruction Set Manual, Volume II: Privileged Architecture, Version 1.10)
    \item 通过 IRAM/DRAM 接口零等待周期访问片上 SRAM 和缓存中的程序和数据
    \item 具有静态分支预测功能的分支目标缓冲区 (BTB)
    \item 支持用户模式 (user mode) 以及中断委托
    \item 中断控制器 (INTC) 具有多达 28 个外部向量中断，适用于机器模式 (machine mode) 和用户模式，可配置 16 个优先级和阈值级别
    \item 专用于各个特权模式的核心本地中断 (CLINT)
    \item 调试模块 (DM) 符合 RISC-V 外部调试支持规范 (RISC-V External Debug Support) v0.13，支持通过行业标准的 JTAG/USB 端口连接外部调试器
    \item 支持指令追踪
    \item 调试器通过系统总线 (SBA) 直接访问存储器和外设
    \item 硬件触发器符合 RISC-V 调试规范 v0.13，具有多达 4 个断点/观察点
    \item 物理存储器保护 (PMP) 和物理存储器属性 (PMA)，最多可配置 16 个区域
    \item 32 位 AHB 系统总线，用于访问外设
    \item 可配置的核心性能指标事件
\end{itemize}

\section{中断控制器}
\label{sec:riscvcpu-int-ctrl}

\subsection{特性}
\label{sec:riscvcpu-int-features}
中断控制器能够捕获、屏蔽来自 RISC-V CPU 外部的中断源，并对中断源的优先级进行动态仲裁。中断控制器具有以下特性：

\begin{itemize}
    \item 多达 28 个外部异步中断和 4 个核心本地中断源 (CLINT)，皆有唯一 ID (0-31)
    \item 可通过读写存储器映射寄存器配置中断控制器
    \item 可委托给用户模式
    \item 15 个优先级级别，可以分配给不同的中断
    \item 支持电平触发或边沿触发的中断源
    \item 可配置的全局阈值，用于屏蔽优先级较低的中断
    \item 与陷阱向量地址偏移量匹配的中断 ID
\end{itemize}

中断寄存器的完整列表及详细描述请见章节 \ref{mod:intmtrx} \textit{\nameref{mod:intmtrx}} > 章节 \ref{sec:intpri-reg-summ}。

\subsection{功能描述}
\label{sec:riscvcpu-int-func-descr}

每个中断 ID 都有 6 个属性，对于 28 个外部中断（1-2、5-6、8-31）来说，这些属性皆可配置，但是对于 4 个本地 CLINT 中断（0、3、4、7）来说，这些属性是静态的（模式除外）。这些属性如下：

\begin{enumerate}
    \item 模式 (M/U)：
    \begin{itemize}
        \item 决定中断在哪种模式下被处理。
        \item 通过设置或清除 \hyperref[regdesc:MIDELEG]{mideleg} CSR 中的相应位进行配置。
        \item 如果某个中断在 \hyperref[regdesc:MIDELEG]{mideleg} CSR 中的对应位被清除，则该中断将在机器模式下被捕获。
        \item 如果某个中断在 \hyperref[regdesc:MIDELEG]{mideleg} CSR 中的对应位被设置，则该中断将被委托到用户模式。
    \end{itemize}
    \item 使能状态 (0-1)：
    \begin{itemize}
        \item 决定是否允许由 CPU 捕获和处理中断。
        \item 通过写入 \hyperref[regdesc:INTPRICORE0CPUINTENABLEREG]{INTPRI\_CORE0\_CPU\_INT\_ENABLE\_REG} 相应的位进行配置。
        \item 由于本地 CLINT 中断在存储器映射寄存器中保留了相应的位，因此这些中断在中断控制器级别始终为使能状态。
        \item 机器模式下的外部中断和本地中断还需要通过在 \hyperref[regdesc:MIE]{mie} CSR 中设置相应的位来在核心级别取消屏蔽。
        \item 用户模式下的外部中断和本地中断还需要通过在 \hyperref[regdesc:UIE]{uie} CSR 中设置相应的位来在核心级别取消屏蔽。
    \end{itemize}
    \item 类型 (0-1)：
    \begin{itemize}
        \item 在中断信号的上升沿锁存中断状态。
        \item 通过写入 \hyperref[regdesc:INTPRICORE0CPUINTTYPEREG]{INTPRI\_CORE0\_CPU\_INT\_TYPE\_REG} 相应的位进行配置。
        \item 类型保持为 0 的中断称为电平触发中断。
        \item 类型保持为 1 的中断称为边沿触发中断。
        \item 本地 CLINT 中断始终为电平触发中断，因此在上述寄存器中保留了相应的位。
    \end{itemize}
    \item 优先级 (1-15)：
    \begin{itemize}
        \item 当有多个中断在等待时，决定 CPU 先处理哪一个中断。
        \item 通过写入外部中断 ID \regindex{n} 的 \hyperref[regdesc:INTPRICORE0CPUINTPRINREG]{INTPRI\_CORE0\_CPU\_INT\_PRI\_\regindex{n}\_REG} 进行配置。
        \item 优先级小于 \hyperref[regdesc:INTPRICORE0CPUINTTHRESHREG]{INTPRI\_CORE0\_CPU\_INT\_THRESH\_REG} 指定阈值的外部中断将被屏蔽。
        \item 优先级最低为 0，最高为 15。
        \item 具有相同优先级的中断通过其 ID 静态确定优先级，ID 越小，优先级越高。
        \item 本地 CLINT 中断具有静态的优先级，因此保留了相应的优先级寄存器。
        \item 本地 CLINT 中断在两种模式下均不会被优先级阈值屏蔽。
    \end{itemize}
    \item 等待状态 (0-1)：
    \begin{itemize}
        \item 反映已使能且未被屏蔽的外部中断信号被捕获时的状态。
        \item 通过读取 \hyperref[regdesc:INTPRICORE0CPUINTEIPSTATUSREG]{INTPRI\_CORE0\_CPU\_INT\_EIP\_STATUS\_REG} 中的相应位获得每个外部中断 ID 的等待状态。
        \item 也可通过读取机器模式中断的 \hyperref[regdesc:MIP]{mip} CSR 或用户模式中断的 \hyperref[regdesc:UIP]{uip} CSR 中的相应位来获取外部中断和本地中断 ID 的等待状态。
        \item 如果没有更高优先级的中断在等待，则当前在等待的中断将导致 CPU 进入陷阱。
        \item 如果在等待的中断抢占 CPU 并导致其跳转到相应的陷阱向量地址，则称该中断为“已声明”。
        \item 所有在等待的中断都为“未声明”。
    \end{itemize}
    \item 清除状态 (0-1)：
    \begin{itemize}
        \item 切换此属性将仅清除已声明的边沿类型中断的等待状态。
        \item 通过先置位然后清零 \hyperref[regdesc:INTPRICORE0CPUINTCLEARREG]{INTPRI\_CORE0\_CPU\_INT\_CLEAR\_REG} 中的相应位进行切换。
        \item 电平触发中断的等待状态不受切换操作的影响，而必须从中断源中清除。
        \item 未声明的边沿类型中断的等待状态可以被清空，方法是先清零 \hyperref[regdesc:INTPRICORE0CPUINTENABLEREG]{INTPRI\_CORE0\_CPU\_INT\_ENABLE\_REG} 中的相应位再置位 \hyperref[regdesc:INTPRICORE0CPUINTCLEARREG]{INTPRI\_CORE0\_CPU\_INT\_CLEAR\_REG} 中的相同位。
    \end{itemize}
\end{enumerate}

有关核心本地中断源的详细说明，请参阅章节 \ref{sec:riscvcpu-clint}。

当 CPU 处理在等待的机器模式或用户模式中断时，会进行以下操作：
\begin{itemize}
    \item 将当前未执行指令的地址保存在 \hyperref[regdesc:MEPC]{mepc} 或 \hyperref[regdesc:UEPC]{uepc} 中，以便之后恢复执行。
    \item 将 \hyperref[regdesc:MCAUSE]{mcause} 或 \hyperref[regdesc:UCAUSE]{ucause} 的值更新为正在处理的中断 ID。
    \item 将 \hyperref[fielddesc:MSTATUSMIE]{MIE} 或 \hyperref[fielddesc:USTATUSUIE]{UIE} 的状态复制到 \hyperref[fielddesc:MSTATUSMPIE]{MPIE} 或 \hyperref[fielddesc:USTATUSUPIE]{UPIE}，然后清零 \hyperref[fielddesc:MSTATUSMIE]{MIE} 或 \hyperref[fielddesc:USTATUSUIE]{UIE}，从而全局禁用中断。
    \item 通过跳转到 \hyperref[regdesc:MTVEC]{mtvec} 或 \hyperref[regdesc:UTVEC]{utvec} 中存储地址的字对齐偏移量进入陷阱。
\end{itemize}

机器模式中断 $ID=i$ 按字对齐的陷阱地址 = $(\hyperref[fielddesc:MTVECBASE]{mtvec} + 4i)$，用户模式中断 $ID=i$ 按字对齐的陷阱地址 = $(\hyperref[fielddesc:UTVECBASE]{utvec} + 4i)$。

在跳转到对应模式的陷阱向量之后，执行流程取决于软件实现，但一般来说该中断将在某个中断服务程序 (ISR) 中被处理（并清除），然后在 CPU 遇到 MRET 或 URET 指令后恢复正常程序流。

执行 MRET 或 URET 指令后，CPU 将进行以下操作：
\begin{itemize}
    \item 将 \hyperref[fielddesc:MSTATUSMPIE]{MPIE} 或 \hyperref[fielddesc:MSTATUSUPIE]{UPIE} 的状态复制回 \hyperref[fielddesc:MSTATUSMIE]{MIE} 或 \hyperref[fielddesc:USTATUSUIE]{UIE}，然后清零 \hyperref[fielddesc:MSTATUSMPIE]{MPIE} 或 \hyperref[fielddesc:USTATUSUPIE]{UPIE}。这意味着，如果之前置位了 \hyperref[fielddesc:MSTATUSMPIE]{MPIE} 或 \hyperref[fielddesc:USTATUSUPIE]{UPIE}，则执行 MRET 或 URET 后 \hyperref[fielddesc:MSTATUSMIE]{MIE} 或 \hyperref[fielddesc:USTATUSUIE]{UIE} 将被置位，进而全局使能中断。
    \item 跳转到 \hyperref[regdesc:MEPC]{mepc} 或 \hyperref[regdesc:UEPC]{uepc}中存储的地址，然后恢复执行。
\end{itemize}

软件可以在 ISR 内部实现中断嵌套，具体请参考章节 \ref{sec:riscvcpu-int-suggest-op}。

中断控制器具有以下行为特点：

\begin{itemize}
    \item 仅当中断优先级大于或等于阈值寄存器中的值时，它才会反映在 \hyperref[regdesc:INTPRICORE0CPUINTEIPSTATUSREG]{INTPRI\_CORE0\_CPU\_INT\_EIP\_STATUS\_REG} 中。
    \item 如果一个中断反映在 \hyperref[regdesc:INTPRICORE0CPUINTEIPSTATUSREG]{INTPRI\_CORE0\_CPU\_INT\_EIP\_STATUS\_REG} 中但是还未被处理，则可以通过降低它的优先级或提高全局阈值将其屏蔽（进而防止 CPU 对其进行处理）。
    \item 如果一个中断反映在\\ \hyperref[regdesc:INTPRICORE0CPUINTEIPSTATUSREG]{INTPRI\_CORE0\_CPU\_INT\_EIP\_STATUS\_REG} 中，要清除它（防止被处理），则必须将其禁用（如果是边沿属性的中断则需要清除）。
\end{itemize}

\subsection{建议操作}
\label{sec:riscvcpu-int-suggest-op}

\subsubsection{延迟}
\label{sec:riscvcpu-int-latency}

配置中断控制器时应考虑延迟问题。

在稳态操作中，中断控制器的等待时间固定为 4 个周期。稳态操作的意思是最近没有对中断控制器寄存器作任何更改。这意味着一个中断从被中断控制器触发 (assert) 到被 CPU 开始处理刚好消耗 4 个周期。这也意味着，在抢占发生之前，CPU 最多可以执行 5 条指令。

当寄存器被修改时，中断控制器会进入临时状态，然后需要最多 4 个周期才能再次进入稳态。在临时状态期间，中断的顺序可能无法预测，因此，需要软件采取一些安全措施以避免任何同步问题。

还须注意的是，中断控制器的配置寄存器位于 APB 地址范围内，因此对这些寄存器的读写访问可能需要消耗几个周期。

考虑到上述特征，建议用户在修改中断控制器寄存器时遵循以下操作顺序：

\begin{enumerate}
    \item 保存 \hyperref[fielddesc:MSTATUSMIE]{MIE} 的状态，然后将其清零
    \item 通过“读-修改-写”的方式写中断控制器寄存器
    \item 执行 FENCE 指令以等待所有未完成的写操作完成
    \item 最后，恢复 \hyperref[fielddesc:MSTATUSMIE]{MIE} 的状态
\end{enumerate}

如上述步骤显示，建议用户在配置中断控制器寄存器之前先全局禁用中断 (\hyperref[fielddesc:MSTATUSMIE]{MIE}=0)，然后立即恢复 \hyperref[fielddesc:MSTATUSMIE]{MIE}。

执行完上述操作后，中断控制器将恢复稳态操作。

\subsubsection{配置流程}
\label{sec:riscvcpu-int-config}

默认情况下，\hyperref[regdesc:MSTATUS]{mstatus} 里的 \hyperref[fielddesc:MSTATUSMIE]{MIE} 为 0，即全局禁用中断。在中断堆栈初始化（包括将 \hyperref[regdesc:MTVEC]{mtvec} 设置为中断向量地址）完成之后，软件必须将 \hyperref[fielddesc:MSTATUSMIE]{MIE} 置为 1。

在 \hyperref[regdesc:INTPRICORE0CPUINTTHRESHREG]{INTPRI\_CORE0\_CPU\_INT\_THRESH\_REG} 中，外部中断的阈值默认为 0。对于基于优先级的中断屏蔽，可在 CPU 复位后将对应位初始化为 1。这样所有默认优先级为 0 的中断源都将被屏蔽。

在正常情况下，如果要使能某个外部中断 \regindex{n}，可以遵循以下步骤：
\begin{enumerate}
    \item 保存 \hyperref[fielddesc:MSTATUSMIE]{MIE} 的状态，然后将其清零
    \item 根据中断的类型（边沿/电平），置位或清零 \hyperref[regdesc:INTPRICORE0CPUINTTYPEREG]{INTPRI\_CORE0\_CPU\_INT\_TYPE\_REG} 中的第 \regindex{n} 个位
    \item 通过写入 \hyperref[regdesc:INTPRICORE0CPUINTPRINREG]{INTPRI\_CORE0\_CPU\_INT\_PRI\_\regindex{n}\_REG} 指定优先级（最低为 1，最高为 15）
    \item 置位 \hyperref[regdesc:INTPRICORE0CPUINTENABLEREG]{INTPRI\_CORE0\_CPU\_INT\_ENABLE\_REG} 中的第 \regindex{n} 个位
    \item 执行 FENCE 指令
    \item 恢复 \hyperref[fielddesc:MSTATUSMIE]{MIE} 的状态
\end{enumerate}

当一个或多个中断在等待时，CPU 将确认（声明）最高优先级的中断，然后跳转到与该中断 ID 相对应的陷阱向量地址。软件可以通过读取 \hyperref[regdesc:MCAUSE]{mcause} 来推断陷阱类型（\hyperref[fielddesc:MCAUSEINT]{mcause(31)} 为 1 代表中断，为 0 代表异常）和中断 ID（\hyperref[fielddesc:MCAUSECODE]{mcause(4-0)} 提供中断或异常的 ID）。如果陷阱向量中的每个表项都是指向不同陷阱处理程序的跳转指令，则软件无需做此推断。最后，陷阱处理程序会将程序指引到该中断相应的 ISR。

进入 ISR 后，如果是边沿触发中断，则软件必须置位 \hyperref[regdesc:INTPRICORE0CPUINTCLEARREG]{INTPRI\_CORE0\_CPU\_INT\_CLEAR\_REG} 中的第 \regindex{n} 个位。如果是电平触发中断，则必须清除相应的中断源。

软件还可以更新 \hyperref[regdesc:INTPRICORE0CPUINTTHRESHREG]{INTPRI\_CORE0\_CPU\_INT\_THRESH\_REG} 的值并置位 \hyperref[fielddesc:MSTATUSMIE]{MIE} 来让更高优先级的中断抢占当前 ISR（即嵌套），但是，在此之前，必须先保存所有状态 CSR（\hyperref[regdesc:MEPC]{mepc}、\hyperref[regdesc:MSTATUS]{mstatus}、\hyperref[regdesc:MCAUSE]{mcause} 等），这是由于发生嵌套时状态 CSR 的值会被覆盖。之后，在退出 ISR 时，再恢复这些 CSR 的值。

最后，程序从 ISR 返回到陷阱处理程序之后，可以执行 MRET 指令以恢复正常程序流。

如果不再需要中断 \regindex{n} 并且需要将其禁用，则可以遵循以下操作步骤：
\begin{enumerate}
    \item 保存 \hyperref[fielddesc:MSTATUSMIE]{MIE} 的状态，然后将其清零
    \item 读取 \hyperref[regdesc:INTPRICORE0CPUINTEIPSTATUSREG]{INTPRI\_CORE0\_CPU\_INT\_EIP\_STATUS\_REG} 检查中断是否在等待
    \item 置位/复位 \hyperref[regdesc:INTPRICORE0CPUINTENABLEREG]{INTPRI\_CORE0\_CPU\_INT\_ENABLE\_REG} 中的第 \regindex{n} 个位
    \item 如果中断属于边沿类型并且在等待，则必须切换 \hyperref[regdesc:INTPRICORE0CPUINTCLEARREG]{INTPRI\_CORE0\_CPU\_INT\_CLEAR\_REG} 中的第 \regindex{n} 个位以清空它的等待状态
    \item 执行 FENCE 指令
    \item 恢复 \hyperref[fielddesc:MSTATUSMIE]{MIE} 的状态
\end{enumerate}

以上只是建议的操作方案，实际操作由软件实现决定。

\section{核心本地中断 (CLINT)}
\label{sec:riscvcpu-clint}

\subsection{概述}
\label{sec:riscvcpu-clint-overview}

CPU 支持 4 个本地电平触发中断源，它们具有静态的优先级。

\begin{longtable}{ | c | c | r | }
\caption{核心本地中断 (CLINT) 源} \label{tab:riscvcpu-clint-table}
\\ \hline
\rowcolor{lightgray}
ID & 描述 & 优先级 \\ \hline
0 & 用户模式软件中断 & 1 \\ \hline
3 & 机器模式软件中断 & 3 \\ \hline
4 & 用户模式定时器中断 & 0 \\ \hline
7 & 机器模式定时器中断 & 2 \\ \hline
\end{longtable}

这些中断源具有固定的 ID 和优先级，在两种模式下均不会被中断控制器的阈值寄存器屏蔽。

根据 \hyperref[regdesc:MIDELEG]{mideleg} CSR 中相应位的复位值，其中有两个中断（0 和 4）默认委托给 用户模式。

必须注意的是，虽然 CLINT 中断有固定的优先级，但是等待中的外部中断源总是比 CLINT 源具有更高的优先级。

\subsection{特性}
\begin{itemize}
     \item 4 个具有静态优先级和 ID 的电平触发中断源
     \item 映射到存储器的配置和状态寄存器
     \item 支持机器模式和用户模式下的中断
     \item 64 位定时器，支持中断溢出标志
     \item 软件中断
\end{itemize}

\subsection{软件中断}
\label{sec:riscvcpu-clint-software}

机器模式和用户模式软件中断源分别通过设置或清除存储器映射寄存器 \hyperref[regdesc:CLINTMSIPREG]{MSIP} 或 \hyperref[regdesc:CLINTUSIPREG]{USIP} 来控制。

若需在某一模式下使能核心级别的中断，则必须设置 \hyperref[regdesc:MIE]{mie}/\hyperref[regdesc:UIE]{uie} CSR 中的 \hyperref[fielddesc:MIEMSIE]{MSIE}/\hyperref[fielddesc:UIEUSIE]{USIE}。

若需获取某一模式下的中断等待状态，可读取 \hyperref[regdesc:MIP]{mip}/\hyperref[regdesc:UIP]{uip} CSR 中的 \hyperref[fielddesc:MIPMSIP]{MSIP}/\hyperref[fielddesc:UIPUSIP]{USIP}。

请注意，ID 为 0 的用户模式软件中断在 \hyperref[regdesc:MIDELEG]{mideleg} CSR 中的相应位默认置 1，即中断 0 默认委托给用户模式，但可以通过切换该位的值在机器模式下使用该中断。同样地，也可以设置机器模式软件中断对应的位，以便在用户模式下使用。

\subsection{定时器计数器与中断}
\label{sec:riscvcpu-clint-timer}

\hyperref[regdesc:CLINTMTIMEREG]{MTIME} 是 CPU 本地的机器模式定时器计数器寄存器，是一个可读可写的 64 位存储器映射寄存器。定时器计数器可以通过设置 \hyperref[regdesc:CLINTMTIMECTLREG]{MTIMECTL} 中的 \hyperref[fielddesc:CLINTMTIMECTLMTCE]{MTCE} 位来使能。

\hyperref[regdesc:CLINTUTIMEREG]{UTIME} 是一个只读存储器映射寄存器，可从用户模式读取定时器计数器，不过读取的值总是和 \hyperref[regdesc:CLINTMTIMEREG]{MTIME} 相同。

若需使能机器模式/用户模式的定时器中断，请设置 \hyperref[regdesc:CLINTMTIMECTLREG]{MTIMECTL}/\hyperref[regdesc:CLINTUTIMECTLREG]{UTIMECTL} 中的 \hyperref[fielddesc:CLINTMTIMECTLMTIE]{MTIE}/\hyperref[fielddesc:CLINTUTIMECTLUTIE]{UTIE}。若需在核心级别使能该中断，还必须设置 \hyperref[regdesc:MIE]{mie} 中的 \hyperref[fielddesc:MIEMTIE]{MTIE}/\hyperref[fielddesc:UIEUTIE]{UTIE}。

当 64 位定时器的值超过 \hyperref[regdesc:CLINTMTIMECMPREG]{MTIMECMP}/\hyperref[regdesc:CLINTUTIMECMPREG]{UTIMECMP} 中写入的 64 位比较值时，机器模式/用户模式的中断被触发 (assert)。

若需获取机器模式/用户模式定时器中断的等待状态，请读取 \hyperref[regdesc:CLINTMTIMECTLREG]{MTIMECTL}/\hyperref[regdesc:CLINTUTIMECTLREG]{UTIMECTL} 中的 \hyperref[fielddesc:CLINTMTIMECTLMTIP]{MTIP}/\hyperref[fielddesc:CLINTUTIMECTLUTIP]{UTIP}。

为了在机器模式/用户模式下解除 (de-assert) 定时器中断，必须清除 \hyperref[fielddesc:CLINTMTIMECTLMTIE]{MTIE}/\hyperref[fielddesc:CLINTUTIMECTLUTIE]{UTIE} 位或更新 \hyperref[regdesc:CLINTMTIMECMPREG]{MTIMECMP}/\hyperref[regdesc:CLINTUTIMECMPREG]{UTIMECMP} 寄存器。

若需获取不同模式中断在核心级别的等待状态，可读取 \hyperref[regdesc:MIP]{mip}/\hyperref[regdesc:UIP]{uip} 中的 \hyperref[fielddesc:MIPMTIP]{MTIP}/\hyperref[fielddesc:UIPUTIP]{UTIP}。

64 位定时器计数器溢出时，\hyperref[regdesc:CLINTMTIMECTLREG]{MTIMECTL}/\hyperref[regdesc:CLINTUTIMECTLREG]{UTIMECTL} 中的 \hyperref[fielddesc:CLINTMTIMECTLMTOF]{MTOF}/\hyperref[fielddesc:CLINTUTIMECTLUTOF]{UTOF} 会立即置位，在处理完溢出情况后，可以将其清除。

注意，ID 为 4 的用户模式定时器中断在 \hyperref[regdesc:MIDELEG]{mideleg} CSR 中的相应位默认置 1，即中断 4 默认委托给用户模式，但可以通过切换该位的值在机器模式下使用该中断。同样地，也可以设置机器模式软件中断对应的位，以便在用户模式下使用。

\section{物理存储器保护}
\label{sec:riscvcpu-pmp}

\subsection{概述}
\label{sec:riscvcpu-pmp-overview}
CPU 内核包含一个物理存储器保护单元，完全遵循 RISC\-/V 指令集手册 V1.10 第二卷“特权架构”，可以供软件设置存储器访问特权（读、写、执行权限）。除了标准的 PMP 检查之外，CPU 内核还实现了自定义物理存储器属性 (PMA) 检查器，从而根据预定义的属性提供额外的权限检查。

\subsection{特性}
\label{sec:riscvcpu-pmp-features}
PMP 单元用于控制对物理存储器的访问。它支持 16 个区域，最小可以对 4 个字节大小的区域进行权限设定，支持的最大 NAPOT 范围是 4 GB。

\subsection{功能描述}
\label{sec:riscvcpu-pmp-func-descr}

软件可以设置 PMP 单元的配置和地址寄存器，以保存错误并确保安全执行。配置 PMP CSR 可使能 PMP 单元， 但只能在机器模式下进行配置，一旦 PMP 单元被使能，则将根据 16 个 pmpcfgX 和 pmpaddrX 寄存器（见 \hyperref[sec:riscvcpu-pmp-reg-summ]{寄存器列表}）中的配置值对用户模式下的所有存储器访问进行写入、读取、访问权限检查。

默认情况下，PMP 允许机器模式下的所有存储器访问，而撤销用户模式下的所有访问。这意味着必须通过 pmpcfg 和 pmpaddr 寄存器（见\hyperref[sec:riscvcpu-pmp-reg-summ]{寄存器列表}）设置用户模式可以访问的地址范围和有效权限，以确保访问成功。但是在机器模式下没有此要求，因为机器模式下默认 PMP 允许所有访问。如果在机器模式下也需要 PMP 检查，则软件可以将所需 PMP 表项的锁定位置位来使能权限检查。锁定位一旦置位，就只能通过 CPU 复位被清零。

如果从存储器区域提取指令而没有执行权限，则会在处理器级别生成异常，并且在 \hyperref[regdesc:MCAUSE]{mcause} CSR 中异常原因被设置为指令访问错误。同样，任何没有有效读/写权限的读写访问都将生成异常，并且 \hyperref[regdesc:MCAUSE]{mcause} 会更新为读取存储器访问错误或写入存储器访问错误。如果发生存储器读写异常，则存储器访问地址会更新到 \hyperref[regdesc:MTVAL]{mtval} CSR 中。

\section{物理存储器属性检查器 (PMAC)}
\label{sec:riscvcpu-pma}

\subsection{概述}
\label{sec:riscvcpu-pma-overview}
CPU 核心还实现了自定义物理内存属性检查器 (PMAC)，可根据 CSR 预先定义的存储器类型提供额外的权限检查。

\subsection{特性}
\label{sec:riscvcpu-pma-features}

PMAC 支持以下特性：

\begin{itemize}
    \item 可为指定的存储器空间配置存储器类型
    \item 可为指定的存储器空间配置属性
\end{itemize}

\subsection{功能描述}
\label{sec:riscvcpu-pma-func-descr}

软件可以设置 PMAC 单元的配置和地址寄存器，避免因访问无效存储器区域造成的错误。PMAC CSR 只能在机器模式下进行配置。一旦 PMAC 被使能，则将根据 16 个 pma\_cfgX 和 pma\_addrX 寄存器（见\hyperref[sec:riscvcpu-pma-reg-summ]{Register Summary}）中的配置值对各个模式下的所有存储器访问进行写入、读取、访问权限检查。访问被设置为无效的存储器的表项将导致获取错误或读写错误异常。

PMA 检查相关的的异常生成及错误处理与 PMP 检查类似。如果从配置为 null 或无效的存储器区域提取指令，则会在处理器级别生成异常，并且在 \hyperref[regdesc:MCAUSE]{mcause} CSR 中将异常原因设置为指令访问错误。同样，读写任何 null 或无效的存储器区域都将生成异常，并且 \hyperref[regdesc:MCAUSE]{mcause} 会更新为读取存储器访问错误或写入存储器访问错误。如果发生存储器读写异常，则存储器访问地址会更新到 \hyperref[regdesc:MTVAL]{mtval} CSR 中。对于配置为有效存储器区域的 PMA 检查表项，处理方式与 PMP 检查相同。

如果软件想要关闭对 PMAC 寄存器的写入，可使用表项的锁定位。一旦任何 pma\_cfgX 寄存器中的锁定位被设置，相应的 pma\_cfgX 和 pma\_addrX 寄存器就不能被写入，除非 CPU 复位。

PMAC CSR 中还提供了一个 4 位的域来定义存储器区域的属性。这些位不为 CPU 内核使用，而是基于地址匹配在读写接口上作为边带信号，供 cache 控制器用于其内部操作。

\section{调试}
\label{sec:riscvcpu-dbg}


\subsection{概述}
\label{sec:riscvcpu-dbg-overview}
本节介绍如何调试和测试在 ESP-RISC-V CPU 内核上运行的软件。调试功能由标准 JTAG 管脚提供，并符合 RISC-V 外部调试支持规范 v0.13。

图 \ref{fig:riscvcpu-dbg-sys-overview} 为外部调试系统架构图。

\begin{figure}[h]
\centering
\includegraphics[width=\textwidth/2,keepaspectratio]{\modulefiles/figures/debug_H2.png}
\caption{调试系统架构}\label{fig:riscvcpu-dbg-sys-overview}
\end{figure}

用户与运行调试器（Debugger，例如 GDB）的调试主机（DEBUG HOST，例如笔记本电脑）进行交互。调试器通过调试转换器（Debug Translator，可能包含硬件驱动，例如 OPENOCD）与调试传输硬件（DEBUG TRANSPORT，例如 ESP-Prog 适配器）进行通信。调试传输硬件通过标准 JTAG 接口将调试主机连接到 ESP-RISC-V 内核的调试传输模块 (JTAG DTM)。JTAG DTM 使用调试模块接口 (DMI) 提供对调试模块 (DM) 的访问。

DM 允许调试器暂停选中的内核。抽象命令提供对 GPR（通用寄存器）的访问。程序缓冲区允许调试器在内核上执行任意代码，从而读取 CPU 内核的其他运行状态。CPU 内核的其他运行状态也可以由其他抽象命令读取。ESP-RISC-V 内核带有一个支持 4 个触发器的触发器模块。当满足触发条件时，内核将自发暂停并通知调试模块。

系统总线访问模块无需使用 RISC-V 内核即可访问存储器和外设寄存器。


\subsection{特性}
\label{sec:riscvcpu-dbg-features}
基础调试功能具有以下特性：

\begin{itemize}
    \item {向调试器提供有关实现的必要信息}
    \item {支持暂停和恢复 CPU 内核}
    \item {CPU 内核寄存器（包括 CSR）可以由调试器读取/写入}
    \item {CPU 可以从复位后执行的第一条指令开始就被调试}
    \item {可以通过调试器复位 CPU 内核}
    \item {可以通过软件断点指令暂停 CPU}
    \item {硬件单步调试}
    \item {通过程序缓冲区在暂停的 CPU 中执行任意指令。支持 16 字的程序缓冲区}
    \item {支持系统总线的字对齐地址访问}
    \item {支持 4 个硬件触发器（可用作断点/观察点），具体见章节 \ref{sec:riscvcpu-hwt}}
\end{itemize}

\subsection{功能描述}
\label{sec:riscvcpu-dbg-func-descr}

调试机制遵守 RISC-V 外部调试支持规范 v0.13。有关调试功能的详细介绍，请参考 RISC-V 外部调试支持规范。

\section{硬件触发器}
\label{sec:riscvcpu-hwt}

\subsection{特性}
\label{sec:riscvcpu-hwt-features}

硬件触发器模块提供了断点和观察点功能，供调试使用。硬件触发器具有以下特性：

\begin{itemize}

    \item 可以通过引起断点异常来抢占执行
    \item 可以暂停执行并将控制权转移给调试器
    \item 支持 NAPOT（2 的幂次方对齐区域）地址编码
\end{itemize}

\subsection{功能描述}
\label{sec:riscvcpu-hwt-func-descr}

硬件触发器模块提供了 4 个 CSR，见 \hyperref[sec:riscvcpu-hwt-reg-summ]{寄存器列表}。其中，\hyperref[regdesc:TDATA1]{tdata1} 和 \hyperref[regdesc:TDATA2]{tdata2} 是抽象 CSR，也就是说它们是用于访问某个触发单元中的内部寄存器的影子寄存器，一次访问一个触发单元。

要选择特定的触发单元，需要将相应的编号 (0-3) 写入 \hyperref[regdesc:TSELECT]{tselect} CSR。当写入有效数值时，抽象 CSR \hyperref[regdesc:TDATA1]{tdata1} 和 \hyperref[regdesc:TDATA2]{tdata2} 将自动匹配该触发单元的内部寄存器。每个触发单元都有两个内部寄存器，即 \hyperref[regdesc:MCONTROL]{mcontrol} 和 \hyperref[regdesc:MADDRESS]{maddress}，它们分别与 tdata1 和 tdata2 匹配。

向 tselect 写入超过最大编号的数值时会导致该数值被裁剪为最大的编号，此编号可以被读回。这个特性可用于枚举初始化期间或使用调试器时可用的触发器。

由于软件或调试器可能需要知道所选触发器的类型以便正确解读 tdata1 和 tdata2，因此 tdata1 的 4 个位 (31-28) 对所选触发器的类型进行了编码。此域为只读访问属性，并且值始终为 0x2，代匹配类型触发器，因此，可以推断 tdata1 和 tdata2 会通过 mcontrol 和 maddress 被解读。RISC-V 调试规范 v0.13 提供了其他可能值的信息，但是该触发模块仅支持 0x2 类型。

一旦选定了触发单元，就可以通过置位 mcontrol CSR (tdata1) 中相应的域并将目标地址写入 maddress CSR (tdata2) 来对该触发单元进行配置。

通过写入 mcontrol 的 action 域，可以将每个触发单元配置为引起断点异常或进入调试模式。该域只能从调试器写入，因此默认情况下，触发器（如果启用）将引起断点异常。

每个触发单元的 mcontrol 都有一个 \hyperref[fielddesc:MCONTROLHIT]{hit} 域。在 CPU 暂停或进入异常后，通过读取该域可以查明是否是触发单元触发了。触发器触发后该域会立即被置位，但在恢复操作之前必须被手动清零，不过不清零不会影响正常执行。

每个触发单元仅支持地址匹配，该地址可以是存储器访问地址，也可以是指令的虚拟地址。通过写入所选触发单元的 maddress (tdata2) CSR，可以指定区域的地址和大小。大于 1 个字节的区域大小通过 NAPOT 编码（见表 \ref{tab:riscvcpu-hwt-table}）指定，并通过置位 mcontrol 中 match 域来使能。注意，根据定义，NAPOT 编码地址的起始地址与区域大小对齐（即，是区域大小的整数倍）。

\begin{longtable}{ | c | c | r | }

\\ \hline
\rowcolor{lightgray}

\hyperref[regdesc:TCONTROL]{tcontrol} CSR 对所有触发单元都是通用的。在机器模式下，当陷阱处理程序运行时，该寄存器可用于阻止触发器重复引起异常，而且默认会禁用 ISR 内部的断点异常，但是，出于调试目的，可以在进入 ISR 之前手动使能断点异常。如果将触发器配置为进入调试模式，则此 CSR 不相关。

\subsection{触发执行流程}
\label{sec:riscvcpu-hwt-suggest-op}

当触发器触发引起 hart 暂停并进入调试模式时 (\hyperref[fielddesc:MCONTROLACTION]{action} = 1)：
\begin{itemize}
    \item \hyperref[regdesc:DPC]{dpc} 在解码阶段被设置为当前程序计数器 (PC)
    \item \hyperref[regdesc:DCSR]{dcsr} 的 cause 域被设置为 2，表示暂停是由于触发器触发引起
    \item 与触发的触发器对应的 \hyperref[fielddesc:MCONTROLHIT]{hit} 域被置位
\end{itemize}

当触发器触发引起 hart 进入陷阱时 (\hyperref[fielddesc:MCONTROLACTION]{action} = 0)：
\begin{itemize}
    \item \hyperref[regdesc:MEPC]{mepc} 在解码阶段被设置为当前 PC
    \item \hyperref[regdesc:MCAUSE]{mcause} 被设置为 3，即断点异常
    \item \hyperref[fielddesc:TCONTROLMPTE]{mpte} 被设置为陷阱发生之前的 \hyperref[fielddesc:TCONTROLMTE]{mte} 的值
    \item \hyperref[fielddesc:TCONTROLMTE]{mte} 被设置为 0
    \item 与触发的触发器对应的 \hyperref[fielddesc:MCONTROLHIT]{hit} 域被置位
\end{itemize}

\textit{说明：如果两个触发器同时触发，一个 action = 0，action = 1，则 hart 会暂停并进入调试模式。}

\section{追踪}
\label{sec:riscvcpu-trace}

\subsection{概述}
\label{sec:riscvcpu-trace-overview}

为了支持非侵入式软件调试，CPU 内核提供了指令追踪接口，为离线调试提供相关信息。追踪编码器 (Trace Encoder) 模块可通过指令追踪接口获取这些信息，并将其压缩后存储在分配的存储器中。软件解码器可直接从存储器中读取这些信息，无需中断 CPU 内核，便可重新生成 CPU 内核实际执行的程序。


\subsection{特性}
\label{sec:riscvcpu-trace-features}

CPU 内核支持指令追踪功能，并按照 RISC-V 处理器追踪规范 v1.0 (RISC-V Processor Trace v1.0) 中的规定向追踪编码器提供以下信息：

\begin{itemize}
    \item 退役指令数量
    \item 异常和中断是否发生及其原因、陷阱值
    \item hart 当前的特权级别
    \item 跳转、分支和返回退役指令的指令类型
    \item 程序计数器更改之前和之后退役指令的地址
\end{itemize}

\subsection{功能描述}
\label{sec:riscvcpu-trace-func-descr}

ESP-RISC-V CPU 核心实现了强制指令 delta 追踪，也称为分支追踪。它的工作原理是通过发送程序获取的 delta 信息来追踪从已知起始地址开始的执行情况。delta 通常由跳转、调用、返回、分支类型的指令、中断以及异常引入。所有此类 delta 以及其他详细信息，如原因和实际指令/地址，都通过高带宽指令追踪接口从 CPU 输出给追踪编码器，追踪编码器对这些信息进行操作，将信息压缩后存入存储器，供解码器离线调试。有关编码的更多信息，请参阅章节 \ref{mod:riscvtracenc} \textit{\nameref{mod:riscvtracenc}}。

CPU 没有任何内部寄存器来控制指令追踪接口，所有相关控制寄存器都在 \ref{mod:riscvtracenc} \textit{\nameref{mod:riscvtracenc}} 模块中。

\section{专用 IO}
\label{sec:riscvcpu-fastio}

\subsection{概述}
\label{sec:riscvcpu-fastio-overview}

GPIO 通常是一个 APB 外围设备，这意味着对输出的更改和对输入的读取可能会卡在写入缓冲区或其他传输之后，并且通常速度较慢，因为 APB 总线的运行速度通常低于 CPU。
作为替代方案，CPU 内核实现了 I/O 处理器特定的 CPU 寄存器 (CSR)，直接连接到 GPIO 矩阵或 IO 管脚。这些寄存器只需一条指令即可访问，因此十分迅速。

\subsection{特性}
\label{sec:riscvcpu-fastio-features}
 \begin{itemize}
    \item 8 个直接映射到 GPIO 的专用 IO
    \item 驱动输出端口无延迟
    \item 读取输入值有两个 CPU 周期的延迟
\end{itemize}

\subsection{功能描述}
\label{sec:riscvcpu-fastio-func-descr}

CPU 核心有一组 8 个输入和输出（管脚值 + 管脚输出使能），这些输入和输出端口直接连接到 GPIO 矩阵，可通过 GPIO 矩阵映射到最上层的管脚。更多相关信息请参考 \ref{mod:iomuxgpio} \textit{\nameref{mod:iomuxgpio}}。

CPU 自定义了 3 个 CSR：
\begin{itemize}
    \item GPIO\_IN 只读，表示输入值。
    \item GPIO\_OUT 可读可写，表示 GPIO 输出值。
    \item GPIO\_OEN 可读可写，表示 GPIO 输出使能状态，控制管脚方向，拉高表示管脚配置为输出模式，拉低表示配置为输入模式。
\end{itemize}

\section{原子 (A) 扩展}
\label{sec:riscvcpu-atomic}

\subsection{概述}
\label{sec:riscvcpu-atomic-overview}

对原子 (A) 扩展的支持符合 RISC\-/V 指令集手册 V2.2 第一卷“非特权架构”，重点是保证向前推进，即功能上避免出现任何数据存储器被无限期锁定的情况。

原子指令目前忽略 aq（获取）和 rl（释放）位，因为这些位对于目前这种可以保证存储器排序的架构无关。

\subsection{功能描述}
\label{sec:riscvcpu-atomic-func-descr}

\subsubsection{加载保留字 (LR.W) 指令}
\label{sec:riscvcpu-atomic-lr}

LR.W 指令只是锁定正在读取的 32 位对齐存储器地址。 一旦某个 4 字节的存储器空间被锁定，它将保持锁定状态，其他 hart 无法访问该区域，直到在执行过程中遇到以下任何一种情况：

\begin{itemize}
     \item 任何读操作
     \item 任何写操作
     \item 任何中断/异常
     \item 向后跳转/向后分支跳转
     \item JALR 指令
     \item ECALL/EBREAK/MRET/URET 指令
     \item FENCE/FENCE.I 指令
     \item 调试模式
     \item 临界区超过 64 字节
     \item SC.W 指令中的数据地址与 LR.W 指令中的不匹配
\end{itemize}

如果发生上述任何一种情况，存储器锁将立刻被释放。如果遇到 SC.W 指令，则存储器锁不会立即被释放，但最终会被释放，有关具体如何释放的信息，请参考章节 \ref{sec:riscvcpu-atomic-sc}。

如果地址未对齐，则会导致异常 \hyperref[regdesc:MCAUSE]{mcause} = 6。

\subsubsection{条件存入字 (SC.W) 指令}
\label{sec:riscvcpu-atomic-sc}

SC.W 指令首先检查内存锁是否仍然有效，以及地址是否与上一条 LR.W 指令指定的地址相同，只有以上条件同时满足时，SC.W 指令才会对存储器执行写操作，并且在收到存储器发出的写操作完成确认信息后立即释放锁。

如果锁已失效（失效原因见章节 \ref{sec:riscvcpu-atomic-lr}），SC.W 指令将在目标寄存器 rd 中设置失败代码，目前只能设置为 1。

如果地址未对齐，则会导致异常 \hyperref[regdesc:MCAUSE]{mcause} = 6。

\subsubsection{AMO 指令}
\label{sec:riscvcpu-atomic-amo}

AMO (原子性存储器操作) 指令分三步执行：

\begin{enumerate}
     \item 从 rs1 给定的存储器地址读取数据，并将数据保存到目标寄存器 rd。
     \item 将 rd 和 rs2 中的数据进行运算。
     \item 将上面第 2 步得到的结果写入 rs1 给定的存储器地址。
\end{enumerate}

AMO 运算共有 9 种类型：SWAP、ADD、AND、OR、XOR、MAX、MIN、MAXU 和 MINU。

在整个过程中，存储器地址保持锁定状态，其他 hart 无法访问。如果遇到未对齐的地址，则会导致异常 \hyperref[regdesc:MCAUSE]{mcause} = 6。

AMO 操作的读/写访问 (PMP/PMA) 都在第一步中进行检查。若发生错误，则 \hyperref[regdesc:MCAUSE]{mcause} = 7。
