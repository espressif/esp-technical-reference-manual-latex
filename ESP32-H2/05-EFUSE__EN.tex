\chapter{eFuse Controller (EFUSE)}
\label{mod:efuse}

\section{Overview}

\chipname{} contains a 4096-bit eFuse memory to store user data and hardware parameters, including control parameters for hardware modules, calibration parameters, the MAC address, and keys used for the encryption and decryption module. Once an eFuse bit is programmed to 1, it can never be reverted to 0. Users cannot directly access the eFuse memory. They can only use the eFuse controller to read and write eFuse memory bits. For confidential data in eFuse memory, read protection can be enabled by programming the corresponding read protection bit. So, the data cannot be accessed via the controller.


\section{Features}

\begin{itemize}
    \item 4096-bit one-time-programmable memory including 1792 bits reserved for custom use
    \item Configurable write protection
    \item Configurable read protection
    \item Various hardware encoding schemes against data corruption
\end{itemize}


\section{Functional Description}

\subsection{Structure}

The eFuse system consists of the eFuse controller and eFuse memory. Data flow in this system is shown in Figure \ref{fig:efuse-data-flow}.

To program data to the eFuse memory, write the data to the programming register first and then execute the programming instruction. For detailed programming steps, please refer to Section \ref{sec:efuse-software-programming-para}.

Users cannot directly read data from the eFuse memory. They need to use the eFuse controller to take the data into the reading data register of the corresponding address segment. During the reading process, if the data is inconsistent with that in the eFuse memory, the eFuse controller can automatically correct it through the hardware encoding mechanism (see Section \ref{sec:efuse-data-storage} for details), and send the error message to the error report register. For detailed steps to read parameters, please refer to Section \ref{sec:efuse-software-read-para}.\\

Among these blocks, BLOCK4 \verb+~+ 9 can be used to store KEY0 \verb+~+ 5. Up to six 256-bit keys can be written into eFuse. Whenever a key is written, its purpose value should also be written (see table \ref{tab:efuse-key-purpose}). For example, when a key for the JTAG function in HMAC Downstream mode is written to KEY\textcolor{myred}{3} (i.e., BLOCK7), its key purpose value 6 should also be written to EFUSE\_KEY\_PURPOSE\_\textcolor{myred}{3}.

\begin{tiplisting}
Do not program the XTS-AES key or ECDSA\_DS key into the KEY5 block, i.e., BLOCK9. Otherwise, the key may be unreadable. Instead, program them into the preceding blocks, i.e., BLOCK4 \verb+~+ BLOCK8. The last block, BLOCK9, is used to program other keys.
\end{tiplisting}

BLOCK1 \verb+~+ BLOCK10 use the RS coding scheme, so there are some limitations on writing to these parameters. For more detailed information, please refer to Section \ref{sec:efuse-data-storage} and Section \ref{sec:efuse-software-programming-para}.

\subsubsection{Parameters Used by Hardware Modules}
\label{sec:para-hw-modules}

Parameters used by hardware modules are marked with ``Y" in Table \ref{tab:efuse-block-1-10-para} > Column ``Accessible by Hardware". They are used by hardware modules through circuit connections. That is to say, changes to the parameter value will directly affect how the peripheral controlled by it behaves, and this process cannot be intervened by the user.

\subsubsection{\hyperref[fielddesc:EFUSEWRDIS]{EFUSE\_WR\_DIS}}
\label{sec:efuse-wr-dis}

Parameter \hyperref[fielddesc:EFUSEWRDIS]{EFUSE\_WR\_DIS} determines whether individual eFuse parameters are write-protected. After \hyperref[fielddesc:EFUSEWRDIS]{EFUSE\_WR\_DIS} has been programmed, execute an eFuse read operation to make the new values take effect.

Column ``Write Protection by \hyperref[fielddesc:EFUSEWRDIS]{EFUSE\_WR\_DIS} Bit Number" in Table  \ref{tab:efuse-block0-para} and Table \ref{tab:efuse-block-1-10-para} list the specific bits in \hyperref[fielddesc:EFUSEWRDIS]{EFUSE\_WR\_DIS} that disable writing.

When a write protection bit is set to 0, the corresponding parameter is not write-protected and can be programmed, unless it has been programmed before.

When the write protection bit is set to 1, the corresponding parameter is write-protected and none of its bits can be modified, with non-programmed bits always remaining 0 and programmed bits always remaining 1. That is to say, if a parameter is write-protected, it will always remain in this state and cannot be changed.

\subsubsection{\hyperref[fielddesc:EFUSERDDIS]{EFUSE\_RD\_DIS}}
\label{sec:efuse-rd-dis}

Only the parameters in BLOCK4 \verb+~+ BLOCK10 can be set to be read-protected from users, as shown in column ``Read Protection by \hyperref[fielddesc:EFUSERDDIS]{EFUSE\_RD\_DIS} Bit Number" of Table \ref{tab:efuse-block-1-10-para}. After \hyperref[fielddesc:EFUSERDDIS]{EFUSE\_RD\_DIS} has been programmed, execute an eFuse read operation to make the new values take effect.

If an \hyperref[fielddesc:EFUSERDDIS]{EFUSE\_RD\_DIS} bit is 0, the corresponding parameter is not read-protected from users. If it is 1, the parameter is read-protected.

Other parameters that are not in BLOCK4 \verb+~+ BLOCK10 can always be read by users.

When BLOCK4 \verb+~+ BLOCK10 are set to be read-protected, the data in them can still be read by hardware cryptography modules if the EFUSE\_KEY\_PURPOSE\_\textcolor{myred}{\it{n}} bit is set accordingly.

\subsubsection{Data Storage}
\label{sec:efuse-data-storage}

Internally, eFuse uses the hardware encoding scheme to protect data from corruption. The scheme and the encoding process are invisible to users.

All BLOCK0 parameters except for \hyperref[fielddesc:EFUSEWRDIS]{EFUSE\_WR\_DIS} are stored with four backups, meaning each bit is stored four times. This backup scheme is not visible to users.

In BLOCK0, \hyperref[fielddesc:EFUSEWRDIS]{EFUSE\_WR\_DIS} occupies 32 bits, and other parameters takes 152 bits each. So, the total eFuse memory space occupied by BLOCK0 is 32 + 152 * 4 = 640 bits.

In BLOCK1 \verb+~+ BLOCK10, data are coded using Reed-Solomon's RS (44, 32) coding scheme that supports up to 6 bytes of automatic error correction. The primitive polynomial of RS (44, 32) is \(p(x)=x^8+x^4+x^3+x^2+1\).

\begin{figure}[ht!]
    \centering
    \fbox{\includegraphics[width=1.0\textwidth]{\modulefiles/figures/05-eFuse-shift-reg-cir-1.eps}}
    \caption{Shift Register Circuit (first 32 output)}
    \label{fig:efuse-lfsr-circuit-1}
\end{figure}

\begin{figure}[ht!]
    \centering
    \fbox{\includegraphics[width=1.0\textwidth]{\modulefiles/figures/05-eFuse-shift-reg-cir-2.eps}}
    \caption{Shift Register Circuit (last 12 output)}
    \label{fig:efuse-lfsr-circuit-2}
\end{figure}

As shown in Figure \ref{fig:efuse-lfsr-circuit-1} and \ref{fig:efuse-lfsr-circuit-2}, the shift register circuit  processes the 32-byte parameters using RS (44, 32) and encodes them into 44 bytes:

\begin{itemize}
    \item Bytes [0:31] are the data itself
    \item Bytes [32:43] are the encoded parity bytes stored in 8-bit flip-flops DFF1, DFF2, ..., DFF12 (gf\_mul\_n is the result of multiplying a byte of data in \(GF(2^8)\) by \(\alpha^n\), where n is an integer).
\end{itemize}

After that, the hardware programs into eFuse the 44-byte codeword consisting of the data bytes and the parity bytes. When the eFuse block is read, the eFuse controller automatically decodes the codeword and applies error correction if needed.

Because the RS check codes are generated on the entire 32-byte eFuse block, each block can only be written once.

Since the size of BLOCK1 is less than 32 bytes, the unused bits will be treated as 0 by hardware during the RS (44, 32) encoding. Thus, the final coding result will not be affected.

Among blocks using the RS (44, 32) coding scheme, the parameters in BLOCK1 is 24 bytes, and the RS check code is 12 bytes, so BLOCK1 occupies 24 + 12 = 36 bytes in eFuse memory.

The parameter in other blocks (Block2 \verb+~+ 10) is 32 bytes respectively, and the RS check code is 12 bytes, so they occupy (32 + 12) * 9 = 396 bytes in eFuse memory.

\subsection{Programming of Parameters}
\label{sec:efuse-software-programming-para}

The eFuse controller can only program eFuse parameters to one block at a time. BLOCK0 \verb+~+ BLOCK10 share the same address range for storing the parameters to be programmed. Configure parameter \hyperref[fielddesc:EFUSEBLKNUM]{EFUSE\_BLK\_NUM} to indicate which block should be programmed.

Since there is a one-to-one correspondence between the reading data registers and the programming data registers (see table \ref{tab:efuse-softw-read-para} for details), users can find out where the data to be programmed is located in programming registers by checking the parameter description and the parameter location in the corresponding reading data registers.

For example, if the user wants to program the parameter \hyperref[fielddesc:EFUSEDISICACHE]{EFUSE\_DIS\_ICACHE} in BLOCK0 to 1, they can first search the reading data registers EFUSE\_RD\_REPEAT\_DATA0 \verb+~+ 4\_REG in BLOCK0 for where the parameter is located, namely, the 8th bit in \hyperref[regdesc:EFUSERDREPEATDATA0REG]{EFUSE\_RD\_REPEAT\_DATA0\_REG}. So, the user can set the 8th bit of \hyperref[regdesc:EFUSEPGMDATA1REG]{EFUSE\_PGM\_DATA1\_REG}
to 1 and follow the programming steps below. After the steps are completed, the corresponding bit in the eFuse memory will be programmed to 1.

{\bfseries Programming preparation}

\begin{itemize}
    \item {\bfseries Programming BLOCK0}
    \begin{enumerate}
        \item Set \hyperref[fielddesc:EFUSEBLKNUM]{EFUSE\_BLK\_NUM} to 0.
        \item Write into \hyperref[regdesc:EFUSEPGMDATA0REG]{EFUSE\_PGM\_DATA0\_REG} \verb+~+ \hyperref[regdesc:EFUSEPGMDATA1REG]{EFUSE\_PGM\_DATA5\_REG} the data to be programmed to BLOCK0. \\
        The data in \hyperref[regdesc:EFUSEPGMDATA6REG]{EFUSE\_PGM\_DATA6\_REG} \verb+~+ \hyperref[regdesc:EFUSEPGMDATA7REG]{EFUSE\_PGM\_DATA7\_REG} and \hyperref[regdesc:EFUSEPGMCHECKVALUE0REG]{EFUSE\_PGM\_CHECK\_VALUE0\_REG} \verb+~+ \hyperref[regdesc:EFUSEPGMCHECKVALUE2REG]{EFUSE\_PGM\_CHECK\_VALUE2\_REG} does not affect the programming of BLOCK0.
    \end{enumerate}
    \item {\bfseries Programming BLOCK1}
    \begin{enumerate}
        \item Set \hyperref[fielddesc:EFUSEBLKNUM]{EFUSE\_BLK\_NUM} to 1.
        \item Write into \hyperref[regdesc:EFUSEPGMDATA0REG]{EFUSE\_PGM\_DATA0\_REG} \verb+~+ \hyperref[regdesc:EFUSEPGMDATA5REG]{EFUSE\_PGM\_DATA5\_REG} the data to be programmed to BLOCK1. Write into \hyperref[regdesc:EFUSEPGMCHECKVALUE0REG]{EFUSE\_PGM\_CHECK\_VALUE0\_REG} \verb+~+ \hyperref[regdesc:EFUSEPGMCHECKVALUE2REG]{EFUSE\_PGM\_CHECK\_VALUE2\_REG} the corresponding RS check code.\\
        The data in \hyperref[regdesc:EFUSEPGMDATA0REG]{EFUSE\_PGM\_DATA6\_REG} \verb+~+ \hyperref[regdesc:EFUSEPGMDATA5REG]{EFUSE\_PGM\_DATA7\_REG} does not affect the programming of BLOCK1. When calculating RS check of BLOCK1 using software, please treat the 8 bytes as 0.
    \end{enumerate}
    \item {\bfseries Programming BLOCK2 \verb+~+ 10}
    \begin{enumerate}
        \item Set \hyperref[fielddesc:EFUSEBLKNUM]{EFUSE\_BLK\_NUM} to the block number.
        \item Write into \hyperref[regdesc:EFUSEPGMDATA0REG]{EFUSE\_PGM\_DATA0\_REG} \verb+~+ \hyperref[regdesc:EFUSEPGMDATA7REG]{EFUSE\_PGM\_DATA7\_REG} the data to be programmed. Write into \hyperref[regdesc:EFUSEPGMCHECKVALUE0REG]{EFUSE\_PGM\_CHECK\_VALUE0\_REG}  \verb+~+ \hyperref[regdesc:EFUSEPGMCHECKVALUE2REG]{EFUSE\_PGM\_CHECK\_VALUE2\_REG} the corresponding RS code.
    \end{enumerate}
\end{itemize}

{\bfseries Programming process}

The process of programming parameters is as follows:

\begin{enumerate}
\item Configure the value of parameter \hyperref[fielddesc:EFUSEBLKNUM]{EFUSE\_BLK\_NUM} to determine the block to be programmed.

\item Write parameters to be programmed to registers \hyperref[regdesc:EFUSEPGMDATA0REG]{EFUSE\_PGM\_DATA0\_REG} \verb+~+ \hyperref[regdesc:EFUSEPGMDATA7REG]{EFUSE\_PGM\_DATA7\_REG} and  \hyperref[regdesc:EFUSEPGMCHECKVALUE0REG]{EFUSE\_PGM\_CHECK\_VALUE0\_REG} \verb+~+ \hyperref[regdesc:EFUSEPGMCHECKVALUE2REG]{EFUSE\_PGM\_CHECK\_VALUE2\_REG}.

\item Make sure the eFuse programming voltage VDDQ is configured correctly as described in Section \ref{sec:efuse-vddq-timing}.

\item Configure the field \hyperref[fielddesc:EFUSEOPCODE]{EFUSE\_OP\_CODE} of register \hyperref[regdesc:EFUSECONFREG]{EFUSE\_CONF\_REG} to 0x5A5A.

\item Configure the field \hyperref[fielddesc:EFUSEPGMCMD]{EFUSE\_PGM\_CMD} of register \hyperref[regdesc:EFUSECMDREG]{EFUSE\_CMD\_REG} to 1.

\item Poll register \hyperref[regdesc:EFUSECMDREG]{EFUSE\_CMD\_REG} until it is 0x0, or wait for a PGM\_DONE interrupt. For more information on how to identify a PGM\_DONE or READ\_DONE interrupt, please see the end of Section \ref{sec:efuse-software-read-para}.

\item Clear the parameters in \hyperref[regdesc:EFUSEPGMDATA0REG]{EFUSE\_PGM\_DATA0\_REG} \verb+~+ \hyperref[regdesc:EFUSEPGMDATA7REG]{EFUSE\_PGM\_DATA7\_REG} and \\ \hyperref[regdesc:EFUSEPGMCHECKVALUE0REG]{EFUSE\_PGM\_CHECK\_VALUE0\_REG} \verb+~+ \hyperref[regdesc:EFUSEPGMCHECKVALUE2REG]{EFUSE\_PGM\_CHECK\_VALUE2\_REG}.

\item Trigger an eFuse read operation (see Section \ref{sec:efuse-software-read-para}) to update eFuse registers with the new values.
\item Check error record registers. If the values read in error record registers are not 0, the programming process should be performed again following above steps 1 \verb+~+ 7. Please check the following error record registers for different eFuse blocks:
\begin{itemize}
    \item BLOCK0: \hyperref[regdesc:EFUSERDREPEATERR0REG]{EFUSE\_RD\_REPEAT\_ERR0\_REG} \verb+~+ \hyperref[regdesc:EFUSERDREPEATERR4REG]{EFUSE\_RD\_REPEAT\_ERR4\_REG}
    \item BLOCK1: \hyperref[fielddesc:EFUSEMACSPI8MERRNUM] {EFUSE\_MAC\_SPI\_8M\_ERR\_NUM}, \hyperref[fielddesc:EFUSEMACSPI8MFAIL] {EFUSE\_MAC\_SPI\_8M\_FAIL}
    \item BLOCK2: \hyperref[fielddesc:EFUSESYSPART1NUM] {EFUSE\_SYS\_PART1\_NUM}, \hyperref[fielddesc:EFUSESYSPART1FAIL] {EFUSE\_SYS\_PART1\_FAIL}
    \item BLOCK3: \hyperref[fielddesc:EFUSEUSRDATAERRNUM] {EFUSE\_USR\_DATA\_ERR\_NUM}, \hyperref[fielddesc:EFUSEUSRDATAFAIL] {EFUSE\_USR\_DATA\_FAIL}
    \item BLOCK4: \hyperref[fielddesc:EFUSEKEY0ERRNUM] {EFUSE\_KEY0\_ERR\_NUM}, \hyperref[fielddesc:EFUSEKEY0FAIL] {EFUSE\_KEY0\_FAIL}
    \item BLOCK5: \hyperref[fielddesc:EFUSEKEY1ERRNUM] {EFUSE\_KEY1\_ERR\_NUM}, \hyperref[fielddesc:EFUSEKEY1FAIL] {EFUSE\_KEY1\_FAIL}
    \item BLOCK6: \hyperref[fielddesc:EFUSEKEY2ERRNUM] {EFUSE\_KEY2\_ERR\_NUM}, \hyperref[fielddesc:EFUSEKEY2FAIL] {EFUSE\_KEY2\_FAIL}
    \item BLOCK7: \hyperref[fielddesc:EFUSEKEY3ERRNUM] {EFUSE\_KEY3\_ERR\_NUM}, \hyperref[fielddesc:EFUSEKEY3FAIL] {EFUSE\_KEY3\_FAIL}
    \item BLOCK8: \hyperref[fielddesc:EFUSEKEY4ERRNUM] {EFUSE\_KEY4\_ERR\_NUM}, \hyperref[fielddesc:EFUSEKEY4FAIL] {EFUSE\_KEY4\_FAIL}
    \item BLOCK9: \hyperref[fielddesc:EFUSEKEY5ERRNUM] {EFUSE\_KEY5\_ERR\_NUM}, \hyperref[fielddesc:EFUSEKEY5FAIL] {EFUSE\_KEY5\_FAIL}
    \item BLOCK10: \hyperref[fielddesc:EFUSESYSPART2ERRNUM] {EFUSE\_SYS\_PART2\_ERR\_NUM}, \hyperref[fielddesc:EFUSESYSPART2FAIL] {EFUSE\_SYS\_PART2\_FAIL}
\end{itemize}
\end{enumerate}

{\bfseries Limitations}

In BLOCK0, each bit can be programmed separately. However, we recommend to minimize programming cycles and program all the bits of a parameter in one programming action. In addition, after all parameters controlled by a certain bit of \hyperref[fielddesc:EFUSEWRDIS]{EFUSE\_WR\_DIS} are programmed, that bit should be immediately programmed. The programming of parameters controlled by a certain bit of \hyperref[fielddesc:EFUSEWRDIS]{EFUSE\_WR\_DIS},
and the programming of the bit itself can even be completed at the same time in one programming action.

BLOCK1 cannot be programmed by users as it has been programmed at manufacturing.

BLOCK2 \verb+~+ 10 can only be programmed once. Repeated programming is not allowed.

Users cannot read eFuse bits directly. The eFuse controller reads eFuse bits and stores the result to the corresponding reading data registers (whose name is prefixed with EFUSE\_RD\_). Then, users can read the eFuse bits from those registers. Details are provided in Table \ref{tab:efuse-softw-read-para}.

{\bfseries Updating reading data registers}
\label{sec:efuse-update-reg}

The eFuse controller reads eFuse memory to update corresponding registers. This read operation happens at system reset and can also be triggered manually by users as needed (e.g., if new eFuse values have been programmed). The process of triggering a read operation by users is as follows:

\begin{enumerate}
    \item Configure the field \hyperref[fielddesc:EFUSEOPCODE]{EFUSE\_OP\_CODE} in register \hyperref[regdesc:EFUSECONFREG]{EFUSE\_CONF\_REG} to 0x5AA5.
    \item Configure the field \hyperref[fielddesc:EFUSEREADCMD]{EFUSE\_READ\_CMD} in register \hyperref[regdesc:EFUSECMDREG]{EFUSE\_CMD\_REG} to 1.
    \item Poll register \hyperref[regdesc:EFUSECMDREG]{EFUSE\_CMD\_REG} until it is 0x0, or wait for a READ\_DONE interrupt. Information on how to identify a PGM\_DONE or READ\_DONE interrupt is provided below in this section.
    \item Read the values of each parameter from eFuse memory.
\end{enumerate}
The eFuse reading data registers will hold all values until the next read operation.

{\bfseries Error detection}

Error record registers allow users to detect if there is any inconsistency between the parameter read by eFuse controller and that in eFuse memory.

Registers EFUSE\_RD\_REPEAT\_ERR0 \verb+~+ 3\_REG indicate if there are any errors in programming parameters (except \hyperref[fielddesc:EFUSEWRDIS]{EFUSE\_WR\_DIS}) to BLOCK0. The value 1 indicates an error is detected in programming the corresponding bit. The value 0 indicates no error.

Registers EFUSE\_RD\_RS\_ERR0 \verb+~+ 1\_REG store the number of corrected bytes as well as the result of RS decoding when eFuse controller reads BLOCK1 \verb+~+ BLOCK10.

The values of the above registers will be updated every time the reading data registers of eFuse controller have been updated.

{\bfseries Identifying program/read operation}

    The methods to identify the completion of a program/read operation are described below. Please note that bit 1 corresponds to a program operation, and bit 0 corresponds to a read operation.
    \begin{itemize}
    \item Method one: Poll bit 1/0 in register \hyperref[regdesc:EFUSEINTRAWREG]{EFUSE\_INT\_RAW\_REG} until it becomes 1, which represents the completion of a program/read operation.
    \item Method two:
        \begin{enumerate}
            \item Set bit 1/0 in register \hyperref[regdesc:EFUSEINTENAREG]{EFUSE\_INT\_ENA\_REG} to 1 to enable the eFuse controller to post a PGM\_DONE or READ\_DONE interrupt.
            \item Configure the Interrupt Matrix to enable the CPU to respond to eFuse interrupt signals. See Chapter \ref{mod:intmtrx} \textit{\nameref{mod:intmtrx}}.
            \item Wait for the PGM\_DONE or READ\_DONE interrupt.
            \item Set bit 1/0 in register \hyperref[regdesc:EFUSEINTCLRREG]{EFUSE\_INT\_CLR\_REG} to 1 to clear the PGM\_DONE or READ\_DONE interrupt.
    \end{enumerate}
\end{itemize}


{\bfseries Note}


When eFuse controller is updating its registers, it will use \hyperref[regdesc:EFUSEPGMDATA0REG]{EFUSE\_PGM\_DATA\textcolor{red}{\it{n}}\_REG} (n=0, 1, ... ,7) again to store data. So please do not write important data into these registers before this updating process is initiated. \\
During the chip boot process, eFuse controller will automatically update data from eFuse memory into the registers that can be accessed by users. Users can get programmed eFuse data by reading corresponding registers. Thus, there is no need to update the reading data registers in such case.

\subsection{eFuse VDDQ Timing}
\label{sec:efuse-vddq-timing}

The eFuse controller operates at the clock frequency of 20 MHz, and its programming voltage VDDQ should be configured as follows:

\begin{itemize}
    \item \hyperref[fielddesc:EFUSEDACNUM]{EFUSE\_DAC\_NUM} (the rising period of VDDQ): The default value of VDDQ is 2.5 V and the voltage increases by 0.01 V in each clock cycle. The default value of this parameter is 255.
    \item \hyperref[fielddesc:EFUSEDACCLKDIV]{EFUSE\_DAC\_CLK\_DIV} (the clock divisor of VDDQ): The clock period to program VDDQ should be larger than 1 $\mu$s.
    \item \hyperref[fielddesc:EFUSEPWRONNUM]{EFUSE\_PWR\_ON\_NUM} (the power-up time for VDDQ): The programming voltage should be stabilized after this time, which means the value of this parameter should be configured to exceed the result of \hyperref[fielddesc:EFUSEDACCLKDIV]{EFUSE\_DAC\_CLK\_DIV} times \hyperref[fielddesc:EFUSEDACNUM]{EFUSE\_DAC\_NUM}.
    \item \hyperref[fielddesc:EFUSEPWROFFNUM]{EFUSE\_PWR\_OFF\_NUM} (the power-out time for VDDQ): The value of this parameter should be larger than 10 $\mu$s.
\end{itemize}
