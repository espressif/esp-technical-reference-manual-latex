\chapter{ESP-RISC-V CPU}
\label{mod:riscvcpu}

\section{概述}
\chipname{} 采用了乐鑫第二代 32 位 RISC-V 指令集架构 (ISA) 的 CPU 核心系统。核心系统包含单核 RISC-V CPU、专用的核心本地中断控制器 (CLIC)、调试模块以及核心本地中断 (CLINT) 定时器。图 \ref{fig:riscvcpu-core-complex-block-diagram} 为 CPU 核心系统的框图。

\begin{figure}[H]
\centering
\includegraphics[scale=1,width=0.8\textwidth]{\modulefiles/figures/C61_block_diag.png}
\caption{CPU 核心系统框图}
\end{figure}
\label{fig:riscvcpu-core-complex-block-diagram}

在 CPU 核心系统中，RISC-V 核心使用 5 级顺序标量流水线架构，针对面积、功耗和性能进行了优化，内置专用的核心本地中断控制器 (CLIC) 和系统总线 (SYS BUS) 接口，用于访问存储器和外设。调试模块将 JTAG 接口连接至 RISC-V 核心。RISC-V 核心还提供用于离线调试的追踪接口。

\section{特性}
\label{sec:riscvcpu-features}
CPU 核心系统中的 RISC-V HP CPU 核心实现了以下标准 RISC-V 扩展：
\begin{itemize}
    \item 必须实现的基本整数 (I)
    \item 乘法/除法 (M)
    \item 原子 (A)
    \item 压缩 (C)
    \item 代码体积优化 (Zc)
    \item 位操作 (Zb)
\end{itemize}


除了上述 RISC-V 指令扩展，CPU 核心还支持以下特性：
\begin{itemize}
    \item 具有五级流水线，支持高达 160 MHz 的工作时钟频率
    \item 兼容《RISC-V 指令集手册 v2.2 第一卷“RISC-V 用户级 ISA”》(RISC-V Instruction Set Manual, Volume I: RISC-V User-Level ISA, Version 2.2)
    \item 兼容《RISC-V 指令集手册 v1.10 第二卷“特权架构”》(RISC-V Instruction Set Manual, Volume II: RISC-V Privileged Architecture, Version 1.10)
    \item 兼容《核心本地中断控制器 (CLIC) v0.9》(RISC-V Core-Local Interrupt Controller (CLIC) Version 0.9)
    \item 兼容《RISC-V 外部调试支持规范 v0.13》(RISC-V External Debug Support Version 0.13.2)
    \item 通过 IRAM/DRAM 接口零等待周期访问片上 SRAM 和缓存中的程序和数据
    \item 32 位 AHB 系统总线用于访问外设
    \item 核心本地中断 (CLINT)
    \item 支持用户 (U) 特权模式
    \item 通过 CLIC 实现用户 (U) 模式中断委派
    \item 总线错误响应
    \item 标准物理存储器保护 (PMP) 和自定义物理存储器属性 (PMA)，最多可配置 16 个区域以确保安全
    \item 分支目标缓冲器 (BTB) 具有 BHT 和 RAS 以提高性能
    \item 调试模块 (DM) 符合《RISC-V 外部调试支持规范 v0.13》，支持通过行业标准的 JTAG/USB 端口连接外部调试器
    \item 调试器通过核心数据总线以及系统总线 (SBA) 直接访问存储器和外设
    \item 支持指令追踪以进行离线调试
    \item 硬件触发器符合《RISC-V 调试规范 v0.13.2》(RISC-V External Debug Support Version 0.13.2)，具有多达 3 个断点/观察点
    \item 支持多达 8 个专用 IO
    \item 可配置的核心性能指标事件

\end{itemize}

\subsection{乘法/除法 (M) 扩展}
\label{sec:riscvcpu-mext}
\subsubsection{概述}
\label{sec:riscvcpu-mext-overview}
除了基本的 RV32I 指令集外，HP 核心还实现了标准整数乘法和除法扩展，并完全符合《RISC-V 指令集手册 v2.2 第一卷“RISC-V 用户级 ISA”》 中指定的 M 标准扩展的 2.0 版本。

\subsubsection{功能描述}

\label{sec:riscvcpu-mext-func-descr}
M 扩展中的指令对两个整数寄存器中的值进行乘法和除法操作。以下是乘法和除法指令在流水线中没有数据冒险的情况下所需的 CPU 周期。
\begin{itemize}
    \item MUL 操作需要 2 个周期
    \item DIV 操作需要 1-19 个周期，具体取决于操作数的情况
\end{itemize}

\subsection{压缩 (C) 扩展}
\label{sec:riscvcpu-cext}
\subsubsection{概述}

\label{sec:riscvcpu-cext-overview}

HP 核心还实现了 RISC-V 标准压缩 (C) 指令集扩展，并完全符合 《RISC-V 指令集手册 v2.2 第一卷“RISC-V 用户级 ISA”》 中 C 标准扩展的 2.0 版本。RVC 是 RISC-V 压缩扩展的简称。RVC 可以为常见操作添加较短的 16 位指令编码，减少静态和动态代码大小。通常，程序中 50-60\% 的 RISC-V 指令可以替换为 RVC 指令，从而可以减少 25-35\% 的代码量。

\subsubsection{功能描述}
\label{sec:riscvcpu-cext-func-descr}
RVC 采用的压缩方案较为简单，为 32 位 RISC-V 指令提供较短的 16 位版本，适用情景如下：
\begin{enumerate}
    \item 立即数和地址偏移量较小
    \item 其中一个寄存器是零寄存器 (x0)、ABI 链接寄存器 (x1) 或 ABI 堆栈指针 (x2)
    \item 目标寄存器和第一个源寄存器相同
    \item 使用的寄存器是 8 个最常用的寄存器，即 x8 $\sim$ x15
\end{enumerate}

\subsubsection{概述}
\label{sec:riscvcpu-zc-overview}

\subsubsection{功能描述}
\label{sec:riscvcpu-zc-func-descr}






\subparagraph{Zcmt 扩展}

Zcmt 扩展也称为表跳转，它从跳转表中读取跳转目标地址，然后跳转到该地址。

表跳转使用指令存储器中的 256 项 32 位宽的表来存储函数地址。表必须至少 64 字节对齐。作为字典压缩的一种形式，表跳可减少 jal、auipc+jalr、jr、auipc+jr 指令的代码大小。
表跳转允许链接器用 cm.jalt 编码和表中的一个条目替换以下指令序列：
\begin{itemize}
    \item 32 位 j 调用
    \item 32 位 jal ra 调用
\end{itemize}

\subsection{位操作 (Zb) 扩展}
\label{sec:riscvcpu-zb}

\subsubsection{概述}
\label{sec:riscvcpu-zb-overview}

位操作（bit manipulation，简称 bitmanip）扩展，也称为 Zb 扩展集，包含多个子扩展，是对 RISC-V 基础架构的补充。这些扩展旨在提升代码密度、性能及效率。HP 核心完全支持 Zb 扩展。有关 Zb 扩展的详细规范，请参阅 \href{https://github.com/riscv/riscv-b}{Zb-specification v1.0.0}。


\subsubsection{功能描述}
\label{sec:riscvcpu-zb-func-descr}

虽然位操作指令是为通用场景设计的，但某些指令在特定场景中更为适用。因此，bitmanip 扩展集被划分为多个较小的子扩展，而非单一的大扩展。每个子扩展都有其对应的 Zb 扩展名称，其包含的指令有在功能上相关，在使用场景上相似，甚至有时具有相同的实现逻辑。一些指令仅存在于某个子扩展中，而另一些指令可能存在于多个子扩展中。这些指令的助记符和编码独立于它们所属的子扩展。因此，在实现具有重复指令的多个子扩展时，逻辑或编码不会冗余。

\subsection{原子 (A) 扩展}
\label{sec:riscvcpu-atomic}

\subsubsection{概述}
\label{sec:riscvcpu-atomic-overview}

对原子 (A) 扩展的支持符合《RISC-V 指令集手册 v2.2 第一卷“RISC-V 用户级 ISA”》，重点是保证向前推进，即功能上避免出现任何数据存储器被无限期锁定的情况。

原子指令目前忽略 aq（获取）和 rl（释放）位，因为这些位对于目前这种可以保证存储器排序的架构无关。

\subsubsection{功能描述}
\label{sec:riscvcpu-atomic-func-descr}

原子 (A) 扩展指令集包含可对存储器执行原子读-改-写操作的指令，用于同步同一存储器空间内运行的多个 RISC-V 硬件线程。原子指令包括两种形式：保留加载/条件存储（load-reserved/store-conditional）指令，以及原子获取并操作（fetch-and-op）存储器指令。目前，这些原子指令会忽略 aq（获取）和 rl（释放）位，因为在当前架构中，存储器顺序总是可以保证，因此这些位无实际作用。

\subsubsection{加载保留字 (LR.W) 指令}
\label{sec:riscvcpu-atomic-lr}

LR.W 指令只是锁定正在读取的 32 位对齐存储器地址。 一旦某个 4 字节的存储器空间被锁定，它将保持锁定状态，其他 hart 无法访问该区域，直到在执行过程中遇到以下任何一种情况：

\begin{itemize}
    \item 任何读操作
    \item 任何写操作
    \item 任何中断/异常
    \item 向后跳转/向后分支跳转
    \item JALR 指令
    \item ECALL/EBREAK/MRET/URET 指令
    \item FENCE/FENCE.I 指令
    \item 调试模式
    \item 临界区超过 64 字节
    \item SC.W 指令中的数据地址与 LR.W 指令中的不匹配
\end{itemize}

如果发生上述任何一种情况，存储器锁将立刻被释放。如果遇到 SC.W 指令，则存储器锁不会立即被释放，但最终会被释放，有关具体如何释放的信息，请参考章节 \ref{sec:riscvcpu-atomic-sc}。

如果地址未对齐，则会导致异常 \hyperref[regdesc:MCAUSE]{mcause} = 6。

\subsubsection{条件存入字 (SC.W) 指令}
\label{sec:riscvcpu-atomic-sc}

SC.W 指令首先检查存储器锁是否仍然有效，以及地址是否与上一条 LR.W 指令指定的地址相同，只有以上条件同时满足时，SC.W 指令才会对存储器执行写操作，并且在收到存储器发出的写操作完成确认信息后立即释放锁。

如果锁已失效（失效原因见章节 \ref{sec:riscvcpu-atomic-lr}），SC.W 指令将在目标寄存器 rd 中设置失败代码，目前只能设置为 1。

如果地址未对齐，则会导致异常 \hyperref[regdesc:MCAUSE]{mcause} = 6。

\subsubsection{AMO 指令}
\label{sec:riscvcpu-atomic-amo}

AMO (原子性存储器操作) 指令分三步执行：

\begin{enumerate}
    \item 从 rs1 给定的存储器地址读取数据，并将数据保存到目标寄存器 rd。
    \item 将 rd 和 rs2 中的数据进行运算。
    \item 将上面第 2 步得到的结果写入 rs1 给定的存储器地址。
\end{enumerate}

AMO 运算共有 9 种类型：SWAP、ADD、AND、OR、XOR、MAX、MIN、MAXU 和 MINU。

在整个过程中，存储器地址保持锁定状态，其他 hart 无法访问。如果遇到未对齐的地址，则会导致异常 \hyperref[regdesc:MCAUSE]{mcause} = 6。

AMO 操作的读/写访问 (PMP/PMA) 都在第一步中进行检查。若发生错误，则 \hyperref[regdesc:MCAUSE]{mcause} = 7。

\subsection{CLIC}
\label{sec:riscvcpu-clic}

\subsubsection{概述}
\label{sec:riscvcpu-clic-overview}


HP 核心中实现的 CLIC 遵循 \href{https://github.com/riscv/riscv-fast-interrupt/blob/v0.9/src/clic.adoc}{核心本地中断控制器 (CLIC) RISC-V 特权架构扩展 (10/10/2023)}。

HP 核心仅支持 CLIC 操作模式，即 \hyperref[fielddesc:MTVECMODE]{mtvec.MODE} 硬连线为 0x3。因此，RISC-V 基本的中断处理方案和相关的 CSR（如 mie、mip、mideleg、uie 和 uip）不可用。

\subsubsection{CLIC CSRs}
\label{sec:riscvcpu-clic-csrs}

以下机器模式 CSR 与 CLIC 方案中的中断处理相关：

\begin{multicols}{4}
\begin{itemize}
\item \hyperref[regdesc:MSTATUS]{mstatus}
\item \hyperref[regdesc:MTVEC]{mtvec}
\item \hyperref[regdesc:MTVT]{mtvt}
\item \hyperref[regdesc:MSCRATCH]{mscratch}
\item \hyperref[regdesc:MEPC]{mepc}
\item \hyperref[regdesc:MCAUSE]{mcause}
\item \hyperref[regdesc:MNXTI_REG]{mnxti}
\item \hyperref[regdesc:MINTTHRESH]{mintthresh}
\item \hyperref[regdesc:MINTSTATUS]{mintstatus}
\item \hyperref[regdesc:MCLICBASE_REG]{mclicbase}
\item \hyperref[regdesc:MSCRATCHCSW_REG]{mscratchcsw}
\item \hyperref[regdesc:MSCRATCHCSWL_REG]{mscratchcswl}
\end{itemize}
\end{multicols}


在 CLIC 模式下，机器模式的全局中断启用仍由 \hyperref[fielddesc:MSTATUSMIE]{mstatus.MIE} 位决定。


每个中断可以是向量化的或非向量化的，具体取决于 \hyperref[fielddesc:MCLICINTATTR_i_SHV]{clicintattr[i].SHV} 域的值。

对于非向量化中断，HP 核心跳转到 \hyperref[fielddesc:MTVECBASE]{mtvec.BASE} 域中配置的陷阱处理程序基地址。

对于向量化中断，HP 核心跳转到向量表中相应索引对应的 32 位字地址，向量表的基地址在 \hyperref[regdesc:MTVT]{mtvt} 中配置。例如，对于第 i 个机器模式中断，首先加载存储器中 mtvt + 4*i 位置的 32 位地址，然后 HP 核心将加载的数据视为地址并跳转到该地址。

在接受中断后，HP 核心会立刻将中断的指令地址存储在 \hyperref[regdesc:MEPC]{mepc} 中。此外，HP 核心将根据中断的 ID 更新 \hyperref[regdesc:MCAUSE]{mcause} 的值。

\subsection{标准物理存储器保护}
\label{sec:riscvcpu-pmp}
\subsubsection{概述}
\label{sec:riscvcpu-pmp-overview}
CPU 核心包含物理存储器保护 (PMP) 单元，完全符合《RISC-V 指令集手册 v1.10 第二卷“特权架构”》。软件可以使用它为所需的存储器区域设置存储器访问特权（读、写和执行权限）。除了标准 PMP 检查外，CPU 核心还实现了自定义的物理存储器属性 (PMA) 检查器，可根据预定义的属性提供额外的权限检查。

\subsubsection{特性}
\label{sec:riscvcpu-pmp-features}

PMP 单元用于控制对物理存储器的访问，主要特性包括：

\begin{itemize}
    \item 最多支持 16 个 PMP 表项
    \item 最小地址颗粒度为 128 字节
    \item 三种地址匹配模式：OFF、TOR（范围顶部）和 NAPOT（自然对齐的 2 的幂）。不支持 NA4（自然对齐的 4 字节区域）地址匹配模式
    \item 读、写和执行的权限控制
    \item 每个 PMP 表项均支持锁定功能
    \item 最大物理空间地址为 4 GB
\end{itemize}

\subsubsection{功能描述}
\label{sec:riscvcpu-pmp-func-descr}

软件可以设置 PMP 单元的配置和地址寄存器，以保存错误并确保安全执行。配置 PMP CSR 可使能 PMP 单元， 但只能在机器模式下进行配置，一旦 PMP 单元被使能，则将根据 pmpcfg\regindex{n} (\regindex{n}: 0-3) 和 pmpaddr\regindex{n} (\regindex{n}: 0-15) 寄存器（见 \ref{sec:riscvcpu-pmp-reg-summ}）中的配置值，对用户模式下的所有存储器访问进行写入、读取、访问权限检查。

默认情况下，PMP 允许机器模式下的所有存储器访问，而撤销用户模式下的所有访问。这意味着必须通过 pmpcfg 和 pmpaddr 寄存器（见 \ref{sec:riscvcpu-pmp-reg-summ}）设置用户模式可以访问的地址范围和有效权限，以确保访问成功。但是在机器模式下没有此要求，因为机器模式下默认 PMP 允许所有访问。如果在机器模式下也需要 PMP 检查，则软件可以将所需 PMP 表项的锁定位置位来使能权限检查。锁定位一旦置位，就只能通过 CPU 复位被清零。

如果从没有执行权限的存储器区域获取指令，则处理器会生成异常，并且在 \hyperref[regdesc:MCAUSE]{mcause} CSR 中将异常原因设置为指令访问错误。同样，任何没有有效读/写权限的读写访问都将生成异常， \hyperref[regdesc:MCAUSE]{mcause} 会更新为读取存储器访问错误或写入存储器访问错误，并将则存储器访问地址更新到 \hyperref[regdesc:MTVAL]{mtval} CSR 中。

\subsection{自定义物理存储器属性 (PMA) 检查器}
\label{sec:riscvcpu-pma}
\subsubsection{概述}
\label{sec:riscvcpu-pma-overview}
CPU 核心还实现了自定义物理存储器属性检查器 (PMAC)，可根据 CSR 预先定义的存储器类型提供额外的权限检查。请注意，PMA 检查不区分核心的特权模式，即不区分机器和用户模式。


\subsubsection{特性}
\label{sec:riscvcpu-pma-features}

PMAC 支持以下特性：

\begin{itemize}
    \item 最多支持 16 个 PMA 表项
    \item 指定的存储器区域可配置存储器类型
    \item 最小地址颗粒度为 128 字节
    \item 三种地址匹配模式：OFF、TOR（范围顶部）和 NAPOT（自然对齐的 2 的幂）。不支持 NA4（自然对齐的 4 字节区域）地址匹配模式
    \item 读、写和执行的权限控制
    \item 每个 PMA 表项均支持锁定功能
    \item 最大物理空间地址为 4 GB
    \item 指定的存储器区域可配置属性
\end{itemize}

\subsubsection{功能描述}
\label{sec:riscvcpu-pma-func-descr}

软件可以设置 PMAC 单元的配置和地址寄存器，避免因访问无效存储器区域造成的错误。PMAC CSR 只能在机器模式下进行配置。一旦 PMAC 被使能，则将根据 16 个 pma\_cfgX 和 pma\_addrX 寄存器（见 \ref{sec:riscvcpu-pma-reg-summ}）中的配置值对各个模式下的所有存储器访问进行写入、读取、访问权限检查。访问被设置为无效的存储器的表项将导致获取错误或读写错误异常。

PMA 检查相关的的异常生成及错误处理与 PMP 检查类似。如果从配置为 null 或无效的存储器区域提取指令，则会在处理器级别生成异常，并且在 \hyperref[regdesc:MCAUSE]{mcause} CSR 中将异常原因设置为指令访问错误。同样，读写任何 null 或无效的存储器区域都将生成异常，并且 \hyperref[regdesc:MCAUSE]{mcause} 会更新为读取存储器访问错误或写入存储器访问错误。如果发生存储器读写异常，则存储器访问地址会更新到 \hyperref[regdesc:MTVAL]{mtval} CSR 中。对于配置为有效存储器区域的 PMA 检查表项，处理方式与 PMP 检查相同。

如果软件想要关闭对 PMAC 寄存器的写入，可使用表项的锁定位。一旦任何 pma\_cfgX 寄存器中的锁定位被设置，相应的 pma\_cfgX 和 pma\_addrX 寄存器就不能被写入，除非 CPU 复位。

PMAC CSR 中还提供了一个 4 位的域来定义存储器区域的属性。这些位不为 CPU 核心使用，而是基于地址匹配在读写接口上作为边带信号，供 cache 控制器用于其内部操作。

\subsection{调试}
\label{sec:riscvcpu-dbg}

\subsubsection{概述}
\label{sec:riscvcpu-dbg-overview}

本节介绍如何调试在 HP 核心上运行的软件。调试功能由标准 JTAG 管脚提供，并符合 \href{https://riscv.org/wp-content/uploads/2019/03/riscv-debug-release.pdf}{《RISC-V 外部调试支持规范 v0.13.2》}。

HP 核心配备了独立的 JTAG DTM（调试传输模块），用于连接调试模块 (Debug Module)。

图 \ref{fig:riscvcpu-dbg-sys-overview} 显示了外部调试支持的主要组件。

\begin{figure}[H]
\centering
\includegraphics[width=0.8\textwidth,keepaspectratio]{\modulefiles/figures/debug_bigfonts.png}
\caption{调试系统概述}\label{fig:riscvcpu-dbg-sys-overview}
\end{figure}


用户与运行调试器（Debugger，例如 GDB）的调试主机（DEBUG HOST，例如笔记本电脑）进行交互。调试器通过调试转换器（Debug Translator，可能包含硬件驱动，例如 OPENOCD）与调试传输硬件（DEBUG TRANSPORT，例如 ESP-Prog 适配器）进行通信。调试传输硬件通过标准 JTAG 接口将调试主机连接到 HP 核心的调试传输模块 (JTAG DTM)。JTAG DTM 使用调试模块接口 (DMI) 提供对调试模块 (DM) 的访问。

DM 允许调试器暂停 HP 核心。抽象命令提供对 GPR（通用寄存器）的访问。程序缓冲区允许调试器在核心上执行任意代码，从而读取 CPU 核心的其他运行状态。CPU 核心的其他运行状态也可以由其他抽象命令读取。

HP 核心的触发器模块支持 3 个触发器。当满足触发条件时，核心将自发暂停并通知调试模块。

系统总线访问模块可以在不影响任一 HP 核心的情况下访问存储器和外设寄存器。


\subsubsection{功能}
\label{sec:riscvcpu-dbg-features}

HP CPU 核心支持以下基本调试功能：

\begin{itemize}
    \item 向调试器提供有关实现的必要信息
    \item 支持暂停和恢复核心
    \item 核心寄存器（包括 CSR）可以由调试器读取/写入
    \item 可以从复位后执行的第一条指令开始调试核心
    \item 可以通过调试器复位核心
    \item 可以通过软件断点指令暂停核心
    \item 硬件单步调试
    \item 通过程序缓冲区在暂停的核心中执行任意指令。支持 2 个字的程序缓冲区
    \item 支持系统总线的字对齐地址访问
    \item HP 核心支持 3 个硬件触发器（可用作断点/观察点），详情见章节 \ref{sec:riscvcpu-hwt}
\end{itemize}

\subsubsection{功能描述}
\label{sec:riscvcpu-dbg-func-descr}

如前所述，调试机制遵守 \href{https://riscv.org/wp-content/uploads/2019/03/riscv-debug-release.pdf}{《RISC-V 外部调试支持规范 v0.13.2》}。有关功能操作的详细信息，请参阅该规范。

\subsection{硬件触发器}
\label{sec:riscvcpu-hwt}

\subsubsection{概述}
\label{sec:riscvcpu-hwt-overview}

HP 核心实现一个硬件触发器模块，提供调试的断点和监视点功能。该模块遵循 \href{https://riscv.org/wp-content/uploads/2019/03/riscv-debug-release.pdf}{《RISC-V 外部调试支持规范 v0.13.2》}。有关具体功能和操作方式，请参考该规范文档。

\subsubsection{特性}
\label{sec:riscvcpu-hwt-features}

HP 核心硬件触发器模块支持以下特性：

\begin{itemize}
    \item 3 个独立的可配置触发单元，可匹配程序计数器地址或加载/存储访问地址
    \item 可以通过引起断点异常来抢占执行
    \item 可以暂停执行并将控制权转移给调试器
    \item 支持 NAPOT（2 的幂次方对齐区域）地址编码
\end{itemize}

\subsubsection{功能描述}
\label{sec:riscvcpu-hwt-func-descr}

硬件触发器模块提供了 7 个 CSR，见 \hyperref[sec:riscvcpu-hwt-reg-summ]{寄存器列表}。其中，\hyperref[regdesc:TDATA1]{tdata1} 和 \hyperref[regdesc:TDATA2]{tdata2} 是抽象 CSR，也就是说它们是用于访问某个触发单元中的内部寄存器的影子寄存器，一次访问一个触发单元。

要选择特定的触发单元，需要将相应的编号 (0-2) 写入 \hyperref[regdesc:TSELECT]{tselect} CSR。当写入有效数值时，抽象 CSR \hyperref[regdesc:TDATA1]{tdata1} 和 \hyperref[regdesc:TDATA2]{tdata2} 将自动匹配该触发单元的内部寄存器。每个触发单元都有两个内部寄存器，即 \hyperref[regdesc:MCONTROL]{mcontrol} 和 \hyperref[regdesc:MADDRESS]{maddress}，它们分别与 tdata1 和 tdata2 匹配。

向 tselect 写入超过最大编号的数值时会导致该数值被裁剪为最大的编号，此编号可以被读回。这个特性可用于枚举初始化期间或使用调试器时可用的触发器。

由于软件或调试器可能需要知道所选触发器的类型以便正确解读 tdata1 和 tdata2，因此 tdata1 的 4 个位 (31-28) 对所选触发器的类型进行了编码。此域为只读访问属性，并且值始终为 0x2，代匹配类型触发器，因此，可以推断 tdata1 和 tdata2 会通过 mcontrol 和 maddress 被解读。RISC-V 调试规范 v0.13.2 提供了其他可能值的信息，但是该触发模块仅支持 0x2 类型。

一旦选定了触发单元，就可以通过置位 mcontrol CSR (tdata1) 中相应的域并将目标地址写入 maddress CSR (tdata2) 来对该触发单元进行配置。

通过写入 mcontrol 的 action 域，可以将每个触发单元配置为引起断点异常或进入调试模式。该域只能从调试器写入，因此默认情况下，触发器（如果启用）将引起断点异常。

每个触发单元的 mcontrol 都有一个 \hyperref[fielddesc:MCONTROLHIT]{hit} 域。在 CPU 暂停或进入异常后，通过读取该域可以查明是否是触发单元触发了。触发器触发后该域会立即被置位，但在恢复操作之前必须被手动清零，不过不清零不会影响正常执行。

每个触发单元仅支持地址匹配，该地址可以是存储器访问地址，也可以是指令的虚拟地址。通过写入所选触发单元的 maddress (tdata2) CSR，可以指定区域的地址和大小。大于 1 个字节的区域大小通过 NAPOT 编码（见表 \ref{tab:riscvcpu-hwt-table}）指定，并通过置位 mcontrol 中 match 域来使能。注意，根据定义，NAPOT 编码地址的起始地址需与区域大小对齐（即，是区域大小的整数倍），最大支持 128 字节。

\hyperref[regdesc:TCONTROL]{tcontrol} CSR 对所有触发单元都是通用的。在机器模式下，当陷阱处理程序运行时，该寄存器可用于阻止触发器重复引起异常，而且默认会禁用 ISR 内部的断点异常，但是，出于调试目的，可以在进入 ISR 之前手动使能断点异常。如果将触发器配置为进入调试模式，则此 CSR 不相关。

\subsubsection{触发执行流程}
\label{sec:riscvcpu-hwt-suggest-op}

当触发器触发引起 hart 暂停并进入调试模式时 (\hyperref[fielddesc:MCONTROLACTION]{action} = 1)：
\begin{itemize}
    \item \hyperref[regdesc:DPC]{dpc} 在解码阶段被设置为当前程序计数器 (PC)
    \item \hyperref[regdesc:DCSR]{dcsr} 的 cause 域被设置为 2，表示暂停是由于触发器触发引起
    \item 与触发的触发器对应的 \hyperref[fielddesc:MCONTROLHIT]{hit} 域被置位
\end{itemize}

当触发器触发引起 hart 进入陷阱时 (\hyperref[fielddesc:MCONTROLACTION]{action} = 0)：
\begin{itemize}
    \item \hyperref[regdesc:MEPC]{mepc} 在解码阶段被设置为当前 PC
    \item \hyperref[regdesc:MCAUSE]{mcause} 被设置为 3，即断点异常
    \item \hyperref[fielddesc:TCONTROLMPTE]{mpte} 被设置为陷阱发生之前的 \hyperref[fielddesc:TCONTROLMTE]{mte} 的值
    \item \hyperref[fielddesc:TCONTROLMTE]{mte} 被设置为 0
    \item 与触发的触发器对应的 \hyperref[fielddesc:MCONTROLHIT]{hit} 域被置位
\end{itemize}

\textit{说明：如果两个触发器同时触发，一个 action = 0，另一个action = 1，则 hart 会暂停并进入调试模式。}

\subsection{追踪}
\label{sec:riscvcpu-trace}

\subsubsection{概述}
\label{sec:riscvcpu-trace-overview}
为了支持非侵入式软件调试，CPU 核心提供了指令追踪接口，为离线调试提供相关信息。追踪编码器 (Trace Encoder) 模块可通过指令追踪接口获取这些信息，并将其压缩后存储在分配的存储器中。软件解码器可直接从存储器中读取这些信息，无需中断 CPU 核心，便可重新生成 CPU 核心实际执行的程序。


\subsubsection{特性}
\label{sec:riscvcpu-trace-features}

CPU 核心支持指令追踪功能，并按照 RISC-V 处理器追踪规范 v1.0 (RISC-V Processor Trace v1.0) 中的规定向追踪编码器提供以下信息：

\begin{itemize}
    \item 退役指令数量
    \item 异常和中断是否发生及其原因、陷阱值
    \item hart 当前的特权级别
    \item 跳转、分支和返回退役指令的指令类型
    \item 程序计数器更改之前和之后退役指令的地址
    \item 根据 \hyperref[regdesc:MCONTROL]{mcontrol} 中的 \hyperref[regdesc:MCONTROL]{action} 位在触发命中时启用追踪
\end{itemize}


\subsubsection{功能描述}
\label{sec:riscvcpu-trace-func-descr}

HP 核心实现了强制指令 delta 追踪，也称为分支追踪。它的工作原理是通过发送程序获取的 delta 信息来追踪从已知起始地址开始的执行情况。delta 通常由跳转、调用、返回、分支类型的指令、中断以及异常引入。所有此类 delta 以及其他详细信息，如原因和实际指令/地址，都通过高带宽指令追踪接口从 CPU 输出给追踪编码器，追踪编码器对这些信息进行操作，将信息压缩后存入存储器，供解码器离线调试。有关编码的更多信息，请参阅章节 \ref{mod:riscvtracenc} \textit{\nameref{mod:riscvtracenc}}。

CPU 没有任何内部寄存器来控制指令追踪接口，所有相关控制寄存器都在 \ref{mod:riscvtracenc} \textit{\nameref{mod:riscvtracenc}} 模块中。

\subsection{分支预测}
\label{sec:riscvcpu-perf-bp}

\subsubsection{概述}
\label{sec:riscvcpu-perf-bp-overview}
在分支指令执行前，核心会预测分支是否跳转以及跳转的目标地址。然后，基于预测的结果，从相应的 PC 获取指令。在执行指令时，验证预测是否正确，如果预测错误，则刷新流水线，并从正确的 PC 开始获取指令。预测能提高吞吐量，因为核心会根据预测结果继续获取指令，而不是等待分支指令执行完成。下面的章节会详细描述核心使用的预测技术。

\subsubsection{特性}
\label{sec:riscvcpu-perf-bp-features}
\begin{itemize}
    \item 预测条件/无条件分支类型指令的结果和目标地址
    \item 支持分支目标预测，指令包括 BEQ、BNE、BLT、BLTU、BGE、BGEU、C.BEQZ、C.BNEZ、JAL、J、CJ
    \item 支持条件分支预测，指令包括：BEQ、BNE、BLT、BLTU、BGE、BGEU、C.BEQZ、C.BNEZ
    \item 使用 2 位饱和计数器，提高准确性
    \item 使用条件分支的全局历史进行预测，提高性能
    \item 对于无条件分支，目标预测在一个时钟周期内完成，即当预测或推测正确时，0 周期惩罚
\end{itemize}

\subsubsection{功能描述}
\label{sec:riscvcpu-perf-bp-func-descr}

核心使用分支历史表 (BHT) 和分支目标缓冲区 (BTB) 来预测分支指令和跳转指令的行为。

BHT 是 HP CPU 核心中的一个 512x16 位 SRAM 存储器。在预测时，使用虚拟分支历史寄存器 (VBHR) 作为索引。在更新时，分支指令执行结果出来之后，由全局分支历史寄存器 (GBHR) 作为索引。GBHR 包含前 13 条分支指令执行的结果，VBHR 包含前 12 条结果和当前分支指令的预测结果。因此，这两个寄存器都保存了先前执行的分支轨迹。基于 VBHR/GBHR 的 13 位，BHT 中存储的 2 位饱和计数器值分别被读取或写入。

如图 \ref{fig:riscvcpu-bp_sat-scheme} 所示，2 位饱和计数器表示分支是否为：
\begin{itemize}
    \item 弱不跳转 (weakly not taken, 00)
    \item 强不跳转(strongly not taken, 01)
    \item 弱跳转 (weakly taken, 10)
    \item 强跳转 (strongly taken, 11)
\end{itemize}

如果预测计数器处于任意“未跳转”状态，则该分支会被预测为未跳转；同样地，如果预测计数器处于任意“跳转”状态，则会被预测为跳转。当分支的实际执行结果确定后，会根据下面的 2 位饱和计数器状态转换规则来更新对应的 BHT 表项。

BTB 存储 32 位分支目标地址，最多可存储 16 条地址，以条件或无条件分支类型指令 PC 的最后 16 位作为索引。因此，核心根据 BHT 来预测分支类型指令的结果，如果预测为跳转，跳转到 BTB 中的目标地址。对于无条件分支，核心直接跳转到 BTB 中的目标地址，因为这种情况不需要进行预测。如果预测错误，核心在执行过程中，会获得分支类型指令的实际结果和目标地址，同时也会刷新流水线，跳转到正确的地址，并更新 BTB 和 BHT 表项。

图 \ref{fig:riscvcpu-bp_sat-scheme} 详细介绍了 HP 核心中使用的 2 位饱和方案。

\subsection{RAS}
\label{sec:riscvcpu-perf-ras}

\subsubsection{概述}
\label{sec:riscvcpu-perf-ras-overview}
返回地址栈 (RAS) 是核心维护的一个栈，用于推送跳转和链接 (JAL/JALR) 类型指令的返回地址。因此，核心可以在 RET 指令被预解码时立即跳转到返回地址，从而提高函数调用返回的性能。

\subsubsection{特性}
\label{sec:riscvcpu-perf-ras-features}

\begin{itemize}
    \item 避免函数调用返回时流水线停顿，提高核心性能
    \item 最多可存储四级函数调用嵌套
\end{itemize}


\subsubsection{功能描述}
\label{sec:riscvcpu-perf-ras-func-descr}
当核心预解码 JAL/JALR/C.JALR 指令时，下一条指令的 32 位 PC 会被推送到 RAS。

当核心预解码返回类型指令 (JALR, C.JR, C.JALR) 时，地址从 RAS 中弹出。然后，核心开始从弹出的地址获取指令。当核心执行返回指令时，如果弹出的地址与链接寄存器中存储的地址不匹配，则刷新流水线并从链接寄存器中的地址开始获取指令。RAS 最多可以存储四个返回地址，即支持四级函数调用嵌套。

RAS 预测取决于以下条件：

\begin{longtable}{ | c | c | c | }
\hline\rowcolor{lightgray}
目标寄存器 (Rd) & 源寄存器 (Rs) & RAS 操作 \\ \hline
\endfirsthead
\hline
rd != x1 & rs1 != x1 & 无操作 \\ \hline
rd != x1 & rs1 == x1 & 弹出 \\ \hline
rd == x1 & rs1 != x1 & 推入 \\ \hline
rd == x1 & rs1 == x1 & 推入并弹出 \\ \hline
\end{longtable}

\subsection{总线错误响应}
\label{sec:riscvcpu-bus-err-resp}
\subsubsection{概述}
\label{sec:riscvcpu-buserr-overview}

在 HP 核心中，加载/存储总线错误被处理为异步异常。因此，在检测到此类异常前，CPU 可能已经执行了引起异常的指令之后的指令。

在处理同步异常时，标准 CSR \hyperref[regdesc:MCAUSE]{mcause}、\hyperref[regdesc:MEPC]{mepc} 和 \hyperref[regdesc:MTVAL]{mtval} 提供的上下文足够。但是，对于异步异常来说，这些信息并不够。因此，需要额外的 CSR 来给软件提供程序所需的必要上下文，来处理和恢复遇到异步总线错误异常的程序。

下一小节将介绍这些总线错误 CSR 以及获取总线错误异常上下文的推荐方法。

\subsubsection{功能描述}
\label{sec:riscvcpu-buserr-func-descr}

有两种方法来处理加载/存储总线错误：

\begin{itemize}
    \item 同步总线错误模式： \\
    当启用此模式时（通过设置 \hyperref[fielddesc:MHINT_SBE]{mhint.SBE} 位），总线错误异常可以像其他同步异常一样进行处理。因此，执行会在发生总线错误的 PC 处触发异常，而且 \hyperref[regdesc:MEPC]{mepc} 持有相同的 PC。\\
    启用此模式的缺点是增加了所有加载/存储指令的延迟。
    \item 异步总线错误模式（默认）：
    \begin{itemize}
    \item 为了在异步模式下处理总线错误，建议（但不是必须）在启动时清除 \hyperref[fielddesc:MEXSTATUS_NMFT]{mexstatus.NMFT} 位。\\
    \item 在异步模式下，当（一个或多个）加载/存储指令导致总线错误时，CPU 会进入陷阱处理程序，MCAUSE 值会更新为 5 或 7（取决于导致错误的操作类型），\hyperref[regdesc:MEPC]{mepc} 保存本应在该加载/存储指令之后执行的指令的 PC。\\
    \item 进入原因 5 或 7 的异常处理程序时，必须检查 \hyperref[fielddesc:MEXSTATUS_BUS_ERR]{mexstatus.BUS\_ERR} 位，确认是否是由于加载/存储总线错误造成的异常。如果是，则必须清除此位进行确认。\\
    \item 然后，必须检查位 \hyperref[fielddesc:LDPC0_VLD]{ldpc0.vld}、\hyperref[fielddesc:LDPC1_VLD]{ldpc1.vld}、\hyperref[fielddesc:STPC0_VLD]{stpc0.vld}、\hyperref[fielddesc:STPC1_VLD]{stpc1.vld} 和 \hyperref[fielddesc:STPC2_VLD]{stpc2.vld} 是否有效。如果有效，则必须清除这些位，并从这些 CSR 中读取相应的地址域（\hyperref[fielddesc:LDPC0_ADDR]{ldpc0.addr}、\hyperref[fielddesc:LDPC1_ADDR]{ldpc1.addr}、\hyperref[fielddesc:STPC0_ADDR]{stpc0.addr}、\hyperref[fielddesc:STPC1_ADDR]{stpc1.addr} 和 \hyperref[fielddesc:STPC2_ADDR]{stpc2.addr}），以获取这些故障加载/存储指令执行时的程序计数器。\hyperref[regdesc:LDTVAL0]{ldtval0}、\hyperref[regdesc:LDTVAL1]{ldtval1}、\hyperref[regdesc:STTVAL0]{sttval0}、\hyperref[regdesc:STTVAL1]{sttval1} 和 \hyperref[regdesc:STTVAL2]{sttval2} 保存了加载/存储指令相对应的存储器访问地址。注意，序号较低的 CSR 记录了最新的总线错误，序号较高的 CSR 保存了更早的总线错误。\\
    \item 注意，即使原因是 5 或 7，加载和存储总线错误 CSR 都必须进行检查，因为有时有多个加载/存储指令（或未对齐访问被拆分为多个加载/存储操作）已执行并正在进行中。在首次遇到总线错误之前进入陷阱处理程序时，CPU 会完成所有挂起的存储器操作，并记录它们引起的任何总线错误。因此，检查所有 CSR 非常重要，因为这可以捕获和处理所有此类异常。\\
    \item 注意，跟同步总线错误异常相比，异步总线错误异常发生后，恢复正常的程序执行更加复杂，因为程序可能执行了问题指令之后的很多指令。
    \end{itemize}
\end{itemize}

\subsection{专用 IO}
\label{sec:riscvcpu-fastio}
\subsubsection{概述}
\label{sec:riscvcpu-fastio-overview}

GPIO 通常为 APB 外设，这意味着对输出的更改和对输入的读取，可能会卡在写缓冲区或其他传输之后，并且速度通常较慢，因为 APB 总线的运行速度通常低于 CPU。
作为替代方案，CPU 核心实现了 I/O 处理器特定的 CPU 寄存器 (CSR)，直接连接到 GPIO 矩阵或 IO 管脚。这些寄存器可以通过单条指令访问，从而能够快速控制 IO 管脚。

\subsubsection{特性}
\label{sec:riscvcpu-fastio-features}
 \begin{itemize}
    \item 8 个直接映射到 GPIO 的专用 IO
    \item 驱动输出端口无延迟
    \item 读取输入值有两个 CPU 周期的延迟
\end{itemize}

\subsubsection{功能描述}
\label{sec:riscvcpu-fastio-func-descr}

CPU 核心有一组 8 个输入和输出（管脚值 + 管脚输出使能），这些输入和输出端口直接连接到 GPIO 矩阵，可通过 GPIO 矩阵映射到最上层的管脚。更多相关信息请参考 \ref{mod:iomuxgpio} \textit{\nameref{mod:iomuxgpio}}。

CPU 自定义了 3 个 CSR：
\begin{itemize}
    \item GPIO\_IN 只读，表示输入值。
    \item GPIO\_OUT 可读可写，表示 GPIO 输出值。
    \item GPIO\_OEN 可读可写，表示 GPIO 输出使能状态，控制管脚方向，拉高表示管脚配置为输出模式，拉低表示配置为输入模式。
\end{itemize}

\subsection{运行暂停 (RunStall)}
\label{sec:riscvcpu-custom-runstall}
\subsubsection{概述}
\label{sec:riscvcpu-custom-runstall-overview}

运行暂停 (RunStall) 信号允许外部代理暂停处理器，同时关闭大部分时钟树，以节省运行功耗。

RunStall 输入信号允许外部逻辑暂停处理器的内部流水线。RunStall 输入信号激活后，流水线中正在进行的所有指令将完成运行。流水线成功暂停后，核心将输出握手信号 corestalled，表示成功暂停 CPU 流水线。


\subsubsection{功能描述}
\label{sec:riscvcpu-custom-runstall-feature-func-descr}

RunStall 输入信号可用于以下两种场景：
\begin{itemize}
    \item 在复位后初始化指令和数据 RAM。如果在复位期间激活RunStall，并在复位移除后仍然保持激活状态，那么处理器将处于暂停状态。可以初始化指令和数据 RAM，在初始化完成后，可以释放 RunStall，允许处理器开始执行代码。这种机制可以让处理器的复位向量指向本地指令 RAM中的地址，并且复位代码可以在处理器开始执行代码之前，由外部代理写入指令 RAM。
    \item 冻结处理器的内部流水线。激活 RunStall 减少了处理器的活动功耗，也无需使用中断和 WAITI 选项。然而，使用 RunStall 信号所节省的功耗没有 WAITI 节省得那么多，因为 RunStall 逻辑需要处理器的某些部分在流水线暂停时，仍然保持活跃状态。
\end{itemize}

图 \ref{fig:riscvcpu-runstall} 显示了 RunStall 和 CoreStalled 信号的时序图以及 CPU 核心状态的切换。

\caption{使用 RunStall 时 CPU 核心状态的切换}\label{fig:riscvcpu-runstall}
\end{figure}

\label{sec:riscvcpu-custom-runstall-reg-summ}

\subsection{核心锁定}
\label{sec:riscvcpu-custom-lockup-features}
\subsubsection{概述}
\label{sec:riscvcpu-custom-lockup-overview}
目前，RISC-V 编程模型中没有定义异常使能寄存器。而且，从异常服务程序返回到正常程序流时，MCAUSE 寄存器中的异常向量 ID 未被清除。因此，软件开发者无法仅通过编程模型中定义的寄存器，轻松判断 CPU 当前是在处理中断异常，还是在执行正常的程序流程。为了解决这一问题，CPU 实现了一个自定义的 LOCKUP 域，可以通过自定义 CSR 读取。

\subsubsection{功能描述}
\label{sec:riscvcpu-custom-lockup-feature-func-descr}
当 CPU 响应异常时，会通过 MEXSTATUS 域中的 EXPT\_VLD 确定当前是否正在运行异常服务程序。如果 CPU 正在处理异常，并且从异常服务程序返回之前又触发了新的异常，CPU 将被锁定。当 CPU 被锁定时：

\begin{itemize}
    \item 锁定信号会被设置为高，来通知 SoC CPU 处于锁定状态
    \item CPU 停止获取和执行指令
    \item PC 停留在触发锁定的指令上
    \item MEXSTATUS 寄存器中的 LOCKUP 域置为 1
\end{itemize}

在锁定状态下，调试器仍然可以介入并执行调试命令。建议在检测到锁定状态时重置核心。

锁定规则有一些例外，当 ECALL 或 EBREAK 引起的异常嵌套在任何异常类型中时，不会触发 CPU 锁定。
