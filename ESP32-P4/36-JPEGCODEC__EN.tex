\chapter{JPEG Codec}
\label{mod:jpegcodec}
\chipname{}'s JPEG codec is an image codec, which is based on the JPEG baseline standard, for compressing (encoding) and decompressing (decoding) images to reduce the bandwidth required to transmit images or the space required to store images, making it possible to process large-resolution images. The image encoding process mainly includes discrete cosine transform (DCT), zigzag scanning, quantization, and Huffman coding. The image decoding process, which is the inverse operation of the encoding process, mainly includes Huffman decoding, inverse quantization, inverse zigzag scanning, and inverse discrete cosine transform (IDCT).

\section{Terminology}
\begin{longtable}[c]{ R{4cm} p{12cm} }
\textbf{MCU} & Minimum coded unit. The basic unit of codec processing \\
\textbf{baseline} & A particular sequential DCT-based encoding and decoding process specified in the standard, which is required for all DCT-based decoding processes \\
\textbf{image} & A set of two-dimensional arrays of integer data \\
\textbf{component} & One of the two-dimensional arrays that comprise an image \\
\textbf{data unit} & An 8 × 8 block of pixels of one component \\
\textbf{marker} & A two-byte code in which the first byte is hexadecimal FF (0xFF) and the second byte is a value between 1 and hexadecimal FE (0xFE). It is served to identify the various structural parts of the compressed data formats \\
\textbf{restart marker} & Used to isolate coded data segments and for application-specific uses, such as decoder re-synchronization and error recovery \\
\textbf{restart interval} & The integer number of MCUs processed between two restart markers \\
\textbf{scan} & A single pass through the data for one or more of the components in an image \\
\end{longtable}

\section{Introduction}
The JPEG codec is based on the JPEG baseline standard, which specifies that an encoding process is completed through one or more scans. An scan contains one or more MCUs. MCU is the sequence of data units defined by the sampling factors of the image components in the scan, which means MCU of different image formats contains different sequence of data units, as shown in Table \ref{tab:data_unit_of_mcu}.


\begin{ThreePartTable}
\begin{longtable}{ | l | l | l | l |}
\caption{The Number of Data Units in the MCU of Different Image Format} \label{tab:data_unit_of_mcu}
\\\hline

\rowcolor{lightgray}
 & \multicolumn{3}{c}{\textbf{MCU\tnote{1}}} \\\cline{2-4}

\rowcolor{lightgray}
\multirow{-2}*{\textbf{Image Format}}  & \textbf{Luminance Component Y }  &  \textbf{Chrominance Component U}  &  \textbf{Chrominance Component V}\\ \hline

YUV444  &  1 x 1 &  1 x 1 &  1 x 1  \\ \hline
YUV422  &  2 x 1 &  1 x 1 &  1 x 1  \\ \hline

GRAY    &  1 x 1 &  none  &  none  \\ \hline

\item[1] The MCU of each image format is represented by the number of data units in the horizontal direction x the number of data units in the vertical direction of each component.
\end{tablenotes}

\end{ThreePartTable}

At a time, the JPEG codec can only be configured as an encoder or a decoder, that is, it cannot work as an encoder and a decoder at the same time.
\begin{itemize}
    \item When used as an encoder, it obtains the image to be compressed through 2D DMA and sends the encoded bitstream to the 2D DMA.
    \item When used as a decoder, it obtains the coded bitstream through 2D DMA and sends the decoded image to 2D DMA.
\end{itemize}

\section{Features}
\label{sec:jpegcodec-features}

When used as an encoder, the JPEG codec has the following features:
\begin{itemize}
    \item integrated discrete cosine transform algorithm
    \item integrated canonical Huffman coding
    \item RGB888, RGB565, YUV422 and GRAY as original input image formats
    \item conversion of RGB888 and RGB565 into YUV444, YUV422 or YUV420 (the only formats supported by impression) for image compression
    \item 4 x configurable quantization coefficient tables with 8-bit or 16-bit precision
    \item performance:
    \begin{itemize}
        \item still image compression: up to 4K resolution
        \item dynamic image compression: up to 1080P@40fps,720P@70fps (excluding header encoding time)
    \end{itemize}
    \item automatically added stuffed zero byte
    \item automatically added EOI marker
\end{itemize}

When used as a decoder, the JPEG codec has the following features:
\begin{itemize}
    \item integrated inverse discrete cosine transform algorithm
    \item integrated  Huffman decoding
    \item supported image formats for compressed bitstream decoding: YUV444, YUV422, YUV420, and GRAY.
    \item 4 x configurable quantization coefficient tables with 8-bit or 16-bit precision
    \item 2 x DC and 2 x AC Huffman tables
    \item supports image decoding of any resolution. However, the resolution of output decoded image differs from the format of the input image:
    \begin{itemize}
        \item YUV444, GRAY: both the horizontal and vertical resolutions of the output decoded image are multiples of 8, i.e., 150 × 150 images with an output resolution of 152 × 152
        \item YUV422: the horizontal resolution of the output decoded image is the multiples of 16 and the vertical resolution is multiples of 8, i.e., 150 × 150 images with an output resolution of 160 × 152
        \item YUV420: both the horizontal and vertical resolutions of the output decoded image are multiples of 16, i.e., 150 × 150 images with an output resolution of 160 × 160
    \end{itemize}
    \item performance:
    \begin{itemize}
        \item still image decoding: up to 4K resolution
        \item dynamic image decoding: up to 1080P@40fps,720P@70fps (excluding header parsing time)
    \end{itemize}
\end{itemize}

\section{Architectural Overview}
\label{sec:jpegcodec-arch-overview}

When the JPEG codec is configured as an encoder, its architecture is shown in Figure \ref{fig:jpeg-enc-arc}.
\begin{figure}[H]
    \centering
    \includegraphics[width=1\textwidth]{\modulefiles/figures/jpeg-enc-arc}
    \caption{JPEG encoder architecture}
    \label{fig:jpeg-enc-arc}
\end{figure}

\begin{itemize}
    \item \textbf{rgb2ycrcb} implements the color space conversion, and converts the original image into an image format supported by the codec for compressing.
    \item \textbf{dct\_idct} implements two-dimensional DCT transformation of data units.
    \item \textbf{zigzag} performs zigzag scanning and sorting on the DCT transformed data units.
    \item \textbf{quant\_table} stores the quantization coefficient table delivered by the software.
    \item \textbf{quantizer} quantifies the zigzag sorted data units based on the quantization coefficient table stored in \textbf{quant\_table}.
    \item \textbf{huffman\_encoder} performs Huffman encoding on the quantized result of the data units.
    \item \textbf{packer} packs the Huffman-encoded data into groups of 32 bits, and outputs the final encoded bitstream.
\end{itemize}

When the JPEG codec is configured as a decoder, its architecture is shown in figure \ref{fig:jpeg-dec-arc}.
\begin{figure}[H]
    \centering
    \includegraphics[width=1.0\textwidth]{\modulefiles/figures/jpeg-dec-arc}
    \caption{JPEG decoder architecture}
    \label{fig:jpeg-dec-arc}
\end{figure}

\begin{itemize}
    \item \textbf{header\_dec} implements bitstream extraction and marker SOS and EOI parsing, and controls the decoding process.
    \item \textbf{huffman\_decoder} stores the Huffman table delivered by the software, and implements Huffman decoding of the bitstream.
    \item \textbf{quant\_table} stores the quantization coefficient table delivered by the software.
    \item \textbf{dequantizer} dequantizes the Huffman decoded data based on the quantization coefficient table stored in \textbf{quant\_table}.
    \item \textbf{dezigzag} implements reverse zigzag scanning and sorting on the dequantized data units, and combines them into dequantized data units
    \item \textbf{dct\_idct} implements the two-dimensional inverse DCT transformation of the dequantized data units, and outputs the final decoded data units.
\end{itemize}

Since the JPEG codec cannot be used as an encoder and decoder at the same time, the encoder and decoder share the \textbf{quant\_table} and \textbf{dct\_idct} modules to save hardware resources.

\subsubsection{Pause}
\label{sec:jpegcodec-enc-pause}

The JPEG encoder can be paused during the operation for other higher priority tasks. This feature is controlled by register field \hyperref[fielddesc:JPEGPAUSEEN]{JPEG\_PAUSE\_EN}.

\begin{itemize}
    \item Set \hyperref[fielddesc:JPEGPAUSEEN]{JPEG\_PAUSE\_EN} to 1, the JPEG encoder will no longer output the bitstream to the 2D DMA RX channel.
    \item Set \hyperref[fielddesc:JPEGPAUSEEN]{JPEG\_PAUSE\_EN} back to 0, the JPEG encoder will be resumed, and output the bitstream normally.
\end{itemize}

\subsubsection{Color Space Conversion}
\label{sec:jpegcodec-enc-csc}

According to the standard, the image formats supported by the JPEG encoder are YUV444, YUV422, YUV420, and GRAY. However, \chipname{}'s JPEG encoder has integrated a color space conversion function to convert RGB888 and RGB565 formats into the above-mentioned image formats supported by the encoder before encoding. In this way, the range of supported image formats is expanded.

The color space conversion formulas for converting RGB to YUV are:
\begin{center}
$Y = (0.299 * R) + (0.587 * G) + (0.114 * B)$ \\
$U = (-0.1687 * R) + (-0.3313 * G) + (0.5 * B) + 128$ \\
$V = (0.5 * R) + (-0.4187 * G) + (-0.0813 * B) + 128$ \\
\end{center}

\subsubsection{Configurable Quantization Coefficient Table}
\label{sec:jpegcodec-enc-qnr}

\chipname{} JEPG encoder has implemented four 8 x 8 quantization coefficient tables in its hardware, which allows 64 software-configurable coefficient values per table.

The precision of each quantization table is configurable via \hyperref[fielddesc:JPEGQNRPRECISION]{JPEG\_QNR\_PRECISION}:
\begin{itemize}
    \item 0: low 8-bit effective
    \item 1: low 16-bit effective
\end{itemize}

The coefficient values for each table can be configured in FIFO mode or non-FIFO mode depending on the value of \hyperref[fielddesc:JPEGQNRFIFOEN]{JPEG\_QNR\_FIFO\_EN}:
\begin{itemize}
    \item 0: non-FIFO mode, in which each coefficient is written to a specific address
    \item 1: FIFO mode, in which all coefficients are written to the same address
\end{itemize}

In different modes, the configuration of the quantization coefficient table is different. Note, \regindex{n} (\regindex{n} = 0, 1, 2, 3) indicates the number of each quantization coefficient table:
\begin{itemize}
    \item non-FIFO mode: each quantization coefficient in a specific position of a quantization coefficient table \regindex{n} must be written to a specific address, which is (JPEG codec base address + \regindex{n} * 0x100 + \regindex{quantization coefficient position number} * 0x4), in which the \regindex{quantization coefficient position number} can be 0, 1, 2, ..., 63 (see Table \ref{tab:example_qnr}). In this mode, since each coefficient is written to a specific address, configuring the 64 quantization coefficients in sequence is not required.
    \item FIFO mode: all 64 quantization coefficients of the quantization table \regindex{n} are written to the same address, which is \hyperref[regdesc:JPEGT0QNRREG]{JPEG\_T\regindex{n}QNR\_REG}. In this mode, since all coefficients are written to the same address, configuring the 64 quantization coefficients one by one in sequence is required. Afterwards, the hardware will automatically store these 64 quantization coefficients in sequence as shown in Table \ref{tab:example_qnr}.
\end{itemize}

If there are less than 4 quantization tables, the corresponding quantization table will not be written.

To use a specific quantization coefficient table \regindex{n} for the luminance component or the chrominance component, just set the corresponding register fields to \regindex{n}:
\begin{itemize}
    \item luminance component: \hyperref[fielddesc:JPEGLQNRTBLSEL]{JPEG\_LQNR\_TBL\_SEL}
    \item chrominance component: \hyperref[fielddesc:JPEGCQNRTBLSEL]{JPEG\_CQNR\_TBL\_SEL}
\end{itemize}

\subsubsection{Stuffed Zero Byte}
\label{sec:jpegcodec-enc-zero_byte}

As required in the JPEG baseline standard to avoid any marker within a Huffman-coded segment, any 0xFF byte generated by the Huffman coding should be followed by a ``stuffed" zero byte.

JPEG encoder has implemented a mechanism to automatically add this ``stuffed" 0x00 byte after each 0xFF byte in the Huffman-coded segment of bitstream.

Users can configure whether to add 0x00 byte after each 0xFF byte via \hyperref[fielddesc:JPEGFFCHECKEN]{JPEG\_FF\_CHECK\_EN}:
\begin{itemize}
    \item 0: not add
    \item 1: add (by default)
\end{itemize}

\subsubsection{EOI Marker}
\label{sec:jpegcodec-enc-eoi}

The end of Image (EOI) marker is used to mark the end of the bitstream of a compressed image. The EOI marker code is 0xFFD9.

JPEG encoder has implemented a mechanism to automatically add the EOI marker at the end of the bitstream.

Users can configure whether to add EOF marker via \hyperref[fielddesc:JPEGTAILEREN]{JPEG\_TAILER\_EN}:
\begin{itemize}
    \item 0: not add
    \item 1: add
\end{itemize}

\subsection{JPEG Decoder}
\label{sec:jpegcodec-dec-mode}

The information in the bitstream must be parsed by the software, then sent to the hardware through the APB configuration bus. Note that the hardware does not support parsing any marker segments, except the SOS, RST and EOI marker segments.

\subsubsection{Multiple Chrominance Components}
\label{sec:jpegcodec-dec-color_components}

The frame header contains information about the number of chrominance components of the image, the ID of each component, and the sampling factors in the horizontal and vertical directions of each component.

JPEG decoder's software must parse such information and pass it to respective registers \hyperref[fielddesc:JPEGCOMPONENTNUM]{JPEG\_COMPONENT\_NUM}, \hyperref[regdesc:JPEGC0REG]{JPEG\_C\regindex{n}\_ID}, \hyperref[fielddesc:JPEGC0xFACTOR]{JPEG\_C\regindex{n}\_X\_FACTOR}, and \hyperref[fielddesc:JPEGC0YFACTOR]{JPEG\_C\regindex{n}\_Y\_FACTOR}. Here, \regindex{n} can be 0, 1, 2, indicating that up to 3 chrominance components are supported by hardware. Note that if less than 3 components are involved, just do not configure the rest of the registers.

According to the value of register \hyperref[regdesc:JPEGC0REG]{JPEG\_C\regindex{n}\_ID} of each chrominance component, the hardware will check whether it is consistent with the ID in the scan header information. The sampling factor indicates the number of data units of this component that an MCU contains in both the horizontal and vertical directions. \subsubsection{Parsing RST Marker}
\label{sec:jpegcodec-dec-rst_marker}

The bitstream may have a restart (RST) marker in the scan segment. The RST marker code is 0xFFD\regindex{x} (\regindex{x} = 0, 1, ..., 7).

JPEG decoder's hardware can parse RST markers and check whether the actual number of MCUs between two RST markers is the same as the restart interval defined in the DRI segment of the Tables/Miscellaneous segment.

Firstly, the software parses the DRI segment and pass the defined restart interval to register \hyperref[fielddesc:JPEGRESTARTINTERVAL]{JPEG\_RESTART\_INTERVAL}. Then, the hardware will check whether the subsequent bitstream has an RST marker every time \hyperref[fielddesc:JPEGRESTARTINTERVAL]{JPEG\_RESTART\_INTERVAL} MCUs are decoded.

Configuring \hyperref[fielddesc:JPEGRESTARTINTERVAL]{JPEG\_RESTART\_INTERVAL} to 0 indicates the bitstream does not contain any RST marker. In this case, the hardware will not check RST markers.

\subsubsection{Configurable Quantization Coefficient Table}
\label{sec:jpegcodec-dec-qnr}

\chipname{} JPEG decoder also supports 4 configurable quantization coefficient tables. Since the JPEG codec cannot be used as an encoder and decoder at the same time, decoder can share the same four configurable quantization coefficient tables with the encoder to save hardware resources.

Therefore, the configuration of decoder's quantization coefficient tables is similar to that of the encoder (see Section \ref{sec:jpegcodec-enc-qnr}) with some exceptions described below:
\begin{itemize}
    \item The software needs to parse the DQT segment in the Tables/Miscellaneous segment to obtain the quantization coefficient tables of the decoder. The recommended quantization coefficient tables are the same as the encoder shown in Table \ref{tab:luminance_qnr} and Table \ref{tab:chrominance_qnr}.
    \item The precision of the quantization table\regindex{n} is configured via higher 4 bits of register \hyperref[regdesc:JPEGDQTINFOREG]{JPEG\_T\regindex{n}\_DQT\_INFO}:
    \begin{itemize}
        \item 0: low 8-bit effective
        \item 1: low 16-bit effective
    \end{itemize}
    The precision is also obtained from the DQT segment in the Tables/Miscellaneous segment.
    \item For each chrominance component, a different quantization coefficient table can be selected. The lower 4 bits of register \hyperref[regdesc:JPEGDQTINFOREG]{JPEG\_T\regindex{n}\_DQT\_INFO} specify the ID of each quantization coefficient table, and each chrominance component selects the quantization coefficient table whose ID is the value of register \hyperref[fielddesc:JPEGC0DQTTBLSEL]{JPEG\_C\regindex{n}\_DQT\_TBL\_SEL}. The ID of each quantization coefficient table is also obtained from the DQT segment in the Tables/Miscellaneous segment.
\end{itemize}

\subsubsection{Configurable Huffman Table}
\label{sec:jpegcodec-dec-huffman}

\chipname{} supports up to 2 DC Huffman tables and 2 AC Huffman tables. Users can configure as many Huffman tables as needed within the limits.

Each Huffman table can be established with three types of information, which can be obtained from the DHT segment in the Tables/Miscellaneous segment:
\begin{itemize}
    \item the number of codeword with a codeword length of 1 to 16 bits,
    \item the symbols corresponding to the decoded codeword
    \begin{itemize}
        \item DC table: up to 16 symbols
        \item AC table: up to 256 symbols
    \end{itemize}
    \item the minimum codeword with a codeword length of 1 to 16 bits.
\end{itemize}

Both of the DC and AC Huffman tables can be configured in a FIFO fashion or a non-FIFO fashion depending on the value of \hyperref[fielddesc:JPEGDHTFIFOEN]{JPEG\_DHT\_FIFO\_EN}:
\begin{itemize}
    \item 0: non-FIFO mode
    \item 1: FIFO mode
\end{itemize}

In different modes, the configuration of the Huffman table is different. Here is the configuration of DC tables\regindex{n} (\regindex{n} = 0, 1):
\begin{itemize}
    \item non-FIFO mode: each piece in different type of information used to establish the DC\regindex{n} Huffman table must be written to a specific address.
    \begin{itemize}
        \item the number of codeword with a codeword length of 1 to 16 bits: JPEG codec base address + 0x500 + \regindex{n} * 0x80 + (\regindex{codeword length} - 1) * 0x4, in which \regindex{codeword length} = 1, 2, ..., 16.
        \item the symbols corresponding to the decoded codeword: JPEG codec base address + 0x600 + \regindex{n} * 0x40 + \regindex{symbol number} * 0x4, in which \regindex{symbol number} = 0, 1, ..., 15.
        \item the minimum codeword with a codeword length of 1 to 16 bits: JPEG codec base address + 0xF00 + \regindex{n} * 0x80 + (\regindex{codeword length} - 1) * 0x4, in which \regindex{codeword length} = 1, 2, ..., 16.
    \end{itemize}
    In this mode, since each piece of different type of information is written to a specific address, configuring each piece of information in sequence is not required.
    \item FIFO mode: all pieces of the same type of information used to establish the DC\regindex{n} Huffman table must be written to the same address, respectively.
    \begin{itemize}
        \item the number of codeword with a codeword length of 1 to 16 bits: \hyperref[regdesc:JPEGDHTTOTLENDC0REG]{JPEG\_DHT\_TOTLEN\_DC\regindex{n}\_REG}.
        \item all symbols corresponding to the decoded codeword (DC table has up to 16 symbols): \hyperref[regdesc:JPEGDHTVALDC0REG]{JPEG\_DHT\_VAL\_DC\regindex{n}\_REG}.
        \item the minimum codeword with a codeword length of 1 to 16 bits:  \hyperref[regdesc:JPEGDHTCODEMINDC0REG]{JPEG\_DHT\_CODEMIN\_DC\regindex{n}\_REG}.
    \end{itemize}
    In this mode, since all pieces of the same type of information are written to the same address, configuring each piece of same type of information one by one in sequence is required. Afterwards, the hardware will automatically store such information in sequence.
\end{itemize}

The configuration of AC\regindex{n} (\regindex{n} = 0, 1) tables is very similar to that of the DC\regindex{n} (\regindex{n} = 0, 1) tables. Here below is the configuration of AC\regindex{n} (\regindex{n} = 0, 1) tables:

\begin{itemize}
    \item non-FIFO mode: each piece in different type of information used to establish the AC\regindex{n} Huffman table must be written to a specific address.
    \begin{itemize}
        \item the number of codeword with a codeword length of 1 to 16 bits: JPEG codec base address + 0x540 + \regindex{n} * 0x80 + (\regindex{codeword length} - 1) * 0x4, in which \regindex{codeword length} = 1, 2, ..., 16.
        \item the symbols corresponding to the decoded codeword: JPEG codec base address + 0x680 + \regindex{n} * 0x400 + \regindex{symbol number} * 0x4, in which \regindex{symbol number} = 0, 1, ..., 256.
        \item the minimum codeword with a codeword length of 1 to 16 bits: JPEG codec base address + 0xF40 + \regindex{n} * 0x80 + (\regindex{codeword length} - 1) * 0x4, in which \regindex{codeword length} = 1, 2, ..., 16.
    \end{itemize}
    In this mode, since each piece of different type of information is written to a specific address, configuring each piece of information in sequence is not required.
    \item FIFO mode: all pieces of the same type of information used to establish the AC\regindex{n} Huffman table must be written to the same address, respectively.
    \begin{itemize}
        \item the number of codeword with a codeword length of 1 to 16 bits: \hyperref[regdesc:JPEGDHTTOTLENAC0REG]{JPEG\_DHT\_TOTLEN\_AC\regindex{n}\_REG}.
        \item all symbols corresponding to the decoded codeword (AC table has up to 256 symbols): \hyperref[regdesc:JPEGDHTVALAC0REG]{JPEG\_DHT\_VAL\_AC\regindex{n}\_REG}.
        \item the minimum codeword with a codeword length of 1 to 16 bits:  \hyperref[regdesc:JPEGDHTCODEMINAC0REG]{JPEG\_DHT\_CODEMIN\_AC\regindex{n}\_REG}.
    \end{itemize}
    In this mode, since all pieces of the same type of information are written to the same address, configuring each piece of same type of information one by one in sequence is required. Afterwards, the hardware will automatically store such information in sequence.
\end{itemize}

For the minimum codeword with a codeword length of 1 to 16 bits, it should be moved left to the high bit of the 16-bit configuration data. If a codeword length does not exist, the configuration value is 0xFFFF. For example, if a minimum codeword is 0x7C, the 16-bit configuration data should be 0x7C00.

The JPEG baseline standard recommended DC and AC Huffman tables can be found in \href{https://www.itu.int/rec/T-REC-T.81}{ITU T.81 Recommendation}.

The software can obtain the IDs of DC\regindex{n} and AC\regindex{n} Huffman tables by parsing the DHT segment in the Tables/Miscellaneous segment, then pass these ID to the corresponding register \hyperref[regdesc:JPEGDHTINFOREG]{JPEG\_DC\regindex{n}\_DHT\_ID} and \hyperref[regdesc:JPEGDHTINFOREG]{JPEG\_AC\regindex{n}\_DHT\_ID}.

The hardware will automatically use the DC\regindex{n} and AC\regindex{n} Huffman tables with the same ID from \hyperref[regdesc:JPEGDHTINFOREG]{JPEG\_DC\regindex{n}\_DHT\_ID} and \hyperref[regdesc:JPEGDHTINFOREG]{JPEG\_AC\regindex{n}\_DHT\_ID}.

\subsubsection{Timeout Detection}

Additionally, timeout detection is supported when the JPEG codec is working as a decoder. The hardware triggers the timeout interrupt JPEG\_DECODE\_TIMEOUT\_INT:
\begin{itemize}
    \item when the decoder stops reading the bitstream for more than ($ 2 ^ {\hyperref[fielddesc:JPEGDECODETIMEOUTTHRES]{JPEG\_DECODE\_TIMEOUT\_THRES}} - 1 $) clock cycles,
    \item or when the number of clock cycles consumed for decoding an image exceeds the value of ($ 2 ^ {\hyperref[fielddesc:JPEGDECODETIMEOUTTHRES]{JPEG\_DECODE\_TIMEOUT\_THRES}} - 1 $)
\end{itemize}

In this case, the decoder will be reset depending on the configuration of \hyperref[fielddesc:JPEGDECODETIMEOUTTASKSEL]{JPEG\_DECODE\_TIMEOUT\_TASK\_SEL}:
\begin{itemize}
    \item 0: the hardware will not perform any operation, and the software is required to configure the \hyperref[fielddesc:JPEGSOFTRST]{JPEG\_SOFT\_RST} or \hyperref[fielddesc:JPEGFSMRST]{JPEG\_FSM\_RST} to perform a reset operation.
    \item 1: the hardware will automatically reset the decoder to exit the timeout state and enter the initial state.
\end{itemize}

\section{Interrupts}

\chipname{}'s JPEG Codec can generate the JPEG\_INTR interrupt signal that will be sent to the \hyperref[mod:intmtrx]{Interrupt Matrix}.

Interrupt signal JPEG\_INTR can be generated by the following internal interrupt sources:
\begin{itemize}
\item JPEG\_DONE\_INT: Triggered when a frame of image encoding or decoding ends.
\item JPEG\_RLE\_PARALLEL\_ERR\_INT: Triggered when a run-length encoding error occurs.
\item JPEG\_CID\_ERR\_INT: Triggered when the decoded component ID is different from the component ID configured by the software.
\item JPEG\_C\_DHT\_DC\_ID\_ERR\_INT: Triggered when the decoded DC Huffman table ID of each component is not the software configured DC0 Huffman table ID or DC1 Huffman table ID.
\item JPEG\_C\_DHT\_AC\_ID\_ERR\_INT: Triggered when the decoded AC Huffman table ID of each component is not the software configured AC0 Huffman table ID or AC1 Huffman table ID.
\item JPEG\_C\_DQT\_ID\_ERR\_INT: Triggered when the decoded quantization table ID of each component is different from the software configured quantization table ID.
\item JPEG\_RST\_UXP\_ERR\_INT: Triggered when the \hyperref[fielddesc:JPEGRESTARTINTERVAL]{JPEG\_RESTART\_INTERVAL} configured by the software is 0 but the RST marker is parsed by the decoder.
\item JPEG\_RST\_CHECK\_NONE\_ERR\_INT: Triggered when the \hyperref[fielddesc:JPEGRESTARTINTERVAL]{JPEG\_RESTART\_INTERVAL} configured by the software is non-0 but the RST marker cannot be parsed by the decoder.
\item JPEG\_RST\_CHECK\_POS\_ERR\_INT: Triggered when the MCU number between two parsed RST markers is not equal to the \hyperref[fielddesc:JPEGRESTARTINTERVAL]{JPEG\_RESTART\_INTERVAL} configured by the software.
\item JPEG\_OUT\_EOF\_INT: Triggered when the EOF marker is read from the TX channel of 2D DMA.
\item JPEG\_SR\_COLOR\_MODE\_ERR\_INT: This interrupt is invalid.
\item JPEG\_DCT\_DONE\_INT: Triggered when the DCT or IDCT calculation of one data unit is completed.
\item JPEG\_BS\_LAST\_BLOCK\_EOF\_INT: Triggered when the encoding of the last data unit is completed.
\item JPEG\_SCAN\_CHECK\_NONE\_ERR\_INT: Triggered when an image frame has multiple scans to be decoded and the SOS marker is not parsed within (\hyperref[fielddesc:JPEGSOSCHECKBYTENUM]{JPEG\_SOS\_CHECK\_BYTE\_NUM} + 1) bytes in any scan header information.
\item JPEG\_SCAN\_CHECK\_POS\_ERR\_INT: Triggered when the position of the scan header information parsed by the decoder is wrong.
\item JPEG\_UXP\_DET\_INT: Triggered when the marker parsed by the decoder is not supported by the hardware.
\item JPEG\_EN\_FRAME\_EOF\_ERR\_INT: Triggered when the number of data units to be encoded read from 2D DMA is less than the number of data units calculated based on the image resolution configured by the software.
\item JPEG\_EN\_FRAME\_EOF\_LACK\_INT: Triggered when a frame of image to be encoded is completely read from 2D DMA but the EOF marker is not read.
\item JPEG\_DE\_FRAME\_EOF\_ERR\_INT: Triggered when the number of data units obtained after decoding a frame of image is different from the number of data units calculated based on the image resolution configured by the software.
\item JPEG\_DE\_FRAME\_EOF\_LACK\_INT: Triggered when the bitstream of a image is completely read from 2D DMA but the EOF marker or EOI marker is not read.
\item JPEG\_SOS\_UNMATCH\_ERR\_INT: Triggered when the number of components in the scan header information parsed by the decoder is 0 or the header length in the scan header information parsed by the decoder does not match the actual header length.
\item JPEG\_MARKER\_ERR\_FST\_SCAN\_INT: Triggered when there is an error in the first scan header information parsed by the decoder. The errors include errors triggering JPEG\_CID\_ERR\_INT, errors triggering JPEG\_C\_DHT\_DC\_ID\_ERR\_INT, errors triggering JPEG\_C\_DHT\_AC\_ID\_ERR\_INT, errors triggering JPEG\_C\_DQT\_ID\_ERR\_INT and errors triggering JPEG\_SOS\_UNMATCH\_ERR\_INT.
\item JPEG\_MARKER\_ERR\_OTHER\_SCAN\_INT: Triggered when there is an error in the non-first scan header information parsed by the decoder. The error type is the same as the errors that trigger JPEG\_MARKER\_ERR\_FST\_SCAN\_INT.
\item JPEG\_UNDET\_INT: Triggered when the bitstream of an image is completely read from 2D DMA but the SOS marker is not read.
\item JPEG\_DECODE\_TIMEOUT\_INT: Triggered when the decoder is timeout. Section \ref{sec:jpegcodec-dec-prog} describes how this interrupt will occur.
\end{itemize}

The above interrupt sources can only be triggered when the interrupt enable register \regindex{interrupt\_source\_name}\_ENA is set to 1. Write 1 to \regindex{interrupt\_source\_name}\_CLR will clear the interrupt.

\begin{tiplisting}
For definitions of \textit{interrupt}, \textit{interrupt signal}, \textit{interrupt source}, and their correlations, please refer to Chapter \ref{mod:intmtrx} \textit{\nameref{mod:intmtrx}} > Section \ref{sec:intmtx-int-terminology} \textit{\nameref{sec:intmtx-int-terminology}}.
\end{tiplisting}

Each interrupt source can be configured by a common set of registers that are described in Section \hyperref[interrupt-config-registers]{\textit{Interrupt Configuration Registers}}. The specific registers can be found in Section \ref{sec:jpegcodec-reg-summ} \textit{\nameref{sec:jpegcodec-reg-summ}}.

\subsection{JPEG Decoder}

\label{sec:jpegcodec-dec-prog}
When the JPEG codec is working as a decoder, the configuration process is:
\begin{enumerate}
    \item Activate the JPEG codec:
    \begin{itemize}
        \item Set \hyperref[fielddesc:HPSYSCLKRSTJPEGSYSCLKEN]{HP\_SYS\_CLKRST\_JPEG\_SYS\_CLK\_EN} to 1 to enable the JPEG codec clock
        \item Set \hyperref[fielddesc:HPSYSCLKRSTRSTENJPEG]{HP\_SYS\_CLKRST\_RST\_EN\_JPEG} to 0 to release the JPEG codec system level reset.
    \end{itemize}
    \item Configure the JPEG codec:
    \begin{itemize}
        \item Set \hyperref[fielddesc:JPEGSOFTRST]{JPEG\_SOFT\_RST} to 1 to reset the JPEG codec, and then set \hyperref[fielddesc:JPEGSOFTRST]{JPEG\_SOFT\_RST} to 0 to release the JPEG codec reset.
        \item Set \hyperref[fielddesc:JPEGMODE]{JPEG\_MODE} to 1 to configure the JPEG codec to be used as a decoder.
        \item Enable the RST marker check following the instructions in Section \ref{sec:jpegcodec-dec-rst_marker}, if required.
        \item Configure the quantization coefficient tables by following the instructions in Section \ref{sec:jpegcodec-dec-qnr}.
        \item Configure the Huffman tables by following the instructions in Section \ref{sec:jpegcodec-dec-huffman}.
        \item Configure the width and height of the decoded image:
        \begin{itemize}
            \item Width: write width of the decoded image to \hyperref[fielddesc:JPEGHA]{JPEG\_HA}.
            \item Height: write the height of the decoded image to \hyperref[fielddesc:JPEGVA]{JPEG\_VA}.
            \item According to different decoded image formats, \hyperref[fielddesc:JPEGHA]{JPEG\_HA} and \hyperref[fielddesc:JPEGVA]{JPEG\_VA} must meet the requirements of Table \ref{tab:de_jpeg_ha_va}.

\subsection{Reset}
Users can \textbf{reset} JPEG codec entirely or partially, depending on the actual scenarios, at any time during the encoding or decoding process:
\begin{itemize}
    \item The entire JPEG codec, including all the RAMs and FIFOs as well as all the state machines: set \hyperref[fielddesc:JPEGSOFTRST]{JPEG\_SOFT\_RST} to 1. In this case, the register configuration value will not be reset.
    \item Only all the RAMs and FIFOs inside the JPEG codec, for example, changing the quantization coefficient tables or Huffman tables: set \hyperref[fielddesc:JPEGFIFORST]{JPEG\_FIFO\_RST} to 1.
    \item Only all the state machines inside the JPEG codec, for example, only stop the current encoding or decoding process: set \hyperref[fielddesc:JPEGFSMRST]{JPEG\_FSM\_RST} to 1.
\end{itemize}
