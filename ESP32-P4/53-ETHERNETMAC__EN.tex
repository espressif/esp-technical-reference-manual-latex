\chapter{Ethernet Media Access Controller (EMAC)}
\label{mod:ethernetmac}

\section{Overview}


By using the external Ethernet PHY (physical layer), \chipname{} can send and receive data via Ethernet MAC (Media Access Controller) according to the IEEE 802.3 standard, as Figure \ref{fig:ethernetmac-overview} shows. Ethernet is currently the most commonly used network protocol that controls how data is transmitted over local- and wide-area networks, abbreviated as LAN and WAN, respectively.

\begin{figure}[H]
    \centering
    \includegraphics[width=0.9\textwidth]{\modulefiles/figures/ehternet_mac_overview_p4}
    \caption{Ethernet MAC Functionality Overview}
    \label{fig:ethernetmac-overview}
\end{figure}

\chipname{} Ethernet MAC complies with the following standards:
\begin{itemize}
	\item IEEE 802.3-2002 for Ethernet MAC
	\item IEEE 1588-2008 standard for precise  networked clock synchronization
	\item IEEE 802.3 standard Media Independent Interface (MII) and Reduced Media Independent Interface (RMII)
        \item IEEE 802.3az-2010 for Energy Efficient Ethernet
        \item IEEE 802.1Q for VLAN frame format
\end{itemize}

\section{Features}

\begin{itemize}
	\item Data rates of 10/100 Mbit/s through an external PHY interface
	\item Communication with an external Fast Ethernet PHY through IEEE 802.3-compliant MII and RMII interfaces
	\item Full-duplex and half-duplex modes
	\begin{itemize}
    	\item Carrier Sense Multiple Access or Collision Detection (CSMA/CD) protocol in half-duplex mode
    	\item IEEE 802.3x flow control in full-duplex mode
    	\item Optional forwarding of received pause control frame to the user application in full-duplex mode
    	\item Back-pressure flow control in half-duplex mode
    	\item Automatic transmission of zero-quanta pause frame on deassertion of flow control input in full-duplex mode
	\end{itemize}
	\item Preamble and start-of-frame data (SFD) insertion in Transmit, and deletion in Receive paths
	\item Automatic CRC and padding (all 0) generation controllable on a per-frame basis
	\item Options for automatic padding generation for data below the minimum frame length
	\item Programmable frame length supporting jumbo frames of up to 16 KB
	\item Programmable inter-frame gap (IFG) from 40 to 96 bit times in steps of 8)
	\item Supports a variety of flexible address filtering modes:
    \begin{itemize}
    	\item Up to eight 48-bit perfect address filters with masks for each byte
    	\item Up to eight 48-bit source address (SA) address comparison check with masks for each byte
        \item Option to pass all multicast addressed frames
        \item Promiscuous mode support to pass all frames without any filtering for network monitoring
        \item Passes all incoming packets (as per filter) with a status report
    \end{itemize}
	\item Separate 32-bit status returned for transmission and reception packets
        \item Supports IEEE 802.1Q VLAN tag detection for reception frames
	\item Separate transmission, reception, and control interfaces for the application
	\item Management Data Input/Output (MDIO) interface for PHY device configuration and management
	\item Supports checksum off-load for received IPv4 and TCP packets encapsulated by the Ethernet frame
	\item Supports checking IPv4 header checksum and TCP, UDP, or ICMP checksum encapsulated in IPv4 or IPv6 datagrams
	\item Supports 64-bit timestamp on each transmit and receive frame (see IEEE 1588-2008)
        \item Supports Energy Efficient Ethernet (see IEEE 802.3az-2010)
        \item CRC replacement, SA insertion or replacement, and VLAN insertion, replacement or deletion in transmit frames
	\item Two FIFOs: 256-byte TX FIFO and 256-byte RX FIFO
        \item Receive status vectors inserted into RX FIFO after the EOF (end of frame) transfer, which enables multiple-frame storage in RX FIFO without requiring another FIFO to store those frames' receive status
        \item Option to forward good runt frames
	\item Supports statistics by generating pulses for frames dropped or corrupted due to overflow in RX FIFO
	\item Automatic re-transmission of collision frames  (subject to certain conditions, see Section \ref{sec:ethernetmac-retrans-dur-collision})
	\item Discarding frames in cases of late collisions, excessive collisions, excessive deferrals, and underflow conditions
	\item Software control to flush TX FIFO
\end{itemize}

\section{Ethernet MAC Architecture}

Figure \ref{fig:ethernetmac_block_diagram} shows the block diagram of the Ethernet MAC.

\begin{figure}[H]
    \centering
    \includegraphics[width=0.8\textwidth]{\modulefiles/figures/Ethernet_block_diagram}
    \caption{Ethernet MAC Block Diagram}
    \label{fig:ethernetmac_block_diagram}
\end{figure}

Ethernet MAC consists of three layers: EMAC\_CORE (MAC Core Layer), EMAC\_MTL (MAC Transition Layer), and EMAC\_DMA (Direct Memory Access). Each of these three layers has two directions: TX and RX. They are connected to the system through the Advanced High Performance Bus (AHB) and the Advanced Peripheral Bus (APB) on the chip. Off the chip, they communicate with the external PHY through the MII and RMII interfaces to establish an Ethernet connection.

\section{Functional Description}

\subsection{EMAC\_CORE}
The MAC supports two interfaces (see Section \ref{sec:ethernetmac-phy-interface}) towards the PHY chip. The PHY interface can be selected only once after the chip reset. The MAC core communicates with the application side (DMA side) using the MAC Transmit Interface (MTI), MAC Receive Interface (MRI), and the MAC Control Interface (MCI).

\subsubsection{Transmission}

Transmission is initiated when the MTL application pushes in data with the SOF signal (start of frame) asserted.  When the SOF signal is detected, the MAC accepts the data and begins transmitting to the RMII or MII. The time required to transmit the frame data to the RMII or MII after the application initiates transmission is variable, depending on delay factors like IFG delay, time to transmit preamble or SFD (Start Frame Delimiter), and any back-off delays for half-duplex mode. Until then, the MAC does not accept the data received from MTL by deasserting the ready signal.

After the EOF (end of frame) is transmitted to the MAC, the MAC completes normal transmission and then gives the Transmit Status to the MTL. If a normal collision (in half-duplex mode) occurs during transmission, the MAC core makes valid the Transmit Status to the MTL. It will then accept and drop all further data until the next SOF is received. The MTL block should retransmit the same frame from SOF on observing a retry request (in the Status) from the MAC.

The MAC issues an underflow status if the MTL is not able to provide the data continuously during the transmission. During the normal transmission of a frame from the MTL, if the MAC receives an SOF without getting an EOF for the previous frame, then the SOF is ignored and the new frame is considered as a continuation of the previous frame.

\subparagraph{Transmit Flow Control}
In full-duplex mode, when the Transmit Flow Control Enable bit (\hyperref[regdesc:EMACFLOWCONTROLREG]{TFCE}) is set to 1, the MAC generates pause frames and transmits them as necessary, with the calculated CRC appended.

Pause frame generation can be initiated in two ways. When the application sets the Flow Control Busy bit (\hyperref[regdesc:EMACFLOWCONTROLREG]{FCBBA}) to 1, or when the RX FIFO is full, a pause frame is transmitted.

\begin{itemize}
    \item If the application has requested flow control by setting the \hyperref[regdesc:EMACFLOWCONTROLREG]{FCBBA} bit to 1, the MAC will generate and transmit a single pause frame. The value of the pause time in the generated frame is the pause time value programmed in the \hyperref[regdesc:EMACFLOWCONTROLREG]{PAUSE\_TIME} field. If the application wants to extend or end the pause prior to the time specified in the previously transmitted pause frame, it must request another pause frame transmission after programming the pause time field \hyperref[regdesc:EMACFLOWCONTROLREG]{PAUSE\_TIME} with an appropriate value.
    \item If the application has requested flow control when the RX FIFO is full, the MAC will generate and transmit a pause frame. The value of the pause time in the generated frame is the pause time value programmed in the \hyperref[regdesc:EMACFLOWCONTROLREG]{PAUSE\_TIME} field. If the RX FIFO remains full at a configurable number of slot times (\hyperref[regdesc:EMACFLOWCONTROLREG]{PLT}) before the pause time runs out, a second pause frame will be transmitted. The process will be repeated as long as the RX FIFO remains full. If the FIFO is no longer full prior to the sampling time, the MAC will send a pause frame with zero pause time to indicate to the remote end that the RX buffer is ready to receive new data frames.
\end{itemize}

\subparagraph{Retransmission During Collision}
\label{sec:ethernetmac-retrans-dur-collision}
In half-duplex mode, a collision may occur on the MAC line interface when frames are transmitted to the MAC. The MAC may indicate a retry attempt by giving the status even before the EOF is transferred. Then the MAC will enable the retransmission by popping out the frame again from the FIFO. After more than 96 bytes are popped towards the MAC core, the FIFO controller frees the space in the FIFO, and makes it available to the DMA to push in more data. This means that the retransmission is not possible after this threshold is crossed or when the MAC core indicates a late-collision event.

The MAC transmitter may abort the transmission of a frame because of collision, TX FIFO underflow, loss of carrier, jabber timeout, no carrier, excessive deferral, or late collision. When frame transmission is aborted because of collision, the MAC requests retransmission of the frame.

\subparagraph{Transmit Status Word}
At the end of the Ethernet frame transfer, the MAC outputs the transmit status to the application. The detailed description of the Transmit Status is the same as for \hyperref[tab:ethernetmac-TDES0]{TDES0}.

\subsubsection{Reception}
A receive operation is initiated when the MAC detects an SFD on the RMII or MII. The MAC strips the preamble and SFD before proceeding to process the frame. The header fields are checked for filtering and the FCS (Frame Check Sequence) field used to verify the CRC for the frame. The received frame is stored in a shallow buffer until the address filtering is performed. The frame is dropped in the MAC if it fails the address filtering.

The frames received by the MAC are pushed into the RX FIFO. Once the RX FIFO crosses the configured Receive threshold (\hyperref[regdesc:DMAOPERATIONMODEREG]{RX\_THRESH\_CTRL}), its status is indicated to the DMA, so that the DMA can initiate a pre-configured burst transfers towards the AHB interface.

In the default cut-through mode, when the FIFO receives 64 bytes or a full packet of data, it pops out the data and indicates the availability to the DMA. Once the DMA initiates the transfer to the AHB interface, the data transmission continues from the FIFO until a complete packet has been transferred. Upon the completion of the EOF frame transfer, the status word will be popped out and transmitted to the DMA controller.


\subparagraph{Receive Protocol}
The receive module strips the preamble and SFD of the received frame. Once the SFD is detected, the MAC begins sending the Ethernet frame data to the RX FIFO, starting from the first byte following the SFD (destination address). If the IEEE 1588 timestamp feature is enabled, the snapshot of system time will be captured whenever an SFD of any frame is detected on MII, and sent to the application unless the MAC filters out and drops the frame.

If the Length/Type field of the receiving frame is less than 0x600 and if the MAC is programmed for the automatic CRC/paddingstripping option, the MAC  sends the frame data up to the count specified in the Length/Type field to the RX FIFO, and then starts dropping bytes (including the FCS field). If the Length/Type field is greater than or equal to 0x600, the MAC will send all received Ethernet frame data to the RX FIFO, regardless of the value on the programmed automatic CRC stripping option. By default, the MAC is programmed for the watchdog timer to be enabled, that is frames above 2048 bytes (including DA, SA, LT, data, padding, and FCS) are cut off. This feature can be disabled by programming the Watchdog Disable bit \hyperref[regdesc:EMACCONFIGREG]{EMACWATCHDOG}. However, even if the watchdog timer is disabled, frames longer than 16 KB will still be cut off and the watchdog timeout status will be indicated.

\subparagraph{Receive Frame Controller}
If the \hyperref[regdesc:EMACFFREG]{RECEIVE\_ALL} bit in the MAC Frame Filter Register is reset, the MAC performs frame filtering based on the destination and source addresses. The application still needs to perform another level of filtering if it decides not to receive any bad frames like runt, CRC error frames, etc. On detecting a filter failure, the frame is dropped and not transmitted to the application. When the filter parameters change dynamically, if a DA and SA filter failure occurs, the reset of the frame is dropped and the Receive Status word (with zero frame length, CRC Error, and Runt Error bits set) is updated immediately indicating the filter failure.

\subparagraph{Receive Flow Control}
The MAC detects the receiving pause frame and pauses the frame transmission for the delay specified within the received pause frame (in full-duplex mode only). The Pause Frame Detection Function can be enabled or disabled with the \hyperref[regdesc:EMACFLOWCONTROLREG]{RFCE} bit. Once the receive flow control is enabled, the MAC starts monitoring the received frame destination address for any match with the multicast address of the control frame (0x0180 C200 0001). If a match is detected (i.e., the destination address of the received frame matches the destination address of the reserved control frame), the MAC decides whether to transmit the received control frame to the application, according to the \hyperref[regdesc:EMACFFREG]{PCF} field.

The MAC also decodes the type, the opcode, and the pause timer field of the receiving control Frame. If the byte count of the status indicates 64 bytes, and if there is no CRC error, the MAC transmitter will pause the transmission of any data frame for the duration of the decoded pause time value multiplied by the slot time (64 byte times for 10/100 Mbit/s mode). Meanwhile, if another pause frame is detected with a zero pause time, the MAC will reset the pause time and give another pause request.

If the received control frame matches neither the type field (0x8808), the opcode (0x00001), nor the byte length (64 bytes), or if there is a CRC error, the MAC will not generate a pause.

In the case of a pause frame with a multicast destination address, the MAC filters the frame according to the address match.

For a pause frame with a unicast destination address, the filtering depends on whether the DA matched the contents of the MAC Address Register 0 and the \hyperref[regdesc:EMACFLOWCONTROLREG]{UPFD} (detecting a pause frame even with a unicast destination address) is set. The \hyperref[regdesc:EMACFFREG]{PCF} register controls the filtering for control frames in addition to the address filter.

\subparagraph{Receive Operation Multiframe Handling}
Since the status is available immediately following the data, the MAC is capable of storing any number of frames into the FIFO, as long as it is not full.

\subparagraph{Error Handling}
If the RX FIFO is full before it receives the EOF data from the MAC, an overflow will be declared and the whole frame will be dropped. The status bit \hyperref[tab:ethernetmac-RDES0]{RDES0}[11] will reflect the fact that this frame is a partial frame due to overflow. The RX FIFO can perform the filtering of error frames and runt frames, if this function is enabled via \hyperref[regdesc:DMAOPERATIONMODEREG]{FLUSH\_TX\_FIFO} and \hyperref[regdesc:DMAOPERATIONMODEREG]{FWD\_UNDER\_GF}. 

In cut-through mode, if a frame's status and length are available when reading an SOF from the RX FIFO, the whole error frame can be dropped. The DMA can flush the error frame being read from the FIFO by clearing \hyperref[regdesc:DMAOPERATIONMODEREG]{FWD\_ERR\_FRAME} to 0. The DMA then stops transferring data to the application, internally reads out the rest of the frame, and drops it. The MTL will then start the transfer of the next frame, if it is available. If FIFO is available, the transmission of the next frame will be initiated.

\subparagraph{Receive Status Word}
At the end of the Ethernet frame transfer, the MAC outputs the receive status to the application via EMAC\_DMA. The detailed description of the receive status is the same for bit[31:0] in \hyperref[tab:ethernetmac-RDES0]{RDES0}.

\subsection{EMAC\_MTL (MAC Transaction Layer)}
The MAC Transaction Layer provides FIFO memory to buffer and regulate the frames between the application system memory and the MAC. It also enables the data to be transmitted between the application clock domain and the MAC clock domains. The MTL layer has two data paths, namely the Transmit path and the Receive path. The data path for both directions is 32-bit wide and operates with a simple FIFO protocol.

\subsection{EMAC\_DMA}
EMAC has its own DMA module, with independent Transmit and Receive engines and a CSR (Control and Status Registers) space. The Transmit engine transfers data from the system memory to the device port (MTL), while the Receive engine transmits data from the device port to the system memory. The controller uses descriptors to efficiently move data from source to destination with minimal Host CPU intervention. The DMA is designed for packet-oriented data transfer such as frames in Ethernet. The controller can be programmed to interrupt the Host CPU for normal situations, such as the completion of frame transmission or reception, or error conditions.

\subsection{PHY Interface}\label{sec:ethernetmac-phy-interface}
The MAC and the external PHY communicate through two interfaces:
\begin{itemize}
\item Media Independent Interface (MII)
\item Reduced Media Independent Interface (RMII)
\end{itemize}


\subsubsection{Media Independent Interface: MII}
The Media Independent Interface (MII) defines the interconnection between MAC sublayers and PHYs at 10 Mbit/s and 100 Mbit/s.

\subparagraph{MII Signals to PHY}
The MII interface signals are shown in Figure \ref{fig:ethernetmac-Mii_IF}.

\begin{figure}[H]
    \centering
    \includegraphics[width=0.6\textwidth]{\modulefiles/figures/Mii_IF_P4}
    \caption{MII Interface}
    \label{fig:ethernetmac-Mii_IF}
\end{figure}

The MII interface signals are described below:
\begin{itemize}
    \item MII\_TX\_CLK: TX clock signal. This signal provides the reference timing for TX data transmission. It has two frequencies: 2.5 MHz at 10 Mbit/s, and 25 MHz at 100 Mbit/s.
    \item MII\_TXD[3:0]: A bundle of four transmit data signals driven by the MAC sublayer, and qualified (valid data) on the assertion of the MII\_TX\_EN signal. MII\_TXD[0] is the lowest significant bit and MII\_TXD[3] is the highest significant bit. When the MII\_TX\_EN signal is deasserted, the transmit data have no effect on the PHY.
    \item MII\_TX\_EN: Transmission enable signal. This signal indicates that the MAC is presenting nibbles (4 bits) on the MII for transmission. It must be asserted synchronously (MII\_TX\_CLK) with the first nibble of the preamble and must remain asserted while all nibbles to be transmitted are presented to the MII.
    \item MII\_TX\_ERR: Transmit error signal. The signal must be asserted for one or more clock periods (MII\_TX\_CLK) to indicate to the PHY layer that an error was detected somewhere in the frame.
    \item MII\_RX\_CLK: RX clock signal. This signal provides the reference timing for RX data transmission. It has two frequencies: 2.5 MHz at 10 Mbit/s, and 25 MHz at 100 Mbit/s.
    \item MII\_RXD[3:0]: A bundle of four receive data signals driven by the MAC sublayer, and qualified (valid data) on the assertion of the MII\_RX\_DV signal. MII\_RXD[0] is the lowest significant bit and MII\_RXD[3] is the highest significant bit. When MII\_RX\_DV is deasserted and MII\_RX\_ERR is asserted, a specific MII\_RXD[3:0] value is used to transfer specific information from the PHY.
    \item MII\_RX\_DV: Receive data valid signal. This signal indicates that the PHY is presenting recovered and decoded nibbles on the MII for reception. It must be asserted synchronously (MII\_RX\_CLK) with the first recovered nibble of the frame and must remain asserted through the final recovered nibble. It must be deasserted prior to the first clock cycle that follows the final nibble. In order to receive the frame correctly, the MII\_RX\_DV signal must encompass the frame, starting no later than the SFD field.
    \item MII\_CRS: Carrier sense signal. When the transmit or receive medium is non-idle, the signal is asserted by the PHY. When the transmit or receive medium is idle, the signal is deasserted by the PHY. The PHY must ensure that the MII\_CRS signal remains asserted throughout the duration of a collision condition. This signal does not need to be synchronized with the TX and RX clocks.  In full duplex mode the state of this signal is don't care for the MAC sublayer.
    \item MII\_COL: Collision detection signal. This signal must be asserted by the PHY upon detection of a collision on the medium and must remain asserted while the collision condition persists. This signal is not required to transition synchronously with respect to the TX and RX clocks. In full duplex mode, the state of this signal is don't care for the MAC sublayer.
    \item MII\_RX\_ERR: Receive error signal. The signal must be asserted for one or more clock periods (MII\_RX\_CLK) to indicate to the MAC sublayer that an error was detected somewhere in the frame.
    \item MDIO and MDC: Management data input/output and management data clock signal. The two signals constitute a serial bus defined for the Ethernet family of IEEE 802.3 standards, used to transfer control and data information to the PHY, see section \hyperref[section:Station Management Agent (SMA) Interface]{Station Management Agent (SMA) Interface}.
\end{itemize}

\subparagraph{RMII Signals}

The Reduced Media Independent Interface (RMII) specification reduces the number of pins between the microcontroller's Ethernet MAC and the external PHY at 10 Mbit/s or 100 Mbit/s. According to the IEEE 802.3u standard, an MII contains 16 pins for data and control. The RMII specification reduces the pin count to 7.

RMII has the following features:
\begin{itemize}
    \item Supports an operating rate of 10 Mbit/s or 100 Mbit/s
    \item The clock reference must be 50 MHz.
    \item The same clock reference must be sourced externally both to the MAC and the external Ethernet PHY.
    \item Provides independent 2-bit wide TX and RX data paths.
\end{itemize}

\begin{itemize}
\item Read a PHY register:
\begin{itemize}

    \item Configure \hyperref[regdesc:EMACMIIADDRREG]{MIIREG} to specify the address of the PHY register to access.

    \item Read \hyperref[regdesc:EMACMIIDATAREG]{EMACMIIDATA\_REG} to retrieve the value of the PHY register.
\end{itemize}
\item Write a PHY register:
\begin{itemize}
    \item Read \hyperref[regdesc:EMACMIIADDRREG]{MIIBUSY} until the MII is in an idle state.
    \item Configure \hyperref[regdesc:EMACMIIDATAREG]{EMACMIIDATA\_REG} to set the value to be written to the PHY register.
    \item Configure \hyperref[regdesc:EMACMIIADDRREG]{MIIDEV} to specify the PHY device to access.
    \item Configure \hyperref[regdesc:EMACMIIADDRREG]{MIIREG} to specify the address of the PHY register to access.
    \item Configure \hyperref[regdesc:EMACMIIADDRREG]{MIICSRCLK} to set the APB clock frequency.
    \item Set \hyperref[regdesc:EMACMIIADDRREG]{MIIWRITE} to 1 for write access.
    \item Read \hyperref[regdesc:EMACMIIADDRREG]{MIIBUSY} until the MII is in an idle state, which indicates the completion of the write operation.
\end{itemize}
\end{itemize}

\subsection{MAC Address Filtering}

Address filtering checks the destination and source addresses on all received frames and reports the address filtering status accordingly. Address checking is based on different parameters (Frame filter register) chosen by the application. The filtered frame can also be identified: multicast or broadcast frame. Address filtering uses the station's physical (MAC) address for address checking.

\subsubsection{Unicast Destination Address Filtering}
The MAC supports up to 9 MAC addresses for unicast perfect filtering.  MAC compares all 48 bits of the received unicast address with the programmed MAC address for any match. By default, EMACADDR0 is always enabled, and the other addresses (EMACADDR0 \verb+~+ EMACADDR8) are selected with an individual enable bit. Each byte of these other addresses (EMACADDR0 \verb+~+ EMACADDR8) can be
masked during comparison with the corresponding received DA byte by setting the
corresponding Mask Byte Control bit in the register. This helps group address filtering for the
DA.

\subsubsection{Multicast Destination Address Filtering}
The MAC can be programmed to pass all multicast frames by setting the Pass All Multicast (\hyperref[regdesc:EMACFFREG]{PAM}) bit in the Frame Filter Register to 1. If the \hyperref[regdesc:EMACFFREG]{PAM} bit is reset, the MAC  the filtering for multicast addresses.

In perfect filtering mode, the multicast address is compared with the programmed MAC Destination Address Registers (EMACADDR0 \verb+~+ EMACADDR8). Group address filtering is also supported.

\subsubsection{Broadcast Address Filter}
The MAC does not filter any broadcast frames in the default mode. However, if the MAC is programmed to reject all broadcast frames by setting the Disable Broadcast Frames (\hyperref[regdesc:EMACFFREG]{DBF}) bit in the Frame Filter Register to 1, all broadcast frames will be dropped.

\subsubsection{Unicast Source Address Filter}
The MAC can also perform perfect filtering based on the source address field of the received frames. By default, the Address Filtering Module (AFM) compares the Source Address (SA) field with the values programmed in the SA register. The MAC address registers [1:3] can be configured to contain SA instead of DA for comparison, by setting bit[30] in the corresponding register. Group filtering with SA is also supported. The frames that fail the SA filter are dropped by the MAC if the Source Address Filter Enable (\hyperref[regdesc:EMACFFREG]{SAFE}) bit in the Frame filter register is set. Otherwise, the result of the SA filter is given as a status bit in the Receive Status word (see Table \ref{tab:ethernetmac-RDES0}).

When the \hyperref[regdesc:EMACFFREG]{SAFE} bit is set to 1, the result of the SA filtering and DA filtering is AND'ed to decide whether the
frame needs to be forwarded. This means that either of the filter failure results will drop the
frame. Both filters have to pass the frame for the frame to be forwarded to the application.

\subsubsection{Inverse Filtering Operation}
For both destination and source address filtering, there is an option to invert the filter-match
result at the final output. These are controlled by the \hyperref[regdesc:EMACFFREG]{DAIF} and \hyperref[regdesc:EMACFFREG]{SAIF} bits in the Frame filter register, respectively. The \hyperref[regdesc:EMACFFREG]{DAIF} bit is applicable for both Unicast and Multicast DA frames.
When the \hyperref[regdesc:EMACFFREG]{DAIF} bit is set, the result of the unicast/multicast destination address filter is inverted.
Similarly, when the \hyperref[regdesc:EMACFFREG]{SAIF} bit is set, the result of the unicast SA filter is inverted.

\subsection{Energy Efficient Ethernet (EEE) }
Energy Efficient Ethernet (EEE) is an optional operating mode that enables the MAC sublayer along with a series of physical layers to operate in the Low Power Idle (LPI) mode. The EEE mode supports 100 Mbps MAC operations compliant with the IEEE 802.3az-2010 standard. The EEE is supported only in the full-duplex mode when MAC uses MII.

The LPI mode allows power saving by switching off parts of the communication device functionality when there is no data to be transmitted and received. The system on both sides of the link can disable some functionalities and save power during the period of low-link utilization. The MAC controls whether the system should enter or exit the LPI mode and communicate this to the PHY.

\subsubsection{Transmission}

To enter the LPI mode, the software must set the \hyperref[regdesc:EMACLPICSRREG]{LPIEN} bit to
indicate to the MAC to stop transmission and initiate the LPI protocol. The MAC completes the transmission in progress, generates its transmission status, and then starts transmitting the LPI pattern instead of the IDLE pattern during the Interframe gap (IFG). The MAC then updates \hyperref[regdesc:EMACLPICSRREG]{TLPIEN} and generates an interrupt.

To exit the LPI mode, the software must clear the \hyperref[regdesc:EMACLPICSRREG]{LPIEN} bit to
indicate to the MAC. The MAC stops transmitting the LPI pattern, and starts transmitting the IDLE pattern. When the \hyperref[regdesc:EMACLPITIMERSCONTROLREG]{LPI\_TW\_TIMER} expires, the MAC updates \hyperref[regdesc:EMACLPICSRREG]{TLPIEX} and generates an interrupt.


\subsubsection{Reception}

When the PHY receives the signals from the link partner to enter into the LPI state, the PHY starts transmitting the LPI pattern. The MAC updates \hyperref[regdesc:EMACLPICSRREG]{RLPIEN} and generates an interrupt.

When the PHY receives signals from the link partner to exit the LPI state, the PHY stops transmitting the LPI pattern. The MAC updates \hyperref[regdesc:EMACLPICSRREG]{RLPIEX} and generates an interrupt.

\subsection{Source Address, VLAN, and CRC Control}

\subsubsection{Source Address Control}

The software can enable source address insertion or replacement for all or some frames:
\begin{itemize}
    \item Enable source address insertion or replacement for all frames by configuring \hyperref[regdesc:EMACCONFIGREG]{SAIRC}.
    \item Enable source address insertion or replacement for the current frame by configuring \hyperref[tab:ethernetmac-TDES1]{TDES1[31:29]} of the frame. When \hyperref[tab:ethernetmac-TDES1]{TDES1[31]} is set, \hyperref[regdesc:EMACADDR1HIGHREG]{MAC\_ADDRESS1\_HI} and \hyperref[regdesc:EMACADDR1LOWREG]{EMACADDR1LOW\_REG} are used for insertion and replacement. When \hyperref[tab:ethernetmac-TDES1]{TDES1[31]} is cleared, \hyperref[regdesc:EMACADDR0HIGHREG]{MAC\_ADDRESS0\_HI} and \hyperref[regdesc:EMACADDR0LOWREG]{EMACADDR0LOW\_REG} are used for insertion and replacement. When \hyperref[regdesc:EMACADDR1HIGHREG]{ADDRESS\_ENABLE1} is not set, \hyperref[regdesc:EMACADDR0HIGHREG]{MAC\_ADDRESS0\_HI} and  \hyperref[regdesc:EMACADDR0LOWREG]{EMACADDR0LOW\_REG} are used for insertion and replacement irrespective of \hyperref[tab:ethernetmac-TDES1]{TDES1[31]}.
\end{itemize}

\subsubsection{VLAN Control}

The software can enable VLAN insertion,  replacement, or deletion for all or some frames:

\begin{itemize}
    \item Enable VLAN insertion,  replacement, or deletion for all frames by configuring \hyperref[regdesc:EMACTXVLANTAGCONTROLREG]{VLC}.
    \item Enable VLAN insertion, replacement, or deletion for the current frame by configuring \hyperref[tab:ethernetmac-TDES0]{TDES0[19:18]} of the frame.
\end{itemize}

\subsubsection{CRC Control}

CRC replacement is a feature that the software can use to indicate the MAC to replace the FCS field of the frame being transmitted with the calculated CRC. This feature can be enabled for the current frame by configuring \hyperref[tab:ethernetmac-TDES0]{TDES0[24]}.

CRC replacement is valid only when MAC does not append CRC (\hyperref[tab:ethernetmac-TDES0]{TDES0[27]} is set), i.e., when FCS is provided by software in frames transmitted. If source address or VLAN control is enabled, the MAC will replace or append FCS fields with CRC when \hyperref[tab:ethernetmac-TDES0]{TDES0[27]} is set or cleared respectively.

\subsection{Time Stamp}

The IEEE 1588-2008 standard defines the Precision Time Protocol (PTP) that allows precise clock
synchronization in measurement and control systems
implemented with technologies such as local computing, network communication and distributed objects. it supports system-wide synchronization accuracy in the sub-microsecond range with a minimum network and local clock computing resources.

\subsubsection{Transmit Path}

The MAC captures a timestamp when the start-of-frame data (SFD) is sent on the MII interface. The frames for which timestamps are captured can be controlled on a per-frame basis. In other words, each transmit frame can be marked to indicate whether a timestamp should be captured for that frame. The MAC does not process the transmitted to identity PTP frames. Rather, it captures the timestamp or not according to the configurations of \hyperref[tab:ethernetmac-TDES0]{TDES0[25]} by software. The MAC returns the timestamp to the software inside the corresponding transmit descriptor, thus connecting the timestamp automatically to the specific PTP frame. The 64-bit timestamp information is written to the \hyperref[tab:ethernetmac-TDES6]{TDES6} and \hyperref[tab:ethernetmac-TDES7]{TDES7} fields.

\subsubsection{Receive Path}

The MAC processes the received frames to identify valid PTP frames. The snapshot of the time to be sent to the application can be controlled by software via \hyperref[regdesc:EMACTIMESTAMPCONTROLREG]{EMACTSTPCTRL\_REG}.

The MAC returns the timestamp to the software inside the corresponding receive descriptor. The extended status (including the timestamp status and IPC status) is written to \hyperref[tab:ethernetmac-RDES4]{RDES4}, and the timestamp snapshot is written to \hyperref[tab:ethernetmac-RDES6]{RDES6} and \hyperref[tab:ethernetmac-RDES7]{RDES7}.

\subsection{Remote Wakeup}
The MAC supports remote wakeup. When the receiver receives a remote wakeup frame or a magic packet, an interrupt will be triggered, and the PMU will be notified to wake up the SoC system.

The software can control whether to use remote wakeup frames to trigger interrupts by configuring  \hyperref[regdesc:PMTCSRREG]{RWKPKTEN} , whether to use magic packets to trigger interrupts by configuring \hyperref[regdesc:PMTCSRREG]{MGKPKTEN}, and whether the MAC drops all received frames in power-down mode by configuring \hyperref[regdesc:PMTCSRREG]{PWRDWN}.

\subsection{Good Transmitted and Received Frames}

A frame that has been successfully transmitted is considered a ``good frame". In other words, a transmitted frame is considered to be ``good", if the frame transmission is not aborted due to the following errors:
\begin{itemize}
\item Jabber timeout (\hyperref[tab:ethernetmac-TDES0]{TDES0[14]})
\item No carrier or loss of carrier (\hyperref[tab:ethernetmac-TDES0]{TDES0[11:10]})
\item Late collision (\hyperref[tab:ethernetmac-TDES0]{TDES0[9]})
\item Frame underflow  (\hyperref[tab:ethernetmac-TDES0]{TDES0[1]})
\item Excessive deferral (\hyperref[tab:ethernetmac-TDES0]{TDES0[2]})
\item Excessive collision (\hyperref[tab:ethernetmac-TDES0]{TDES0[8]})
\end{itemize}
A received frame is considered to be ``good" if the following errors do not occur:
\begin{itemize}
\item CRC error (\hyperref[tab:ethernetmac-RDES0]{RDES0[1]})
\item Runt frames (frames shorter than 64 bytes)
\item Alignment error (in 10/100 Mbps modes only) (\hyperref[tab:ethernetmac-RDES0]{RDES0[2]})
\item Length error (non-type frames only) (\hyperref[tab:ethernetmac-RDES0]{RDES0[12]})
\item Frame size over the maximum size (for non-type frames over the maximum frame size only)) (\hyperref[tab:ethernetmac-RDES0]{RDES0[7]})
\item MII\_RXER input error (\hyperref[tab:ethernetmac-RDES0]{RDES0[3]})
\end{itemize}
The maximum frame size depends on the frame type:
\begin{itemize}
\item The maximum size of untagged frames = 1518 bytes
\item The maximum size of VLAN frames = 1522 bytes
\end{itemize}

\subsection{EMAC Initial Configuration}

Initialize EMAC\_DMA:
\begin{itemize}
    \item Reset EMAC by setting \hyperref[regdesc:DMABUSMODEREG]{SW\_RST} to 1.
    \item Wait for the completion of reset until reading 0 from \hyperref[regdesc:DMABUSMODEREG]{SW\_RST}, which indicates  that the reset is done.
    \item Read \hyperref[regdesc:DMAHBSTATUSREG]{AHB\_ST}; a value of 0 indicates that all AHB bus transactions have been completed.
    \item Configure the AHB bus mode for EMAC\_DMA via \hyperref[regdesc:DMABUSMODEREG]{DMABUSMODE\_REG}.
    \item Prepare the linked list and configure the linked list base address via \hyperref[regdesc:DMARXBASEADDRREG]{DMARXBASEADDR\_REG} and \hyperref[regdesc:DMATXBASEADDRREG]{DMATXBASEADDR\_REG}.
    \item Configure the operating mode for EMAC\_DMA and EMAC\_MTL via \hyperref[regdesc:DMAOPERATIONMODEREG]{DMAOPERATION\_MODE\_REG}.
    \item Configure whether to enable DMA interrupts via \hyperref[regdesc:DMAINENREG]{DMAIN\_EN\_REG}.
\end{itemize}

Initialize EMAC\_CORE:
\begin{itemize}
    \item Read \hyperref[regdesc:EMACMIIADDRREG]{EMACMIIADDR\_REG} and \hyperref[regdesc:EMACMIIDATAREG]{EMACMIIDATA\_REG} for the PHY's connection status, operating frequency, and operating mode.
    \item Configure \hyperref[regdesc:EMACADDR0HIGHREG]{EMACADDR0HIGH\_REG} and \hyperref[regdesc:EMACADDR0LOWREG]{EMACADDR0LOW\_REG} to specify the MAC address.
    \item Configure \hyperref[regdesc:EMACFFREG]{EMACFF\_REG} to select the packet filtering mode.
    \item Configure \hyperref[regdesc:EMACFLOWCONTROLREG]{EMACFC\_REG} for flow control.
    \item Configure whether to mask MAC interrupts via \hyperref[regdesc:EMACINTMASKREG]{EMACINTMASK\_REG}.
    \item Configure the MAC's operating mode for transmission and reception via \hyperref[regdesc:EMACCONFIGREG]{EMACCONFIG\_REG}.
\end{itemize}

\subsection{Starting Transmission}

\begin{itemize}
    \item Start EMAC\_DMA transmission by setting \hyperref[regdesc:DMAOPERATIONMODEREG]{START\_STOP\_TRANSMISSION\_COMMAND} to 1.
    \item Start EMAC\_CORE transmission by setting \hyperref[regdesc:EMACCONFIGREG]{EMACTX} to 1.
    \item Detect the TRANS\_INT interrupt if enabled, and wait for the frame transmission to complete.
\end{itemize}

\subsection{Starting Reception}

\begin{itemize}
    \item Start EMAC\_DMA reception by setting \hyperref[regdesc:DMAOPERATIONMODEREG]{START\_STOP\_RX} to 1.
    \item Start EMAC\_CORE reception by setting \hyperref[regdesc:EMACCONFIGREG]{EMACRX} to 1.
    \item Detect the RECV\_INT interrupt if enabled, and wait for the frame reception to complete.
\end{itemize}


\subsection{TX Entering and Exiting the LPI State}
\begin{itemize}
\item Configure the LPI timer counter via \hyperref[regdesc:EMACLPITIMERSCONTROLREG]{EMACLPITIMERSCONTROL\_REG}
\item Make the transmitter to enter the LPI state via \hyperref[regdesc:EMACLPICSRREG]{EMACLPI\_CSR\_REG}
\item Detect the LPI\_INT interrupt, and wait for \hyperref[regdesc:EMACLPICSRREG]{TLPIEN} to be set, which indicates that the transmitter has entered the LPI state
\item Bring the transmitter out of the LPI state by clearing \hyperref[regdesc:EMACLPICSRREG]{LPIEN}
\item Detect the LPI\_INT interrupt, and wait for  \hyperref[regdesc:EMACLPICSRREG]{TLPIEX} to be set, which indicates that the transmitter has exited the LPI state
\end{itemize}

\subsection{RX Entering and Exiting the LPI State}
\begin{itemize}
\item Detect the LPI\_INT interrupt, and wait for \hyperref[regdesc:EMACLPICSRREG]{RLPIEN} to be set, which indicates that the receiver has entered the LPI state
\item Detect the LPI\_INT interrupt, and wait for \hyperref[regdesc:EMACLPICSRREG]{RLPIEX} to be set, which indicates that the receiver has exited the LPI state
\end{itemize}


\section{Interrupts}

\chipname{}'s EMAC can generate the following interrupt signals that will be sent to the \hyperref[mod:intmtrx]{Interrupt Matrix}.

    \begin{itemize}
        \item ETH\_MAC\_INTR
        \item PMT\_INTR
        \item LPT\_INTR
    \end{itemize}

There are several internal interrupt sources from EMAC that can generate the above interrupt signals.

The following interrupt sources can generate the ETH\_MAC\_INTR interrupt signal:
\begin{itemize}
\item \hyperref[regdesc:DMASTATUSREG]{ABN\_INT\_SUMM}: Triggered when \hyperref[regdesc:DMAINENREG]{DMAIN\_AISE} is 1 and the following interrupts are triggered.
    \begin{itemize}
    \item \hyperref[regdesc:DMASTATUSREG]{TRANS\_PROC\_STOP}: Triggered when \hyperref[regdesc:DMAINENREG]{DMAIN\_TSE} is 1 and the transmit process stops.
    \item \hyperref[regdesc:DMASTATUSREG]{TRANS\_JABBER\_TO}: Triggered when \hyperref[regdesc:DMAINENREG]{DMAIN\_TJTE} is 1 and the frame size exceeds 2048 bytes (10240 bytes if the jumbo frame is enabled).
    \item \hyperref[regdesc:DMASTATUSREG]{RECV\_OVFLOW}: Triggered when\hyperref[regdesc:DMAINENREG]{DMAIN\_OIE}
     is 1 and the receive buffer overflows during frame reception.
    \item \hyperref[regdesc:DMASTATUSREG]{TRANS\_UNDFLOW}: Triggered when\hyperref[regdesc:DMAINENREG]{DMAIN\_UIE} is 1 and the transmit buffer underflows during frame transmission.
    \item \hyperref[regdesc:DMASTATUSREG]{RECV\_BUF\_UNAVAIL}: Triggered when \hyperref[regdesc:DMAINENREG]{DMAIN\_RBUE} is 1, the Host owns the Next Descriptor in the Receive List and the DMA cannot acquire it.
    \item \hyperref[regdesc:DMASTATUSREG]{RECV\_PROC\_STOP}: Triggered when \hyperref[regdesc:DMAINENREG]{DMAIN\_RSE} is 1 and the receive process stops.
    \item \hyperref[regdesc:DMASTATUSREG]{RECV\_WDT\_TO}: Triggered when \hyperref[regdesc:DMAINENREG]{DMAIN\_RWTE} is 1 and the Receive Watchdog Timer has expired while receiving the current frame.
    \item \hyperref[regdesc:DMASTATUSREG]{EARLY\_TRANS\_INT}: Triggered when \hyperref[regdesc:DMAINENREG]{DMAIN\_ETIE} is 1 and the frame has been completely transmitted to MTL TX FIFO.
    \item \hyperref[regdesc:DMASTATUSREG]{FATAL\_BUS\_ERR\_INT}: Triggered when \hyperref[regdesc:DMAINENREG]{DMAIN\_FBEE} is 1 and a bus error described in  \hyperref[regdesc:DMAINENREG]{ERROR\_BITS} occurs.
    \end{itemize}

\item \hyperref[regdesc:DMASTATUSREG]{NORM\_INT\_SUMM}: Triggered when \hyperref[regdesc:DMAINENREG]{DMAIN\_NISE} is 1 and the following interrupts are triggered.
    \begin{itemize}
    \item \hyperref[regdesc:DMASTATUSREG]{TRANS\_INT}: Triggered when \hyperref[regdesc:DMAINENREG]{DMAIN\_TIE} is 1 and the transmission completes.
    \item \hyperref[regdesc:DMASTATUSREG]{TRANS\_BUF\_UNAVAIL}: Triggered when \hyperref[regdesc:DMAINENREG]{DMAIN\_TBUE} is 1, the Host owns the Next Descriptor in the Transmit List and the DMA cannot acquire it.
    \item \hyperref[regdesc:DMASTATUSREG]{RECV\_INT}: Triggered when \hyperref[regdesc:DMAINENREG]{DMAIN\_RIE} is 1 and the frame has been received.
    \item \hyperref[regdesc:DMASTATUSREG]{EARLY\_RECV\_INT}: Triggered when \hyperref[regdesc:DMAINENREG]{DMAIN\_ERIE} is 1 and the DMA has received the first buffer of the data packet.
    \end{itemize}

\hyperref[regdesc:DMASTATUSREG]{EMAC\_PMT\_INT}: Triggered when \hyperref[regdesc:EMACINTMASKREG]{PMTINTMASK} is 0, and a magic packet or a remote wakeup frame has been received in power-down mode.

\hyperref[regdesc:DMASTATUSREG]{TS\_TRI\_INT}: Triggered when \hyperref[regdesc:EMACINTMASKREG]{TSINTMASK} is 0 and any of the following conditions is true:
\begin{itemize}
\item  The system time value equals or exceeds the value specified in the Target Time registers
\item There is an overflow in the System Time Seconds Register
\end{itemize}

\hyperref[regdesc:DMASTATUSREG]{EMAC\_LPI\_INT}: Triggered when \hyperref[regdesc:EMACINTMASKREG]{LPIINTMASK} is 0 and  any of the following conditions is true:
\begin{itemize}
\item The MAC Transmitter enters the LPI state
\item The MAC Transmitter exits the LPI state
\item The MAC Receiver enters the LPI state
\item The MAC Receiver exits the LPI state
\end{itemize}
\end{itemize}

The following interrupt sources can generate the PMT\_INTR interrupt signal:
\begin{itemize}
\item \hyperref[regdesc:PMTCSRREG]{RWKPRCVD}: Triggered when a remove wakeup frame is received.
\item \hyperref[regdesc:PMTCSRREG]{MGKPRCVD}: Triggered when a magic packet is received.
\end{itemize}

The following interrupt source can generate the LPI\_INTR interrupt signal:

\begin{itemize}
\item \hyperref[regdesc:EMACLPICSRREG]{RLPIEX}: Triggered when the MAC receiver has stopped receiving the LPI pattern on PHY and has exited the LPI state.
\end{itemize}
