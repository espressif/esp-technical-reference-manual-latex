\chapter{LCD and Camera Controller (LCD\_CAM)}
\label{mod:camlcdctrl}

\section{Overview}

The LCD and Camera controller (LCD\_CAM) on the \chipname{}, consisting of an independent LCD control module and a camera control module, is a versatile component designed to facilitate interfacing with both LCDs and cameras.

The LCD module provides an LCD interface that can connect to RGB, Motorola 6800 (MOTO6800), and Intel 8080 (I8080) compatible LCD devices. The LCD module is responsible for sending video data to the display.

The Camera module provides one parallel camera interface that can connect to external digital video port (DVP) cameras in 8/16-bit modes. It manages communication with the camera to capture images or video frames.

The LCD\_CAM controller offers flexibility for projects that involve both displaying information on screens and working with cameras for image or video processing.

\section{Features}

LCD\_CAM has the following features:

\begin{itemize}
    \item Supports the following operation modes:
    \begin{itemize}
        \item LCD master TX mode
        \item Camera slave RX mode
        \item Camera master RX mode
    \end{itemize}
    \item Supports simultaneous connection to an external LCD and a camera
    \item When interfacing with an external LCD, the following is supported:
    \begin{itemize}
        \item 8/16/24-bit parallel output modes
        \item RGB, MOTO6800, and I8080 LCD formats
        \item LCD data retrieved from internal memory or external memory via GDMA
    \end{itemize}
    \item When interfacing with an external camera (i.e., DVP image sensor), the following is supported:
    \begin{itemize}
        \item 8/16-bit parallel input modes
        \item Camera data stored in internal or external memory via GDMA
    \end{itemize}
    \item Supports interrupts
\end{itemize}

\section{Functional Description}

\subsection{Block Diagram}

Figure \ref{Figure:lcdcam_arch} shows the structure of LCD\_CAM which includes:

\begin{itemize}
    \item One TX control unit (LCD\_Ctrl)
    \item One RX control unit (Camera\_Ctrl)
    \item One asynchronous TX FIFO (Async TX FIFO) for communicating with external devices, e.g., LCDs
    \item One asynchronous RX FIFO (Async RX FIFO) for communicating with external devices, e.g., cameras
    \item Two clock generators (LCD\_Clock Generator and CAM\_Clock Generator) for generating clocks for each module
    \item Two format converters (RGB/YUV Converter) for converting video data into various formats
\end{itemize}

\subsection{LCD\_CAM Module Clocks}\label{sec:lcdcam-module-clk}

\subsubsection{LCD Clock}\label{sec:lcdcam-lcd-clock}

The clocks used in the LCD module are generated from clock sources by the LCD\_Clock Generator, see Figure \ref{fig:lcd_clk}. The clocks include:
\begin{itemize}
    \item Master clock: LCD\_CLK, divided from the clock sources
    \item Pixel clock: LCD\_PCLK, divided from LCD\_CLK
\end{itemize}

\hyperref[fielddesc:HPSYSCLKRSTLCDCLKEN]{HP\_SYS\_CLKRST\_LCD\_CLK\_EN} is used to enable LCD\_CLK\_SRC (the clock source), and \hyperref[fielddesc:HPSYSCLKRSTLCDCLKSRCSEL]{HP\_SYS\_CLKRST\_LCD\_CLK\_SRC\_SEL} is used to select LCD\_CLK\_SRC from one of the following clock sources:
\begin{itemize}
    \item 0: XTAL\_CLK
    \item 1: PLL\_F160M\_CLK
    \item 2: APLL\_CLK
    \item 3: Disable LCD clock source
\end{itemize}

\begin{figure}[H]
    \centering   \fbox{\includegraphics[width=0.8\textwidth]{39-CAMLCDCTRL/figures/lcd_clock.pdf}}
    \caption{LCD Clock}
    \label{fig:lcd_clk}
\end{figure}

The following formula shows the relation between the frequency of LCD\_CLK ($f_{\textrm{LCD\_CLK}}$) and LCD\_CLK\_SRC ($f_{\textrm{LCD\_CLK\_SRC}}$):
$$f_{\textrm{LCD\_CLK}}=\frac{f_{\textrm{LCD\_CLK\_SRC}}}{N + \frac{b}{a}}$$

The values of \textit{N}, \textit{a}, and \textit{b} are related to \hyperref[fielddesc:HPSYSCLKRSTLCDCLKDIVNUM]{HP\_SYS\_CLKRST\_LCD\_CLK\_DIV\_NUM}, \hyperref[fielddesc:HPSYSCLKRSTLCDCLKDIVNUMERATOR]{HP\_SYS\_CLKRST\_LCD\_CLK\_DIV\_NUMERATOR}, and \hyperref[fielddesc:HPSYSCLKRSTLCDCLKDIVDENOMINATOR]{HP\_SYS\_CLKRST\_LCD\_CLK\_DIV\_DENOMINATOR}. Specifically,

\begin{itemize}
    \item When \hyperref[fielddesc:HPSYSCLKRSTLCDCLKDIVNUM]{HP\_SYS\_CLKRST\_LCD\_CLK\_DIV\_NUM} = 0, \textit{N} = 1. The values of \textit{a} and \textit{b} do not take effect. The divider is always 1.
    \item When \hyperref[fielddesc:HPSYSCLKRSTLCDCLKDIVNUM]{HP\_SYS\_CLKRST\_LCD\_CLK\_DIV\_NUM} >=1, \textit{N} = \hyperref[fielddesc:HPSYSCLKRSTLCDCLKDIVNUM]{HP\_SYS\_CLKRST\_LCD\_CLK\_DIV\_NUM} + 1.
    \begin{itemize}
        \item For integer divider, please clear \hyperref[fielddesc:HPSYSCLKRSTLCDCLKDIVNUMERATOR]{HP\_SYS\_CLKRST\_LCD\_CLK\_DIV\_NUMERATOR} and \hyperref[fielddesc:HPSYSCLKRSTLCDCLKDIVDENOMINATOR]{HP\_SYS\_CLKRST\_LCD\_CLK\_DIV\_DENOMINATOR}.
        \item For fractional divider, \textit{b} = \hyperref[fielddesc:HPSYSCLKRSTLCDCLKDIVNUMERATOR]{HP\_SYS\_CLKRST\_LCD\_CLK\_DIV\_NUMERATOR}, \textit{a} = \hyperref[fielddesc:HPSYSCLKRSTLCDCLKDIVDENOMINATOR]{HP\_SYS\_CLKRST\_LCD\_CLK\_DIV\_DENOMINATOR}. The value of \hyperref[fielddesc:HPSYSCLKRSTLCDCLKDIVNUMERATOR]{HP\_SYS\_CLKRST\_LCD\_CLK\_DIV\_NUMERATOR} should be smaller than the value of  \hyperref[fielddesc:HPSYSCLKRSTLCDCLKDIVDENOMINATOR]{HP\_SYS\_CLKRST\_LCD\_CLK\_DIV\_DENOMINATOR}.
    \end{itemize}
\end{itemize}

The following formula shows the relation between the frequencies of LCD\_PCLK ($f_{\textrm{LCD\_PCLK}}$ ) and LCD\_CLK ($f_{\textrm{LCD\_CLK}}$ ):

$$f_{\textrm{LCD\_PCLK}} = \frac{f_{\textrm{LCD\_CLK}}}{\textrm{MO}}$$


MO is determined by  \hyperref[fielddesc:LCDCAMLCDCLKEQUSYSCLK]{LCD\_CAM\_LCD\_CLK\_EQU\_SYSCLK} and  \hyperref[fielddesc:LCDCAMLCDCLKCNTN]{LCD\_CAM\_LCD\_CLKCNT\_N}, specifically:

\begin{itemize}
    \item When  \hyperref[fielddesc:LCDCAMLCDCLKEQUSYSCLK]{LCD\_CAM\_LCD\_CLK\_EQU\_SYSCLK} = 1, MO = 1.
    \item When  \hyperref[fielddesc:LCDCAMLCDCLKEQUSYSCLK]{LCD\_CAM\_LCD\_CLK\_EQU\_SYSCLK} = 0, MO = \hyperref[fielddesc:LCDCAMLCDCLKCNTN]{LCD\_CAM\_LCD\_CLKCNT\_N} + 1.
\end{itemize}

{\bfseries Notes: }
\begin{itemize}
    \item \hyperref[fielddesc:LCDCAMLCDCLKCNTN]{LCD\_CAM\_LCD\_CLKCNT\_N} must not be configured as 0.
    \item Using fractional divider may introduce clock jitters. In case LCD\_CLK and LCD\_PCLK can not be generated from PLL\_F160M\_CLK by integer divider, APLL\_CLK can be used as the clock source. For more information, please refer to Chapter \ref{mod:resclk} \textit{\nameref{mod:resclk}}.
\end{itemize}

\subsubsection{Camera Clock}

The clocks used in the Camera module are generated from clock sources by the CAM\_Clock Generator, see Figure \ref{Figure:camera_clk}. The clocks include:
\begin{itemize}
    \item Master clock: CAM\_CLK, master clock output from the Camera module, divided from the clock sources.
    \item Pixel clock: CAM\_PCLK, clock input from camera slave.
\end{itemize}

\hyperref[fielddesc:HPSYSCLKRSTCAMCLKEN]{HP\_SYS\_CLKRST\_CAM\_CLK\_EN} is used to enable the CAM\_CLK\_SRC (the clock source), and \hyperref[fielddesc:HPSYSCLKRSTCAMCLKSRCSEL]{HP\_SYS\_CLKRST\_CAM\_CLK\_SRC\_SEL} is used to select CAM\_CLK\_SRC from one of the following clock sources:
\begin{itemize}
    \item 0: XTAL\_CLK
    \item 1: PLL\_F160M\_CLK
    \item 2: APLL\_CLK
    \item 3: Disable camera clock source
\end{itemize}

\begin{figure}[H]
    \centering
    \fbox{\includegraphics[width=0.8\textwidth]{39-CAMLCDCTRL/figures/cam_clock.pdf}}
    \caption{Camera Clock}
    \label{Figure:camera_clk}
\end{figure}

The following formula shows the relation between the frequencies of CAM\_CLK ($f_{\textrm{CAM\_CLK}}$ ) and the divider's clock source ($f_{\textrm{CAM\_CLK\_SRC}}$ ):
$$f_{\textrm{CAM\_CLK}}=\frac{f_{\textrm{CAM\_CLK\_SRC}}}{N + \frac{b}{a}}$$

\textit{N} is an integer value between 2 and 256. The value of \textit{N} corresponds to the value of  \hyperref[fielddesc:HPSYSCLKRSTCAMCLKDIVNUM]{HP\_SYS\_CLKRST\_CAM\_CLK\_DIV\_NUM} as follows:

\begin{itemize}
    \item When \hyperref[fielddesc:HPSYSCLKRSTCAMCLKDIVNUM]{HP\_SYS\_CLKRST\_CAM\_CLK\_DIV\_NUM} = 0, \textit{N} = 256.
    \item When \hyperref[fielddesc:HPSYSCLKRSTCAMCLKDIVNUM]{HP\_SYS\_CLKRST\_CAM\_CLK\_DIV\_NUM} = 1, \textit{N} = 2.
    \item When \hyperref[fielddesc:HPSYSCLKRSTCAMCLKDIVNUM]{HP\_SYS\_CLKRST\_CAM\_CLK\_DIV\_NUM} = other value, \textit{N} = \hyperref[fielddesc:HPSYSCLKRSTCAMCLKDIVNUM]{HP\_SYS\_CLKRST\_CAM\_CLK\_DIV\_NUM}.
\end{itemize}

\textit{b} corresponds to the value of \hyperref[fielddesc:HPSYSCLKRSTCAMCLKDIVNUMERATOR]{HP\_SYS\_CLKRST\_CAM\_CLK\_DIV\_NUMERATOR}, and \textit{a} to the value of  \hyperref[fielddesc:HPSYSCLKRSTCAMCLKDIVDENOMINATOR]{HP\_SYS\_CLKRST\_CAM\_CLK\_DIV\_DENOMINATOR}.
\begin{itemize}
    \item For integer divider, please clear \hyperref[fielddesc:HPSYSCLKRSTCAMCLKDIVNUMERATOR]{HP\_SYS\_CLKRST\_CAM\_CLK\_DIV\_NUMERATOR} and \hyperref[fielddesc:HPSYSCLKRSTCAMCLKDIVDENOMINATOR]{HP\_SYS\_CLKRST\_CAM\_CLK\_DIV\_DENOMINATOR}.
    \item For fractional divider, the value of \hyperref[fielddesc:HPSYSCLKRSTCAMCLKDIVNUMERATOR]{HP\_SYS\_CLKRST\_CAM\_CLK\_DIV\_NUMERATOR} should be smaller than the value of \hyperref[fielddesc:HPSYSCLKRSTCAMCLKDIVDENOMINATOR]{HP\_SYS\_CLKRST\_CAM\_CLK\_DIV\_DENOMINATOR}.
\end{itemize}

\subsection{LCD\_CAM Reset} \label{sec:lcdcam_reset}

LCD\_CAM module has the following reset registers that can reset different parts of the module:

\begin{itemize}
    \item \hyperref[fielddesc:HPSYSCLKRSTRSTENLCDCAM]{HP\_SYS\_CLKRST\_RST\_EN\_LCDCAM}: Setting this register will reset the whole LCD\_CAM module.
    \item \hyperref[fielddesc:LCDCAMLCDRESET]{LCD\_CAM\_LCD\_RESET}: Setting this register will reset the whole LCD module, including the LCD control unit (LCD\_Ctrl), the RGB/YUV Converter, and Async TX FIFO. Configuring this register will not affect the Camera module.
    \item \hyperref[fielddesc:LCDCAMCAMRESET]{LCD\_CAM\_CAM\_RESET}: Setting this register will reset the whole Camera module, including the Camera control unit (Camera\_Ctrl), the RGB/YUV Converter, and Async RX FIFO. Configuring this register will not affect the LCD module.
    \item \hyperref[fielddesc:LCDCAMLCDAFIFORESET]{LCD\_CAM\_LCD\_AFIFO\_RESET}: Setting this register will reset the Async TX FIFO without affecting other parts in the module. For example, setting this reset register every time the LCD finishes sending a frame can prevent transmission errors of the previous frame from affecting the transmission of the next frame.
    \item \hyperref[fielddesc:LCDCAMCAMAFIFORESET]{LCD\_CAM\_CAM\_AFIFO\_RESET}: Setting this register will reset the Async RX FIFO without affecting other parts in the module. For example, setting this reset register every time the Camera finishes receiving a frame can prevent transmission errors of the previous frame from affecting the receiving of the next frame.
\end{itemize}

{\bfseries Notes: }
\begin{itemize}
    \item The reset register \hyperref[fielddesc:HPSYSCLKRSTRSTENLCDCAM]{HP\_SYS\_CLKRST\_RST\_EN\_LCDCAM} is not self-clearing, meaning writing 1 enables the reset, and writing 0 releases the reset.
    \item All the above reset registers, except for \hyperref[fielddesc:HPSYSCLKRSTRSTENLCDCAM]{HP\_SYS\_CLKRST\_RST\_EN\_LCDCAM}, are self-clearing, meaning that after writing 1, the hardware will generate a reset pulse and automatically clear the register.
    \item The module clocks LCD\_CLK and CAM\_CLK must be configured first before the module and FIFO are reset.
\end{itemize}

\subsection{LCD\_CAM Data Format Control}
\subsubsection{LCD Data Format Control} \label{lcd_data_control}

When the data is transmitted by the LCD module, it undergoes three stages of processing:

\begin{enumerate}
    \item Stage One: Data preprocessing\\
   The LCD module processes the input data from GDMA and sends it to the YUV-RGB converter.
    \item Stage Two: Data color space conversion\\
    The YUV-RGB converter processes the data. If the YUV-RGB converter is disabled, the data remains unchanged at this stage.
    \item Stage Three: Data post-processing\\
    The LCD module gets data from the YUV-RGB converter for post-processing and then transmits the data to GPIO.
\end{enumerate}

\begin{tiplisting}
    The output data from the previous stage is the input data for the subsequent stage.
\end{tiplisting}

The detailed configuration of each stage is described below.

\textbf{Stage One: Data Preprocessing}

At this stage, configure the following registers to adjust the bit width, bit order, and byte order of the data from GDMA.

\begin{itemize}
    \item \hyperref[fielddesc:LCDCAMLCDBYTEMODE]{LCD\_CAM\_LCD\_BYTE\_MODE}
    \begin{itemize}
        \item 0: The data width of the LCD input from GDMA is 8 bits.
        \item 1: The data width of the LCD input from GDMA is 16 bits.
        \item 2: The data width of the LCD input from GDMA is 24 bits.
    \end{itemize}
    \item \hyperref[fielddesc:LCDCAMLCDBITORDER]{LCD\_CAM\_LCD\_BIT\_ORDER}
    \begin{itemize}
        \item 0: Do not invert.
        \item 1: Invert the input data bit order.
    \end{itemize}
    \item \hyperref[fielddesc:LCDCAMLCDBYTEORDER]{LCD\_CAM\_LCD\_BYTE\_ORDER}
    \begin{itemize}
        \item 0: Do not invert.
        \item 1: Invert data byte order, only valid in 16/24-bit modes.
    \end{itemize}
\end{itemize}

\textbf{Stage Two: Data Color Space Conversion}\\
At this stage, the data undergoes color space conversion. For detailed configurations, please refer to Section \ref{sec:yuvrgb_config}.

\begin{tiplisting}
Users can skip Stage Two by configuring \hyperref[fielddesc:LCDCAMLCDCONVENABLE]{LCD\_CAM\_LCD\_CONV\_ENABLE} as 0, in which case the output of Stage One will directly become the input of Stage Three.
\end{tiplisting}

\textbf{Stage Three: Data Post-processing}\\
At this stage, configure the following registers to adjust the bit and byte order of the output data.

\begin{itemize}
    \item \hyperref[fielddesc:LCDCAMLCDWIREMODE]{LCD\_CAM\_LCD\_WIRE\_MODE}
    \begin{itemize}
        \item 0: The bit width of the data transmitted to GPIO is 8 bits.
        \item 1: The bit width of the data transmitted to GPIO 16 bits.
        \item 2: The bit width of the data transmitted to GPIO is 24 bits.
    \end{itemize}
    \item \hyperref[fielddesc:LCDCAMLCDDOUTBITORDER]{LCD\_CAM\_LCD\_DOUT\_BIT\_ORDER}
    \begin{itemize}
        \item 0: Do not invert.
        \item 1: Invert the output data bit order.
    \end{itemize}
    \item \hyperref[fielddesc:LCDCAMLCDDOUTBYTESWIZZLEENABLE]{LCD\_CAM\_LCD\_DOUT\_BYTE\_SWIZZLE\_ENABLE}
    \begin{itemize}
        \item 0: Disable LCD output data byte reordering.
        \item 1: Enable LCD output data byte reordering by configuring \hyperref[fielddesc:LCDCAMLCDDOUTBYTESWIZZLEMODE]{LCD\_CAM\_LCD\_DOUT\_BYTE\_SWIZZLE\_MODE}. For the reordering results see Table \ref{tab:lcd-byte-swizzle-mode}.
    \end{itemize}
\end{itemize}

\subsection{YUV-RGB Data Format Conversion}\label{sec:yuvrgb_config}

LCD\_CAM is capable of converting data formats between YUV and RGB. The LCD module and Camera module each have a data format converter. The converters support format conversion:
\begin{itemize}
    \item under BT601 and BT709 standards
    \untagged{ESP32-P4-latest}{\item between RGB565 (full/limited range) and YUV422/420/411 (full/limited range) formats}
    \tagged{ESP32-P4-latest}{\item from RGB565 to YUV422/420/411 (full/limited range)}
    \untagged{ESP32-P4-latest}{\item between YUV422/420/411 (full/limited range) formats}

\section{Software Configuration Process}
\begin{tiplisting}
Updating the configurations of the LCD module or the Camera module requires setting the LCD\_CAM\_LCD\_UPDATE and LCD\_CAM\_CAM\_UPDATE respectively to synchronize registers from the APB clock domain to the LCD/camera clock domain. Check the configuration examples below for more details.
\end{tiplisting}
\vspace{-1em}
\subsection{Configure LCD (RGB Format) as TX Mode}

Follow the steps below to configure LCD (RGB format) as TX mode via software:

\begin{enumerate}
    \item Configure clock according to Section \ref{sec:lcdcam-module-clk}.

    \item Configure signal pins according to Table \ref{tab:lcdcam-signal-description} and Chapter \ref{mod:iomuxgpio} \textit{\nameref{mod:iomuxgpio}}.

    \item \label{LCDCAM-TX-INTERRUPT1}Enable corresponding interrupts, see Section \ref{sec:lcdcam-interrupts}.

    \item Enable RGB format by setting \hyperref[fielddesc:LCDCAMLCDRGBMODEEN]{LCD\_CAM\_LCD\_RGB\_MODE\_EN}.

    \item Configure frame structure according to Section \ref{sec:lcd_timing}.

    \item Set \hyperref[fielddesc:LCDCAMLCDUPDATE]{LCD\_CAM\_LCD\_UPDATE} so that the configurations take effect.

    \item Reset TX control unit (LCD\_Ctrl) and Async TX FIFO as described in Section \ref{sec:lcdcam_reset}.

    \item Configure GDMA outlink, see Chapter \ref{mod:vdma} \textit{\nameref{mod:vdma}}.

    \item Start transmitting data:
    \begin{itemize}
        \item Wait till LCD slave gets ready.
        \item Set \hyperref[fielddesc:LCDCAMLCDSTART]{LCD\_CAM\_LCD\_START} to start transmitting data.
    \end{itemize}

    \item Wait for the interrupt signals set in Step \ref{LCDCAM-TX-INTERRUPT1}.

    \item In frame intervals during data transmission, check if \hyperref[fielddesc:LCDCAMLCDNEXTFRAMEEN]{LCD\_CAM\_LCD\_NEXT\_FRAME\_EN} is set:
    \begin{itemize}
        \item If yes, set \hyperref[fielddesc:LCDCAMLCDUPDATE]{LCD\_CAM\_LCD\_UPDATE} (repeating the steps above) to continue transmitting next frame.
        \item If not, LCD stops transmitting data.
    \end{itemize}
    \item Clear \hyperref[fielddesc:LCDCAMLCDSTART]{LCD\_CAM\_LCD\_START} if data transmission is done.
\end{enumerate}

\subsection{Configure LCD (I8080/MOTO6800 Format) as TX Mode}

Follow the steps below to configure LCD (I8080/MOTO6800 format) as TX mode via software:

\begin{enumerate}
    \item Configure clock according to Section \ref{sec:lcdcam-module-clk}.

    \item Configure signal pins according to Table \ref{tab:lcdcam-signal-description} and Chapter \ref{mod:iomuxgpio} \textit{\nameref{mod:iomuxgpio}}.

    \item Enable corresponding interrupts, see Section \ref{sec:lcdcam-interrupts}.

    \item Clear \hyperref[fielddesc:LCDCAMLCDRGBMODEEN]{LCD\_CAM\_LCD\_RGB\_MODE\_EN} to disable RGB format.
    \item Configure CMD phase with \hyperref[fielddesc:LCDCAMLCDCMD]{LCD\_CAM\_LCD\_CMD}, \hyperref[fielddesc:LCDCAMLCDCMD2CYCLEEN]{LCD\_CAM\_LCD\_CMD\_2\_CYCLE\_EN}, and \hyperref[fielddesc:LCDCAMLCDLATTERCMDVALUE]{LCD\_CAM\_LCD\_LATTER\_CMD\_VALUE}.
    \item Configure DUMMY phase with \hyperref[fielddesc:LCDCAMLCDDUMMY]{LCD\_CAM\_LCD\_DUMMY} and \hyperref[fielddesc:LCDCAMLCDDUMMYCYCLELEN]{LCD\_CAM\_LCD\_DUMMY\_CYCLELEN}.
    \item Configure DOUT phase with \hyperref[fielddesc:LCDCAMLCDDOUT]{LCD\_CAM\_LCD\_DOUT}, depending on the output mode:
    \begin{itemize}
        \item In a fixed-length output\hyperref[i2s-note-1]{$^1$}, configure data length in \hyperref[fielddesc:LCDCAMLCDDOUTCYCLELEN]{LCD\_CAM\_LCD\_DOUT\_CYCLELEN}.
        \item In a continuous output\hyperref[i2s-note-2]{$^2$}, set \hyperref[fielddesc:LCDCAMLCDALWAYSOUTEN]{LCD\_CAM\_LCD\_ALWAYS\_OUT\_EN}. Users do not need to configure \hyperref[fielddesc:LCDCAMLCDDOUTCYCLELEN]{LCD\_CAM\_LCD\_DOUT\_CYCLELEN}.
    \begin{tiplisting}
    \vspace{-2em}
    \begin{enumerate}
        \item \label{camlcdctrl-note-1} In a fixed-length output, the LCD module stops sending data once the data length reaches the value set in \hyperref[fielddesc:LCDCAMLCDDOUTCYCLELEN]{LCD\_CAM\_LCD\_DOUT\_CYCLELEN}.
        \item \label{camlcdctrl-note-2} In a continuous output, LCD module keeps sending data till:
            \begin{itemize}
                \item \hyperref[fielddesc:LCDCAMLCDSTART]{LCD\_CAM\_LCD\_START} is cleared;
                \item or \hyperref[fielddesc:LCDCAMLCDRESET]{LCD\_CAM\_LCD\_RESET} is set;
                \item or all the data in GDMA is sent out.
            \end{itemize}
    \end{enumerate}
    \end{tiplisting}
    \end{itemize}

    \item Configure the CD signal mode, including the default value of the CD signal and the values at each phase, see the description of \hyperref[regdesc:LCDCAMLCDMISCREG]{LCD\_CAM\_LCD\_MISC\_REG}.

    \item Set \hyperref[fielddesc:LCDCAMLCDUPDATE]{LCD\_CAM\_LCD\_UPDATE} so that the configurations take effect.

    \item Reset TX control unit (LCD\_Ctrl) and Async TX FIFO as described in Section \ref{sec:lcdcam_reset}.

    \item Configure GDMA outlink, see Chapter \ref{mod:vdma} \textit{\nameref{mod:vdma}}.

    \item Start transmitting data:
    \begin{itemize}
        \item Wait till LCD slave gets ready.
        \item Set \hyperref[fielddesc:LCDCAMLCDSTART]{LCD\_CAM\_LCD\_START} to start transmitting data.
    \end{itemize}

    \item Wait for the interrupt signals set in Step 2.

    \item Clear \hyperref[fielddesc:LCDCAMLCDSTART]{LCD\_CAM\_LCD\_START} if data transmission is done.
\end{enumerate}

\textbf{Notes:}\\
No matter in which format, RGB or I8080/MOTO6800, the rules below must be followed when accessing internal and external memory via GDMA:

\begin{itemize}
    \item If LCD data bus is configured to 8-bit parallel output mode, then
    \begin{itemize}
        \item The pixel clock frequency must be less than 80 MHz.
        \item If YUV-RGB format conversion is being used at the same time, the pixel clock frequency must be less than 60 MHz.
    \end{itemize}
    \item If LCD data bus is configured to 16-bit parallel output mode, then
    \begin{itemize}
        \item The pixel clock frequency must be less than 40 MHz.
        \item If YUV-RGB format conversion is being used at the same time, the pixel clock frequency must be less than 30 MHz.
    \end{itemize}
\end{itemize}

\subsection{Configure Camera as RX Mode}

Follow the steps below to configure the Camera as RX mode via software:

\begin{enumerate}
    \item Configure clock according to Section \ref{sec:lcdcam-module-clk}. Note that in slave mode, the module clock frequency should be twice as fast as the PCLK frequency of the image sensor.

    \item Configure signal pins according to Table \ref{tab:lcdcam-signal-description} and Chapter \ref{mod:iomuxgpio} \textit{\nameref{mod:iomuxgpio}}.

    \item Set or clear  \hyperref[fielddesc:LCDCAMCAMVHDEMODEEN]{LCD\_CAM\_CAM\_VH\_DE\_MODE\_EN} according to the control signal HSYNC.

    \item Set needed RX data format according to Section \ref{sec:cam_data_control}.
    \item Set \hyperref[fielddesc:LCDCAMCAMUPDATE]{LCD\_CAM\_CAM\_UPDATE} so that the configurations take effect.

    \item Reset RX control unit (Camera\_Ctrl) and Async RX FIFO as described in Section \ref{sec:lcdcam_reset}.
    \item Enable corresponding interrupts, see Section  \ref{sec:lcdcam-interrupts}.\label{LCDCAM-RX-INTERRUPT}

    \item Configure GDMA inlink, and set the length of RX data in \hyperref[fielddesc:LCDCAMCAMRECDATABYTELEN]{LCD\_CAM\_CAM\_REC\_DATA\_BYTELEN}.

    \item Start receiving data:
    \begin{itemize}
        \item In master mode, when the slave is ready, set \hyperref[fielddesc:LCDCAMCAMSTART]{LCD\_CAM\_CAM\_START} to start receiving data.
        \item In slave mode, set \hyperref[fielddesc:LCDCAMCAMSTART]{LCD\_CAM\_CAM\_START}. Receiving data starts after the master provides clock signal and control signal.
    \end{itemize}
    \item Receive data and store the data to GDMA. Then corresponding interrupts set in Step \ref{LCDCAM-RX-INTERRUPT} will be generated.

\end{enumerate}

\textbf{Notes:}
\begin{itemize}
    \item No matter in which operation mode, camera master RX mode or camera slave RX mode, the rules below must be followed when accessing internal memory via GDMA:
\begin{itemize}

    \item If 8-bit parallel data input mode is selected, then
        \begin{itemize}
        \item The pixel clock frequency must be less than 80 MHz.
        \item If YUV-RGB format is being used at the same time, the pixel clock frequency must be less than 60 MHz.
    \end{itemize}
    \item If 16-bit parallel data input mode is selected, then
        \begin{itemize}
        \item The pixel clock frequency must be less than 40 MHz.
        \item If YUV-RGB format conversion is being used at the same time, the pixel clock frequency must be less than 30 MHz.
    \end{itemize}
\end{itemize}
\end{itemize}

\begin{tiplisting}
   If both LCD and camera are connected externally at the same time, it is necessary to ensure that the maximum data throughput on the interface is less than the total data bandwidth of GDMA when accessing internal/external storage. The default frequency of APB\_CLK is 80 MHz in this scenario. For more information about APB\_CLK, see Chapter \ref{mod:resclk} \textit{\nameref{mod:resclk}}.
\end{tiplisting}
