\chapter{High-Performance CPU}
\label{mod:riscvcpu}

\section{Overview}
ESP32-P4 is powered by Espressif's second-generation 32-bit CPU core complex based upon RISC-V Instruction Set Architecture (ISA). The CPU Core complex consists of dual RISC-V CPU cores with a dedicated core-local interrupt controller (CLIC), a debug block, and a core-local interrupt (CLINT) timer. Figure \ref{fig:riscvcpu-core-complex-block-diagram} shows the block diagram of the CPU core complex.

\begin{figure}[H]
\centering
\includegraphics[scale=1,width=0.8\textwidth]{\modulefiles/figures/P4_CPU CoreComplex.png}
\caption{CPU Core Complex Block Diagram}
\end{figure}
\label{fig:riscvcpu-core-complex-block-diagram}

Each RISC-V core in CPU core complex uses a 5-stage, in-order, scalar pipeline architecture optimized for area, power, and performance. Each RISC-V core has a built-in dedicated core-local interrupt controller (CLIC) and system bus (SYS BUS) interfaces for memory and peripheral access. The CLINT timer is shared between two cores. The DEBUG block provides a connection between the JTAG interface and the dual cores. It also supports LP core debug. Each core also provides an instruction trace interface for offline debugging.

\section{Features}
\label{sec:riscvcpu-features}
Each RISC-V CPU core in the CPU core complex implements the following standard RISC-V extensions:
\begin{itemize}
    \item Mandatory base integer (I)
    \item Multiplication/Division (M)
    \item Atomic (A)
    \item Floating (F)
    \item Compressed (C)
    \item Code size reduction (Zc)
    \item Bit Manipulation (Zb)
\end{itemize}

In addition to the above standard extensions, each RISC-V CPU core also implements custom instructions (X) which can be broadly classified in the following two categories:
\begin{itemize}
    \item Custom instructions to accelerate AI algorithms performance
    \item Custom hardware loop instructions to further improve performance and reduce code size for iterative AI algorithms execution
\end{itemize}

Apart from the above RISC-V instruction extensions support, each CPU core supports the features below:
\begin{itemize}
    \item RISC-V RV32IMAFCX ISA with a five-stage pipeline that supports an operating clock frequency up to 360 MHz
    \item Compatible with RISC-V Instruction Set Manual, Volume I: RISC-V User-Level ISA, Version 2.2
    \item Compatible with RISC-V Instruction Set Manual, Volume II: RISC-V Privileged Architecture, Version 1.10
    \item Compatible with Core-Local Interrupt Controller (CLIC) RISC-V Privileged Architecture Extensions (10/10/2023)
    \item Compatible with RISC-V External Debug Support Version 0.13.2
    \item Zero wait cycle access to on-chip SRAM and cache for program and data access over IRAM/DRAM interface
    \item 32-bit AHB system bus for peripheral access
    \item Core-local interrupts (CLINT) dedicated for each privilege mode
    \item User (U) privilege mode execution
    \item Interrupt delegation to user mode
    \item Bus error response
    \item Standard physical memory protection (PMP) configurable up to 32 regions and custom attributes (PMA) configurable up to 16 regions for security
    \item Branch target buffer (BTB) with BHT and RAS for performance improvement
    \item Debug module (DM) compliant with the specification RISC-V External Debug Support Version 0.13.2 with external debugger support over an industry-standard JTAG/USB port
    \item Debugger with a direct system bus access (SBA) to memory and peripherals via Core 0 data bus
    \item Support for instruction trace for offline debug
    \item Hardware trigger compliant with the specification RISC-V External Debug Support Version 0.13.2 with up to 3 breakpoints/watchpoints
    \item Fast GPIO support (up to 8)
    \item Configurable events for core performance metrics
\end{itemize}

\subsection{M Extension}
\label{sec:riscvcpu-mext}
\subsubsection{Overview}
\label{sec:riscvcpu-mext-overview}
In addition to base RV32I instruction set, each HP core also implements standard integer multiplication and division extension and is fully compliant with version 2.0 of standard extension "M" specified RISC-V Instruction Set Manual, Volume I: RISC-V User-Level ISA, Version 2.2.

\subsubsection{Functional Description}
\label{sec:riscvcpu-mext-func-descr}

Instructions specified in "M" extension perform multiply and divide operations on the values held in two integer registers. Below are CPU cycle details taken by multiply and divide instructions in the absence of any data hazard in the pipeline.

\begin{itemize}
    \item MUL operation takes 2 cycles
    \item DIV operation takes 1-19 cycles depending on operands
\end{itemize}

\subsection{C Extension}
\label{sec:riscvcpu-cext}

\subsubsection{Overview}
\label{sec:riscvcpu-cext-overview}

Each HP core also implements RISC-V standard compressed instruction set extension named "C" and is fully compliant with version 2.0 of standard extension "C" specified in RISC-V Instruction Set Manual, Volume I: RISC-V User-Level ISA, Version 2.2. RVC is generic term used to specify C extension support. RVC reduces static and dynamic code size by adding short 16-bit instruction encoding for common operations. Typically, 50-60\% of the RISC-V instructions in a program can be replaced with RVC instructions, resulting in a 25-35\% code-size reduction.

\subsubsection{Functional Description}
\label{sec:riscvcpu-cext-func-descr}

RVC uses a simple compression scheme that offers a shorter 16-bit version of 32-bit RISC-V instructions for any of the below scenarios:

\begin{enumerate}
    \item Immediate and address offset is small
    \item One of the registers is a zero register (x0), the ABI lint register (x1), or the ABI stack pointer (x2)
    \item Destination register and the first source register are identical, or
    \item The registers used are the 8 most popular ones i.e. x8-x15.
\end{enumerate}

\subsection{F Extension}
\label{sec:riscvcpu-fpu}

\subsubsection{Overview}
\label{sec:riscvcpu-fpu-overview}

Each HP core implements the RISC-V standard single-precision floating-point instruction set extension, "RV32F" extension in short. The RV32F extension also provides its own register file and various CSRs for configuration and flag checking. Please refer to the RISC-V Instruction Set Manual, Volume I: RISC-V User-Level ISA, Version 2.2, F Standard Extension Version 2.0" for complete information.

\subsubsection{Functional Description}
\label{sec:riscvcpu-fpu-func-descr}

\begin{itemize}
    \item Various instructions to perform floating-point arithmetic operations, conversion/move operations to/from integer format, comparison operations, classification operations, and memory load/store operations.
    \item Single-precision 32-bit floating-point operations compliant with IEEE 754-2008 standard.
    \item Separate 32-bit wide floating-point register file with 32 registers.
    \item Configurable rounding modes.
    \item Flags for checking inexact, overflow, underflow, invalid and divide by zero outcomes.
    \item Subnormal and NaN (not a number) arithmetic
    \item Status bits to control the availability of the extension and track its usage in the context of a program execution.
\end{itemize}

\subsubsection{Initialization}
\label{sec:riscvcpu-fpu-init}

The RV32F extension is disabled by default and must be manually enabled using the \hyperref[fielddesc:MSTATUS_FS]{mstatus.FS} bits. Otherwise, an illegal instruction exception will be generated upon executing any of the instructions, or accessing any of the CSRs, which are part of the RV32F extension.

Please refer to the RISC-V Instruction Set Manual, Volume II: Privileged Architecture, Version 1.10, Extension Context Status in mstatus Register for complete information.

\subsection{Zc (Z) Extension}
\label{sec:riscvcpu-zc}

\subsubsection{Overview}
\label{sec:riscvcpu-zc-overview}

Zc extension is useful for code size reduction. Each HP core implements this extension compatible with \href{https://github.com/riscvarchive/riscv-code-size-reduction/blob/v1.0.4-3/Zc-specification/Zc.adoc}{Zc-specification v1.0.4-3}. The Zc-specification has multiple subset extensions, and the \chipname HP CPU has implemented Zcb, Zcmp, and Zcmt extensions.

\subsubsection{Functional Description}
\label{sec:riscvcpu-zc-func-descr}

\subparagraph{Zcb Extension}

The Zcb extension depends on the C extension and provides more compressed instructions for the following scenarios:

\begin{itemize}
    \item Immediate and address offset is small.
    \item Destination register and the first source register are identical.
    \item The registers used are the 8 most popular ones (x8-x15).
\end{itemize}

\subparagraph{Zcmp Extension}

The Zcmp extension is a set of instructions which may be executed as a series of existing 32-bit RISC-V instructions.

\begin{itemize}
    \item PUSH, POP, POPRET operations used to reduce the size of function prologues and epilogues
        \begin{itemize}
            \item The PUSH operation
                \begin{itemize}
                    \item adjusts the stack pointer to create the stack frame
                    \item pushes (stores) the registers specified in the register list to the stack frame
                \end{itemize}
            \item The POP operation
                \begin{itemize}
                    \item pops (loads) the registers in the register list from the stack frame
                    \item adjusts the stack pointer to destroy the stack frame
                \end{itemize}
            \item The POPRET operation
                \begin{itemize}
                    \item pops (loads) the register in the register list from the stack frame
                    \item cm.popretz also moves zero into a0 as the return value
                    \item adjusts the stack pointer to destroy the stack frame
                    \item executes a ret instruction to return from the function
                \end{itemize}
        \end{itemize}
    \item Move two a0-a1 into two registers of s0-s7
    \item Move two s0-s7 into a0-a1
\end{itemize}

\subparagraph{Zcmt Extension}

The Zcmt extension is also referred to as table jump. It reads the jump target address from the jump table and then jumps to it.

Table jump uses a 256-entry 32-bit wide table in instruction memory to contain function address. The table must be a minimum of 64-byte aligned. It is used as a form of dictionary compression to reduce the code size of jal, auipc+jalr, jr, auipc+jr instructions.

Table jump allows the linker to replace the following instruction sequences with a cm.jalt encoding, and an entry in the table:

\begin{itemize}
    \item 32-bit j calls
    \item 32-bit jal ra calls
\end{itemize}

\subsection{Bit Manipulation (Zb) Extension}
\label{sec:riscvcpu-zb}

\subsubsection{Overview}
\label{sec:riscvcpu-zb-overview}

The bit-manipulation (bitmanip) extension, also known as the Zb extension set, consists of several component extensions to the base RISC-V architecture. These extensions aim to improve code density, performance, and energy efficiency. The HP core fully supports the Zb extension. For Zb extension specifications, please refer to \href{https://github.com/riscv/riscv-b}{Zb-specification v1.0.0}.

\subsubsection{Functional Description}
\label{sec:riscvcpu-zb-func-descr}

Although the bitmanip instructions are designed for general-purpose use, certain instructions are more applicable to specific domains. Hence, the bitmanip extension set is divided into several smaller extensions instead of a single large one. Each extension has its own Zb*-extension name, and comprises instructions that share related functionality, use cases, and often the same implementation logic. Some instructions are available in only one extension, while others are available in several. The instructions have mnemonics and encodings that are independent of the extensions in which they appear. Thus, when implementing extensions with overlapping instructions, there is no redundancy in logic or encoding.

\subsection{A Extension}
\label{sec:riscvcpu-atomic}

\subsubsection{Overview}
\label{sec:riscvcpu-atomic-overview}

Support for atomic (A) extension is available in compliance with RISC-V Instruction Set Manual, Volume I: RISC-V User-Level ISA, Version 2.2, with an emphasis to guarantee forward progress, i.e. any situation that may cause data memory lock for an indefinite amount of time is prevented by their very functionality.

\subsubsection{Functional Description}
\label{sec:riscvcpu-atomic-func-descr}

Atomic (A) Extension contains instructions that atomically read-modify-write memory to support synchronization between multiple RISC-V harts running in the same memory space. The two forms of atomic instruction provided are load-reserved/store-conditional instructions and atomic fetch-and-op memory instructions. The atomic instructions currently ignore the aq (acquire) and rl (release) bits as they are irrelevant to the current architecture, in which memory ordering is always guaranteed.

\subsubsection{Load-Reserve (LR.W) Instruction}
\label{sec:riscvcpu-atomic-lr}

The LR.W instruction simply locks a 32-bit aligned memory address to which the load access is being performed. Once a 4-byte memory region is locked, it will remain locked, i.e. other harts won't be able to access this same memory location, until any of the following scenarios is encountered during execution:

\begin{itemize}
    \item any load operation
    \item any store operation
    \item any interrupts/exceptions
    \item backward jump/taken backward branch
    \item JALR
    \item ECALL/EBREAK/MRET/URET
    \item FENCE/FENCE.I
    \item debug mode
    \item critical section exceeding 64 bytes
    \item data address in SC.W instruction not matching that in LR.W instruction
\end{itemize}

If any of the above happens, except SC.W, the memory lock will be released immediately. If an SC instruction is encountered instead, the lock will be released eventually (not immediately) in the manner described in Section \ref{sec:riscvcpu-atomic-sc}.

If a misaligned address is encountered, it will cause an exception with \hyperref[regdesc:MCAUSE]{mcause} = 6.

\subsubsection{Store-Conditional (SC.W) Instruction}
\label{sec:riscvcpu-atomic-sc}

The SC.W instruction first checks if the memory lock is still valid, and the address is the same as specified during the last LR.W instruction. If so, only then will it perform the store to memory, and later release the lock as soon as it gets an acknowledgement of operation completion from the memory.

On the other hand, if the lock is found to have been invalidated (due to any of the situations as described in Section \ref{sec:riscvcpu-atomic-lr}), it will set a fail code (currently always 1) in the destination register rd.

If a misaligned address is encountered, it will cause an exception with \hyperref[regdesc:MCAUSE]{mcause} = 6.

\subsubsection{AMO Instructions}
\label{sec:riscvcpu-atomic-amo}

An AMO instruction executes in three steps:

\begin{enumerate}
    \item Read data from memory address given by rs1, and save it to destination register rd.
    \item Combine the data in rd and rs2 according to the operation type and keep the result for Step 3 below.
    \item Write the result obtained in Step 2 above to memory address given by rs1.
\end{enumerate}

There are nine different AMO operations: SWAP, ADD, AND, OR, XOR, MAX, MIN, MAXU, and MINU.

During this whole process, the memory address is kept locked from being accessed by other harts. If a misaligned address is encountered, it will cause an exception with \hyperref[regdesc:MCAUSE]{mcause} = 6.

For AMO operations both load and store access faults (PMP/PMA) are checked in the 1st step itself. For such cases \hyperref[regdesc:MCAUSE]{mcause} = 7.

\subsection{Hardware Loop}
\label{sec:riscvcpu-hwloop}

\subsubsection{Overview}
\label{sec:riscvcpu-hwloop-overview}

HWLP (hardware loop) is the implementation of “For” loop in hardware. A loop is configured with a start and an end address and the number of iterations before its execution. When instruction execution encounters the start address, the loop execution starts. When the end address is executed, it is considered as one loop count and the start address is fetched again. This procedure is repeated until the loop count is equal to the number of iterations configured.

\subsubsection{Features}
\label{sec:riscvcpu-hwloop-feature}

HWLP has the following features:

\begin{itemize}
\item Execution with no extra penalty compared to conditional branch instructions
\item Support for two nested HWLPs
\item Configurable using a single instruction (lp.setupi) in case of end address within 1 KB from the configuration instruction
\end{itemize}

\subsubsection{Functional Description}
\label{sec:riscvcpu-hwloop-func-descr}

The configuration of HWLP is done either through CSRs or through custom instructions of HWLP, which will be described in detail in subsequent sections. Any HWLP needs to be configured with a start, an end address, and an iteration count.

When a single loop is to be set up, either Loop0 or Loop1 CSRs can be configured. Before setting up a HWLP, configure \hyperref[regdesc:mhwlp_state_reg]{STATE} bits in \hyperref[sec:riscvcpu-hwloop-reg-desc]{mhwloop\_state\_reg} CSR to any value other than 0 (OFF, default state). Otherwise, accessing HWLP configuration CSRs will cause an illegal instruction exception. To set up and execute HWLP in user mode, \hyperref[regdesc:mhwlp_state_reg]{URW} bit in \hyperref[sec:riscvcpu-hwloop-reg-desc]{mhwloop\_state\_reg} CSR should be set to 1.

The core supports a maximum of two HWLPs, which can be executed in nested fashion. Each loop is configured separately in this case. Note that, the inner loop has higher priority than the outer loop in case the end instruction of both loops is common. Hardware Loop0 has to be considered as the inner loop and Loop1 as the outer loop while configuring nested loops, as Loop0 has a higher priority than Loop1 in hardware implementation.

The advantage of a HWLP over a conditional branch instruction is that it has zero penalty for its execution. Branch/JUMP instructions use prediction logic for evaluating branch condition and branch/jump address. If a miss-prediction happens, the core pipeline will be flushed and the correct address will be fetched, and thus a few cycles have already been wasted in fetch of the incorrect instructions before flush. As the HWLP does not need prediction of result or target address of conditional branch, it has no penalty, and thus has a higher throughput.

\subsubsection{Instructions/Operations/Modes Supported in HWLP}
\label{sec:riscvcpu-hwloop-inst}

\begin{itemize}
    \item Load type instructions
    \item Store type instructions
    \item Arithmetic, logical, AIA instructions
    \item Zc-extension instructions
    \item Any interrupts/exceptions within the HWLP
    \item ECALL/EBREAK
    \item FENCE/FENCE.I/CSRx instructions
    \item Atomic instructions
    \item Debug mode
    \item Data address in SC.W instruction not matching that in LR.W instruction
\end{itemize}

\subsubsection{HWLP Constraints}
\label{sec:riscvcpu-hwloop-constraints}

Please note that branch or jump instructions at the end of HWLP may cause unexpected behavior.

\subsection{Processor Instruction Extension}
\label{sec:riscvcpu-pie}

\subsubsection{Overview}
\label{sec:riscvcpu-pie-overview}

The Processor Instruction Extension (PIE) ISA entails various custom 128-bit wide SIMD (Single Instruction Multiple Data) and complex instructions for accelerating edge inference and digital signal processing applications. These instructions are fully integrated into the CPU pipeline for seamless use with the standard RV32IMAFC ISA with which the HP core is compliant. PIE Extension support is compliant to Espressif's PIE V2.2.0 version.

\subsubsection{Functional Description}
\label{sec:riscvcpu-pie-func-descr}

\begin{itemize}
    \item 8 x 128-bit wide integer vector register files
    \item Each 128-bit wide vector register can be used as 16 x 8-bit elements, 8 x 16-bit elements, or 4 x 32-bit elements for SIMD operations
    \item Up to 128-bit wide memory load/store operations directly to/from the vector registers
    \item Fused instructions for parallel execution of arithmetic and memory operations
    \item Vector and scalar MAC units capable of operating on 16 x 8-bit elements or 8 x 16-bit elements per cycle
    \item 512-bit wide vector register for accumulating results of 8/16-bit vector MAC operations
    \item 40-bit wide scalar register for accumulating results of 8/16-bit scalar MAC operations
    \item Instructions for parallel 16-bit FFT/complex math operations
    \item Final results are generated after rounding and saturation
    \item Configurable rounding modes and scaling (arithmetic right shift) parameter
\end{itemize}

For a complete description of all the available instructions and configurations, please refer to Chapter \ref{mod:pie} \textit{\nameref{mod:pie}}.

\subsubsection{Initialization, Context Switching, and Exceptions}
\label{sec:riscvcpu-pie-init}

The PIE extension is disabled by default and must be manually enabled using the \hyperref[fielddesc:MEXT_PIE_STATE]{mext\_pie\_status.STATE} field. Otherwise, an illegal instruction exception will be generated upon executing any of the instructions that are part of the PIE extension.

To indicate that an illegal instruction exception, i.e. exception with \hyperref[regdesc:MCAUSE]{mcause} = 0x2, has occurred due to disabled PIE extension, the HP core will set the \hyperref[fielddesc:MEXT_ILL_PIE_ILL]{mext\_ill.PIE\_ILL} bit.

When the PIE extension is enabled, i.e. \hyperref[fielddesc:MEXT_PIE_STATE]{mext\_pie\_status.STATE} bits are non-zero, the execution of any PIE instructions would always cause the \hyperref[fielddesc:MEXT_PIE_STATE]{mext\_pie\_status.STATE} bits to change to DIRTY (11).

To enable the PIE extension, follow the steps below:

\begin{enumerate}
    \item Set the \hyperref[fielddesc:MEXT_PIE_STATE]{mext\_pie\_status.STATE} bits to the INITIAL (01) state.
    \item Use the PIE instructions to initialize the PIE register file and accumulator registers to zero, which will cause the \hyperref[fielddesc:MEXT_PIE_STATE]{mext\_pie\_status.STATE} bits to change to DIRTY (11) automatically.
    \item Since the value of the PIE registers is known at this point, set the \hyperref[fielddesc:MEXT_PIE_STATE]{mext\_pie\_status.STATE} bits to CLEAN (10).
    \item Execute the application program.
    \item During execution of the application code, if any PIE instruction is encountered, the \hyperref[fielddesc:MEXT_PIE_STATE]{mext\_pie\_status.STATE} bits will automatically change to DIRTY (11).
\end{enumerate}

During context switching, whether to save/restore the PIE extension registers can be decided based on the value of the \hyperref[fielddesc:MEXT_PIE_STATE]{mext\_pie\_status.STATE} bits. If the value is DIRTY, the state of the PIE register must be saved to the stack. After that, the state would be updated to CLEAN or OFF, depending on whether the next context (thread) has the PIE extension enabled or not.

Some PIE instructions have constraints on the register fields or immediate fields. e.g. for fused instructions which are performing load and arithmetic, with the same destination register for the two parallel operations, it will be considered ambiguous and thus cause a PIE illegal exception with \hyperref[regdesc:MCAUSE]{mcause} = 0x1f.

\label{sec:riscvcpu-int-ctrl}

\subsection{Overview}
\label{sec:riscvcpu-intr-overview}

Each HP core uses the CLIC + CLINT scheme for implementing and supporting interrupts.

CLINT stands for core-local interrupts, which are sources local to the HP core subsystem for generating timer and software interrupts for each HP core.

CLIC stands for core-local interrupt controller, which provides low-latency, vectored, pre-emptive interrupts for each HP core. Each HP core CLIC unit supports 32 external interrupts and 2 CLINT interrupts.

\subsection{CLIC}
\label{sec:riscvcpu-clic}

\subsubsection{Overview}
\label{sec:riscvcpu-clic-overview}

The HP core CLIC implementation follows the official specification: \href{https://github.com/riscv/riscv-fast-interrupt/blob/v0.10/src/clic.adoc}{Core-Local Interrupt Controller (CLIC) RISC-V Privileged Architecture Extensions (10/10/2023)}.

Only the CLIC mode of operation is supported by the HP core, i.e. \hyperref[fielddesc:MTVECMODE]{mtvec.MODE} is hardwired to 0x3. Hence, the basic RISC-V interrupt handling scheme and the associated CSRs (such as mie, mip, mideleg, uie, and uip) are unavailable.

HP core supports machine mode interrupt as well as user mode interrupt delegation.

\subsubsection{CLIC CSRs}
\label{sec:riscvcpu-clic-csrs}

The following machine mode CSRs are relevant for interrupt handling in the CLIC scheme:

\begin{multicols}{4}
\begin{itemize}
\item \hyperref[regdesc:MSTATUS]{mstatus}
\item \hyperref[regdesc:MTVEC]{mtvec}
\item \hyperref[regdesc:MTVT]{mtvt}
\item \hyperref[regdesc:MSCRATCH]{mscratch}
\item \hyperref[regdesc:MEPC]{mepc}
\item \hyperref[regdesc:MCAUSE]{mcause}
\item \hyperref[regdesc:MNXTI_REG]{mnxti}
\item \hyperref[regdesc:MINTTHRESH]{mintthresh}
\item \hyperref[regdesc:MINTSTATUS]{mintstatus}
\item \hyperref[regdesc:MCLICBASE_REG]{mclicbase}
\item \hyperref[regdesc:MSCRATCHCSW_REG]{mscratchcsw}
\item \hyperref[regdesc:MSCRATCHCSWL_REG]{mscratchcswl}
\end{itemize}
\end{multicols}


As mentioned earlier, the CLIC mode is decided by the \hyperref[fielddesc:MTVECMODE]{mtvec.MODE} field. In each HP core, these bits are hardwired to 0x3, indicating that only the CLIC mode is supported.

In CLIC mode, the global interrupt enable for machine mode is still dictated by the \hyperref[fielddesc:MSTATUSMIE]{mstatus.MIE} bit.

interrupt can be either vectored or non-vectored depending on the \hyperref[fielddesc:MCLICINTATTR_i_SHV]{clicintattr[i].SHV} bit.

For a non-vectored interrupt, the HP core jumps to the trap handler base address as configured in \hyperref[fielddesc:MTVECBASE]{mtvec.BASE} field.

For a vectored interrupt, the HP core jumps to the 32-bit word address stored at the corresponding index in the vector table. For example, for the "i"th machine mode interrupt, first the 32-bit address stored at the memory location "mtvt + 4*i" will be loaded, after which the HP core will treat the loaded data as an address and jump to it. The base address of the vector table is configured in \hyperref[regdesc:MTVT]{mtvt}.

Upon taking an interrupt, the HP core will store the address of the instruction that was interrupted in \hyperref[regdesc:MEPC]{mepc}. Also, the HP core will update the value of \hyperref[regdesc:MCAUSE]{mcause} based on the interrupt's ID.

\hyperref[fielddesc:MCLICCFG_MNLBITS]{mcliccfg.MNLBITS} decides the number of higher bits in the \hyperref[regdesc:MCLICINTCTL_REG_i]{clicintctl[i]} registers which are to be used to encode the level (indicated as \texttt{'l'} in the above table), while the remaining lower bits are used to encode the priority (indicated as \texttt{'p'} in the above table). \subsubsection{Timer Counter and Interrupt}

Pending state of timer interrupt is reflected as IP bit of clicintip register of interrupt 7 in CLIC.

For de-asserting the pending timer interrupt in M mode, either IE bit of clicintie register of interrupt 7 in CLIC has to be cleared or the value of the \hyperref[regdesc:CLINTMTIMECMPLOREG]{mtimecmplo} and \hyperref[regdesc:CLINTMTIMECMPHIREG]{mtimecmphi} register needs to be updated.

\subsection{Standard Physical Memory Protection}
\label{sec:riscvcpu-pmp}
\subsubsection{Overview}
\label{sec:riscvcpu-pmp-overview}
The CPU core includes a physical memory protection (PMP) unit, which is fully compliant with RISC-V Instruction Set Manual, Volume II: Privileged Architecture, Version 1.10. It can be used by software to set memory access privileges (read, write and execute permissions) for required memory regions. In addition to standard PMP checks, the CPU core also implements custom physical memory attributes (PMA) checkers to provide additional permission checks based on pre-defined attributes.

\subsubsection{Features}
\label{sec:riscvcpu-pmp-features}

The PMP unit can be used to restrict access to physical memory. The main features are:

\begin{itemize}
    \item Up to 32 PMP entries
    \item Minimum address granularity of 128 bytes
    \item Three address matching modes: OFF, TOR (Top of Range), and NAPOT (Naturally Aligned Power-of-Two). NA4 (Naturally Aligned 4-Byte Region) address matching mode is not supported
    \item Permission controls for read, write, and execute
    \item Lock function for each PMP entry
    \item Maximum physical space address of 4 GB
\end{itemize}

\subsubsection{Functional Description}
\label{sec:riscvcpu-pmp-func-descr}

Software can program the PMP unit's configuration and address registers in order to contain faults and support secure execution. PMP CSRs can only be programmed in machine mode. Once the PMP unit is enabled by configuring PMP CSRs, write, read and execute permission checks are applied to all the accesses in user mode as per programmed values of the enabled 16 pmpcfgX and pmpaddrX registers (refer to Section \ref{sec:riscvcpu-pmp-reg-summ}).

By default, PMP grants permission to all accesses in machine mode and revokes permission of all access in user mode. This implies that it is mandatory to program the address range and valid permissions in pmpcfg and pmpaddr registers (refer to Section \ref{sec:riscvcpu-pmp-reg-summ}) for any valid access to pass through in user mode. However, it is not required for machine mode as PMP permits all accesses to go through by default. In cases where PMP checks are also required in machine mode, software can set the lock bit of required PMP entry to enable permission checks on it. Once the lock bit is set, it can only be cleared through CPU reset.

When any instruction is being fetched from a memory region without execute permissions, an exception is generated at processor level and exception cause is set as instruction access fault in \hyperref[regdesc:MCAUSE]{mcause} CSR. Similarly, any load/store access without valid read/write permissions will result in an exception generation with \hyperref[regdesc:MCAUSE]{mcause} updated as load access and store access fault respectively. In case of load/store access faults, violating address is captured in \hyperref[regdesc:MTVAL]{mtval} CSR.

\subsection{Custom Physical Memory Attribute (PMA) Checker}
\label{sec:riscvcpu-pma}
\subsubsection{Overview}
\label{sec:riscvcpu-pma-overview}
CPU core also implements a custom physical memory attributes checker (PMAC) to provide additional permission checks based on pre-defined memory types configured through custom CSRs. Please note that the PMA checks are applied irrespective of the core's privilege mode, i.e., it doesn't differentiate between machine and user mode.


\subsubsection{Features}
\label{sec:riscvcpu-pma-features}

PMAC supports below features:

\begin{itemize}
    \item 16 PMA entries
    \item Configurable memory type for defined memory regions
    \item Minimum address granularity of 128 bytes
    \item Three address matching mode: OFF, TOR (Top of Range) and NAPOT (Naturally Aligned Power-of-Two). NA4 (Naturally Aligned 4-Byte Region) address matching mode is not supported
    \item Permission controls for read, write and execute
    \item Lock function for each PMA entry
    \item Maximum physical space address of 4 GB
    \item Configurable attribute for defined memory regions
\end{itemize}

\subsubsection{Functional Description}
\label{sec:riscvcpu-pma-func-descr}

Software can program the PMAC unit's configuration and address registers in order to avoid faults due to access to invalid memory regions. PMAC CSRs can only be programmed in machine mode. Once enabled, write, read and execute permission checks are applied to all the accesses irrespective of privilege mode as per programmed values of enabled 16 pma\_cfgX and pma\_addrX registers (refer to Section \ref{sec:riscvcpu-pma-reg-summ}). Access to entries marked as invalid memory types will result in fetch fault or load/store fault exception, as the case may be.

Exception generation and handling for PMAC related faults will be handled in similar way to PMP checks. When any instruction is being fetched from a memory region configured as a null or an invalid memory region, an exception is generated at the processor level and the exception cause is set as instruction access fault in \hyperref[regdesc:MCAUSE]{mcause} CSR. Similarly, any load/store access to a null or an invalid memory region will result in an exception generation with \hyperref[regdesc:MCAUSE]{mcause} updated as load access or store access fault respectively. In case of load/store access faults, violating address is captured in \hyperref[regdesc:MTVAL]{mtval} CSR. For the PMAC entries configured as valid memory, the handling is same as for PMP checks.

A lock bit per entry is also provided in case software wants to disable programming of PMAC registers. Once the lock bit in any pma\_cfgX register is set, respective pma\_cfgX and pma\_addrX registers can not be programmed further, unless a CPU reset cycle is applied.

A 4-bit field in PMAC CSRs is also provided to define attributes for memory regions. These bits are not used internally by the CPU core for any purpose. Based on address match, these attributes are provided on load/store interface as side-band signals and are used by cache controller block for its internal operation.

\subsection{Debug}
\label{sec:riscvcpu-dbg}

\subsubsection{Overview}
\label{sec:riscvcpu-dbg-overview}

This section describes how to debug software running on HP and LP CPU cores. Debug support is provided through standard JTAG pins and complies to the \href{https://riscv.org/wp-content/uploads/2019/03/riscv-debug-release.pdf}{RISC-V External Debug Support Version 0.13.2} specification.

HP cores and LP core have their own dedicated JTAG DTM (debug transport module) in order to connect with respective Debug Modules. Both these JTAG DTM's are daisy-chained to provide debugger access to respective cores via a single JTAG interface. Depending on its debug requirements, Debugger can keep the unused DTM in a bypass state while debugging the required core. Both HP cores can be debugged through a single debug module (DM) connected to JTAG DTM (HP Cores), while the LP core is debugged through a separate debug module (DM) connected to JTAG DTM (LP Core).

Please note it is not possible to debug HP cores and LP core at same time as they are accessed through separate debug modules.

Figure \ref{fig:riscvcpu-dbg-sys-overview} below shows the main components of External Debug Support.

\begin{figure}[H]
\centering
\includegraphics[width=0.8\textwidth,keepaspectratio]{\modulefiles/figures/debug_p4.png}
\caption{Debug System Overview}\label{fig:riscvcpu-dbg-sys-overview}
\end{figure}

This section focuses on HP core debug features. For LP core debug features, please refer to Chapter \ref{mod:ulp} \textit{\nameref{mod:ulp}}.

The user interacts with the Debug Host (e.g. laptop), which is running a debugger (e.g. GDB). The debugger communicates with a Debug Translator (e.g. OpenOCD, which may include a hardware driver) to communicate with Debug Transport Hardware (e.g. ESP-Prog adapter). The Debug Transport Hardware connects the Debug Host to the HP core's Debug Transport Module (DTM) through standard JTAG interface (bypassing LP core DTM). The DTM provides access to the Debug Module (DM) using the Debug Module Interface (DMI).

The DM allows the debugger to halt selected HP cores. Abstract commands provide access to GPRs (general-purpose registers). The Program Buffer allows the debugger to execute arbitrary code on the core, which allows access to additional CPU core state. Alternatively, additional abstract commands can provide access to additional CPU core state.

Each HP core contains Trigger Module supporting 3 triggers. When trigger conditions are met, respective core will halt spontaneously and inform the debug module that they have halted.

System bus access block allows memory and peripheral register access without affecting either HP core.


\subsubsection{Features}
\label{sec:riscvcpu-dbg-features}

The HP CPU supports the following basic debugging features for each HP core:

\begin{itemize}
    \item Provides necessary information about the implementation to the debugger
    \item Allows the core to be halted and resumed
    \item Core registers (including CSRs) can be read/written by debugger
    \item Core can be debugged from the first instruction executed after reset
    \item Core can be reset through debugger
    \item Core can be halted on software breakpoint (planted breakpoint instruction)
    \item Hardware single-stepping
    \item Execute arbitrary instructions in the halted core by means of the program buffer. 2-word program buffer is supported
    \item System bus access is supported through word aligned address access
    \item Supports three Hardware Triggers (can be used as breakpoints/watchpoints) per HP core as described in Section \ref{sec:riscvcpu-hwt}
    \item Supports LP core debug through daisy chaining of its JTAG DTM
\end{itemize}

\subsection{Hardware Trigger}
\label{sec:riscvcpu-hwt}

\subsubsection{Overview}
\label{sec:riscvcpu-hwt-overview}

\subsubsection{Features}
\label{sec:riscvcpu-hwt-features}

Each HP core implements a hardware trigger block to provide breakpoint and watchpoint capability for debugging. Each trigger has the following features:

\begin{itemize}
    \item Three independent trigger units, each of which can be configured for matching the address of the program counter or load-store accesses
    \item Preempting execution by causing a breakpoint exception
    \item Halting execution and transferring control to debuggers
    \item Support for NAPOT (naturally aligned power-of-two regions) address encoding
\end{itemize}

\subsubsection{Functional Description}
\label{sec:riscvcpu-hwt-func-descr}

The Hardware Trigger module provides seven CSRs, which are listed under Section \hyperref[sec:riscvcpu-hwt-reg-summ]{register summary}. Among these, \hyperref[regdesc:TDATA1]{tdata1} and \hyperref[regdesc:TDATA2]{tdata2} are abstract CSRs, which means they are shadow registers for accessing internal registers for each of the four trigger units, one at a time.

To choose a particular trigger unit write the index (0-2) of that unit into \hyperref[regdesc:TSELECT]{tselect} CSR. When tselect is written with a valid index, the abstract CSRs \hyperref[regdesc:TDATA1]{tdata1} and \hyperref[regdesc:TDATA2]{tdata2} are automatically mapped to reflect internal registers of that trigger unit. Each trigger unit has two internal registers, namely \hyperref[regdesc:MCONTROL]{mcontrol} and \hyperref[regdesc:MADDRESS]{maddress}, which are mapped to tdata1 and tdata2, respectively.

Writing larger than allowed indexes to tselect will clip the written value to the largest valid index, which can be read back. This property may be used for enumerating the number of available triggers during initialization or when using a debugger.

Since software or debugger may need to know the type of the selected trigger to correctly interpret tdata1 and tdata2, the 4 bits (31-28) of tdata1 encodes the type of the selected trigger. This type field is read-only and always provides a value of 0x2 for every trigger, which stands for match type trigger, hence, it is inferred that tdata1 and tdata2 are to be interpreted as mcontrol and maddress. The information regarding other possible values can be found in the specification, RISC-V External Debug Support Version 0.13.2, but this trigger module only supports type 0x2.

Once a trigger unit has been chosen by writing its index to tselect, it will become possible to configure it by setting the appropriate bits in mcontrol CSR (tdata1) and writing the target address to maddress CSR (tdata2).

Each trigger unit can be configured to either cause breakpoint exception or enter debug mode, by writing to the action field of mcontrol. This bit can only be written from debugger, thus by default a trigger, if enabled, will cause breakpoint exception.

mcontrol for each trigger unit has a \hyperref[fielddesc:MCONTROLHIT]{hit} bit which may be read, after CPU halts or enters exception, to find out if this was the trigger unit that fired. This bit is set as soon as the corresponding trigger fires, but it has to be manually cleared before resuming operation. Although, failing to clear it does not affect normal execution in any way.

Each trigger unit only supports match on address, although this address could either be that of a load/store access or the virtual address of an instruction. The address and size of a region are specified by writing to maddress (tdata2) CSR for the selected trigger unit. Larger than 1 byte region sizes are specified through NAPOT (naturally aligned power-of-two) encoding (see Table \ref{tab:riscvcpu-hwt-table}) and enabled by setting match bit in mcontrol. Note that for NAPOT encoded addresses, by definition, the start address is constrained to be aligned to (i.e. an integer multiple of) the region size. The maximum size supported is 128 bytes.

\begin{longtable}{ | c | c | r | }
\caption{NAPOT encoding for maddress} \label{tab:riscvcpu-hwt-table}
\\ \hline
\rowcolor{lightgray}
maddress(31-0) & Start Address & Size (bytes) \\ \hline
aaa...aaaaaaaaa0 & aaa...aaaaaaaaa0 & $2$ \\ \hline
aaa...aaaaaaaa01 & aaa...aaaaaaaa00 & $4$ \\ \hline
aaa...aaaaaaa011 & aaa...aaaaaaa000 & $8$ \\ \hline
aaa...aaaaaa0111 & aaa...aaaaaa0000 & $16$ \\ \hline
\multicolumn{3}{c}{....} \\ \hline
aaa...0111111111 & aaa...aaa0000000 & $2^{8}$ \\ \hline
\end{longtable}

\hyperref[regdesc:TCONTROL]{tcontrol} CSR is common to all trigger units. It is used for preventing triggers from causing repeated exceptions in machine mode while execution is happening inside a trap handler. This also disables breakpoint exceptions inside ISRs by default, although, it is possible to manually enable this right before entering an ISR, for debugging purposes. This CSR is not relevant if a trigger is configured to enter debug mode.

\subsubsection{Trigger Execution Flow}
\label{sec:riscvcpu-hwt-suggest-op}

When hart is halted and enters debug mode due to the firing of a trigger (\hyperref[fielddesc:MCONTROLACTION]{action} = 1):
\begin{itemize}
    \item \hyperref[regdesc:DPC]{dpc} is set to current PC (in decode stage)
    \item cause field in \hyperref[regdesc:DCSR]{dcsr} is set to 2, which means halt due to trigger
    \item \hyperref[fielddesc:MCONTROLHIT]{hit} bit is set to 1, corresponding to the trigger(s) which fired
\end{itemize}

When hart goes into trap due to the firing of a trigger (\hyperref[fielddesc:MCONTROLACTION]{action} = 0) :
\begin{itemize}
    \item \hyperref[regdesc:MEPC]{mepc} is set to current PC (in decode stage)
    \item \hyperref[regdesc:MCAUSE]{mcause} is set to 3, which means breakpoint exception
    \item \hyperref[fielddesc:TCONTROLMPTE]{mpte} is set to the value in \hyperref[fielddesc:TCONTROLMTE]{mte} right before trap
    \item \hyperref[fielddesc:TCONTROLMTE]{mte} is set to 0
    \item \hyperref[fielddesc:MCONTROLHIT]{hit} bit is set to 1, corresponding to the trigger(s) which fired
\end{itemize}

\textit{Note: If two different triggers fire at the same time, one with action = 0 and another with action = 1, then hart is halted and enters debug mode.}

\subsection{Trace}
\label{sec:riscvcpu-trace}

\subsubsection{Overview}
\label{sec:riscvcpu-trace-overview}
In order to support non-intrusive software debug, the CPU core provides an instruction trace interface which provides relevant information for offline debug purpose. This interface provides relevant information to Trace Encoder block, which compresses the information and stores in memory allocated for it. Software decoders can read this information from trace memory without interrupting the CPU core and re-generate the actual program execution by the CPU core.


\subsubsection{Features}
\label{sec:riscvcpu-trace-features}

The CPU core supports instruction trace feature and provides below information to Trace Encoder as mandated in RISC-V Processor Trace Version 1.0:

\begin{itemize}
    \item Number of instructions being retired.
    \item Occurrence of exception and interrupt along with cause and trap values.
    \item Current privilege level of hart.
    \item Instruction type of retired instructions for jumps, branches and return.
    \item Instruction address for instructions retired before and after program counter changes.
    \item Trace enabling on trigger hit as per \hyperref[regdesc:MCONTROL]{action} bit in \hyperref[regdesc:MCONTROL]{mcontrol}.
\end{itemize}


\subsubsection{Functional Description}
\label{sec:riscvcpu-trace-func-descr}

Each HP core implements mandatory instruction delta tracing, also known as branch tracing. It works by tracking execution from a known start address by sending information about deltas taken by a program. Deltas are typically introduced by jump, call, return, branch type instructions and also by interrupts and exceptions. All such deltas along with additional details about cause and actual instructions/addresses are communicated over high bandwidth instruction trace interface output from the core. Trace Encoder operates on the information on this trace interface and compresses the information for storage in memory for offline debug by a decoder. More information about the encoding is available in Chapter \ref{mod:riscvtracenc} \textit{\nameref{mod:riscvtracenc}}.

The core does not have any internal registers to provide control over instruction trace interface. All register controls are available in \ref{mod:riscvtracenc} \textit{\nameref{mod:riscvtracenc}} block.

\subsection{Branch Prediction}
\label{sec:riscvcpu-perf-bp}

\subsubsection{Overview}
\label{sec:riscvcpu-perf-bp-overview}
A core does branch prediction in terms of branch is taken or not taken and the target address of conditional/unconditional branch type instruction prior to its execution. Based on the prediction, the core starts fetching from the respective PC. Then, it validates the prediction on the execution of the instruction. If it is found to be miss-predicted, then the core flushes the pipeline and starts fetching from the correct PC. The prediction improves the throughput as the core continues fetching instructions based on prediction instead of waiting till the execution of branch instruction. The details about the prediction technique used by the core are described in the following section.

\subsubsection{Features}
\label{sec:riscvcpu-perf-bp-features}
\begin{itemize}
    \item Prediction of both results and target address of conditional/unconditional branch type instructions
    \item Branch target prediction of instructions including BEQ, BNE, BLT, BLTU, BGE, BGEU, C.BEQZ, C.BNEZ, JAL, J, CJ
    \item Conditional branch prediction includes instructions such as: BEQ, BNE, BLT, BLTU, BGE, BGEU, C.BEQZ, C.BNEZ
    \item Uses 2-bit saturating counter for more accuracy
    \item Uses global history of conditional branches for prediction which improves performance
    \item Target prediction in one clock cycle for unconditional branch, that is, 0 cycle penalty when prediction or speculation is correct
\end{itemize}

\subsubsection{Functional Description}
\label{sec:riscvcpu-perf-bp-func-descr}

The core uses Branch History Table (BHT) along with Branch Target Buffer (BTB) for prediction of branch and jump type instructions.

BHT is 512x16-bit SRAM memory for each core. It is indexed by Virtual Branch History Register (VBHR) while performing prediction and by Global Branch History Register (GBHR) while updating or correcting the result (taken or not taken) on execution of branch-type instructions. GBHR contains the results of the last 13 branch-type instructions executed. VBHR contains the results of the last 12 branch-type instructions and predicted results of the current branch instruction. Thus, both registers hold the trajectory of previously executed branches. Based on 13 bits of VBHR and GBHR, 2 bits of saturation counter value stored in BHT are either read or written respectively.

As shown in Figure \ref{fig:riscvcpu-bp_sat-scheme}, 2-bit saturation counter indicates if a branch is：

\begin{itemize}
    \item weakly not taken (00)
    \item strongly not taken (01)
    \item weakly taken (10)
    \item strongly taken (11)
\end{itemize}

If the result of prediction is weakly or strongly not taken, then the branch is predicted as not taken. Similar is the case for weakly/strongly taken.

While updating BHT with the actual result of branch execution, the following transition logic is used to update the respective saturation counter.

BTB stores the 32-bit branch target address. It can hold a maximum of 16 entries of target addresses. It is indexed by the last 16 bits of the PC of conditional/unconditional branch-type instruction. Thus, the core predicts the result of branch-type instructions based on BHT and jumps to the target obtained from BTB if the branch is predicted as taken. For unconditional branches, the core directly jumps to the target obtained from BTB, as it does not need prediction in this case. When the actual result and the target of branch-type instructions are obtained during execution, the core takes a corrective jump by flushing the pipeline in case of misprediction. BTB and BHT entries are corrected accordingly.

Figure \ref{fig:riscvcpu-bp_sat-scheme} shows 2-bit saturation scheme details used in each HP core.

\end{figure}

\subsection{RAS}
\label{sec:riscvcpu-perf-ras}

\subsubsection{Overview}
\label{sec:riscvcpu-perf-ras-overview}
Return Address Stack (RAS) is a stack maintained by the core to push the return addresses of jump and link (JAL/JALR) type instructions. Thus, the core can jump to the return address as soon as the RET instruction is pre-decoded, enhancing the performance of function call return.

\subsubsection{Features}
\label{sec:riscvcpu-perf-ras-features}

\begin{itemize}
    \item Increased core performance by avoiding pipeline stall on return of function call
    \item Able to store up to four levels of function call nesting
\end{itemize}


\subsubsection{Functional Description}
\label{sec:riscvcpu-perf-ras-func-descr}
When JAL/JALR/C.JALR instruction is pre-decoded by the core, then the 32-bit PC of the next instruction is pushed to RAS.

The address is popped out from RAS when the return type instruction (JALR, C.JR, C.JALR) is pre-decoded by the core. The core then starts fetching from the popped address. On execution of the return instruction, if the popped address does not match the address stored in the link register, then the core flushes the pipeline and starts to fetch from the address in the link register. RAS can store a maximum of four entries of return addresses, that is, four levels of function call nesting are supported.

RAS prediction depends on the following conditions:

\begin{longtable}{ | c | c | c | }
\hline\rowcolor{lightgray}
Target Register (Rd) & Source Register (Rs) & RAS action \\ \hline
\endfirsthead
\hline
rd != x1 & rs1 != x1 & No action \\ \hline
rd != x1 & rs1 == x1 & POP \\ \hline
rd == x1 & rs1 != x1 & PUSH \\ \hline
rd == x1 & rs1 == x1 & PUSH and POP \\ \hline

\subsection{Bus Error Response}
\label{sec:riscvcpu-bus-err-resp}
\subsubsection{Overview}
\label{sec:riscvcpu-buserr-overview}

In the HP core, load/store bus errors are handled as asynchronous exceptions, which means that by the time such an exception is detected, the CPU may have executed some of the instructions after the actual load/store instruction that caused the exception.

Due to the asynchronous nature of these exceptions, handling them requires more context than that available via the standard CSRs \hyperref[regdesc:MCAUSE]{mcause}, \hyperref[regdesc:MEPC]{mepc} and \hyperref[regdesc:MTVAL]{mtval}, which are otherwise sufficient for synchronous exceptions. Therefore, additional CSRs have been provided to allow software to gain all the necessary context required to handle and resume a program that has encountered an asynchronous bus-error exception.

In the following sections, these bus-error-specific CSRs are described and the recommended approach is to gain context about a bus-error exception.

\subsubsection{Functional Description}
\label{sec:riscvcpu-buserr-func-descr}

There are two ways to handle load/store bus errors:

\begin{itemize}
    \item Synchronous bus-errors mode: \\
    When this mode is enabled, (by setting \hyperref[fielddesc:MHINT_SBE]{mhint.SBE} bit) bus-error exceptions can be handled just like other synchronous exceptions. Therefore, execution traps at the exact PC corresponding to the bus error causing load/store instruction, and \hyperref[regdesc:MEPC]{mepc} holds the same PC.\\
    The side effect of enabling this mode is that it increases the latency of all load/store instructions.
    \item Asynchronous bus-errors mode (default):
    \begin{itemize}
        \item For handling bus errors in asynchronous mode, it is recommended (although not necessary) to clear the \hyperref[fielddesc:MEXSTATUS_NMFT]{mexstatus.NMFT} bit at startup.\\
        \item In asynchronous mode, bus errors cause the CPU to enter trap handler with MCAUSE value 5 or 7 (depending upon the type of operation which caused the error), and \hyperref[regdesc:MEPC]{mepc} will hold the PC of the last instruction that was about to be executed when CPU detected the bus error(s) due to (one or more) previously executed load/store instruction(s).\\
         \item Upon entering the exception handler for cause 5 or 7, the \hyperref[fielddesc:MEXSTATUS_BUS_ERR]{mexstatus.BUS\_ERR} bit must be checked to confirm if it is due to load/store bus-error. If so, this bit must be cleared to acknowledge.\\
         \item Next, bits \hyperref[fielddesc:LDPC0_VLD]{ldpc0.vld}, \hyperref[fielddesc:LDPC1_VLD]{ldpc1.vld}, \hyperref[fielddesc:STPC0_VLD]{stpc0.vld}, \hyperref[fielddesc:STPC1_VLD]{stpc1.vld} and \hyperref[fielddesc:STPC2_VLD]{stpc2.vld} must be checked to confirm if they are valid. If so, these must be cleared, and the corresponding address fields (\hyperref[fielddesc:LDPC0_ADDR]{ldpc0.addr}, \hyperref[fielddesc:LDPC1_ADDR]{ldpc1.addr}, \hyperref[fielddesc:STPC0_ADDR]{stpc0.addr}, \hyperref[fielddesc:STPC1_ADDR]{stpc1.addr}, and \hyperref[fielddesc:STPC2_ADDR]{stpc2.addr}) can be extracted from these CSRs to get the program counters at which these faulting load/store instructions were executed. The CSRs \hyperref[regdesc:LDTVAL0]{ldtval0}, \hyperref[regdesc:LDTVAL1]{ldtval1}, \hyperref[regdesc:STTVAL0]{sttval0}, \hyperref[regdesc:STTVAL1]{sttval1} and \hyperref[regdesc:STTVAL2]{sttval2} hold the memory access addresses corresponding to the respective loads/stores. Note that the lower indexed CSRs hold the newest bus errors, while the higher indexed CSRs hold the older bus errors.\\
         \item Note that, even if the cause is either 5 or 7, both load and store bus error CSRs must be checked as sometimes there are more than one load/store instructions (or misaligned accesses which are split into multiple loads/stores) that have been executed and are in flight. Prior to entering the trap handler for the first encountered bus error, the CPU will allow all pending memory operations to finish and record any bus errors caused by them, thus it is important to check all the CSRs to make sure that all such exceptions were caught and handled.\\
         \item Also note that recovering a normal program execution after encountering a bus-error exception is not as simple as for synchronous cases, since program execution may have proceeded quite far ahead of the bus-error-causing instructions.
    \end{itemize}
\end{itemize}

\subsection{Dedicated IO}
\label{sec:riscvcpu-fastio}
\subsubsection{Overview}
\label{sec:riscvcpu-fastio-overview}

Normally, GPIOs are an APB peripheral, which means that changes to outputs and reads from inputs can get stuck in write buffers or behind other transfers. They are slower because the APB bus runs at a lower speed than the CPU core.
As an alternative, the CPU core implements I/O processor-specific CPU registers (CSRs)， which are directly connected to the GPIO matrix or IO pads. As these registers can be accessed in one instruction, the speed is fast.

\subsubsection{Features}
\label{sec:riscvcpu-fastio-features}
 \begin{itemize}
    \item 8 dedicated IOs directly mapped on GPIOs
    \item No latency for driving output ports
    \item Two CPU cycle latency for sensing input values
\end{itemize}

\subsubsection{Functional Description}
\label{sec:riscvcpu-fastio-func-descr}

The CPU core has a set of 8 inputs and outputs (pin value + pin output enable). These input and output ports are directly connected to the GPIO matrix, through which they can be mapped on top-level pads. Please refer to Chapter \ref{mod:iomuxgpio} \textit{\nameref{mod:iomuxgpio}} for more details.

The CPU implements three custom CSRs:
\begin{itemize}
    \item GPIO\_IN is read-only and reflects the input value.
    \item GPIO\_OUT is R/W and reflects the output value for the GPIOs.
    \item GPIO\_OEN is R/W and reflects the output enable state for the GPIOs. It controls the pad direction. Programming high would mean the pad should be configured in output mode. Programming low means it should be configured in input mode.
\end{itemize}

\subsection{RunStall Support}
\label{sec:riscvcpu-custom-runstall}
\subsubsection{Overview}
\label{sec:riscvcpu-custom-runstall-overview}

The RunStall signal allows an external agent to stall the processor and shut off much of the clock tree to save operating power.
The RunStall input allows external logic to stall the processor's internal pipeline. Any in-progress instructions in the pipeline will complete after the assertion of the RunStall input line. After the pipeline is stalled, the core outputs a handshake signal "corestalled" to indicate successful stalling of the CPU pipeline.


\subsubsection{Functional Description}
\label{sec:riscvcpu-custom-runstall-feature-func-descr}

The RunStall input can be used in the following two scenarios:
\begin{itemize}
    \item As a mechanism to initialize instruction and data RAMs after a reset. If the RunStall input is asserted during reset and remains asserted after the reset is removed, the processor will be in a stalled state. Instruction and data RAMs can be initialized and after these RAMs have been initialized, the RunStall can be released, allowing the processor to start code execution. This mechanism allows the processor’s reset vector to point to an address in local instruction RAM, and the reset code can be written to the instruction RAM by an external agent before the processor starts executing code.
    \item To freeze the processor’s internal pipeline. Assertion of RunStall reduces the processor’s active power dissipation without using or requiring interrupts and the WAITI option. However, the power savings resulting from use of the RunStall signal will not be as great as for those using WAITI, because the RunStall logic allows some portions of the processor to be active even though the pipeline is stalled.
\end{itemize}

\subsection{Core Lock-up}
\label{sec:riscvcpu-custom-lockup-features}
\subsubsection{Overview}
\label{sec:riscvcpu-custom-lockup-overview}
Currently, there is no exception enable register defined in the RISC-V programming model, and when returning from the exception service program to the normal program flow, the exception vector number in the MCAUSE register is not cleared, so software developers cannot easily know whether the CPU is currently processing an exception or running a normal program track by the registers defined in the programming model. To address this concern, a custom LOCKUP field has been implemented. It is readable through the custom CSR.

\subsubsection{Functional Description}
\label{sec:riscvcpu-custom-lockup-feature-func-descr}
When the CPU responds to an exception, it will determine whether the current program flow is in the exception service program (through EXPT\_VLD in MEXSTATUS field). If the CPU is processing an exception and triggers a new exception before returning from the exception service program, the CPU will be locked. When CPU is locked:

\begin{itemize}
    \item the lock-up signal is set high to inform the SoC that the CPU is in a locked state
    \item the CPU stops instruction fetch and execution
    \item the CPU keeps the PC at the instruction PC that triggers the CPU lock
    \item the LOCKUP field in the MEXSTATUS register is set to 1
\end{itemize}

In a locked-up state, a debugger can still break in and execute debug commands. It is recommended to reset the core when a lock state is detected.

There are some exceptions to the lock-up rule. Exceptions caused by ECALL or EBREAK, when nested in any exception type, will not trigger CPU lock-up.

\subsubsection{Register Summary}
\label{sec:riscvcpu-custom-lockup-reg-summ}

The \hyperref[fielddesc:MEXSTATUS_LOCKUP]{mexstatus.LOCKUP} bit provides lock-up status of individual core.

\subsubsection{Register Description}
\label{sec:riscvcpu-custom-lockup-reg-desc}
Please refer to \hyperref[regdesc:MEXSTATUS]{mexstatus} CSR.

\subsection{External Wakeup}
\label{sec:riscvcpu-custom-ext-wakeup-features}
\subsubsection{Overview}
\label{sec:riscvcpu-custom-ext-wakeup-overview}

Each core features a dedicated input port for wake-up via an external event. This functionality is controlled and configured through registers in the SoC.

\subsubsection{Functional Description}
\label{sec:riscvcpu-custom-ext-wakeup-func-descr}

To save power, each CPU core can enter a sleep state by executing the WFI instruction. It can be awakened by three types of events: an interrupt, a halt request, or a signal from its dedicated wake-up input port. A high-level pulse lasting a single clock cycle on this dedicated port is sufficient to wake the corresponding core.

For details on how to configure this port, please refer to Chapter \ref{mod:sysreg} \textit{\nameref{mod:sysreg}}.
