\chapter{VDMA Controller (VDMA)}
\label{mod:vdma}

%%% Introduction %%%

\section{Overview}

DMA (Direct Memory Access) enables direct access to system memory or peripherals without CPU involvement. The VDMA controller on \chipname{} is a general-purpose DMA that performs high-speed data transfer from memory to memory, from memory to peripheral, and from peripheral to memory. The VDMA complies with the AXI3 protocol and includes two AXI master interfaces. This design allows users to select between the two interfaces for data transfer dynamically.

%%%%%%%%%%%%%%%%%%%%
%%% Terminology %%%
%%%%%%%%%%%%%%%%%%%%

\section{Terminology}

The following terms are defined in the context of the VDMA controller on \chipname{}:

\begin{itemize}
    \item \textbf{AXI}: Advanced eXtensible Interface, a part of AMBA 3 (Advanced Microcontroller Bus Architecture) specification, designed to facilitate high bandwidth and low latency communication between memory and peripheral, and between memory and memory.


    \item \textbf{Peripheral}: A system component that has a handshake interface with VDMA, as listed in Table \ref{tab:VDMA-hw-hs}.

    \item \textbf{Memory}: A system component that is always ready for DMA transfer, which means there is no handshaking interface between VDMA and memory. The memory that supports VDMA data transfer includes HP L2MEM, external flash, and external RAM.

    \item \textbf{Source}: Source memory or peripheral from which VDMA reads data via the AXI interface. VDMA then stores data in channel FIFO.

    \item \textbf{Source transfer}: Transfer from the source to VDMA.

    \item \textbf{Destination}: Destination peripheral or memory to which VDMA writes the data stored in channel FIFO.

    \item \textbf{Destination transfer}: Transfer from VDMA to the destination.

    \item \textbf{Channel}: Data path between source and destination.

    \item \textbf{Master interface}: Refers to the two AXI master interfaces of VDMA.

    \item \textbf{Slave interface}: The APB3 slave interface for programming registers.

    \item \textbf{Handshaking interface}: The handshaking interface performs a handshake between VDMA and the source or destination peripheral. The handshaking interface requests, acknowledges, and controls an VDMA transfer.

    \item \textbf{Flow controller}: The flow controller defines the length of a DMA block transfer and terminates the transfer. The flow controller can be VDMA, a source peripheral, or a destination peripheral, but it cannot be memory.
\end{itemize}

%%% Feature List %%%

\section{Features}

VDMA supports the following features:

\begin{itemize}
    \item Four channels for unidirectional data transfer from source to destination
    \item Two AXI master interfaces
    \item Handshake with MIPI DSI (Display Serial Interface) and ISP (Image Signal Processor)
    \item Memory-to-memory, ISP-to-memory, and MIPI DSI-to-memory transfer types
    \item Multiple levels of DMA transfer hierarchy
    \item Configurable transfer type, transfer length, and transfer size for each channel
    \item Single-block transfer
    \item Multi-block transfer based on contiguous address, automatic reloading register configuration, shadow registers, and linked lists
    \item Independent configuration of multi-block transfer type for source transfer and destination transfer
    \item Channel disabling without data loss
    \item Channel suspension, resume, and abortion
    \item Configurable priorities among arbitration channels
    \item Flow control using VDMA or peripherals
    \item Programmable mapping between peripherals and channels
\end{itemize}

\section{Functional Description}

This section provides a detailed description of the key features and operations of the module.

\subsection{Transfer Hierarchy}

Transfers are organized into a maximum of four levels: DMA transfer level, block transfer level, transaction level, and AXI transfer level.

Figure \ref{fig:VDMA-hier-non-mem} shows four transfer levels between VDMA and peripherals.

Figure \ref{fig:VDMA-hier-mem} shows three transfer levels between VDMA and memory. There is no transaction level because memory is assumed to be always ready for data transfer.

The following descriptions of DMA, block, and AXI transfers apply to the transfer between VDMA and memory, VDMA and peripherals. \textbf{The description of the transaction level applies only to transfers between VDMA and peripherals.}

\begin{itemize}
    \item \textbf{DMA transfer}: Software defines the number of blocks in a DMA transfer. Once a DMA transfer has finished, VDMA disables the channel and generates an interrupt. The channel can then be reprogrammed for a new DMA transfer. A DMA transfer can consist of either a single-block or a multi-block transfer.

    \item \textbf{Block transfer}: A block transfer moves a specified data block through the VDMA. The flow controller controls the block length. There are two types of block transfer: single-block transfer and multi-block transfer. As shown in Figure \ref{fig:VDMA-hier-non-mem}, a block transfer may consist of multiple transactions.

    \item \textbf{Transaction}: A peripheral request may trigger either a single or burst transaction, corresponding to an AXI single transfer or multiple AXI burst transfers, respectively.

        \begin{itemize}
            \item Single transaction: The length of a single transaction is always 1, in the unit of \hyperref[fielddesc:DMACCHNSRCTRWIDTH]{DMAC\_CH\regindex{n}\_SRC\_TR\_WIDTH}.

            \item Burst transaction: The length of a burst transaction is programmed into the VDMA through \hyperref[fielddesc:DMACCHNSRCMSIZE]{DMAC\_CH\regindex{n}\_SRC\_MSIZE} for the source or \hyperref[fielddesc:DMACCHNDSTMSIZE]{DMAC\_CH\regindex{n}\_DST\_MSIZE} for the destination. The burst length typically relates to the FIFO sizes in the source or destination peripherals. For more information, please refer to the chapter on each peripheral.
        \end{itemize}
    \item \textbf{AXI transfer}: Refers to the AXI protocol transfer, which is divided into AXI single transfer and AXI burst transfer.
\end{itemize}

\subsection{Arbitration Scheme}

The arbitration mechanism arbitrates read and write requests for all channels.

VDMA uses a priority and fair-among-equals arbitration mechanism. The priority for each channel is defined in the channel configuration register \hyperref[regdesc:DMACCHNCFG1REG]{DMAC\_CH\regindex{n}\_CFG1\_REG}.

A priority of 3 is the highest, while 0 is the lowest. Multiple channels can share the same priority level.

The arbitration mechanism works as
follows:

\begin{itemize}
    \item The arbitration mechanism determines which request client to grant the authorization signal based on the priority of the request. The request with the highest priority will be granted access to the channel.

    \item When two or more requests have the same priority, request clients will gain access randomly.
\end{itemize}

\subsubsection{Read Arbiter}

The read arbiter arbitrates read requests, including:

\begin{itemize}
    \item VDMA's read requests to obtain data or status from the source
    \item VDMA's read requests to obtain status from the destination
    \item VDMA's read requests to obtain LLI descriptors
\end{itemize}

The block diagram of the read arbiter is shown in Figure \ref{fig:VDMA-read-arb}:

\begin{figure}[H]
     \centering
     \includegraphics[width=1.0\linewidth]{\modulefiles/figures/VDMA-Read-Arb.pdf}
     \caption{VDMA Read Arbiter}
     \label{fig:VDMA-read-arb}
 \end{figure}

Read requests to obtain linked list descriptors will be granted the highest priority, regardless of the channels' priority settings.

\subsubsection{Write Arbiter}

The write arbiter arbitrates write requests, including:

\begin{itemize}
    \item Destination Data Write Request
    \item LLI Descriptor Write-Back Request
\end{itemize}

The block diagram of the write arbiter is shown in Figure \ref{fig:VDMA-write-arb}:

\begin{figure}[H]
     \centering
     \includegraphics[width=1.0\linewidth]{\modulefiles/figures/VDMA-Write-Arb.pdf}
     \caption{VDMA Write Arbiter}
     \label{fig:VDMA-write-arb}
\end{figure}

Within the same channel, Destination Data Write Request has higher priority than LLI Descriptor Write-Back Request. If different channels have the same priority, then Data Write Requests have the highest priority.

\subsection{Handshaking Interface}

A peripheral uses a handshaking interface to inform VDMA that it is ready to transmit or receive data via AXI bus. The operation of the handshaking interface depends on whether the flow controller is the peripheral or VDMA.

VDMA performs a single transaction or a burst transaction per handshake. In some cases, a block transfer cannot be completed using only burst transactions. This situation typically occurs when the block size is not a multiple of the burst transaction length. In this case, the block transfer uses burst transactions until the remaining size of the block is less than the amount of data in the burst transaction. At this point, a single transaction is used to complete the block transfer.

A source's burst transaction size is determined by \hyperref[fielddesc:DMACCHNSRCMSIZE]{DMAC\_CH\regindex{n}\_SRC\_MSIZE}. A destination's burst transaction size is determined by \hyperref[fielddesc:DMACCHNDSTMSIZE]{DMAC\_CH\regindex{n}\_DST\_MSIZE}. The transaction size also corresponds to the burst transaction size used for handshaking. The value of \hyperref[fielddesc:DMACCHNSRCMSIZE]{DMAC\_CH\regindex{n}\_SRC\_MSIZE} or \hyperref[fielddesc:DMACCHNDSTMSIZE]{DMAC\_CH\regindex{n}\_DST\_MSIZE} should match the peripheral configuration. For detailed information, please refer to the chapter on related peripherals.

VDMA has three hardware handshaking interfaces with MIPI DSI, ISP (ISP-to-memory), and ISP (memory-to-ISP). For the mapping relationship please see Table \ref{tab:VDMA-hw-hs}.

\subsection{Transfer Control}

Transfer control logic facilitates the data transfer from a source to a destination. Data from the source is temporarily stored in the channel FIFO before being sent to the destination. If source and destination peripherals use different transfer sizes, VDMA will pack and unpack the data to fit the FIFO configuration.

For a specific DMA transfer, the transfer type and flow control configurations are determined by \hyperref[fielddesc:DMACCHNTTFC]{DMAC\_CH\regindex{n}\_TT\_FC}.

\subsubsection{Single-Block Transfer}

If a DMA transfer consists of a single block, the software can set \hyperref[fielddesc:DMACCHNSRCMULTBLKTYPE]{DMAC\_CH\regindex{n}\_SRC\_MULTBLK\_TYPE} and \hyperref[fielddesc:DMACCHNDSTMULTBLKTYPE]{DMAC\_CH\regindex{n}\_DST\_MULTBLK\_TYPE} to 0 to enable contiguous-address-based single-block transfer. In this case, VDMA disables the channel once the block transfer of size \hyperref[fielddesc:DMACCHNBLOCKTS]{DMAC\_CH\regindex{n}\_BLOCK\_TS} is completed.

\begin{tiplisting}
    Single-block transfer is a special case of multi-block transfer, and there is no strict distinction between the two. In this chapter, single-block transfer refers to the transfer consisting of only one block, while multi-block transfer refers to the transfer consisting of at least two blocks.
\end{tiplisting}

\subsubsection{Multi-Block Transfer}

If a DMA transfer consists of multiple blocks, the software can configure \hyperref[fielddesc:DMACCHNSRCMULTBLKTYPE]{DMAC\_CH\regindex{n}\_SRC\_MULTBLK\_TYPE} and \hyperref[fielddesc:DMACCHNDSTMULTBLKTYPE]{DMAC\_CH\regindex{n}\_DST\_MULTBLK\_TYPE} to choose a multi-block transfer type. There are four types of multi-block transfers, depending on how the transfer control registers (\hyperref[regdesc:DMACCHNSAR0REG]{DMAC\_CH\regindex{n}\_SAR0\_REG}, \hyperref[regdesc:DMACCHNDAR0REG]{DMAC\_CH\regindex{n}\_DAR0\_REG}, \hyperref[fielddesc:DMACCHNBLOCKTS]{DMAC\_CH\regindex{n}\_BLOCK\_TS}, \hyperref[regdesc:DMACCHNCTL0REG]{DMAC\_CH\regindex{n}\_CTL0/1\_REG}) are updated:

\begin{enumerate}
    \item \textbf{Contiguous address}
    \item \textbf{Auto reloading}: Register values reload from initial values
    \item \textbf{Shadow register}: Register values load from shadow registers
    \item \textbf{Linked list} Register values load from the next linked list (LLI)
\end{enumerate}

\subparagraph{Contiguous Address}

In this scenario, the address for each subsequent block continues from the end of the preceding block. To achieve this contiguity, configure \hyperref[fielddesc:DMACCHNSRCMULTBLKTYPE]{DMAC\_CH\regindex{n}\_SRC\_MULTBLK\_TYPE} or \hyperref[fielddesc:DMACCHNDSTMULTBLKTYPE]{DMAC\_CH\regindex{n}\_DST\_MULTBLK\_TYPE} to 0 for the source address or destination address to remain continuous across the blocks.

In the case of multi-block transfer, the values of \hyperref[fielddesc:DMACCHNSRCMULTBLKTYPE]{DMAC\_CH\regindex{n}\_SRC\_MULTBLK\_TYPE} and \hyperref[fielddesc:DMACCHNDSTMULTBLKTYPE]{DMAC\_CH\regindex{n}\_DST\_MULTBLK\_TYPE} cannot both be selected as contiguous addresses, meaning they cannot both be configured to 0 at the same time. If both are configured to 0 during a multi-block transfer, then the current block will be considered as the last block.

During a multi-block transfer, if \hyperref[regdesc:DMACCHNSAR0REG]{DMAC\_CH\regindex{n}\_SAR0\_REG} and \hyperref[regdesc:DMACCHNDAR0REG]{DMAC\_CH\regindex{n}\_DAR0\_REG} need to have contiguous addresses between blocks, you can indirectly achieve this functionality using linked-list or shadow-register-based multi-block transfer.


\begin{tiplisting}
    If you have enabled contiguous-address-based multi-block transfer (i.e., \hyperref[fielddesc:DMACCHNSRCMULTBLKTYPE]{DMAC\_CH\regindex{n}\_SRC\_MULTBLK\_TYPE} or \\ \hyperref[fielddesc:DMACCHNDSTMULTBLKTYPE]{DMAC\_CH\regindex{n}\_DST\_MULTBLK\_TYPE} is 0), ensure there are at least two blocks in the DMA transfer. Otherwise, it will result in unpredictable behavior.
\end{tiplisting}

\subparagraph{Auto Reloading}

In this scenario, \hyperref[regdesc:DMACCHNSAR0REG]{DMAC\_CH\regindex{n}\_SAR0\_REG} and \hyperref[regdesc:DMACCHNDAR0REG]{DMAC\_CH\regindex{n}\_DAR0\_REG} are reloaded with their initial values at the end of each block. VDMA does not proceed to the next block transfer until software clears the corresponding channel’s block transfer complete interrupt by writing 1 to \hyperref[fielddesc:DMACCHNCLEARSRCTRANSCOMPINTSTAT]{DMAC\_CH\regindex{n}\_CLEAR\_SRC\_TRANSCOMP\_INTSTAT} and \hyperref[fielddesc:DMACCHNCLEARDSTTRANSCOMPINTSTAT]{DMAC\_CH\regindex{n}\_CLEAR\_DST\_TRANSCOMP\_INTSTAT}.

\begin{tiplisting}
    If you have enabled auto-reloading-based multi-block transfer, please ensure there are at least two blocks in the DMA transfer. Otherwise, it will result in unpredictable behavior.
\end{tiplisting}

\subparagraph{Shadow Register}

In this scenario, \hyperref[regdesc:DMACCHNSAR0REG]{DMAC\_CH\regindex{n}\_SAR0\_REG}, \hyperref[regdesc:DMACCHNDAR0REG]{DMAC\_CH\regindex{n}\_DAR0\_REG}, \hyperref[fielddesc:DMACCHNBLOCKTS]{DMAC\_CH\regindex{n}\_BLOCK\_TS}, \hyperref[regdesc:DMACCHNCTL0REG]{DMAC\_CH\regindex{n}\_CTL0\_REG}, and \hyperref[regdesc:DMACCHNCTL1REG]{DMAC\_CH\regindex{n}\_CTL1\_REG} synchronize with their corresponding shadow registers when each block finishes. The values of the shadow registers are then used for the subsequent block.

Software writes the block configurations to the corresponding shadow registers. VDMA copies the shadow register contents to \hyperref[regdesc:DMACCHNSAR0REG]{DMAC\_CH\regindex{n}\_SAR0\_REG}, \hyperref[regdesc:DMACCHNDAR0REG]{DMAC\_CH\regindex{n}\_DAR0\_REG}, \hyperref[fielddesc:DMACCHNBLOCKTS]{DMAC\_CH\regindex{n}\_BLOCK\_TS}, \hyperref[regdesc:DMACCHNCTL0REG]{DMAC\_CH\regindex{n}\_CTL0\_REG}, and \hyperref[regdesc:DMACCHNCTL1REG]{DMAC\_CH\regindex{n}\_CTL1\_REG}  before starting the subsequent block transfer.

Read operations to the \hyperref[regdesc:DMACCHNSAR0REG]{DMAC\_CH\regindex{n}\_SAR0\_REG}, \hyperref[regdesc:DMACCHNDAR0REG]{DMAC\_CH\regindex{n}\_DAR0\_REG}, \hyperref[fielddesc:DMACCHNBLOCKTS]{DMAC\_CH\regindex{n}\_BLOCK\_TS}, \hyperref[regdesc:DMACCHNCTL0REG]{DMAC\_CH\regindex{n}\_CTL0\_REG}, and \hyperref[regdesc:DMACCHNCTL1REG]{DMAC\_CH\regindex{n}\_CTL1\_REG} always return the
data corresponding to the current block transfer, not the shadow register contents that correspond to the next block.

\hyperref[fielddesc:DMACCHNSHADOWREGORLLIVALID]{DMAC\_CH\regindex{n}\_SHADOWREG\_OR\_LLI\_VALID} indicates whether the shadow register contents are valid (1) or not (0). If this bit reads as 0 during a shadow register fetch phase, VDMA discards the shadow register contents and generates a \hyperref[int:DMACCHNSHADOWREGORLLIINVALIDERRINT]{DMAC\_CH\regindex{n}\_SHADOWREG\_OR\_LLI\_INVALID\_ERR\_INT} interrupt. VDMA waits until the software writes 1 to \hyperref[fielddesc:DMACCHNBLKTFRRESUMEREQ]{DMAC\_CH\regindex{n}\_BLK\_TFR\_RESUMEREQ}, indicating that the shadow registers are available. After this, VDMA attempts to read the shadow registers again and continues with the next block transfer.

\begin{tiplisting}
    In a shadow-register-based multi-block transfer, the values in \hyperref[regdesc:DMACCHNSAR0REG]{DMAC\_CH\regindex{n}\_SAR0\_REG}, \\\hyperref[regdesc:DMACCHNDAR0REG]{DMAC\_CH\regindex{n}\_DAR0\_REG}, \hyperref[fielddesc:DMACCHNBLOCKTS]{DMAC\_CH\regindex{n}\_BLOCK\_TS}, \hyperref[regdesc:DMACCHNCTL0REG]{DMAC\_CH\regindex{n}\_CTL0\_REG}, and \hyperref[regdesc:DMACCHNCTL1REG]{DMAC\_CH\regindex{n}\_CTL1\_REG} correspond to the current transfer configurations. When software writes to these registers, it is actually writing to the corresponding shadow registers.
\end{tiplisting}

\subparagraph{Linked List}

In this scenario, VDMA retrieves a block descriptor for a specific block from memory to update \hyperref[regdesc:DMACCHNSAR0REG]{DMAC\_CH\regindex{n}\_SAR0\_REG}, \hyperref[regdesc:DMACCHNDAR0REG]{DMAC\_CH\regindex{n}\_DAR0\_REG}, \hyperref[fielddesc:DMACCHNBLOCKTS]{DMAC\_CH\regindex{n}\_BLOCK\_TS}, \hyperref[regdesc:DMACCHNCTL0REG]{DMAC\_CH\regindex{n}\_CTL0\_REG}, and \hyperref[regdesc:DMACCHNCTL1REG]{DMAC\_CH\regindex{n}\_CTL1\_REG}, and then initiates the block transfer. This process is called LLI (Linked List Item) update. The block chaining feature of VDMA uses the linked list pointer register \hyperref[fielddesc:DMACCHNLOC0]{DMAC\_CH\regindex{n}\_LOC0} to store the address of the next linked list item in memory, thereby achieving seamless continuity of data transfer.

VDMA can dynamically extend linked lists, which means you do not need to pre-create the entire linked list in system memory. To enable this feature, set \hyperref[fielddesc:DMACCHNSHADOWREGORLLIVALID]{DMAC\_CH\regindex{n}\_SHADOWREG\_OR\_LLI\_VALID} to 1. If the linked list item is the last one, then also configure \hyperref[fielddesc:DMACCHNSHADOWREGORLLILAST]{DMAC\_CH\regindex{n}\_SHADOWREG\_OR\_LLI\_LAST} to 1.

\hyperref[fielddesc:DMACCHNSHADOWREGORLLIVALID]{DMAC\_CH\regindex{n}\_SHADOWREG\_OR\_LLI\_VALID} indicates whether the linked list item fetched from memory is valid (1) or not (0). If the LLI is invalid, VDMA discards the LLI and generates a \hyperref[int:DMACCHNSHADOWREGORLLIINVALIDERRINT]{DMAC\_CH\regindex{n}\_SHADOWREG\_OR\_LLI\_INVALID\_ERR\_INT} interrupt. This interrupt causes VDMA to halt the corresponding channel and wait until the software writes 1 to \hyperref[fielddesc:DMACCHNBLKTFRRESUMEREQ]{DMAC\_CH\regindex{n}\_BLK\_TFR\_RESUMEREQ} to indicate that the LLI is available. After this, VDMA will attempt to read the LLI again.

\begin{tiplisting}
    For pre-fetched LLIs, if \hyperref[fielddesc:DMACCHNSHADOWREGORLLIVALID]{DMAC\_CH\regindex{n}\_SHADOWREG\_OR\_LLI\_VALID} is 0, then the \\\hyperref[fielddesc:DMACCHNSHADOWREGORLLIINVALIDERRINTSTAT]{DMAC\_CH\regindex{n}\_SHADOWREG\_OR\_LLI\_INVALID\_ERR\_INTSTAT} interrupt will not be generated. VDMA re-attempts to fetch the LLI again after completing the current block transfer and generates a\\ \hyperref[fielddesc:DMACCHNSHADOWREGORLLIINVALIDERRINTSTAT]{DMAC\_CH\regindex{n}\_SHADOWREG\_OR\_LLI\_INVALID\_ERR\_INTSTAT} interrupt only if \hyperref[fielddesc:DMACCHNSHADOWREGORLLIVALID]{DMAC\_CH\regindex{n}\_SHADOWREG\_OR\_LLI\_VALID} still reads as 0.
\end{tiplisting}

    \item CH\regindex{n}\_DAR: The field definition is the same as \hyperref[regdesc:DMACCHNDAR0REG]{DMAC\_CH\regindex{n}\_DAR0\_REG}

\subparagraph{Suspension of Transfers Between Blocks}

At the end of every block transfer, a block transfer completion interrupt \hyperref[int:DMACCHNBLOCKTFRDONEINT]{DMAC\_CH\regindex{n}\_BLOCK\_TFR\_DONE\_INT} is generated if:

\begin{itemize}
    \item Global interrupt is enabled (\hyperref[fielddesc:DMACINTEN]{DMAC\_INT\_EN} = 1)
    \item The channel block transfer completion interrupt is enabled (\hyperref[fielddesc:DMACCHNENABLEBLOCKTFRDONEINTSTAT] {DMAC\_CH\regindex{n}\_ENABLE\_BLOCK\_TFR\_DONE\_INTSTAT} = 1, \hyperref[fielddesc:DMACCHNENABLEBLOCKTFRDONEINTSIGNAL]{DMAC\_CH\regindex{n}\_ENABLE\_BLOCK\_TFR\_DONE\_INTSIGNAL} = 1, \hyperref[fielddesc:DMACCHNIOCBLKTFR]{DMAC\_CH\regindex{n}\_IOC\_BLKTFR} = 1)
\end{itemize}

For contiguous-address and auto-reloading-based multi-block transfers (neither source nor destination peripheral uses shadow-register or linked-list-based multi-block transfers), the DMA transfer automatically stalls after the \hyperref[int:DMACCHNBLOCKTFRDONEINT]{DMAC\_CH\regindex{n}\_BLOCK\_TFR\_DONE\_INT} interrupt is generated. VDMA does not proceed to the next block transfer until software writes 1 to \hyperref[fielddesc:DMACCHNCLEARBLOCKTFRDONEINTSTAT]{DMAC\_CH\regindex{n}\_CLEAR\_BLOCK\_TFR\_DONE\_INTSTAT} to clear the interrupt.

Channel suspension between blocks ensures that the block transfer done ISR (Interrupt Service Routine) of the next-to-last block is serviced before the final block transfer starts. This ensures that the ISR has cleared \hyperref[fielddesc:DMACCHNSRCMULTBLKTYPE]{DMAC\_CH\regindex{n}\_SRC\_MULTBLK\_TYPE} and \hyperref[fielddesc:DMACCHNDSTMULTBLKTYPE]{DMAC\_CH\regindex{n}\_DST\_MULTBLK\_TYPE} before the final block transfer is completed.

\subparagraph{End of Multi-Block Transfers}

If either source or destination peripheral uses shadow-register or linked-list-based multi-block transfers, then \hyperref[fielddesc:DMACCHNSHADOWREGORLLILAST] {DMAC\_CH\regindex{n}\_SHADOWREG\_OR\_LLI\_LAST} indicates whether the current block is the last in the transfer. If this bit is 1, VDMA recognizes that the current block is the final block in the transfer and completes the DMA transfer operation at the end of the current block transfer.

For contiguous-address and auto-reloading-based multi-block transfers (when neither source nor destination peripheral uses shadow-register or linked-list-based multi-block transfers), if the corresponding multi-block type selection fields \hyperref[fielddesc:DMACCHNSRCMULTBLKTYPE]{DMAC\_CH\regindex{n}\_SRC\_MULTBLK\_TYPE} and/or \hyperref[fielddesc:DMACCHNDSTMULTBLKTYPE]{DMAC\_CH\regindex{n}\_DST\_MULTBLK\_TYPE} are 0 at the end of a block transfer, VDMA understands that the previous block was the final block in the transfer and completes the DMA transfer operation.

\subsection{Flow Controller}

The flow controller determines the length of the DMA block transfer. The flow controller can be VDMA, the source peripheral, or the destination peripheral; memory cannot serve as a flow controller.

If the block length is known before enabling the channel, configure VDMA as the flow controller. If the block length is unknown before enabling the channel, use the source or destination peripheral as the flow controller to determine when to terminate the DMA block transfer.

\subsection{Channel Suspend and Resume}

Software can suspend and resume a channel during DMA transfer. To suspend a channel during DMA transfer:

\begin{enumerate}
    \item Software writes 1 to \hyperref[fielddesc:DMACCH1SUSP]{DMAC\_CH\regindex{n}\_SUSP}.
    \item VDMA halts all transfers from the source, after completing all AXI transfers initiated from the source.

    \item VDMA sets \hyperref[int:DMACCHNCHSRCSUSPENDEDINT]{DMAC\_CH\regindex{n}\_CH\_SRC\_SUSPENDED\_INT} to 1 to indicate that source transfer is suspended.

    \item VDMA transfers all the data in channel FIFO to the destination. If \hyperref[fielddesc:DMACCHNSRCTRWIDTH]{DMAC\_CH\regindex{n}\_SRC\_TR\_WIDTH} < \hyperref[fielddesc:DMACCHNDSTTRWIDTH]{DMAC\_CH\regindex{n}\_DST\_TR\_WIDTH} and \hyperref[fielddesc:DMACCH1SUSP]{DMAC\_CH\regindex{n}\_SUSP} is 1, there may still be data in the channel FIFO, but not enough to form a single transfer of \hyperref[fielddesc:DMACCHNDSTTRWIDTH]{DMAC\_CH\regindex{n}\_DST\_TR\_WIDTH}. The remaining data in the channel FIFO will be transferred to the destination if the channel is resumed.

    \item VDMA sets \hyperref[int:DMACCHNCHSUSPENDEDINT]{DMAC\_CH\regindex{n}\_CH\_SUSPENDED\_INT} to 1 to indicate that the channel is suspended.
\end{enumerate}

After a channel suspend, software can resume the channel by setting \hyperref[fielddesc:DMACCH1SUSP]{DMAC\_CH\regindex{n}\_SUSP} to 0. Then, VDMA resumes the DMA transfer from the point where it was suspended.

\subsection{Channel Disable}

Under normal operation, software enables a channel by writing 1 to \hyperref[fielddesc:DMACCH1EN]{DMAC\_CH\regindex{n}\_EN}, and VDMA clears \hyperref[fielddesc:DMACCH1EN]{DMAC\_CH\regindex{n}\_EN} after channel transfer completion.
Software can disable a channel before a transfer completes.

\subsubsection{Disabling a Suspended Channel Before Transfer Completion}

To disable a suspended channel before transfer completion:

\begin{itemize}
    \item Write 0 to \hyperref[fielddesc:DMACCH1EN]{DMAC\_CH\regindex{n}\_EN} after \hyperref[fielddesc:DMACCHNCHSUSPENDEDINTSTAT]{DMAC\_CH\regindex{n}\_CH\_SUSPENDED\_INTSTAT} is set to 1. If \hyperref[fielddesc:DMACCHNSRCTRWIDTH]{DMAC\_CH\regindex{n}\_SRC\_TR\_WIDTH} < \hyperref[fielddesc:DMACCHNDSTTRWIDTH]{DMAC\_CH\regindex{n}\_DST\_TR\_WIDTH} and \hyperref[fielddesc:DMACCH1SUSP]{DMAC\_CH\regindex{n}\_SUSP} is 1, there may still be data in the channel FIFO, but not enough to form a single transfer of \hyperref[fielddesc:DMACCHNDSTTRWIDTH]{DMAC\_CH\regindex{n}\_DST\_TR\_WIDTH}.
    In this case, once the channel is disabled, the remaining data in the channel FIFO is not transferred to the destination peripheral and is lost.
    \item VDMA sets \hyperref[fielddesc:DMACCHNCHDISABLEDINTSTAT]{DMAC\_CH\regindex{n}\_CH\_DISABLED\_INTSTAT} to 1 to indicate that the channel is disabled and generates an interrupt.
\end{itemize}

\subsubsection{Disabling a Non-suspended Channel Before Transfer Completion}

To disable a non-suspended channel before transfer completion:

\begin{itemize}
    \item Software writes 0 to \hyperref[fielddesc:DMACCH1EN]{DMAC\_CH\regindex{n}\_EN} during DMA transfer.
    \item VDMA halts all transfers from the source, after completing all AXI transfers initiated on the source.
    \item VDMA transfers all the data in the channel FIFO to the destination. When \hyperref[fielddesc:DMACCHNSRCTRWIDTH]{DMAC\_CH\regindex{n}\_SRC\_TR\_WIDTH} < \hyperref[fielddesc:DMACCHNDSTTRWIDTH]{DMAC\_CH\regindex{n}\_DST\_TR\_WIDTH} and \hyperref[fielddesc:DMACCH1EN]{DMAC\_CH\regindex{n}\_EN} is 0, there may still be data in the channel FIFO, but not enough to form a single transfer of \hyperref[fielddesc:DMACCHNDSTTRWIDTH]{DMAC\_CH\regindex{n}\_DST\_TR\_WIDTH}.
    In this scenario, once the channel is disabled, the remaining data in the channel FIFO is not transferred to the destination peripheral and is lost.
    \item VDMA sets \hyperref[fielddesc:DMACCHNCHDISABLEDINTSTAT]{DMAC\_CH\regindex{n}\_CH\_DISABLED\_INTSTAT} to 1 to indicate that the channel is disabled and generates an interrupt.
\end{itemize}

\begin{tiplisting}
    Once software initiates a channel disable procedure by writing 0 to \hyperref[fielddesc:DMACCH1EN]{DMAC\_CH\regindex{n}\_EN}, before \hyperref[int:DMACCHNCHDISABLEDINT]{DMAC\_CH\regindex{n}\_CH\_DISABLED\_INT} is set to 1, you cannot re-enable the channel by writing 1 to \hyperref[fielddesc:DMACCH1EN]{DMAC\_CH\regindex{n}\_EN}. VDMA will ignore this write operation.
\end{tiplisting}

\subsection{Low-Power Technique}

VDMA implements a low-power technique that intelligently monitors idle periods across multiple DMA channels, the slave bus interface, and AXI master interface channels in various scenarios. By detecting inactivity, this feature can turn off the clock for these modules, significantly reducing overall power consumption.

When activity is detected, the logic quickly restores the clocks to these modules, ensuring that performance remains uncompromised.

Configure \hyperref[regdesc:DMACLOWPOWERCFG0REG]{DMAC\_LOWPOWER\_CFG0\_REG} to enable the low-power feature and manage the low-power counters.

\subsubsection{Low-Power Technique for DMA Channels}

DMA channels occupy a significant portion of the chip area in the VDMA module, resulting in higher power consumption compared to other components. Therefore, optimizing power usage in this module is essential.

The low-power technique for DMA channels employs a \textbf{DMA channel low-power state machine} to monitor activities. When the state machine detects a low-power condition, it activates a \textbf{DMA channel low-power delay counter}. This counter runs until it expires. Once the counter expires, the DMA channels enter a low-power state by gating the AXI\_DMAC\_CORE\_CLK.

The DMA channels will remain in this low-power state until any activity is detected. If activity is detected while the delay counter is still running, the counter is reset and paused until the low-power condition is detected again.

\subsubsection{Low-Power Technique for Slave Bus Interface}

The slave bus interface unit (SBIU) accesses the internal registers of VDMA via the APB3 slave interface. The low-power technique can optimize the power consumption in SBIU.

The low-power technique for the slave bus interface employs a \textbf{SBIU state machine} to monitor activities. When a low-power condition is detected (i.e., when there are no ongoing APB reads or writes and the SBIU state machine is idle), the \textbf{SBIU low power delay counter} is activated. Once the delay counter expires, SBIU enters the low-power state by gating PCLK or AXI\_DMAC\_CORE\_CLK (based on the clock mode configuration).

SBIU remains in the low-power state until an active read or write transaction is detected on the slave bus interface. If an active transaction is detected while the SBIU low-power delay counter running, the delay counter is reset and paused until the low-power condition is detected again.

If a read or write transaction occurs while SBIU is in a low-power state, the low-power logic immediately exits the low-power state by un-gating PCLK or AXI\_DMAC\_CORE\_CLK.

\subsubsection{Low-Power Technique for AXI Master Interface Channels}

The AXI master interface module implements the AXI bus to transfer data between memory and peripheral.

Whenever a low-power condition is detected on the AXI master interface channels, the \textbf{AXI master interface channel low-power delay counter} starts. The AXI low-power logic waits until the delay counter expires. After the delay counter expires, the AXI master interface channel enters the low-power state by gating the clock of AXI master interface and AXI\_DMAC\_CORE\_CLK. It remains in this state until any activity is detected. If an activity is detected while the AXI master interface channel low-power delay counter is running, the delay counter is reset and paused until the low-power condition is detected again.

\subsubsection{Global Low-Power Technique}

The global low-power technique implements the low-power option based on the idle condition of the DMA channel, slave bus interface, and AXI master interface channels. If all the mentioned modules indicate that the VDMA is idle, the \textbf{global low-power delay counter} is activated.

The global low-power logic waits until the low-power delay counter expires. After the counter expires, VDMA enters the low-power state by gating the clocks of all modules. It remains in the low-power state until any activity is detected on any of the modules. If an activity is detected while the global low-power delay counter is running, the global delay counter is reset and paused until the low-power condition is again detected.

\section{Programming Procedures}
\label{sec:vdma-prog-procedures}

This section outlines the programming steps for different types of transfers.

\subsection{Common Programming Procedures}

The following two steps are common and should be performed before other programming procedures.

\begin{enumerate}
    \item  Set \hyperref[fielddesc:HPSYSCLKRSTGDMASYSCLKEN]{HP\_SYS\_CLKRST\_GDMA\_SYS\_CLK\_EN} to 1 to enable VDMA bus clock.
    \item Configure \hyperref[regdesc:TEEDMAGDMACHNWPMSREG]{TEE\_DMA\_GDMA\_CH\regindex{n}\_W\_PMS} and \hyperref[regdesc:TEEDMAGDMACHNRPMSREG]{TEE\_DMA\_GDMA\_CH\regindex{n}\_R\_PMS} to define the access permission of VDMA. For more details please refer to Chapter \ref{mod:permctrl} \textit{\nameref{mod:permctrl}}.
\end{enumerate}

\subsection{Programming Procedures for Shadow-Register-Based Multi-Block Transfer}

\begin{enumerate}
    \item Read \hyperref[fielddesc:DMACCH1EN]{DMAC\_CH\regindex{n}\_EN} to select an available (unused) channel.

    \item Configure \hyperref[fielddesc:DMACCHNSRCMULTBLKTYPE]{DMAC\_CH\regindex{n}\_SRC\_MULTBLK\_TYPE} and/or \hyperref[fielddesc:DMACCHNDSTMULTBLKTYPE]{DMAC\_CH\regindex{n}\_DST\_MULTBLK\_TYPE} to 2 to enable shadow-register-based multi-block transfer for source transfer and/or destination transfer.


    \item Configure \hyperref[regdesc:DMACCHNSAR0REG]{DMAC\_CH\regindex{n}\_SAR0\_REG} and/or \hyperref[regdesc:DMACCHNDAR0REG]{DMAC\_CH\regindex{n}\_DAR0\_REG}, \hyperref[fielddesc:DMACCHNBLOCKTS]{DMAC\_CH\regindex{n}\_BLOCK\_TS}, \hyperref[regdesc:DMACCHNCTL0REG]{DMAC\_CH\regindex{n}\_CTL0\_REG}, and \hyperref[regdesc:DMACCHNCTL1REG]{DMAC\_CH\regindex{n}\_CTL1\_REG} for the first block. VDMA loads the configured values to the corresponding shadow registers.

    {\bfseries Notice:} The \hyperref[fielddesc:DMACCHNSHADOWREGORLLIVALID]{DMAC\_CH\regindex{n}\_SHADOWREG\_OR\_LLI\_VALID} field in \hyperref[regdesc:DMACCHNCTL1REG]{DMAC\_CH\regindex{n}\_CTL1\_REG} must be the last one to be set to 1.

    \item Set \hyperref[fielddesc:DMACCH1EN]{DMAC\_CH\regindex{n}\_EN} to 1 to enable the corresponding channel.

    \item VDMA initiates the DMA block transfer operation based on the configurations.

        \begin{itemize}
        \item The block transfer might start immediately or after the handshaking request, depending on \hyperref[fielddesc:DMACCHNTTFC]{DMAC\_CH\regindex{n}\_TT\_FC}. Specifically,

            \begin{itemize}
                \item When the source is memory, the source transfer starts immediately; when the destination is memory, the destination transfer starts immediately.

                \item When the source is a peripheral, obtaining data from the source requires waiting for a handshake with the source. When the destination is a peripheral, writing data to the destination requires waiting for a handshake with the destination.
            \end{itemize}

        \item VDMA checks \hyperref[fielddesc:DMACCHNSHADOWREGORLLIVALID]{DMAC\_CH\regindex{n}\_SHADOWREG\_OR\_LLI\_VALID} and if it is seen as 0, VDMA waits until software writes 1 to \hyperref[fielddesc:DMACCHNBLKTFRRESUMEREQ]{DMAC\_CH\regindex{n}\_BLK\_TFR\_RESUMEREQ} to indicate valid shadow registers availability. Then VDMA re-attempts shadow register fetch operation. In this case, VDMA may generate a \hyperref[int:DMACCHNSHADOWREGORLLIINVALIDERRINT]{DMAC\_CH\regindex{n}\_SHADOWREG\_OR\_LLI\_INVALID\_ERR\_INT} interrupt.

        \item VDMA checks \hyperref[fielddesc:DMACCHNSHADOWREGORLLIVALID]{DMAC\_CH\regindex{n}\_SHADOWREG\_OR\_LLI\_VALID} and if it is seen as 1, VDMA copies the shadow register contents to the registers used for executing the DMA block transfer (i.e., \hyperref[regdesc:DMACCHNSAR0REG]{DMAC\_CH\regindex{n}\_SAR0\_REG} and/or \hyperref[regdesc:DMACCHNDAR0REG]{DMAC\_CH\regindex{n}\_DAR0\_REG}, \hyperref[fielddesc:DMACCHNBLOCKTS]{DMAC\_CH\regindex{n}\_BLOCK\_TS}, \hyperref[regdesc:DMACCHNCTL0REG]{DMAC\_CH\regindex{n}\_CTL0\_REG}, and \hyperref[regdesc:DMACCHNCTL1REG]{DMAC\_CH\regindex{n}\_CTL1\_REG}) and clears \hyperref[fielddesc:DMACCHNSHADOWREGORLLIVALID]{DMAC\_CH\regindex{n}\_SHADOWREG\_OR\_LLI\_VALID}.

            \begin{itemize}
            \item If VDMA sees \hyperref[fielddesc:DMACCHNSHADOWREGORLLILAST]{DMAC\_CH\regindex{n}\_SHADOWREG\_OR\_LLI\_LAST} as 1, it understands that the current block is the final block in the transfer and completes the DMA transfer operation at the end of current block transfer.
            \item If VDMA sees \hyperref[fielddesc:DMACCHNSHADOWREGORLLILAST]{DMAC\_CH\regindex{n}\_SHADOWREG\_OR\_LLI\_LAST} as 0, it understands that there are one or more blocks to be transferred and checks \hyperref[fielddesc:DMACCHNSHADOWREGORLLIVALID]{DMAC\_CH\regindex{n}\_SHADOWREG\_OR\_LLI\_VALID} again at the end of current block transfer.
        \end{itemize}
    \end{itemize}

    \item Software waits \hyperref[fielddesc:DMACCHNSHADOWREGORLLIVALID]{DMAC\_CH\regindex{n}\_SHADOWREG\_OR\_LLI\_VALID} to be cleared to 0.

    \item Configure \hyperref[regdesc:DMACCHNSAR0REG]{DMAC\_CH\regindex{n}\_SAR0\_REG} and/or \hyperref[regdesc:DMACCHNDAR0REG]{DMAC\_CH\regindex{n}\_DAR0\_REG}, \hyperref[fielddesc:DMACCHNBLOCKTS]{DMAC\_CH\regindex{n}\_BLOCK\_TS}, \hyperref[regdesc:DMACCHNCTL0REG]{DMAC\_CH\regindex{n}\_CTL0\_REG}, and \hyperref[regdesc:DMACCHNCTL1REG]{DMAC\_CH\regindex{n}\_CTL1\_REG} for the next block. If the next block is the last block, set \hyperref[fielddesc:DMACCHNSHADOWREGORLLILAST]{DMAC\_CH\regindex{n}\_SHADOWREG\_OR\_LLI\_LAST} to 1.

    {\bfseries Notice:} \hyperref[fielddesc:DMACCHNSHADOWREGORLLIVALID]{DMAC\_CH\regindex{n}\_SHADOWREG\_OR\_LLI\_VALID} must be the lase to be set to 1.

    \item Software waits for a \hyperref[int:DMACCHNBLOCKTFRDONEINT]{DMAC\_CH\regindex{n}\_BLOCK\_TFR\_DONE\_INT} interrupt or polls \hyperref[fielddesc:DMACCHNBLOCKTFRDONEINTSTAT]{DMAC\_CH\regindex{n}\_BLOCK\_TFR\_DONE\_INTSTAT} until it reads as 1 and goes back to Step 6.

\end{enumerate}

\begin{tiplisting}
In cases where a \hyperref[int:DMACCHNSHADOWREGORLLIINVALIDERRINT]{DMAC\_CH\regindex{n}\_SHADOWREG\_OR\_LLI\_INVALID\_ERR\_INT} interrupt is generated, follow the recommended flow to resume transfer:

\begin{itemize}
    \item Configure \hyperref[regdesc:DMACCHNSAR0REG]{DMAC\_CH\regindex{n}\_SAR0\_REG} and/or \hyperref[regdesc:DMACCHNDAR0REG]{DMAC\_CH\regindex{n}\_DAR0\_REG}, \hyperref[fielddesc:DMACCHNBLOCKTS]{DMAC\_CH\regindex{n}\_BLOCK\_TS}, \\\hyperref[regdesc:DMACCHNCTL0REG]{DMAC\_CH\regindex{n}\_CTL0\_REG}, and \hyperref[regdesc:DMACCHNCTL1REG]{DMAC\_CH\regindex{n}\_CTL1\_REG} for the next block.

    \item Set \hyperref[fielddesc:DMACCHNCLEARSHADOWREGORLLIINVALIDERRINTSTAT] {DMAC\_CH\regindex{n}\_CLEAR\_SHADOWREG\_OR\_LLI\_INVALID\_ERR\_INTSTAT} to 1 to clear the interrupt.

    \item Set \hyperref[fielddesc:DMACCHNBLKTFRRESUMEREQ]{DMAC\_CH\regindex{n}\_BLK\_TFR\_RESUMEREQ} to 1 to request for resuming block transfer.
\end{itemize}
\end{tiplisting}

\subsection{Programming Procedures for Linked-List-Based Multi-Block Transfer}

\begin{enumerate}
    \item Reads \hyperref[fielddesc:DMACCH1EN]{DMAC\_CH\regindex{n}\_EN} to select an available (unused) channel.

    \item Configure \hyperref[fielddesc:DMACCHNSRCMULTBLKTYPE]{DMAC\_CH\regindex{n}\_SRC\_MULTBLK\_TYPE} and/or \hyperref[fielddesc:DMACCHNDSTMULTBLKTYPE]{DMAC\_CH\regindex{n}\_DST\_MULTBLK\_TYPE} to 3 to enable linked-list-based transfer for source transfer and/or destination transfer.

    \item Configure \hyperref[regdesc:DMACCHNLLP0REG]{DMAC\_CH\regindex{n}\_LLP0\_REG} to define the base address of the first linked list item and the master interface on which the linked list item is available.

    \item Create one or more linked list items in the system memory. You can create the entire linked list item in advance or dynamically extend it using \hyperref[fielddesc:DMACCHNSHADOWREGORLLIVALID]{DMAC\_CH\regindex{n}\_SHADOWREG\_OR\_LLI\_VALID} and \hyperref[fielddesc:DMACCHNSHADOWREGORLLILAST]{DMAC\_CH\regindex{n}\_SHADOWREG\_OR\_LLI\_LAST} fields of the LLI. Follow the steps below to dynamically extend the linked list.
    \begin{itemize}
        \item During the DMA transfer process, when the linked list is not ready, configure \hyperref[fielddesc:DMACCHNSHADOWREGORLLIVALID]{DMAC\_CH\regindex{n}\_SHADOWREG\_OR\_LLI\_VALID} in the linked list to 0 and wait until the linked list is prepared. Then set \hyperref[fielddesc:DMACCHNSHADOWREGORLLIVALID]{DMAC\_CH\regindex{n}\_SHADOWREG\_OR\_LLI\_VALID} to 1 to extend the linked list.
        \item When it is time to end the DMA transfer, set \hyperref[fielddesc:DMACCHNSHADOWREGORLLILAST]{DMAC\_CH\regindex{n}\_SHADOWREG\_OR\_LLI\_LAST} of the last linked list to 1 to indicate the end of the transfer.
    \end{itemize}

    \item Set \hyperref[fielddesc:DMACCH1EN]{DMAC\_CH\regindex{n}\_EN} to 1 to enable the channel.

\begin{tiplisting}
You can swap the sequence of Step 4 and Step 5. However, if Step 5 is performed before Step 4, or if the linked list item for the next block transfer is not
available in system memory during the multi-block transfer (i.e., \hyperref[fielddesc:DMACCHNSHADOWREGORLLIVALID]{DMAC\_CH\regindex{n}\_SHADOWREG\_OR\_LLI\_VALID} of the fetched LLI is 0), VDMA may generate a\\ \hyperref[int:DMACCHNSHADOWREGORLLIINVALIDERRINT]{DMAC\_CH\regindex{n}\_SHADOWREG\_OR\_LLI\_INVALID\_ERR\_INT} interrupt.
\end{tiplisting}

    \item VDMA initiates the DMA block transfer operation based on the configurations. The block transfer might start immediately or after the handshaking request, depending on \hyperref[fielddesc:DMACCHNTTFC]{DMAC\_CH\regindex{n}\_TT\_FC}. VDMA copies the linked list contents to the registers used for executing the DMA block transfer (i.e., \hyperref[regdesc:DMACCHNSAR0REG]{DMAC\_CH\regindex{n}\_SAR0\_REG} and/or \hyperref[regdesc:DMACCHNDAR0REG]{DMAC\_CH\regindex{n}\_DAR0\_REG}, \hyperref[fielddesc:DMACCHNBLOCKTS]{DMAC\_CH\regindex{n}\_BLOCK\_TS}, \hyperref[regdesc:DMACCHNCTL0REG]{DMAC\_CH\regindex{n}\_CTL0\_REG}, and \hyperref[regdesc:DMACCHNCTL1REG]{DMAC\_CH\regindex{n}\_CTL1\_REG}) and initiates the DMA block transfer.

    \item During the linked list fetch phase:
        \begin{itemize}
        \item If VDMA reads \hyperref[fielddesc:DMACCHNSHADOWREGORLLILAST]{DMAC\_CH\regindex{n}\_SHADOWREG\_OR\_LLI\_LAST} as 1, it understands that the current block is the final block in the transfer and completes the DMA transfer operation at the end of the current block transfer.

        \item If VDMA reads \hyperref[fielddesc:DMACCHNSHADOWREGORLLILAST]{DMAC\_CH\regindex{n}\_SHADOWREG\_OR\_LLI\_LAST} as 0, it understands that there are one or more blocks to be transferred and goes to Step 6.

        \item If VDMA reads \hyperref[fielddesc:DMACCHNSHADOWREGORLLIVALID]{DMAC\_CH\regindex{n}\_SHADOWREG\_OR\_LLI\_VALID} as 0, it may generate a \hyperref[int:DMACCHNSHADOWREGORLLIINVALIDERRINT]{DMAC\_CH\regindex{n}\_SHADOWREG\_OR\_LLI\_INVALID\_ERR\_INT} interrupt. VDMA waits until software writes 1 to \hyperref[fielddesc:DMACCHNBLKTFRRESUMEREQ]{DMAC\_CH\regindex{n}\_BLK\_TFR\_RESUMEREQ} to indicate valid LLI availability before re-attempting the LLI read operation.
    \end{itemize}
\end{enumerate}

\subsection{Programming Procedures for Single-Block Transfer}

\begin{enumerate}
    \item Read \hyperref[fielddesc:DMACCH1EN]{DMAC\_CH\regindex{n}\_EN} to choose a free (unused) channel.
    \item Configure \hyperref[fielddesc:DMACCHNSRCMULTBLKTYPE]{DMAC\_CH\regindex{n}\_SRC\_MULTBLK\_TYPE} and \hyperref[fielddesc:DMACCHNDSTMULTBLKTYPE]{DMAC\_CH\regindex{n}\_DST\_MULTBLK\_TYPE} to 0 to enable contiguous-address-based single-block transfer.
    \item Configure \hyperref[regdesc:DMACCHNSAR0REG]{DMAC\_CH\regindex{n}\_SAR0\_REG} and/or \hyperref[regdesc:DMACCHNDAR0REG]{DMAC\_CH\regindex{n}\_DAR0\_REG}, \hyperref[fielddesc:DMACCHNBLOCKTS]{DMAC\_CH\regindex{n}\_BLOCK\_TS}, \hyperref[regdesc:DMACCHNCTL0REG]{DMAC\_CH\regindex{n}\_CTL0\_REG}, and \hyperref[regdesc:DMACCHNCTL1REG]{DMAC\_CH\regindex{n}\_CTL1\_REG} for the block.
    \item Set \hyperref[fielddesc:DMACCH1EN]{DMAC\_CH\regindex{n}\_EN} to 1 to enable the channel.

    \item Software waits for the \hyperref[int:DMACCHNBLOCKTFRDONEINT]{DMAC\_CH\regindex{n}\_BLOCK\_TFR\_DONE\_INT} interrupt or polls \hyperref[fielddesc:DMACCHNBLOCKTFRDONEINTSTAT]{DMAC\_CH\regindex{n}\_BLOCK\_TFR\_DONE\_INTSTAT} until it is 1.
\end{enumerate}
