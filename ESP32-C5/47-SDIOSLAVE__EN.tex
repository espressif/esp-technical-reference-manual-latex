\section{Overview}


The \chipname{} features hardware support for the Secure Digital Input/Output (SDIO) device interface that conforms to the SDIO Specification V2.00. This interface allows an SDIO host to access the \chipname{} using the SDIO bus protocol.

The SDIO host can directly access the \chipname{} SDIO interface registers or use the Direct Memory Access (DMA) engine to access shared memory. This approach reduces processor overhead and maintains high performance.


\section{Features}

The SDIO Slave Controller has the following features:
\begin{itemize}
    \item Compatible with SD Physical Layer Specification V2.00 and SDIO V2.00 specifications
    \item Support for two I/O functions (excluding function 0)
    \item Support for SPI, 1-bit SDIO, and 4-bit SDIO transfer modes
    \item Clock frequency range from 0 to 50 MHz
    \item Configurable sample and drive clock edge
    \item Integrated and SDIO-accessible registers for information exchange
    \item Support for SDIO interrupt mechanism
    \item Automatic padding data and discarding the padded data on the SDIO bus
    \item Block size up to 512 bytes
    \item Bidirectional interrupt vector between host and slave
    \item DMA support for data transfer
    \item Wake-up from sleep mode when connection is retained
\end{itemize}

\begin{tiplisting}
    The SDIO Slave controller can be used together with the USB Serial/JTAG controller in 1-bit SDIO transfer mode, but not in 4-bit SDIO transfer mode.
\end{tiplisting}

\section{Standards Compliance}
The \chipname{} SDIO Slave Controller conforms to the following standards:
\begin{itemize}
    \item SD Specifications Part 1 Physical Layer Specification Version 2.00 (referred to as Physical Layer Specification V2.00 in this chapter)
    \item SD Specifications Part E1 SDIO Specification Version 2.00, January 30, 2007 (referred to as SDIO Specification V2.00 in this chapter)
\end{itemize}

\section{Functional Description}
\subsection{Physical Bus}

\begin{itemize}
      \item Bus modes: SPI, 1-bit, and 4-bit SDIO transfer modes.
      \item Bus signals: Standard SDIO Specification V2.00 signals, including CS/DI/SCLK/DO/IRQ for SPI mode, CMD/CLK/DATA/IRQ for SDIO 1-bit mode, and CMD/CLK/DAT[3:0] for SDIO 4-bit mode
      \item Bus speed modes: Full-speed mode (0–50 MHz) and low-speed mode (0–400 kHz)
      \item I/O functions: Two I/O functions in addition to function 0. Function 0 is used only for CCCR, FBR, and CIS operations. Functions 1 and 2 can be used simultaneously to transfer application data packets (such as Wi-Fi and Bluetooth packets) and to access SLC Host registers.
\end{itemize}

For more information, please refer to the Physical Layer Specification V2.00 and SDIO Specification V2.00.

\subsubsection{Accessing SLC HOST Register Space}

The host can access registers in the contiguous address range from 0x40 to 0x3FF in the slave via I/O functions 1 or 2. To access these registers, the host sets the Register Address field of CMD52 or CMD53 to the low 10 bits of the address. CMD53 also allows the host to access multiple registers in a single operation for higher transfer rates.

From SLCHOST\_CONF\_W0\_REG and SLCHOST\_CONF\_W15\_REG, there are 52 bytes of fields accessible and modifiable by both the host and slave, facilitating information exchange.

Both host and slave software can access the SLC Host register space simultaneously. Therefore, an upper-layer mechanism should be implemented to prevent errors caused by concurrent access.

\subsubsection{Transferring Incremental-Address Packets}

When the host uses addresses 0x400 to 0x1F7FF to transmit multiple application data packets (such as Wi-Fi packets), the address field in CMD53 should be set to increment mode, and the OP Code field should be set to 1.

For example, to transfer (send or receive) three data blocks starting from base address 0x500 using CMD53, the host should:

\begin{itemize}
    \item Set the Block Mode field in CMD53 to 1 (block data unit)
    \item Set the OP Code field to 1 (incremental address mode)
    \item Set the Register Address field to 0x500 (base address)
    \item Set the Byte/Block Count field to 0x3 (three data blocks)
    \item Set other fields according to the SDIO Specification
\end{itemize}

When a packet is transmitted (slave to host or host to slave) through CMD53, the slave determines whether all valid data of the current packet has been transmitted and pads (when sending) or discards (when receiving) invalid data as needed. For more information, see Section \ref{sec:sdioslave-dummy-data}.

\subsubsection{Transferring Fixed-Address Packets}

When the host uses address 0x0 to transmit application data packets (such as Bluetooth packets), both the address field and OP Code field in CMD53 should be set to 0.

For example, to transfer (send or receive) three data blocks starting from fixed address 0x0 using CMD53, the host should:

\begin{itemize}
    \item Set the Block Mode field in CMD53 to 1 (block data unit)
    \item Set the OP Code field to 0 (fixed address mode)
    \item Set the Register Address field to 0x0 (fixed address)
    \item Set the Byte/Block Count field to 0x3 (three data blocks)
    \item Set other fields according to the SDIO Specification
\end{itemize}

When a packet is transmitted between the host and slave through CMD53, the slave determines whether all valid data of the current packet has been transmitted and pads or discards invalid data as needed. For more information, see Section \ref{sec:sdioslave-dummy-data}.

\subsection{DMA}
\label{sec:sdioslave-dma}

The SDIO Slave Controller uses a dedicated DMA to access transfer data from RAM or store it to RAM. As shown in Figure \ref{fig:sdioslave-slave-block-diagram}, RAM is accessed over the AHB. For the RAM space accessible by the Controller, please refer to Chapter \ref{mod:sysmem} \textit{\nameref{mod:sysmem}}. To set the RAM address range that can be accessed for one transfer, please configure the *SHAREMEM*\_REG fields of SLC register described in Section \ref{sec:sdioslave-reg-summ}.

The DMA engine provides two channels: SLC0 and SLC1. SLC0 is used for transferring incremental-address packets, while SLC1 handles fixed-address packets. The SDIO slave supports two I/O functions (Function 1 and Function 2) for data transmission. It is recommended to use Function 1 with SLC0 for incremental-address packets and Function 2 with SLC1 for fixed-address packets. For details on the address ranges of these functions, see Section \ref{sec:sdioslave-fun12-mem-map}.

DMA accesses RAM over AHB. You can configure the burst operation mode and type by setting the relevant fields of \hyperref[regdesc:SDIOSLCCONF0REG]{SDIO\_SLCCONF0\_REG} and \hyperref[regdesc:SDIOSLCBURSTLENREG]{SDIO\_SLC\_BURST\_LEN\_REG}. For more information, see Section \ref{sec:sdioslave-reg-summ}.

\subsubsection{Linked List}
\label{sec:sdioslave-linked-list}

The slave software can use the DMA engine by mounting linked lists. The DMA engine transfers data from the RAM address space specified in the RX (slave-to-host) linked list and stores received data in the RAM address space specified in the TX (host-to-slave) linked list. A linked list is composed of multiple descriptors.

 \begin{figure}[H]
    \centering
    \includegraphics[width=0.5\textwidth]{\modulefiles/figures/47-SDIOSLAVE-descriptor.eps}
    \caption{DMA Linked List Descriptor Structure of the SDIO Slave}
    \label{fig:sdioslave-linked-list-struc}
\end{figure}

The TX and RX linked list descriptors share the same structure, as shown in Figure \ref{fig:sdioslave-linked-list-struc}. Each descriptor consists of three words, with the following fields:

\begin{itemize}
     \item \textbf{owner (DW0) [31]}: Specifies the entity allowed to access the buffer. \\
     0: CPU\\
     1: DMA engine\\
     The slave software sets this field to 1 when creating the descriptor. After the DMA write-back permission is enabled and the corresponding buffer is used by the DMA, the field is cleared to 0.
     \item \textbf{eof (DW0) [30]}: Indicates the end of a data packet. \\
     0: The current descriptor is not the last descriptor of the packet\\
     1: The current descriptor is the last descriptor of the packet\\
     When the host sends a packet to the slave, the slave software should set the field to 0 while creating the descriptor. DMA sets the field of the last descriptor in the packet to 1; When the host receives packets from the slave, the slave software configures the field depending on whether this descriptor is the last descriptor of the packet.
     \item \textbf{reserved (DW0) [29:28]}: Reserved field. \\
    The slave software sets this field to 0x0.
     \item \textbf{length (DW0) [27:14]}: Indicates the number of valid bytes in the corresponding buffer. When the DMA engine is reading data from the buffer, it indicates the number of bytes that can be read; when DMA engine is storing data in the buffer, it indicates the number of bytes of the stored data. \\
     When the host sends a packet to the slave, the slave software should set this field to 0x0 while creating the descriptor. DMA writes back the field after the corresponding buffer is used up; when the host receives the packet from the slave and the slave creates the descriptor, the slave software should set this field to the number of bytes that can be read by the corresponding buffer.
     \item \textbf{size (DW0) [13:0]}: Specifies the size of the buffer in bytes. \\
    This field must be word-aligned. The slave software configures it during descriptor creation.
     \item \textbf{buffer address pointer (DW1)}: Points to the buffer address. \\
    This field must be word-aligned. The slave software configures it during descriptor creation.
     \item \textbf{next descriptor address (DW2)}: Points to the next descriptor in the linked list. \\
    If no next descriptor exists, this field is set to 0. The slave software configures it during descriptor creation.
\end{itemize}

For more information on DMA write-back linked list descriptor fields, see Section \ref{sec:linked-list-wr-back}.

The slave software can combine multiple descriptors into a linked list using the next descriptor address (DW2) field. The SDIO slave DMA linked list is shown in Figure \ref{fig:sdioslave-linked-list}.

 \begin{figure}[H]
    \centering
    \includegraphics[width=0.55\textwidth]{\modulefiles/figures/47-SDIOSLAVE-descriptor-chain}
    \caption{DMA Linked List of the SDIO Slave}
    \label{fig:sdioslave-linked-list}
\end{figure}

The following example demonstrates the use of a linked list and the \texttt{eof} bit. Suppose the slave software creates a linked list with three descriptors:
\begin{itemize}
    \item Descriptor 0 points to 500 bytes of data, with its \texttt{eof} bit set to 0.
    \item Descriptor 1 points to 200 bytes of data, with its \texttt{eof} bit set to 1.
    \item Descriptor 2 points to 200 bytes of data, with its \texttt{eof} bit set to 1.
\end{itemize}

\begin{enumerate}
    \item If the first CMD53 command requests 400 bytes, the DMA sends the first 400 bytes from Descriptor 0 to the host.
    \item If the second CMD53 command requests 400 bytes, the DMA first sends the remaining 100 bytes from Descriptor 0 to the host, followed by 200 bytes from Descriptor 1. Since the \texttt{eof} bit of Descriptor 1 is set to 1, the DMA considers the valid data for the current CMD53 command complete and pads the remaining 100 bytes with invalid data (0x0).
    \item If the third CMD53 command requests 400 bytes, the DMA sends 200 bytes from Descriptor 2 to the host. As the \texttt{eof} bit of Descriptor 2 is set to 1, the DMA considers the valid data for the current CMD53 command complete and pads the remaining 200 bytes with invalid data (0x0).
\end{enumerate}

\subsubsection{Write-Back of Linked List}
\label{sec:linked-list-wr-back}

When sending packets from the host to the slave, if the buffer specified by a linked list descriptor is full or a packet transmission ends, the DMA engine jumps to the next descriptor to store subsequent data. Before jumping, the DMA writes back the current descriptor. In DW0, the DMA updates the \texttt{eof} and \texttt{length} bits to their latest values. The value of the \texttt{owner} bit is determined by SDIO\_SLC0/1\_TX\_LOOP\_TEST in \hyperref[regdesc:SDIOSLCCONF0REG]{SDIO\_SLCCONF0\_REG}.

When receiving packets from the slave, if the host reads all data in the buffer specified by a linked list descriptor, the DMA engine jumps to the next descriptor to read subsequent data. Before jumping, the slave software can set SDIO\_SLC0/1\_RX\_AUTO\_WRBACK in \hyperref[regdesc:SDIOSLCCONF0REG]{SDIO\_SLCCONF0\_REG} to 1 so that the DMA writes back the current descriptor. The value to write to the \texttt{owner} bit is determined by SDIO\_SLC0/1\_RX\_LOOP\_TEST in \hyperref[regdesc:SDIOSLCCONF0REG]{SDIO\_SLCCONF0\_REG}. Other bits in DW0 remain unchanged.

The relevant register fields are described in Section \ref{sec:sdioslave-reg-summ}.

\subsubsection{Data Padding and Discarding}
\label{sec:sdioslave-dummy-data}

To transfer data in blocks, both the host and the slave pad the data sent on the SDIO bus into entire blocks. The slave automatically pads data when sending a packet and discards the padded data after receiving packets.

\begin{itemize}
    \item When the host sends a data packet to the slave through CMD53 and the amount of data reaches the packet length, the SDIO slave considers the valid data of the current packet complete. At this time, the DMA writes back the current linked list descriptor, sets the \texttt{eof} bit of the current descriptor to 1, and generates the \hyperref[int:SLC01TXSUCEOFINT]{SLC0/1\_TX\_SUC\_EOF\_INT} interrupt. After determining that the valid data is complete, the remaining data of the current packet is considered invalid and is not received into the buffer by the DMA. The slave does not restart receiving data into the next buffer until the next CMD53.
    \begin{itemize}
        \item For incremental-address packets, the slave determines valid data based on the address. Data with an address greater than or equal to 0x1F800 is considered invalid and is discarded. Therefore, the host should set the CMD53 start address field to 0x1F800 – Packet\_length (in bytes). The data flow of incremental-address packets on the SDIO bus is shown in Figure \ref{fig:sdioslave-sdio-bus-packet-trans}.
        \begin{figure}[H]
        \centering
        \fbox{\includegraphics[width=0.65\textwidth]{\modulefiles/figures/47-SDIOSLAVE-sdio-bus-packet-tran.eps}}
        \caption{Data Flow of Sending Incremental-address Packets From Host to Slave}
        \label{fig:sdioslave-sdio-bus-packet-trans}
        \end{figure}
         \item For fixed-address packets, the slave interprets the first three bytes of the data packet as the packet length. These three bytes are also stored in the buffer specified by the linked list. After the received data length matches the value indicated by these three bytes, any subsequent data is considered invalid and discarded.
    \end{itemize}
    \item When the host receives data packets (including incremental-address and fixed-address packets) from the slave through a CMD53 command, and the DMA reads the last byte of a buffer where the \texttt{eof} bit of the DMA linked list descriptor is set to 1, the SDIO slave considers the valid data of the current packet complete. At this point, the DMA writes back the current descriptor and generates the \hyperref[int:SLC01RXEOFINT]{SLC0/1\_RX\_EOF\_INT} interrupt. After the SDIO slave determines that the valid data is complete, the remaining bits of the current data packet are padded with invalid data (0x0) and are not read from the buffer via DMA. The slave restarts reading data from the buffer via DMA when the next CMD53 command is sent by the host.

\textbf{Note}: When the host receives either incremental-address or fixed-address data packets from the slave, the \texttt{eof} bit of the DMA linked list descriptor is always used to determine the end of data, rather than the address 0x1F800. Therefore, when the host sends multiple CMD53 commands to obtain multiple data packets, as long as the DMA does not encounter the \texttt{eof} bit set to 1 in the descriptors, the slave obtains the data from buffers in sequence according to the linked list and transmits it to the host. When the DMA encounters the \texttt{eof} bit set to 1, the data is fetched from the corresponding buffer, and then invalid data is padded to complete the current CMD53 command. The next CMD53 command will fetch data from the buffer pointed to by the next descriptor.
\end{itemize}

\subsection{SDIO Bus Timing}\label{sec:sdioslave-sdio-slave-timing}

The SDIO bus operates at high speed, and PCB trace length can affect signal integrity by introducing latency. To ensure proper timing characteristics, the SDIO slave module supports configuration of the input sampling clock edge and output driving clock edge.

When incoming data changes near the rising edge of the clock, the slave samples on the falling edge, and vice versa, as shown in Figure \ref{fig:sdioslave-sampling-timing}.

 \begin{figure}[H]
    \centering
    \fbox{\includegraphics[width=0.8\textwidth]{\modulefiles/figures/47-SDIOSLAVE-sampling.eps}}
    \caption{Sampling Timing Diagram}
    \label{fig:sdioslave-sampling-timing}
\end{figure}

By default, the voltage level of the GPIO25 strapping pin determines the slave pin's sampling edge. You can also configure the sampling edge using the \hyperref[regdesc:SLCHOSTCONFREG]{SLCHOST\_CONF\_REG} register, with the following priority (from high to low): (1) Set SLCHOST\_FRC\_POS\_SAMP to sample the corresponding signal at the rising edge; (2) Set SLCHOST\_FRC\_NEG\_SAMP to sample at the falling edge.

The SLCHOST\_FRC\_POS\_SAMP and SLCHOST\_FRC\_NEG\_SAMP fields are five bits wide, corresponding to the CMD line and four DATA lines (0–3). Setting a bit causes the corresponding line to be sampled at the rising or falling clock edge.

The slave can also select which edge to drive the output lines, to compensate for any latency caused by the physical signal path. The output timing is shown in Figure \ref{fig:sdioslave-output-sequence}.

  \begin{figure}[H]
    \centering
    \fbox{\includegraphics[width=0.8\textwidth]{\modulefiles/figures/47-SDIOSLAVE-driving.eps}}
    \caption{Output Timing Diagram}
    \label{fig:sdioslave-output-sequence}
\end{figure}

By default, the voltage level of the MTDI strapping pin determines the slave pin's output driving edge. You can also configure the output driving edge using the following registers, with priority from high to low: (1) Set SLCHOST\_FRC\_SDIO11 in \hyperref[regdesc:SLCHOSTCONFREG]{SLCHOST\_CONF\_REG} to output the corresponding signal at the falling clock edge; (2) Set SLCHOST\_FRC\_SDIO22 in \hyperref[regdesc:SLCHOSTCONFREG]{SLCHOST\_CONF\_REG} to output at the rising clock edge; (3) Set HINF\_HIGHSPEED\_ENABLE in \hyperref[regdesc:HINFCFGDATA1REG]{HINF\_CFG\_DATA1\_REG} and SLCHOST\_HSPEED\_CON\_EN in \hyperref[regdesc:SLCHOSTCONFREG]{SLCHOST\_CONF\_REG}, then set the EHS (Enable High-Speed) bit in CCCR at the host side to output at the rising clock edge.

The SLCHOST\_FRC\_SDIO11 and SLCHOST\_FRC\_SDIO22 fields are five bits wide, corresponding to the CMD line and four DATA lines (0–3). Setting a bit causes the corresponding line to output at the rising or falling clock edge.

\textbf{Note on priority}: The configuration of strapping pins has the lowest priority for controlling the sampling or driving edge. Lower-priority configurations take effect only when higher-priority configurations are not set. For example, the GPIO25 strapping value determines the sampling edge only when SCLHOST\_FRC\_POS\_SAMP and SCLHOST\_FRC\_NEG\_SAMP are not set.

\section{Interrupt}

The host and slave can interrupt each other via the interrupt vector. There are eight interrupt vectors between the host and each DMA SLC channel of the slave. To send an interrupt, set the enable bit of the interrupt vector register to 1.

\subsection{Host Interrupt}
\begin{itemize}
\item {\it SLCHOST\_SLC0/1\_RX\_NEW\_PACKET\_INT \label{int:SLCHOSTSLC01RXNEWPACKETINT}}: The slave has a packet to send. This interrupt can be triggered when:
        \begin{itemize}
            \item \hyperref[fielddesc:SDIOSLC0RXLINKSTART]{SDIO\_SLC0\_RXLINK\_START} or \hyperref[fielddesc:SDIOSLC1RXLINKSTART]{SDIO\_SLC1\_RXLINK\_START} is set to enable DMA.
            \item A new RX linked list descriptor is available after DMA processes the RX linked list descriptor with the \texttt{eof} bit set to 1.
            \item A packet needs to be retransmitted.
    \end{itemize}
\item {\it SLCHOST\_SLC0/1\_TX\_OVF\_INT \label{int:SLCHOSTSLC01TXOVFINT}}: Slave TX (transmit) buffer overflow interrupt.
\item {\it SLCHOST\_SLC0/1\_RX\_UDF\_INT \label{int:SLCHOSTSLC01RXUDFINT}}: Slave RX (receive) buffer underflow interrupt.
\item {\it SLCHOST\_SLC0/1\_TOHOST\_BIT\textcolor{red}{\it{n}}\_INT \label{int:SLCHOSTSLC01TOHOSTBITnINT}} (\textcolor{red}{\it{n}}: 0–7): Slave interrupts the host.
\end{itemize}

\subsection{Slave Interrupt}
\begin{itemize}
\item {\it SLC0/1\_TX\_DSCR\_ERR\_INT \label{int:SLC01TXDSCRERRINT}}: Slave TX linked list descriptor error.
\item {\it SLC0/1\_RX\_DSCR\_ERR\_INT \label{int:SLC01RXDSCRERRINT}}: Slave RX linked list descriptor error.
\item {\it SLC0/1\_RX\_EOF\_INT \label{int:SLC01RXEOFINT}}: Slave RX operation is finished.
\item {\it SLC0/1\_RX\_DONE\_INT \label{int:SLC01RXDONEINT}}: A single buffer is received by the slave.
\item {\it SLC0/1\_TX\_SUC\_EOF\_INT \label{int:SLC01TXSUCEOFINT}}: Slave TX operation is finished.
\item {\it SLC0/1\_TX\_DONE\_INT \label{int:SLC01TXDONEINT}}: A single buffer is finished during TX operation.
\item {\it SLC0/1\_TX\_OVF\_INT \label{int:SLC01TXOVFINT}}: Slave TX buffer overflow interrupt.
\item {\it SLC0/1\_RX\_UDF\_INT \label{int:SLC01RXUDFINT}}: Slave RX buffer underflow interrupt.
\item {\it SLC0/1\_TX\_START\_INT \label{int:SLC01TXSTARTINT}}: Slave TX start interrupt.
\item {\it SLC0/1\_RX\_START\_INT \label{int:SLC01RXSTARTINT}}: Slave RX start interrupt.
\item {\it SLC\_FRHOST\_BIT\textcolor{red}{\it{n}}\_INT \label{int:SLCFRHOSTBITnINT}} (\textcolor{red}{\it{n}}: 0–15): The host interrupts the slave via the SLC0 channel if interrupt vector Bit[7:0] is set or via the SLC1 channel if interrupt vector Bit[15:8] is set.
\end{itemize}

\section{Packet Sending and Receiving Procedure}

The SDIO host and slave devices must follow specific data transfer procedures to successfully exchange data over the SDIO interface. In addition to SDIO Specifications, \chipname{} should also follow the procedures below to transmit data over higher abstraction layers, such as Wi-Fi and Bluetooth.

\subsection{Sending Packets to SDIO Host}

The transmission of packets from the slave to the host is initiated by the slave. The host is notified with an interrupt (for details, refer to the SDIO Specification). After the host reads the relevant information from the slave, it initiates an SDIO bus transmission accordingly. The procedure is illustrated in Figure \ref{fig:sdioslave-package-sending-procedure}.

 \begin{figure}[H]
    \centering
    \fbox{\includegraphics[width=0.6\textwidth]{\modulefiles/figures/47-SDIOSLAVE-send-to-host__EN.pdf}}
    \caption{Procedure of Slave Sending Packets to Host}
    \label{fig:sdioslave-package-sending-procedure}
\end{figure}
\begin{enumerate}
    \item The slave CPU creates the linked list for the data packets to be sent to the host. For details, see Section \ref{sec:sdioslave-linked-list}.
    \item The slave CPU updates the length of data to be sent using the register \hyperref[regdesc:SDIOSLC0LENCONFREG]{SDIO\_SLC0\_LEN\_CONF\_REG}.
    \item The slave CPU starts DMA by writing the 32-bit address of the first descriptor in linked list to \hyperref[regdesc:SDIOSLC0RXLINKADDRREG]{SDIO\_SLC0RX\_LINK\_ADDR\_REG} or \hyperref[regdesc:SDIOSLC1RXLINKADDRREG]{SDIO\_SLC1RX\_LINK\_ADDR\_REG} and then configuring \hyperref[fielddesc:SDIOSLC0RXLINKSTART]{SDIO\_SLC0\_RXLINK\_START} or \hyperref[fielddesc:SDIOSLC1RXLINKSTART]{SDIO\_SLC1\_RXLINK\_START} to start DMA. For more information on DMA, please refer to Section \ref{sec:sdioslave-dma}.
    \item The slave DMA sends an interrupt to the host.
    \item After the host received the interrupt, it reads from \hyperref[regdesc:SLCHOSTSLC0HOSTINTSTREG]{SLCHOST\_SLC0HOST\_INT\_ST\_REG}, \hyperref[regdesc:SLCHOSTSLC1HOSTINTSTREG]{SLCHOST\_SLC1HOST\_INT\_ST\_REG}, and \hyperref[regdesc:SLCHOSTPKTLENREG]{SLCHOST\_PKT\_LEN\_REG} for the following information:
    \begin{itemize}
        \item SLCHOST\_SLC0/1HOST\_INT\_ST\_REG: Interrupt status register. The SLCHOST\_SLC0/1\_RX\_NEW\_PACKET\_INT\_ST bit set to 1 indicates that cause of the interrupt is the slave sending packets.
        \item SLCHOST\_PKT\_LEN\_REG: Packet length accumulator register. The current value minus the value of last time equals the packet length sent this time.
    \end{itemize}
    \item The host clears the interrupt by writing to the register \hyperref[regdesc:SLCHOSTSLC1HOSTINTCLRREG] {SLCHOST\_SLC0/1HOST\_INT\_CLR\_REG} after the CMD52 command.
    \item The host fetches packets from the slave after the CMD53 command. During transmission, when the slave determines that the valid data of the current packet is complete, the remaining bits are padded with invalid data (0x0). For details on determining the end of valid data, see Section \ref{sec:sdioslave-dummy-data}.
    \item After the packets are transmitted, the slave DMA sends an interrupt to the CPU, and the CPU can recycle the buffer.
\end{enumerate}

\textbf{Notes:}

\begin{itemize}
    \item Do not set all \texttt{eof} bits to 0 in the linked list. Otherwise, the DMA may mistakenly send the next packet's data as part of the current command, causing errors. If all \texttt{eof} bits are set to 0, the slave software should align the length of each packet to the data block size by padding data, to prevent errors. When the host sends CMD53 to read data, it should accurately control the number of data blocks in each packet and be able to identify the padded data.
    \item It is recommended that each CMD53 command transmit only one data packet and each data packet use only one linked list to avoid exceptions caused by complex transmission.
    \item Avoid sending multiple packets through one linked list, as it complicates the host and software's ability to distinguish between packets. If necessary, set the \texttt{eof} bit when creating the linked list to split the packets, so the DMA can pad data accordingly. The length of each packet should be aligned to the data block size to avoid data padding by DMA, and the host should be able to identify the padded data.
\end{itemize}

\subsection{Receiving Packets from SDIO Host}

Transmission of packets from the host to slave is initiated by the host. The slave receives data via DMA and stores it in RAM. After transmission is complete, the CPU is interrupted to process the data. The procedure is shown in Figure \ref{fig:sdioslave-package-receiving-procedure}.

 \begin{figure}[H]
    \centering
    \fbox{\includegraphics[width=0.6\textwidth]{\modulefiles/figures/47-SDIOSLAVE-receive-from-host__EN.pdf}}
    \caption{Procedure of Slave Receiving Packets from Host}
    \label{fig:sdioslave-package-receiving-procedure}
\end{figure}

The host obtains the number of available receiving buffers from the slave by accessing \hyperref[regdesc:SLCHOSTSLC0HOSTTOKENRDATAREG]{SLCHOST\_SLC0HOST\_TOKEN\_RDATA\_REG} or \hyperref[regdesc:SLCHOSTSLC1HOSTTOKENRDATAREG]{SLCHOST\_SLC1HOST\_TOKEN\_RDATA\_REG}. The slave CPU should update the value of the register after the receiving DMA linked list is prepared.

\hyperref[fielddesc:SLCHOSTHOSTSLCHOSTSLC0TOKEN1]{SLCHOST\_HOSTSLCHOST\_SLC0\_TOKEN1} or \hyperref[fielddesc:SLCHOSTHOSTSLCHOSTSLC1TOKEN1]{SLCHOST\_HOSTSLCHOST\_SLC1\_TOKEN1} stores the accumulated number of available buffers. The host can determine the available buffer space by subtracting the number of buffers already used from the register value. If buffers are insufficient, the host should poll the register until enough buffers are available.

During packet transmission to the slave through the CMD53 command, when a buffer specified by a linked list descriptor is full or a packet transmission ends, the DMA jumps to the next buffer to store subsequent data. When the slave determines that the valid data of the current packet is complete, the remaining data is considered invalid and discarded. The DMA writes back the current linked list descriptor, sets the \texttt{eof} bit of the current descriptor to 1, and generates the \hyperref[int:SLC01TXSUCEOFINT]{SLC0/1\_TX\_SUC\_EOF\_INT} interrupt.

For more information about DMA functions, linked list, and data discarding, see Section \ref{sec:sdioslave-dma}.

To ensure sufficient receiving buffers, the slave CPU must continuously load buffers onto the receiving linked list. The process is shown in Figure \ref{fig:sdioslave-receive-load-buffer}.

\begin{figure}[H]
    \centering
    \includegraphics[width=0.3\textwidth]{\modulefiles/figures/47-SDIOSLAVE-load-buffer__EN.eps}
    \caption{Loading Receiving Buffer}
    \label{fig:sdioslave-receive-load-buffer}
\end{figure}

The CPU should append new buffer segments at the end of the linked list used by DMA and available for receiving data.

The CPU must then notify the DMA that the linked list has been updated. This can be done by setting \hyperref[fielddesc:SDIOSLC0TXLINKRESTART]{SDIO\_SLC0\_TXLINK\_RESTART} or \hyperref[fielddesc:SDIOSLC1TXLINKRESTART]{SDIO\_SLC1\_TXLINK\_RESTART}. When the CPU initiates DMA to receive packets for the first time, \hyperref[fielddesc:SDIOSLC0TXLINKSTART]{SDIO\_SLC0\_TXLINK\_START} or \hyperref[fielddesc:SDIOSLC1TXLINKSTART]{SDIO\_SLC1\_TXLINK\_START} should be set to 1.

\textbf{Notes}: Use the *\_RESTART field to restart DMA only in the two scenarios:
\begin{itemize}
    \item DMA is suspended by configuration of the *\_STOP field. You can restart it after configuring the *\_RESTART field.
    \item DMA is suspended due to insufficient linked list descriptors. You can restart it by adding descriptors and configuring the *\_RESTART field.
\end{itemize}

Finally, the CPU refreshes available buffer information by writing to the \hyperref[regdesc:SDIOSLC0TOKEN1REG]{SDIO\_SLC0TOKEN1\_REG} or \hyperref[regdesc:SDIOSLC1TOKEN1REG]{SDIO\_SLC1TOKEN1\_REG} register.
