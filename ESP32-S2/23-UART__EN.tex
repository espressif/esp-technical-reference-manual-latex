\hypertarget{uart}{}
\chapter{UART Controller (UART)}
\label{mod:uart}

\section{Overview}

In embedded system applications, data is required to be transferred in a simple way with minimal system resources. This can be achieved by a Universal Asynchronous Receiver/Transmitter (UART), which flexibly exchanges data with other peripheral devices in full-duplex mode. ESP32-S2 has two UART controllers compatible with various UART devices. They support Infrared Data Association (IrDA) and RS485 transmission.

ESP32-S2 has two UART controllers. Each has a group of registers that function identically. In this chapter, the two UART controllers are referred to as UART\textcolor{red}{\it{n}}, in which \textcolor{red}{\it{n}}  denotes 0 or 1.

\section{Features}
Each UART controller has the following features:
\begin{itemize}
    \item Programmable baud rate
    \item 512 x 8-bit RAM shared by TX FIFOs and RX FIFOs of two UART controllers
    \item Full-duplex asynchronous communication
    \item Automatic baud rate detection
    \item Data bits ranging from 5 to 8
    \item Stop bits whose length can be 1, 1.5, or 2 bits
    \item Parity bits
    \item Special character AT\_CMD detection
    \item RS485 protocol
    \item IrDA protocol
    \item High-speed data communication using DMA
    \item UART as wake-up source
    \item Software and hardware flow control
\end{itemize}

\section{Functional Description}
\subsection{UART Introduction}
A UART is a character-oriented data link for asynchronous communication between devices. Such communication does not add clock signals to the data sent. Therefore, in order to communicate successfully, the transmitter and the receiver must operate at the same baud rate with the same stop bit(s) and parity bit.

A UART data packet usually begins with one start bit, followed by data bits, one parity bit (optional) and one or more stop bits. UART controllers on ESP32-S2 support various lengths of data bits and stop bits. These controllers also support software and hardware flow control as well as DMA for seamless high-speed data transfer. This allows developers to use multiple UART ports at minimal software cost.

\subsection{UART Structure}

\begin{figure}[ht]
    \centerline{\includegraphics[width=1.0\textwidth]{\modulefiles/figures/uart_structure}}
    \centering
    \caption{UART Structure}
    \label{Figure:one_uart_structure}
\end{figure}

Figure \ref{Figure:one_uart_structure} shows the basic structure of a UART controller. It has two possible clock sources: a 80 MHz APB\_CLK and a reference clock REF\_TICK (for details, please refer to Chapter \ref{mod:resclk} \textit{Reset and Clock}), which are selected by configuring \hyperref[fielddesc:UARTTICKREFALWAYSON]{UART\_TICK\_REF\_ALWAYS\_ON}. The selected clock source is divided by a divider to generate clock signals that drive the UART controller. The divisor is configured by \hyperref[regdesc:UARTCLKDIVREG]{UART\_CLKDIV\_REG}: \hyperref[fielddesc:UARTCLKDIV]{UART\_CLKDIV} for the integral part, and \hyperref[fielddesc:UARTCLKDIVFRAG]{UART\_CLKDIV\_FRAG} for the fractional part.

A UART controller is broken down into two parts according to functions: a transmitter and a receiver.

The transmitter contains a TX FIFO, which buffers data to be sent. Software can write data to Tx\_FIFO via the APB bus, or move data to Tx\_FIFO using DMA. Tx\_FIFO\_Ctrl controls writing and reading Tx\_FIFO. When Tx\_FIFO is not empty, Tx\_FSM reads bytes via Tx\_FIFO\_Ctrl, and converts them into a bitstream. The levels of output signal txd\_out can be inverted by configuring the \hyperref[fielddesc:UARTTXDINV]{UART\_TXD\_INV} field.

The receiver contains an RX FIFO, which buffers data to be processed. Software can read data from Rx\_FIFO via the APB bus, or receive data using DMA. The levels of input signal rxd\_in can be inverted by configuring \hyperref[fielddesc:UARTRXDINV]{UART\_RXD\_INV} register, and the signal is then input to the Rx components of the UART Controller: Baudrate\_Detect measures the baud rate of input signal rxd\_in by detecting its minimum pulse width. Start\_Detect detects the start bit in a data frame. If the start bit is detected, Rx\_FSM stores data bits in the data frame into Rx\_FIFO by Rx\_FIFO\_Ctrl.

HW\_Flow\_Ctrl controls rxd\_in and txd\_out data flows by standard UART RTS and CTS flow control signals (rtsn\_out and ctsn\_in). SW\_Flow\_Ctrl controls data flows by automatically adding special characters to outgoing data and detecting special characters in incoming data. When a UART controller is in Light-sleep mode (see Chapter \ref{mod:lowpowm}: \textit{\nameref{mod:lowpowm}} for more details), Wakeup\_Ctrl counts up rising edges of rxd\_in. When the number is equal to or greater than (\hyperref[fielddesc:UARTACTIVETHRESHOLD]{UART\_ACTIVE\_THRESHOLD} + 3), a wake\_up signal is generated and sent to RTC, which then wakes up the ESP32-S2 chip.

\subsection{UART RAM}

\begin{figure}[ht]
    \centerline{\includegraphics[width=0.4\textwidth]{\modulefiles/figures/uart_mem_723}}
    \centering
    \caption{UART Controllers Sharing RAM}
    \label{Figure:uart_mem}
\end{figure}

The two UART controllers on ESP32-S2 share 512 × 8 bits of FIFO RAM. As figure  \ref{Figure:uart_mem} illustrates, RAM is divided into 4 blocks, each having 128 × 8 bits. Figure \ref{Figure:uart_mem} shows by default how many RAM blocks are allocated to TX FIFOs and RX FIFOs of the two UART controllers. UART\textcolor{red}{\it{n}} Tx\_FIFO can be expanded by configuring \hyperref[fielddesc:UARTTXSIZE]{UART\_TX\_SIZE}, while UART\textcolor{red}{\it{n}} Rx\_FIFO can be expanded by configuring \hyperref[fielddesc:UARTRXSIZE]{UART\_RX\_SIZE}. The size of UART0 Tx\_FIFO can be increased to 4 blocks (the whole RAM), the size of UART1 Tx\_FIFO can be increased to 3 blocks (from offset 128 to the end address), the size of UART0 Rx\_FIFO can be increased to 2 blocks (from offset 256 to the end address), but the size of UART1 Rx\_FIFO cannot be increased. Please note that expanding one FIFO may take up the default space of other FIFOs. For example, by setting \hyperref[fielddesc:UARTTXSIZE]{UART\_TX\_SIZE} of UART0 to 2, the size of UART0 Tx\_FIFO is increased by 128 bytes (from offset 0 to offset 255). In this case, UART0 Tx\_FIFO takes up the default space for UART1 Tx\_FIFO, and UART1's transmitting function cannot be used as a result.

When neither of the two UART controllers is active, RAM could enter low-power mode by setting \hyperref[fielddesc:UARTMEMFORCEPD]{UART\_MEM\_FORCE\_PD}.

UART0 Tx\_FIFO and UART1 Tx\_FIFO are reset by setting \hyperref[fielddesc:UARTTXFIFORST]{UART\_TXFIFO\_RST}. UART0 Rx\_FIFO and UART1 Rx\_FIFO are reset by setting \hyperref[fielddesc:UARTRXFIFORST]{UART\_RXFIFO\_RST}.

Data to be sent is written to TX FIFO via the APB bus or using DMA, read automatically, and converted from a frame into a bitstream by hardware Tx\_FSM; data received is converted from a bitstream into a frame by hardware Rx\_FSM, written into RX FIFO, and then stored into RAM via the APB bus or using DMA. The two UART controllers share one DMA controller.

The empty signal threshold for Tx\_FIFO is configured by setting \hyperref[fielddesc:UARTTXFIFOEMPTYTHRHD]{UART\_TXFIFO\_EMPTY\_THRHD}. When data stored in Tx\_FIFO is less than \hyperref[fielddesc:UARTTXFIFOEMPTYTHRHD]{UART\_TXFIFO\_EMPTY\_THRHD}, a UART\_TXFIFO\_EMPTY\_INT interrupt is generated.

The full signal threshold for Rx\_FIFO is configured by setting \hyperref[fielddesc:UARTRXFIFOFULLTHRHD]{UART\_RXFIFO\_FULL\_THRHD}. When data stored in Rx\_FIFO is greater than \hyperref[fielddesc:UARTRXFIFOFULLTHRHD]{UART\_RXFIFO\_FULL\_THRHD}, a UART\_RXFIFO\_FULL\_INT interrupt is generated. In addition, when Rx\_FIFO receives more data than its capacity, a UART\_RXFIFO\_OVF\_INT interrupt is generated.

UART\textcolor{red}{\it{n}} can access FIFO via register \hyperref[regdesc:UARTFIFOREG]{UART\_FIFO\_REG}.

\subsection{Baud Rate Generation and Detection}

\subsubsection{Baud Rate Generation}

Before a UART controller sends or receives data, the baud rate should be configured by setting corresponding registers. A UART Controller baud rate generator functions by dividing the input clock source. It can divide the clock source by a fractional amount. The divisor is configured by \hyperref[regdesc:UARTCLKDIVREG]{UART\_CLKDIV\_REG}: \hyperref[fielddesc:UARTCLKDIV]{UART\_CLKDIV} for the integral part, and \hyperref[fielddesc:UARTCLKDIVFRAG]{UART\_CLKDIV\_FRAG} for the fractional part. When using an 80 MHz input clock, the UART controller supports a maximum baud rate of 5 Mbaud.

The divisor of the baud rate divider is equal to \hyperref[fielddesc:UARTCLKDIV]{UART\_CLKDIV} + (\hyperref[fielddesc:UARTCLKDIVFRAG]{UART\_CLKDIV\_FRAG} / 16), meaning that the final baud rate is equal to
INPUT\_FREQ / (\hyperref[fielddesc:UARTCLKDIV]{UART\_CLKDIV} + (\hyperref[fielddesc:UARTCLKDIVFRAG]{UART\_CLKDIV\_FRAG} / 16)). For example, if \hyperref[fielddesc:UARTCLKDIV]{UART\_CLKDIV} = 694 and \hyperref[fielddesc:UARTCLKDIVFRAG]{UART\_CLKDIV\_FRAG} = 7
then the divisor value is (694 + 7/16) = 694.4375. If the input clock frequency is 80MHz APB\_CLK, the baud rate will be
(80MHz / 69.4375) = 115201.

When \hyperref[fielddesc:UARTCLKDIVFRAG]{UART\_CLKDIV\_FRAG} is zero, the baud rate generator is an integer clock divider where an output pulse is generated every
\hyperref[fielddesc:UARTCLKDIV]{UART\_CLKDIV} input pulses.

When \hyperref[fielddesc:UARTCLKDIVFRAG]{UART\_CLKDIV\_FRAG} is not zero, the divider is fractional and the output baud rate clock pulses are not strictly uniform.
As shown in figure \ref{Figure:uart_frag_div}, for every 16 output pulses, the generator divides either (\hyperref[fielddesc:UARTCLKDIV]{UART\_CLKDIV} + 1) input pulses or \hyperref[fielddesc:UARTCLKDIV]{UART\_CLKDIV} input pulses per
output pulse. A total of \hyperref[fielddesc:UARTCLKDIVFRAG]{UART\_CLKDIV\_FRAG} output pulses are generated by dividing (\hyperref[fielddesc:UARTCLKDIV]{UART\_CLKDIV} + 1) input pulses, and the remaining
(16 - \hyperref[fielddesc:UARTCLKDIVFRAG]{UART\_CLKDIV\_FRAG}) output pulses are generated by dividing \hyperref[fielddesc:UARTCLKDIV]{UART\_CLKDIV} input pulses.

The output pulses are interleaved as shown in figure \ref{Figure:uart_frag_div} below, to make the output timing more uniform:

\begin{figure}[ht]
    \centerline{\includegraphics[width=1.0\textwidth]{\modulefiles/figures/uart_frag_div_en.eps}}
    \centering
    \caption{UART Controllers Division}
    \label{Figure:uart_frag_div}
\end{figure}

To support IrDA (see Section \ref{SubIrDA} \textit{\nameref{SubIrDA}} for details), the fractional clock divider for IrDA data transmission generates clock signals divided by 16× \hyperref[regdesc:UARTCLKDIVREG]{UART\_CLKDIV\_REG}. This divider works similarly as the one elaborated above: it takes \hyperref[fielddesc:UARTCLKDIV]{UART\_CLKDIV}/16 as the integer value and the lowest four bits of \hyperref[fielddesc:UARTCLKDIV]{UART\_CLKDIV} as the fractional value.

\subsubsection{Baud Rate Detection}\label{baud}

Automatic baud rate detection (Autobaud) on UARTs is enabled by setting \hyperref[fielddesc:UARTAUTOBAUDEN]{UART\_AUTOBAUD\_EN}. The Baudrate\_Detect module shown in figure \ref{Figure:one_uart_structure} will measure pulse widths while filtering any noise whose pulse width is shorter than \hyperref[fielddesc:UARTGLITCHFILT]{UART\_GLITCH\_FILT}.

Before communication starts, the transmitter could send random data to the receiver for baud rate detection. \hyperref[fielddesc:UARTLOWPULSEMINCNT]{UART\_LOWPULSE\_MIN\_CNT} stores the minimum low pulse width, \hyperref[fielddesc:UARTHIGHPULSEMINCNT]{UART\_HIGHPULSE\_MIN\_CNT} stores the minimum high pulse width, \hyperref[fielddesc:UARTPOSEDGEMINCNT]{UART\_POSEDGE\_MIN\_CNT} stores the minimum pulse width between two rising edges, and \hyperref[fielddesc:UARTNEGEDGEMINCNT]{UART\_NEGEDGE\_MIN\_CNT} stores the minimum pulse width between two falling edges. These four registers are read by software to determine the transmitter's baud rate.

\begin{figure}[ht]
    \centerline{\includegraphics[width=0.6\textwidth]{\modulefiles/figures/baud_det}}
    \centering
    \caption{The Timing Diagram of Weak UART Signals Along Falling Edges}
    \label{Figure:baud_det}
\end{figure}

The baud rate can be determined in the following three ways:

\begin{enumerate}
\item Normally, to avoid sampling erroneous data along rising or falling edges in a semi-stable state, which results in the inaccuracy of \hyperref[fielddesc:UARTLOWPULSEMINCNT]{UART\_LOWPULSE\_MIN\_CNT} or \hyperref[fielddesc:UARTHIGHPULSEMINCNT]{UART\_HIGHPULSE\_MIN\_CNT}, use a weighted average of these two values to eliminate errors. In this case, the baud rate is calculated as follows:
    \[
    B_{\textrm{uart}}=\frac{f_{\textrm{clk}}} {\textrm{(\hyperref[fielddesc:UARTLOWPULSEMINCNT]{UART\_LOWPULSE\_MIN\_CNT} + \hyperref[fielddesc:UARTHIGHPULSEMINCNT]{UART\_HIGHPULSE\_MIN\_CNT} +2)/2}}
    \]
\item If UART signals are weak along falling edges as shown in figure \ref{Figure:baud_det}, which leads to an inaccurate average of \hyperref[fielddesc:UARTLOWPULSEMINCNT]{UART\_LOWPULSE\_MIN\_CNT} and \hyperref[fielddesc:UARTHIGHPULSEMINCNT]{UART\_HIGHPULSE\_MIN\_CNT}, use \hyperref[fielddesc:UARTPOSEDGEMINCNT]{UART\_POSEDGE\_MIN\_CNT} to determine the transmitter's baud rate as follows:
    \[
    B_{\textrm{uart}}=\frac{f_{\textrm{clk}}} {\textrm{(\hyperref[fielddesc:UARTPOSEDGEMINCNT]{UART\_POSEDGE\_MIN\_CNT} + 1)/2}}
    \]
\item If UART signals are weak along rising edges, use \hyperref[fielddesc:UARTNEGEDGEMINCNT]{UART\_NEGEDGE\_MIN\_CNT} to determine the transmitter's baud rate as follows:
    \[
    B_{\textrm{uart}}=\frac{f_{\textrm{clk}}} {\textrm{(\hyperref[fielddesc:UARTNEGEDGEMINCNT]{UART\_NEGEDGE\_MIN\_CNT} + 1)/2}}
    \]
\end{enumerate}

\subsection{UART Data Frame}

\begin{figure}[ht]
    \centerline{\includegraphics[width=0.9\textwidth]{\modulefiles/figures/uart_data_frame}}
    \centering
    \caption{Structure of UART Data Frame}
    \label{Figure:uart_data_frame}
\end{figure}

Figure \ref{Figure:uart_data_frame} shows the basic structure of a data frame. A frame starts with one START bit, and ends with STOP bits which can be 1, 1.5, or 2 bits long, configured by \hyperref[fielddesc:UARTSTOPBITNUM]{UART\_STOP\_BIT\_NUM} (in RS485 mode turnaround delay may be added. See details in Section \ref{sec:uart-turnaround-delay}). The START bit is logical low, whereas STOP bits are logical high.

The actual data length can be anywhere between 5 \verb+~+ 8 bit, configured by \hyperref[fielddesc:UARTBITNUM]{UART\_BIT\_NUM}. When \hyperref[fielddesc:UARTPARITYEN]{UART\_PARITY\_EN} is set, a parity bit is added after data bits. \hyperref[fielddesc:UARTPARITY]{UART\_PARITY} is used to choose even parity or odd parity. When the receiver detects a parity bit error in the data received, a UART\_PARITY\_ERR\_INT interrupt is generated and the data received is still stored into RX FIFO. When the receiver detects a data frame error, a UART\_FRM\_ERR\_INT interrupt is generated, and data received by default is stored into RX FIFO.

If all data in Tx\_FIFO has been sent, a UART\_TX\_DONE\_INT interrupt is generated. After this, if the \hyperref[fielddesc:UARTTXDBRK]{UART\_TXD\_BRK} bit is set then the transmitter will enter the Break condition and send several NULL characters in which the TX data line is logical low. The number of NULL characters is configured by \hyperref[fielddesc:UARTTXBRKNUM]{UART\_TX\_BRK\_NUM}. Once the transmitter has sent all NULL characters, a UART\_TX\_BRK\_DONE\_INT interrupt is generated. The minimum interval between data frames can be configured using \hyperref[fielddesc:UARTTXIDLENUM]{UART\_TX\_IDLE\_NUM}. If the transmitter stays idle for \hyperref[fielddesc:UARTTXIDLENUM]{UART\_TX\_IDLE\_NUM} or more time, a UART\_TX\_BRK\\\_IDLE\_DONE\_INT interrupt is generated.

The receiver can also detect the Break conditions when the RX data line remains logical low for one NULL character transmission, and a UART\_BRK\_DET\_INT interrupt will be triggered to detect that a Break condition has been completed.

The receiver can detect the current bus state through the timeout interrupt UART\_RXFIFO\_TOUT\_INT. The UART\_RXFIFO\_TOUT\_INT interrupt will be triggered when the bus is in the idle state for more than \hyperref[fielddesc:UARTRXTOUTTHRHD]{UART\_RX\_TOUT\_THRHD} bit time on current baud rate after the receiver has received at least one byte. You can use this interrupt to detect whether all the data from the transmitter has been sent.

\subsection{AT\_CMD Character Structure}

\begin{figure}[ht]
    \centerline{\includegraphics[width=0.8\textwidth]{\modulefiles/figures/uart_at_cmd}}
    \centering
    \caption{AT\_CMD Character Structure}
    \label{Figure:uart_at_cmd}
\end{figure}

Figure \ref{Figure:uart_at_cmd} is the structure of a special character AT\_CMD. If the receiver constantly receives AT\_CMD\_CHAR and the following conditions are met, a UART\_AT\_CMD\_CHAR\_DET\_INT interrupt is generated.

\begin{itemize}
    \item The interval between the first AT\_CMD\_CHAR and the last non AT\_CMD\_CHAR character is at least \hyperref[fielddesc:UARTPREIDLENUM]{UART\\\_PRE\_IDLE\_NUM} single-bit cycles.
    \item The interval between two AT\_CMD\_CHAR characters is less than \hyperref[fielddesc:UARTRXGAPTOUT]{UART\_RX\_GAP\_TOUT} single-bit cycles.
    \item The number of AT\_CMD\_CHAR characters is equal to or greater than \hyperref[fielddesc:UARTCHARNUM]{UART\_CHAR\_NUM}.
    \item The interval between the last AT\_CMD\_CHAR character and next non AT\_CMD\_CHAR character is at least \hyperref[fielddesc:UARTPOSTIDLENUM]{UART\_POST\_IDLE\_NUM} single-bit cycles.
\end{itemize}

\subsection{RS485}

The two UART controllers support the RS485 standard. This standard uses differential signals to transmit data, so it can communicate over longer distances at higher bit rates than RS232. RS485 has two-wire half-duplex mode and four-wire full-duplex mode. UART controllers support two-wire half-duplex transmission and bus snooping. In a two-wire RS485 multidrop network, there can be 32 slaves at most.

\subsubsection{Driver Control}

As shown in figure \ref{Figure:rs485}, in a two-wire multidrop network, an external RS485 transceiver is needed for differential to single-ended conversion. An RS485 transceiver contains a driver and a receiver. When a UART controller is not in transmitter mode, the connection to the differential line can be broken by disabling the driver. When DE is 1, the driver is enabled; when DE is 0, the driver is disabled.

The UART receiver converts differential signals to single-ended signals via an external receiver. RE is the enable control signal for the receiver. When RE is 0, the receiver is enabled; when RE is 1, the receiver is disabled. If RE is configured as 0, the UART controller is allowed to snoop data on the bus, including data sent by itself.

DE can be controlled by either software or hardware. To reduce cost of software, in our design DE is controlled by hardware. As shown in figure \ref{Figure:rs485}, DE is connected to dtrn\_out of UART (please refer to Section\ref{hd_flow_control} \textit{\nameref{hd_flow_control}} for more details).

\begin{figure}[ht]
    \centerline{\includegraphics[width=0.5\textwidth]{\modulefiles/figures/rs485}}
    \centering
    \caption{Driver Control Diagram in RS485 Mode}
    \label{Figure:rs485}
\end{figure}

\subsubsection{Turnaround Delay}\label{sec:uart-turnaround-delay}

By default, the two UART controllers work in receiver mode. When a UART controller is switched from transmitter mode to receiver mode, the RS485 protocol requires a turnaround delay of at least one cycle after the stop bit. The transmitter supports a turnaround delay of two cycles added after the stop bit. When \hyperref[fielddesc:UARTDL1EN]{UART\_DL1\_EN} is set, a turnaround delay of one single-bit cycle is added; when \hyperref[fielddesc:UARTDL0EN]{UART\_DL0\_EN} is set, turnaround delay of a second cycle is added.

\subsubsection{Bus Snooping}

By default, an RS485 device is not allowed to transmit and receive data simultaneously. However, the UART controller peripheral supports snooping this bus by receiving while transmitting. If \hyperref[fielddesc:UARTRS485TXRXEN]{UART\_RS485TX\_RX\_EN} is set and the external RS485 transceiver is configured as in figure \ref{Figure:rs485}, a UART controller may receive data in transmitter mode and snoop the bus. If \hyperref[fielddesc:UARTRS485RXBYTXEN]{UART\_RS485RXBY\_TX\_EN} is set, a UART controller may transmit data in receiver mode.

The two UART controllers can snoop data sent by themselves. In transmitter mode, when a UART controller monitors a collision between data sent and data received, a UART\_RS485\_CLASH\_INT is generated; when a UART controller monitors a data frame error, a UART\_RS485\_FRM\_ERR\_INT interrupt is generated; when a UART controller monitors a polarity error, a UART\_RS485\_PARITY\_ERR\_INT is generated.

\subsection{IrDA} \label{SubIrDA}

IrDA protocol consists of three layers, namely the physical layer, the link access protocol and the link management protocol. The two UART controllers implement IrDA physical layer. In IrDA encoding, a UART controller supports data rates up to 115.2 kbit/s (SIR, or serial infrared mode). As shown in figure \ref{Figure:sir_endec}, the IrDA encoder converts a NRZ (non-return to zero code) signal to a RZI (return to zero inverted code) signal and sends it to the external driver and infrared LED. This encoder uses modulated signals whose pulse width is 3/16 bits to indicate logic ``0”, and low levels to indicate logic ``1”. The IrDA decoder receives signals from the infrared receiver and converts them to NRZ signals. In most cases, the receiver is high when it is idle, and the encoder output polarity is the opposite of the decoder input polarity. If a low pulse is detected, it indicates that a start bit has been received.

When IrDA function is enabled, one bit is divided into 16 clock cycles. If the bit to be sent is zero, then the 9th, 10th and 11th clock cycle are high.

\begin{figure}[ht]
    \centerline{\includegraphics[width=0.8\textwidth]{\modulefiles/figures/sir_endec}}
    \centering
    \caption{The Timing Diagram of Encoding and Decoding in SIR mode}
    \label{Figure:sir_endec}
\end{figure}

The IrDA transceiver is half-duplex, meaning that it cannot send and receive data simultaneously. As shown in figure \ref{Figure:IrDA_arc}, IrDA function is enabled by setting \hyperref[fielddesc:UARTIRDAEN]{UART\_IRDA\_EN}. When \hyperref[fielddesc:UARTIRDATXEN]{UART\_IRDA\_TX\_EN} is set (high), the IrDA transceiver is enabled to send data and not allowed to receive data; when \hyperref[fielddesc:UARTIRDATXEN]{UART\_IRDA\_TX\_EN} is reset (low), the IrDA transceiver is enabled to receive data and not allowed to send data.

\subsection{Wake-up}

UART0 and UART1 can be set as wake-up source. When a UART controller is in Light-sleep mode, Wakeup\_Ctrl counts up the rising edges of rxd\_in. When the number of rising edges is equal to or greater than (\hyperref[fielddesc:UARTACTIVETHRESHOLD]{UART\_ACTIVE\_THRESHOLD} + 3), a wake\_up signal is generated and sent to RTC, which then wakes up ESP32-S2.

After the chip is woken up by UART, it is necessary to clear the wake\_up signal by transmitting data to UART in Active mode or resetting the whole UART, otherwise the number of rising edges required for the next wakeup will be reduced.

\subsection{Flow Control}

UART controllers have two ways to control data flow, namely hardware flow control and software flow control. Hardware flow control is achieved using output signal rtsn\_out and input signal dsrn\_in. Software flow control is achieved by inserting special characters in the data flow sent and detecting special characters in the data flow received.

\subsubsection{Hardware Flow Control}\label{hd_flow_control}


\begin{figure}[ht]
    \centerline{\includegraphics[width=0.7\textwidth]{\modulefiles/figures/uart_flow_control}}
    \centering
    \caption{Hardware Flow Control Diagram}
    \label{Figure:uart_flow_control}
\end{figure}

Figure \ref{Figure:uart_flow_control} shows the hardware flow control of a UART controller. Hardware flow control uses output signal rtsn\_out and input signal dsrn\_in. Figure \ref{Figure:hardware_flow_control_connect_figure} illustrates how these signals are connected between ESP32-S2 UART (hereinafter referred to as IU0) and the external UART (hereinafter referred to as EU0).

When rtsn\_out of IU0 is low, EU0 is allowed to send data; when rtsn\_out of IU0 is high, EU0 is notified to stop sending data until rtsn\_out of IU0 returns to low. Output signal rtsn\_out can be controlled in two ways.

\begin{itemize}
    \item Software control: Enter this mode by setting \hyperref[fielddesc:UARTRXFLOWEN]{UART\_RX\_FLOW\_EN} to 0. In this mode, the level of rtsn\_out is changed by configuring \hyperref[fielddesc:UARTSWRTS]{UART\_SW\_RTS}.
    \item Hardware control: Enter this mode by setting \hyperref[fielddesc:UARTRXFLOWEN]{UART\_RX\_FLOW\_EN} to 1. In this mode, rtsn\_out is pulled high when data in Rx\_FIFO exceeds \hyperref[fielddesc:UARTRXFLOWTHRHD]{UART\_RX\_FLOW\_THRHD}.
\end{itemize}

When ctsn\_in of IU0 is low, IU0 is allowed to send data; when ctsn\_in is high, IU0 is not allowed to send data. When IU0 detects an edge change of ctsn\_in, a UART\_CTS\_CHG\_INT interrupt is generated.

If dtrn\_out of IU0 is high, it indicates that IU0 is ready to transmit data. dtrn\_out is generated by configuring register \hyperref[fielddesc:UARTSWDTR]{UART\_SW\_DTR}. When the IU0 transmitter detects a edge change of dsrn\_in, a UART\_DSR\_CHG\_INT interrupt is generated. After this interrupt is detected, software can obtain the level of input signal dsrn\_in by reading \hyperref[fielddesc:UARTDSRN]{UART\_DSRN}. If dsrn\_in is high, it indicates that EU0 is ready to transmit data.

In a two-wire RS485 multidrop network enabled by setting \hyperref[fielddesc:UARTRS485EN]{UART\_RS485\_EN}, dtrn\_out is used for transmit/receive turnaround. In this case, dtrn\_out is generated by hardware. When data transmission starts, dtrn\_out is pulled high and the external driver is enabled; when data transmission completes, dtrn\_out is pulled low and the external driver is disabled. Please note that when there is a turnaround delay of one cycle added after the stop bit, dtrn\_out is pulled low after the delay.

UART loopback test is enabled by setting \hyperref[fielddesc:UARTLOOPBACK]{UART\_LOOPBACK}. In the test, UART output signal txd\_out is connected to its input signal rxd\_in, rtsn\_out is connected to ctsn\_in, and dtrn\_out is connected to dsrn\_out. If the data sent matches the data received, it indicates that UART controllers are working properly.

\subsubsection{Software Flow Control}

Instead of CTS/RTS lines, software flow control uses XON/XOFF characters to start or stop data transmission. Such flow control is enabled by setting \hyperref[fielddesc:UARTSWFLOWCONEN]{UART\_SW\_FLOW\_CON\_EN} to 1.

When using software flow control, hardware automatically detects if there are XON/XOFF characters in the data flow received, and generate a UART\_SW\_XOFF\_INT or a UART\_SW\_XON\_INT interrupt accordingly. If an XOFF character is detected, the transmitter stops data transmission once the current byte has been transmitted; if an XON character is detected, the transmitter starts data transmission. In addition, software can force the transmitter to stop sending data by setting \hyperref[fielddesc:UARTFORCEXOFF]{UART\_FORCE\_XOFF}, or to start sending data by setting \hyperref[fielddesc:UARTFORCEXON]{UART\_FORCE\_XON}.

Software determines whether to insert flow control characters according to the remaining room in RX FIFO. When \hyperref[fielddesc:UARTSENDXOFF]{UART\_SEND\_XOFF} is set, the transmitter sends an XOFF character configured by \hyperref[fielddesc:UARTXOFFCHAR]{UART\_XOFF\_CHAR} after the current byte in transmission; when \hyperref[fielddesc:UARTSENDXON]{UART\_SEND\_XON} is set, the transmitter sends an XON character configured by \hyperref[fielddesc:UARTXONCHAR]{UART\_XON\_CHAR} after the current byte in transmission. If the RX FIFO of a UART controller stores more data than \hyperref[fielddesc:UARTXOFFTHRESHOLD]{UART\_XOFF\_THRESHOLD}, \hyperref[fielddesc:UARTSENDXOFF]{UART\_SEND\_XOFF} is set by hardware. As a result, the transmitter sends an XOFF character after the current byte in transmission. If the RX FIFO of a UART controller stores less data than \hyperref[fielddesc:UARTXONTHRESHOLD]{UART\_XON\_THRESHOLD}, \hyperref[fielddesc:UARTSENDXON]{UART\_SEND\_XON} is set by hardware. As a result, the transmitter sends an XON character after the current byte in transmission.

\subsection{UDMA}

The two UART controllers on ESP32-S2 share one UDMA (UART DMA), which supports the decoding and encoding of HCI data packets. For more information, please refer to Chapter \ref{mod:dma}: \textit{\nameref{mod:dma}}.

\subsection{UART Interrupts}
\begin{itemize}
\item \label{int:UARTATCMDCHARDETINT}UART\_AT\_CMD\_CHAR\_DET\_INT: Triggered when the receiver detects an AT\_CMD character.
\item \label{int:UARTRS485CLASHINT}UART\_RS485\_CLASH\_INT: Triggered when a collision is detected between the transmitter and the receiver in RS485 mode.
\item \label{int:UARTRS485FRMERRINT}UART\_RS485\_FRM\_ERR\_INT: Triggered when an error is detected in the data frame sent by the transmitter in RS485 mode.
\item \label{int:UARTRS485PARITYERRINT}UART\_RS485\_PARITY\_ERR\_INT: Triggered when an error is detected in the parity bit sent by the transmitter in RS485 mode.
\item \label{int:UARTTXDONEINT}UART\_TX\_DONE\_INT: Triggered when all data in the transmitter's TX FIFO has been sent.
\item \label{int:UARTTXBRKIDLEDONEINT}UART\_TX\_BRK\_IDLE\_DONE\_INT: Triggered when the transmitter stays idle for the minimum interval (threshold) after sending the last data bit.
\item \label{int:UARTTXBRKDONEINT}UART\_TX\_BRK\_DONE\_INT: Triggered when the transmitter sends a NULL character after all data in TX FIFO has been sent.
\item \label{int:UARTGLITCHDETINT}UART\_GLITCH\_DET\_INT: Triggered when the receiver detects a glitch in the middle of the start bit.
\item \label{int:UARTSWXOFFINT}UART\_SW\_XOFF\_INT: Triggered when \hyperref[fielddesc:UARTSWFLOWCONEN]{UART\_SW\_FLOW\_CON\_EN} is set and the receiver receives a XOFF character.
\item \label{int:UARTSWXONINT}UART\_SW\_XON\_INT: Triggered when \hyperref[fielddesc:UARTSWFLOWCONEN]{UART\_SW\_FLOW\_CON\_EN} is set and the receiver receives a XON character.
\item \label{int:UARTRXFIFOTOUTINT}UART\_RXFIFO\_TOUT\_INT: Triggered when the receiver takes more time than \hyperref[fielddesc:UARTRXTOUTTHRHD]{UART\_RX\_TOUT\_THRHD} to receive one byte.
\item \label{int:UARTBRKDETINT}UART\_BRK\_DET\_INT: Triggered when the receiver detects a NULL character (i.e. logic 0 for one NULL character transmission) after stop bits.
\item \label{int:UARTCTSCHGINT}UART\_CTS\_CHG\_INT: Triggered when the receiver detects an edge change of CTSn signals.
\item \label{int:UARTDSRCHGINT}UART\_DSR\_CHG\_INT: Triggered when the receiver detects an edge change of DSRn signals.
\item \label{int:UARTRXFIFOOVFINT}UART\_RXFIFO\_OVF\_INT: Triggered when the receiver receives more data than the capacity of RX FIFO.
\item \label{int:UARTFRMERRINT}UART\_FRM\_ERR\_INT: Triggered when the receiver detects a data frame error.
\item \label{int:UARTPARITYERRINT}UART\_PARITY\_ERR\_INT: Triggered when the receiver detects a parity error.
\item \label{int:UARTTXFIFOEMPTYINT}UART\_TXFIFO\_EMPTY\_INT: Triggered when TX FIFO stores less data than what \hyperref[fielddesc:UARTTXFIFOEMPTYTHRHD]{UART\_TXFIFO\_EMPTY\_THRHD} specifies.
\item \label{int:UARTRXFIFOFULLINT}UART\_RXFIFO\_FULL\_INT: Triggered when the receiver receives more data than what \hyperref[fielddesc:UARTRXFIFOFULLTHRHD]{UART\_RXFIFO\_FULL\_THRHD} specifies.
\item \label{int:UARTWAKEUPINT}UART\_WAKEUP\_INT: Triggered when UART is woken up.
\end{itemize}
