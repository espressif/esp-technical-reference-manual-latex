\hypertarget{efuse}{}
\chapter{eFuse Controller (EFUSE)}
\label{mod:efuse}


\section{Overview}

ESP32-S2 has a 4096-bit eFuse that stores parameters in the SoC. Once an eFuse bit is programmed to 1, it can never be reverted to 0. Users can instruct the eFuse Controller to program individual bits for individual parameters as needed. From outside the chip, eFuse data can only be read via the eFuse Controller. If read-protection for some data is not enabled, that data is readable from outside the chip. If read-protection is enabled, that data can not be read from outside the chip. In all cases, however, some keys stored in eFuse can still be used internally by hardware cryptography modules such as RSA Digital Signature Peripheral (RSA\_DS), HMAC, etc., without exposing this data to the outside world.
 \section{Features}

\begin{itemize}
    \item One-time programmable storage
    \item Configurable write protection
    \item Configurable read protection
    \item Parameters use different hardware encoding schemes to protect against corruption
\end{itemize}


\section{Functional Description}

\subsection{Structure}

There are 11 eFuse blocks (BLOCK0 \verb+~+ BLOCK10).

BLOCK0 holds most core system parameters. Among these parameters, 24 bits are readable but useless to users, 38 bits are reserved for future use.

Table \ref{table:BLOCK0_Parameter} lists all the parameters accessible (readable and usable) to users in BLOCK0 and their offsets, bit widths, functional description, as well as information about whether their configuration is directly accessible by the hardware, and whether they are protected from programming.

The {\bfseries \hyperref[fielddesc:EFUSEWRDIS]{EFUSE\_WR\_DIS}} parameter is used to restrict the programming of other parameters, while {\bfseries \hyperref[fielddesc:EFUSERDDIS]{EFUSE\_RD\_DIS}} is used to restrict users from reading BLOCK4 \verb+~+ BLOCK10. More information on these two parameters can be found in sections \ref{section:EFUSEWRDISABLE} and \ref{section:EFUSERDDISABLE}.

Among these blocks, BLOCK4 \verb+~+ 9 stores KEY0 \verb+~+ 5, respectively. Up to six 256-bit keys can be programmed into eFuse. BLOCK1 \verb+~+ BLOCK10 use the RS coding scheme, so there are some restrictions on writing to these parameters (refer to Section \ref{datastorage}: \textit{Data Storage} and Section \ref{Software Programming of Parameters}: \textit{User Programming of Parameters}).


\subsubsection{EFUSE\_WR\_DIS}\label{section:EFUSEWRDISABLE}

Parameter \hyperref[fielddesc:EFUSEWRDIS]{EFUSE\_WR\_DIS} determines whether individual eFuse parameters are write-protected. After burning \hyperref[fielddesc:EFUSEWRDIS]{EFUSE\_WR\_DIS}, execute an eFuse read operation so the new values will take effect (refer to \textit{Updating eFuse read registers} in Section \ref{updatingRDregister}).

The columns ``Programming-Protected by \hyperref[fielddesc:EFUSEWRDIS]{EFUSE\_WR\_DIS}" in Table  \ref{table:BLOCK0_Parameter} and Table \ref{table:BLOCK1_10_Parameter} list the specific bits of \hyperref[fielddesc:EFUSEWRDIS]{EFUSE\_WR\_DIS} that determine the write protected status of each parameter.

When the corresponding bit is 0, the parameter is not write protected and can be programmed if the parameter has not been programmed.

When the corresponding bit is 1, the parameter is write protected and none of its bits can be modified. The non-programmed bits always remain 0, while programmed bits always remain 1.

\subsubsection{EFUSE\_RD\_DIS}\label{section:EFUSERDDISABLE}

Only the eFuse blocks BLOCK4 \verb+~+ BLOCK10 can be individually read protected to prevent any access from outside the chip. The corresponding bit in \hyperref[fielddesc:EFUSERDDIS]{EFUSE\_RD\_DIS} is shown in Table  \ref{table:BLOCK1_10_Parameter}. After burning \hyperref[fielddesc:EFUSERDDIS]{EFUSE\_RD\_DIS}, execute an eFuse read operation so the new values take effect (refer to \textit{Updating eFuse read registers} in Section \ref{updatingRDregister}).

If the corresponding \hyperref[fielddesc:EFUSERDDIS]{EFUSE\_RD\_DIS} bit is 0, then the eFuse block can be read by users. If the corresponding \hyperref[fielddesc:EFUSERDDIS]{EFUSE\_RD\_DIS} bit is 1, then the parameter controlled by this bit is user read protected.

Other parameters that are not in BLOCK4 \verb+~+ BLOCK10 can always be read by users.

When BLOCK4 \verb+~+ BLOCK10 are set to read-protected, the data in these blocks are not readable by users, but they can still be used internally by hardware cryptography modules, if the \hyperref[fielddesc:EFUSEKEYPURPOSE3]{EFUSE\_KEY\_PURPOSE\_\textcolor{red}{\it{n}}} bit is set accordingly.


\subsubsection{Data Storage}\label{datastorage}

Internal to the SoC, eFuses use hardware encoding schemes to protect against data corruption.

All BLOCK0 parameters except for \hyperref[fielddesc:EFUSEWRDIS]{EFUSE\_WR\_DIS} are stored with four backups, meaning each bit is stored four times. This backup scheme is hardwired to the chip.

BLOCK1 \verb+~+ BLOCK10 use RS (44, 32) coding scheme that supports up to 6 bytes of automatic error correction. The primitive polynomial of RS (44, 32) is \(p(x)=x^8+x^4+x^3+x^2+1\).

    \caption{Shift Register Circuit (first 32 output)}
    \label{Figure:lfsr_circuit_1}
\end{figure}

\begin{figure}[ht]
    \centering
    \fbox{\includegraphics[width=1.0\textwidth]{\modulefiles/figures/lfsr_circuit_20190809_2}}
    \caption{Shift Register Circuit (last 12 output)}
    \label{Figure:lfsr_circuit_2}
\end{figure}

The shift register circuit shown in Figure \ref{Figure:lfsr_circuit_1} and \ref{Figure:lfsr_circuit_2} processes 32 data bytes using RS (44, 32). This coding scheme encodes 32 bytes of  data into 44 bytes:

\begin{itemize}
    \item Bytes [0:31] are the data bytes itself
    \item Bytes [32:43] are the encoded parity bytes stored in 8-bit flip-flops DFF1, DFF2, ..., DFF12 (gf\_mul\_n, where n is an integer, is the result of multiplying a byte of data ...)
\end{itemize}

After that, the hardware burns into eFuse the 44-byte codeword consisting of the data bytes followed by the parity bytes.

When the eFuse block is read back, the eFuse controller automatically decodes the codeword  and applies error correction if needed.

Because the RS check codes are generated across the entire 256-bit eFuse block, each block can only be written to one time.

\subsection{Programming of Parameters}\label{Software Programming of Parameters}

The eFuse controller can only write to eFuse parameters in one block at a time. BLOCK0 \verb+~+ BLOCK10 share the same registers to store the parameters to be programmed. Configure parameter \hyperref[fielddesc:EFUSEBLKNUM]{EFUSE\_BLK\_NUM} to indicate which block is to be programmed.

{\bfseries Programming BLOCK0}

When \hyperref[fielddesc:EFUSEBLKNUM]{EFUSE\_BLK\_NUM} = 0, BLOCK0 is programmed. The \hyperref[regdesc:EFUSEPGMDATANREG]{EFUSE\_PGM\_DATA0\_REG} register stores \hyperref[fielddesc:EFUSEWRDIS]{EFUSE\_WR\_DIS}. \hyperref[regdesc:EFUSEPGMDATANREG]{EFUSE\_PGM\_DATA1\_REG} \verb+~+ \hyperref[regdesc:EFUSEPGMDATANREG]{EFUSE\_PGM\_DATA5\_REG} store the information of new BLOCK0 parameters to be programmed. Note that 24 BLOCK0 bits are readable but useless to users and must always be set to 0 in the programming registers. The specific bits are:

\begin{itemize}
    \item \hyperref[regdesc:EFUSEPGMDATANREG]{EFUSE\_PGM\_DATA1\_REG}[29:31]
    \item \hyperref[regdesc:EFUSEPGMDATANREG]{EFUSE\_PGM\_DATA1\_REG}[20:23]
    \item \hyperref[regdesc:EFUSEPGMDATANREG]{EFUSE\_PGM\_DATA2\_REG}[7:15]
    \item \hyperref[regdesc:EFUSEPGMDATANREG]{EFUSE\_PGM\_DATA2\_REG}[0:3]
    \item \hyperref[regdesc:EFUSEPGMDATANREG]{EFUSE\_PGM\_DATA3\_REG}[16:19]
\end{itemize}

Values written to \hyperref[regdesc:EFUSEPGMDATANREG]{EFUSE\_PGM\_DATA6\_REG} \verb+~+ \hyperref[regdesc:EFUSEPGMDATANREG]{EFUSE\_PGM\_DATA7\_REG} and \hyperref[regdesc:EFUSEPGMCHECKVALUENREG]{EFUSE\_PGM\_CHECK\_VALUE0\_REG} \verb+~+ \hyperref[regdesc:EFUSEPGMCHECKVALUENREG]{EFUSE\_PGM\_CHECK\_VALUE2\_REG} are ignored when programming BLOCK0.

{\bfseries Programming BLOCK1}

When \hyperref[fielddesc:EFUSEBLKNUM]{EFUSE\_BLK\_NUM} = 1, \hyperref[regdesc:EFUSEPGMDATANREG]{EFUSE\_PGM\_DATA0\_REG} \verb+~+ \hyperref[regdesc:EFUSEPGMDATANREG]{EFUSE\_PGM\_DATA5\_REG} store the parameters to be programmed. \hyperref[regdesc:EFUSEPGMCHECKVALUENREG]{EFUSE\_PGM\_CHECK\_VALUE0\_REG} \verb+~+ \hyperref[regdesc:EFUSEPGMDATANREG]{EFUSE\_PGM\_DATA2\_REG} store the corresponding RS check codes. Values written to \hyperref[regdesc:EFUSEPGMDATANREG]{EFUSE\_PGM\_DATA6\_REG} \verb+~+ \hyperref[regdesc:EFUSEPGMDATANREG]{EFUSE\_PGM\_DATA7\_REG} are ignored when programming BLOCK1, the RS check codes should be calculated as if these bits were all 0.

{\bfseries Programming BLOCK2 \verb+~+ 10}

When \hyperref[fielddesc:EFUSEBLKNUM]{EFUSE\_BLK\_NUM} = 2 \verb+~+ 10, \hyperref[regdesc:EFUSEPGMDATANREG]{EFUSE\_PGM\_DATA0\_REG} \verb+~+ \hyperref[regdesc:EFUSEPGMDATANREG]{EFUSE\_PGM\_DATA7\_REG} store the parameters to be programmed to this block. \hyperref[regdesc:EFUSEPGMCHECKVALUENREG]{EFUSE\_PGM\_CHECK\_VALUE0\_REG} \verb+~+ \hyperref[regdesc:EFUSEPGMCHECKVALUENREG]{EFUSE\_PGM\_CHECK\_VALUE2\_REG} store the corresponding RS check codes.

{\bfseries Programming process}

The process of programming parameters is as follows:

\begin{enumerate}
\item Set \hyperref[fielddesc:EFUSEBLKNUM]{EFUSE\_BLK\_NUM} parameter as described above.

\item Write the parameters to be programmed into registers \hyperref[regdesc:EFUSEPGMDATANREG]{EFUSE\_PGM\_DATA0\_REG} \verb+~+ \hyperref[regdesc:EFUSEPGMDATANREG]{EFUSE\_PGM\_DATA7\_REG} and \hyperref[regdesc:EFUSEPGMCHECKVALUENREG]{EFUSE\_PGM\_CHECK\_VALUE0\_REG} \verb+~+ \hyperref[regdesc:EFUSEPGMCHECKVALUENREG]{EFUSE\_PGM\_CHECK\_VALUE2\_REG}.

\item Ensure the eFuse clock registers are set correctly as described in Section \ref{eFuse-Programming Timing}: \textit{eFuse-Programming Timing}.

\item Ensure the eFuse programming voltage VDDQ is set correctly as described in Section \ref{eFuse VDDQ Timing}: \textit{eFuse VDDQ Setting}.

\item Set \hyperref[fielddesc:EFUSEOPCODE]{EFUSE\_OP\_CODE} field in register \hyperref[regdesc:EFUSECONFREG]{EFUSE\_CONF\_REG} to \hexprefix{}5A5A.

\item Set \hyperref[fielddesc:EFUSEPGMCMD]{EFUSE\_PGM\_CMD} field in register \hyperref[regdesc:EFUSECMDREG]{EFUSE\_CMD\_REG} to 1.

\item Poll register \hyperref[regdesc:EFUSECMDREG]{EFUSE\_CMD\_REG} until it is \hexprefix{}0, or wait for a pgm\_done interrupt. Information on how to identify a pgm/read\_done interrupt is provided at the end of Section \ref{softwarereadparameters}.

\item Clear the parameters written into the register.

\item Trigger an eFuse read operation (see Section \ref{softwarereadparameters}: \textit{User Read of Parameters}) to update eFuse registers with the new values.
\end{enumerate}

{\bfseries Limitations}

For BLOCK0, the programming of different parameters and even the programming of different bits of the same parameter does not need to be done at once. It is, however, recommended that users minimize programming cycles
and program all the bits of a parameter in one programming action. In addition, after all parameters controlled by a certain bit of \hyperref[fielddesc:EFUSEWRDIS]{EFUSE\_WR\_DIS} are programmed, that bit should be immediately programmed. The programming of parameters controlled by a certain bit of \hyperref[fielddesc:EFUSEWRDIS]{EFUSE\_WR\_DIS},
and the programming of the bit itself can even be completed at the same time. Repeated programming of already programmed bits is strictly forbidden, otherwise, programming errors will occur.

BLOCK1 cannot be programmed by users as it has been programmed at manufacturing.

BLOCK2 \verb+~+ 10 can only be programmed once. Repeated programming is not allowed.

{\bfseries Updating eFuse read registers}\label{updatingRDregister}

The eFuse Controller hardware populates the read registers from the internal eFuse storage. This read operation takes place on system reset and can also be triggered manually by users as needed, for example if new eFuse values have been programmed.

The process of triggering an eFuse controller read by users is as follows:

\begin{enumerate}
    \item Configure the eFuse read timing registers as described in Section \ref{eFuse-Read Timing}: \textit{eFuse-Read Timing}.
    \item Set the \hyperref[fielddesc:EFUSEOPCODE]{EFUSE\_OP\_CODE} field in register \hyperref[regdesc:EFUSECONFREG]{EFUSE\_CONF\_REG} to \hexprefix{}5AA5.
    \item Set the \hyperref[fielddesc:EFUSEREADCMD]{EFUSE\_READ\_CMD} field in register \hyperref[regdesc:EFUSECMDREG]{EFUSE\_CMD\_REG} to 1.
    \item Poll the \hyperref[regdesc:EFUSECMDREG]{EFUSE\_CMD\_REG} register until it is \hexprefix{}0, or wait for a read\_done interrupt. Information on how to identify a pgm/read\_done interrupt is provided below.
    \item Read the values of each parameter from memory.
\end{enumerate}
The eFuse read registers will now hold updated values for all eFuse parameters.

{\bfseries Error detection}

Error registers allow users to detect an inconsistency in the stored eFuses.

EFUSE\_RD\_REPEAT\_ERR0 \verb+~+ 3\_REG indicate inconsistencies in the stored backup copies of the parameters in BLOCK0 (except for \hyperref[fielddesc:EFUSEWRDIS]{EFUSE\_WR\_DIS}) . Value 1 indicates an error was detected, and the bit became invalid. Value 0 indicates no error.

Registers EFUSE\_RD\_RS\_ERR0 \verb+~+ 1\_REG store the number of corrected bytes as well as the result of RS decoding during eFuse reading BLOCK1 \verb+~+ BLOCK10.

Users can read the values of above registers only after the eFuse read registers have been updated.

{\bfseries Identifying program/read operation completion}\label{tip:program and read done interrupt}

    The two methods to identify the completion of program/read operation are described below. Please note that bit 1 corresponds to program operation, and bit 0 corresponds to read operation.
    \begin{itemize}
    \item Method one:
        \begin{enumerate}
            \item Poll bit 1/0 in register \hyperref[regdesc:EFUSEINTRAWREG]{EFUSE\_INT\_RAW\_REG} until it is 1, which represents the completion of a program/read operation.
        \end{enumerate}
    \item Method two:
        \begin{enumerate}
            \item Set bit 1/0 in register \hyperref[regdesc:EFUSEINTENAREG]{EFUSE\_INT\_ENA\_REG} to 1 to enable eFuse Controller to post a pgm/read\_done interrupt.
            \item Configure Interrupt Matrix to enable the CPU to respond to eFuse interrupt signal.
            \item Wait for the pgm/read\_done interrupt.
            \item Set the bit 1/0 in register \hyperref[regdesc:EFUSEINTCLRREG]{EFUSE\_INT\_CLR\_REG} to 1 to clear the pgm/read\_done interrupt.
    \end{enumerate}
\end{itemize}

\subsection{The Use of Parameters by Hardware Modules}

Hardware modules are directly hardwired to the ESP32-S2 in order to use the parameters listed in Table \ref{table:BLOCK0_Parameter} and \ref{table:BLOCK1_10_Parameter}, specifically those marked with ``Y" in columns ``Accessible by Hardware". Users cannot change this behavior.

\subsection{Interrupts}
\begin{itemize}
\item pgm\_done interrupt: Triggered when eFuse programming has finished. To enable this interrupt, set \hyperref[fielddesc:EFUSEPGMDONEINTENA]{EFUSE\_PGM\_DONE\_INT\_ENA} to 1.
\item read\_done interrupt: Triggered when eFuse reading has finished. To enable this interrupt, set \hyperref[fielddesc:EFUSEREADDONEINTENA]{EFUSE\_READ\_DONE\_INT\_ENA} to 1.
\end{itemize}
