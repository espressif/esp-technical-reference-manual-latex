\chapter{AES Accelerator (AES)}\label{mod:aes}

\section{Introduction}

ESP32-S2 integrates an Advanced Encryption Standard (AES) Accelerator, which is a hardware device that speeds up AES Algorithm significantly, compared to AES algorithms implemented solely in software. The AES Accelerator integrated in ESP32-S2 has two working modes, which are \hyperref[subsection:typicalaes]{Typical AES} and \hyperref[subsection:dma-aes]{DMA-AES}.

\section{Features}

The following functionality is supported:
\begin{itemize}
    \item Typical AES working mode
        \begin{itemize}
            \item AES-128/AES-192/AES-256 encryption and decryption
            \item Four variations of key endianness and four variations of text endianness
        \end{itemize}
    \end{itemize}

    \begin{itemize}
    \item DMA-AES working mode
        \begin{itemize}
            \item Block mode
                \begin{itemize}
                    \item ECB (Electronic Codebook)
                    \item CBC (Cipher Block Chaining)
                    \item OFB (Output Feedback)
                    \item CTR (Counter)
                    \item CFB8 (8-bit Cipher Feedback)
                    \item CFB128 (128-bit Cipher Feedback)
                \end{itemize}
            \item GCM (Galois/Counter Mode)
            \item Interrupt on completion of computation
        \end{itemize}
    \end{itemize}

\section{Working Modes}

The AES Accelerator integrated in ESP32-S2 has two working modes, which are \hyperref[subsection:typicalaes]{Typical AES} and \hyperref[subsection:dma-aes]{DMA-AES}.

\begin{itemize}
    \item Typical AES Working Mode: supports AES-128/AES-192/AES-256 encryption and decryption under \href{https://nvlpubs.nist.gov/nistpubs/FIPS/NIST.FIPS.197.pdf}{NIST FIPS 197}. In this working mode, the plaintext and ciphertext is written and read via CPU directly.
    \item DMA-AES Working Mode: supports block cipher algorithms ECB/CBC/OFB/CTR/CFB8/CFB128 under \href{https://nvlpubs.nist.gov/nistpubs/Legacy/SP/nistspecialpublication800-38a.pdf}{NIST SP 800-38A}, and GCM mode of operation under \href{https://nvlpubs.nist.gov/nistpubs/Legacy/SP/nistspecialpublication800-38d.pdf}{NIST SP 800-38D}. In this working mode, the plaintext and ciphertext is written and read via crypto DMA. An interrupt will be generated when operation completes.
\end{itemize}

Users can choose the working mode for AES accelerator by configuring the \hyperref[regdesc:AESDMAENABLEREG]{AES\_DMA\_ENABLE\_REG} register according to Table \ref{table:AES_DMA_ENABLE} below.

\begin{longtable}[c]{ | c | l | }
\caption{AES Accelerator Working Mode}
    \label{table:AES_DMA_ENABLE}
    \\\hline
    \rowcolor{lightgray}
\hyperref[regdesc:AESDMAENABLEREG]{AES\_DMA\_ENABLE\_REG}       & Working Mode\\
        \hline
        0                       & Typical AES    \\
        \hline
        1                       & DMA-AES        \\
        \hline
\end{longtable}



For detailed introduction on these two working modes, please refer to Section \ref{subsection:typicalaes} and Section \ref{subsection:dma-aes} below.

\begin{importantlisting}
\vspace{-0.5em}

    ESP32-S2's \nameref{mod:digsig} and \hyperref[mod:extmemencr]{External Memory Manual Encryption} modules also call the AES accelerator. Therefore, users cannot access the AES accelerator when these modules are working.
\end{importantlisting}

\section{Typical AES Working Mode}\label{subsection:typicalaes}

In the Typical AES working mode, the AES accelerator is capable of using cryptographic keys of 128, 192, and 256 bits to encrypt and decrypt data, i.e. AES-128/AES-192/AES-256 encryption and decryption. Users can choose the operation type for AES accelerator working in Typical AES working mode by configuring the \hyperref[regdesc:AESMODEREG]{AES\_MODE\_REG} register according to Table \ref{table:AESOperation} below.

\begin{longtable}[c]{ | c | l | }
\caption{Operation Type under Typical AES Working Mode}
\label{table:AESOperation}
\\\hline
\rowcolor{lightgray}
\hyperref[regdesc:AESMODEREG]{AES\_MODE\_REG}[2:0]     & Operation Type\\
    \hline
    0                   & AES-128 encryption    \\
    \hline
    1                   & AES-192 encryption    \\
    \hline
    2                   & AES-256 encryption    \\
    \hline
    4                   & AES-128 decryption    \\
    \hline

    \hline

    \hline

Users can check the working status of the AES accelerator by inquiring the \hyperref[regdesc:AESSTATEREG]{AES\_STATE\_REG} register and comparing the return value against the Table \ref{table:AESStateValue-aes} below.

\begin{longtable}[c]{ | c | l | l| }

\label{table:AESStateValue-aes}
\\\hline

In the Typical AES working mode, the AES accelerator requires 11 \verb+~+ 15 clock cycles to encrypt a message block, and 21 or 22 clock cycles to decrypt a message block.


\subsection{Key, Plaintext, and Ciphertext}

The encryption or decryption key is stored in \hyperref[regdesc:AESKEYnREG]{AES\_KEY\_\regindex{n}\_REG}, which is a set of eight 32-bit registers.

\begin{itemize}
    \item For AES-128 encryption/decryption, the 128-bit key is stored in \hyperref[regdesc:AESKEYnREG]{AES\_KEY\_0\_REG} \verb+~+ \hyperref[regdesc:AESKEYnREG]{AES\_KEY\_3\_REG}.
    \item For AES-192 encryption/decryption, the 192-bit key is stored in \hyperref[regdesc:AESKEYnREG]{AES\_KEY\_0\_REG} \verb+~+ \hyperref[regdesc:AESKEYnREG]{AES\_KEY\_5\_REG}.
    \item For AES-256 encryption/decryption, the 256-bit key is stored in \hyperref[regdesc:AESKEYnREG]{AES\_KEY\_0\_REG} \verb+~+ \hyperref[regdesc:AESKEYnREG]{AES\_KEY\_7\_REG}.
\end{itemize}

The plaintext and ciphertext are stored in \hyperref[regdesc:AESTEXTINnREG]{AES\_TEXT\_IN\_\regindex{m}\_REG} and \hyperref[regdesc:AESTEXTOUTnREG]{AES\_TEXT\_OUT\_\regindex{m}\_REG}, which are two sets of four 32-bit registers.

\begin{itemize}
    \item For AES-128/AES-192/AES-256 encryption, the \hyperref[regdesc:AESTEXTINnREG]{AES\_TEXT\_IN\_\regindex{m}\_REG} registers are initialized with plaintext. Then, the AES Accelerator stores the ciphertext into \hyperref[regdesc:AESTEXTOUTnREG]{AES\_TEXT\_OUT\_\regindex{m}\_REG} after operation.
    \item For AES-128/AES-192/AES-256 decryption, the \hyperref[regdesc:AESTEXTINnREG]{AES\_TEXT\_IN\_\regindex{m}\_REG} registers are initialized with ciphertext. Then, the AES Accelerator stores the plaintext into \hyperref[regdesc:AESTEXTOUTnREG]{AES\_TEXT\_OUT\_\regindex{m}\_REG} after operation.
\end{itemize}


\subsection{Endianness}

\textbf{Text Endianness}

In Typical AES working mode, the AES Accelerator uses cryptograohic keys to encrypt and decrypt data in blocks of 128 bits. The Bit 2 and Bit 3 of the \hyperref[regdesc:AESENDIANREG]{AES\_ENDIAN\_REG} register define the endianness of input text, while the Bit 4 and Bit 5 define the endianness of output text. To be more specific, Bit 2 and Bit 4 control how the four bytes are stored in each word, and Bit 3 and Bit 5 control how the four words are stored in each message block.

Users can choose one of the four text endianness types provided by the AES Accelerator by configuring the \hyperref[regdesc:AESENDIANREG]{AES\_ENDIAN\_REG} register. Details can been seen in Table \ref{table:TypicalAESPlaintextCiphertextEndian}.

\subsection{Operation Process }

\textbf{Single Operation}

\begin{enumerate}
    \item Write 0 to the \hyperref[regdesc:AESDMAENABLEREG]{AES\_DMA\_ENABLE\_REG} register.
    \item Initialize registers \hyperref[regdesc:AESMODEREG]{AES\_MODE\_REG}, \hyperref[regdesc:AESKEYnREG]{AES\_KEY\_\regindex{n}\_REG}, \hyperref[regdesc:AESTEXTINnREG]{AES\_TEXT\_IN\_\regindex{m}\_REG}, and \hyperref[regdesc:AESENDIANREG]{AES\_ENDIAN\_REG}.
    \item Start operation by writing 1 to the \hyperref[regdesc:AESTRIGGERREG]{AES\_TRIGGER\_REG} register.
    \item Wait till the content of the \hyperref[regdesc:AESSTATEREG]{AES\_STATE\_REG} register becomes 0, which indicates the operation is completed.
    \item Read results from the \hyperref[regdesc:AESTEXTOUTnREG]{AES\_TEXT\_OUT\_\regindex{m}\_REG} register.
\end{enumerate}

\textbf{Consecutive Operations}

In consecutive operations, primarily the input \hyperref[regdesc:AESTEXTINnREG]{AES\_TEXT\_IN\_\regindex{m}\_REG} and output \hyperref[regdesc:AESTEXTOUTnREG]{AES\_TEXT\_OUT\_\regindex{m}\_REG} registers are being written and read, while the content of \hyperref[regdesc:AESDMAENABLEREG]{AES\_DMA\_ENABLE\_REG}, \hyperref[regdesc:AESMODEREG]{AES\_MODE\_REG}, \hyperref[regdesc:AESKEYnREG]{AES\_KEY\_\regindex{n}\_REG}, and \hyperref[regdesc:AESENDIANREG]{AES\_ENDIAN\_REG} is kept unchanged. Therefore, the initialization can be simplified during the consecutive operation.

\begin{enumerate}
    \item Write 0 to the \hyperref[regdesc:AESDMAENABLEREG]{AES\_DMA\_ENABLE\_REG} register before starting the first operation.
    \item Initialize registers \hyperref[regdesc:AESMODEREG]{AES\_MODE\_REG}, \hyperref[regdesc:AESKEYnREG]{AES\_KEY\_\regindex{n}\_REG}, and \hyperref[regdesc:AESENDIANREG]{AES\_ENDIAN\_REG} before starting the first operation.
    \item Update the content of \hyperref[regdesc:AESTEXTINnREG]{AES\_TEXT\_IN\_\regindex{m}\_REG}. \label{update register in typical AES continuous calculation}
    \item Start operation by writing 1 to the \hyperref[regdesc:AESTRIGGERREG]{AES\_TRIGGER\_REG} register.
    \item Wait till the content of the \hyperref[regdesc:AESSTATEREG]{AES\_STATE\_REG} register becomes 0, which indicates the operation completes.
    \item Read results from the \hyperref[regdesc:AESTEXTOUTnREG]{AES\_TEXT\_OUT\_\regindex{m}\_REG} register, \label{read result in typical AES continuous calculation} and return to Step \ref{update register in typical AES continuous calculation} to continue the next operation.
\end{enumerate}

\subsection{Key, Plaintext, and Ciphertext}

\textbf{Block Operation}

During the block operations, the AES Accelerator reads source data (in\_stream) from DMA, and write result data (out\_stream) to DMA after the computation.

\begin{itemize}
    \item For encryption, DMA reads plaintext from memory, then passes it to AES as source data. After computation, AES passes ciphertext as result data  back to DMA to write into memory.
    \item For decryption, DMA reads ciphertext from memory, then passes it to AES as source data. After computation, AES passes plaintext as result data back to DMA to write into memory.
\end{itemize}

During block operations, the lengths of the source data and result data are the same. The total computation time is reduced because the DMA data operation and AES computation can happen concurrently.

\label{api:testpadding}The length of source data for AES Accelerator under DMA-AES working mode must be 128 bits or the integral multiples of 128 bits. Otherwise, trailing zeros will be added to the original source data, so the length of source data equals to the nearest integral multiples of 128 bits. Please see details in Table \ref{table:DefineFunctionTextPadding} below.

\subsection{Standard Incrementing Function}\label{Label Label Standard Incrementing Function}

AES accelerator provides two Standard Incrementing Functions for the CTR block operation, which are INC$_{32}$ and INC$_{128}$ Standard Incrementing Functions. By setting the  \hyperref[regdesc:AESINCSELREG]{AES\_INC\_SEL\_REG} register to 0 or 1, users can choose the INC$_{32}$ or INC$_{128}$ functions respectively. For details on the Standard Incrementing Function, please see Chapter B.1 The Standard Incrementing Function in \href{https://nvlpubs.nist.gov/nistpubs/Legacy/SP/nistspecialpublication800-38a.pdf}{NIST SP 800-38A}.

\subsection{ Block Number}\label{Label Block Number}

Register \hyperref[regdesc:AESBLOCKNUMREG]{AES\_BLOCK\_NUM\_REG} stores the Block Number of plaintext $P$ or ciphertext $C$. The length of this register equals to length({\bfseries TEXT-PADDING}($P$))/128 or length({\bfseries TEXT-PADDING}($C$))/128. The AES Accelerator only uses this register when working in the DMA-AES mode.

\subsection{Initialization Vector}\label{Label IV}

\label{memdesc:AESIVMEM}\hyperref[memdesc:AESIVMEM]{AES\_IV\_MEM} is a 16-byte memory, which is only available for AES Accelerator working in block operations. For CBC/OFB/CFB8/CFB128 operations, the \hyperref[memdesc:AESIVMEM]{AES\_IV\_MEM} memory stores the Initialization Vector (IV). For the CTR operation, the \hyperref[memdesc:AESIVMEM]{AES\_IV\_MEM} memory stores the Initial Counter Block (ICB).

Both IV and ICB are 128-bit strings, which can be divided into Byte0, Byte1, Byte2 \begin{math}\cdots\end{math} Byte15 (from left to right). \hyperref[memdesc:AESIVMEM]{AES\_IV\_MEM} stores data following the Endianness pattern presented in Table \ref{table:DMAAESPlaintextCiphertextEndian}, i.e. the most significant (i.e., left-most) byte Byte0 is stored at the lowest address while the least significant (i.e., right-most) byte Byte15 at the highest address.

For more details on IV and ICB, please refer to \href{https://nvlpubs.nist.gov/nistpubs/Legacy/SP/nistspecialpublication800-38a.pdf}{NIST SP 800-38A}.

\subsection{ Block Operation Process}

\begin{enumerate}
    \item Write 0 to the \hyperref[regdesc:CRYPTODMAAESSHASELECTREG]{CRYPTO\_DMA\_AES\_SHA\_SELECT\_REG} register.
    \item Configure Crypto DMA chained list and start DMA. For details, please refer to Chapter \ref{mod:dma} \textit{\nameref{mod:dma}}.
    \item Initialize the AES accelerator-related registers:
    \begin{itemize}
        \item Write 1 to the \hyperref[regdesc:AESDMAENABLEREG]{AES\_DMA\_ENABLE\_REG} register.
        \item Configure the \hyperref[regdesc:AESINTENAREG]{AES\_INT\_ENA\_REG} register to enable or disable the interrupt function.
        \item Initialize registers \hyperref[regdesc:AESMODEREG]{AES\_MODE\_REG}, \hyperref[regdesc:AESKEYnREG]{AES\_KEY\_\regindex{n}\_REG}, and \hyperref[regdesc:AESENDIANREG]{AES\_ENDIAN\_REG}.
        \item Select operation type by configuring the \hyperref[regdesc:AESBLOCKMODEREG]{AES\_BLOCK\_MODE\_REG} register. For details, see Table \ref{table:AESBlockMode}.
        \item Initialize the \hyperref[regdesc:AESBLOCKNUMREG]{AES\_BLOCK\_NUM\_REG} register. For details, see Section \ref{Label Block Number}.
        \item Initialize the \hyperref[regdesc:AESINCSELREG]{AES\_INC\_SEL\_REG} register (only needed when AES Accelerator is working under CTR block operation).
        \item Initialize the \hyperref[memdesc:AESIVMEM]{AES\_IV\_MEM} memory (This is always needed except for ECB block operation).
    \end{itemize}
    \item Start operation by writing 1 to the \hyperref[regdesc:AESTRIGGERREG]{AES\_TRIGGER\_REG} register.
    \item \label{wait ECB/CBC/OFB/CTR/CFB8/CFB128 done}
            Wait for the completion of computation, which happens when the content of \hyperref[regdesc:AESSTATEREG]{AES\_STATE\_REG} becomes 2 or the AES interrupt occurs.
    \item Check if DMA completes data transmission from AES to memory. At this time, DMA had already written the result data in memory, which can be accessed directly. For details on DMA, please refer to Chapter \ref{mod:dma} \textit{\nameref{mod:dma}}.
    \item Clear interrupt by writing 1 to the \hyperref[regdesc:AESINTCLRREG]{AES\_INT\_CLR\_REG} register, if any AES interrupt occurred during the computation.
    \item Release the AES Accelerator by writing 0 to the \hyperref[regdesc:AESDMAEXITREG]{AES\_DMA\_EXIT\_REG} register. After this, the content of the \hyperref[regdesc:AESSTATEREG]{AES\_STATE\_REG} register becomes 0.
    Note that, you can release DMA earlier, but only after Step \ref{wait ECB/CBC/OFB/CTR/CFB8/CFB128 done} is completed.
\end{enumerate}

\subsection{ GCM Operation Process }

\begin{enumerate}
    \item Write 0 to the \hyperref[regdesc:CRYPTODMAAESSHASELECTREG]{CRYPTO\_DMA\_AES\_SHA\_SELECT\_REG} register.
    \item Configure Crypto DMA chained list and start DMA. For details on DMA, please refer to Chapter \ref{mod:dma} \textit{\nameref{mod:dma}}.
    \item Initialize the AES accelerator-related registers:
    \begin{itemize}
        \item Write 1 to the \hyperref[regdesc:AESDMAENABLEREG]{AES\_DMA\_ENABLE\_REG} register.
        \item Configure the \hyperref[regdesc:AESINTENAREG]{AES\_INT\_ENA\_REG} register to enable or disable the interrupt function.
        \item Initialize registers \hyperref[regdesc:AESMODEREG]{AES\_MODE\_REG} and \hyperref[regdesc:AESKEYnREG]{AES\_KEY\_\regindex{n}\_REG}、\hyperref[regdesc:AESENDIANREG]{AES\_ENDIAN\_REG}.
        \item Write 6 to the \hyperref[regdesc:AESBLOCKMODEREG]{AES\_BLOCK\_MODE\_REG} register.
        \item Initialize the \hyperref[regdesc:AESBLOCKNUMREG]{AES\_BLOCK\_NUM\_REG} register. Details about this register are described in Section \ref{Label Block Number}.
        \item Initialize the \hyperref[regdesc:AESAADBLOCKNUMREG]{AES\_AAD\_BLOCK\_NUM\_REG} register. Details about this register are described in Section \ref{Label AAD Block Number}.
        \item Initialize the \hyperref[regdesc:AESREMAINDERBITNUMREG]{AES\_REMAINDER\_BIT\_NUM\_REG} register. Details about this register is described in Section \ref{Label Remainder Bit Number}.
    \end{itemize}
    \item Start operation by writing 1 to the \hyperref[regdesc:AESTRIGGERREG]{AES\_TRIGGER\_REG} register.
    \item Wait for the completion of computation, which happens when the content of \hyperref[regdesc:AESSTATEREG]{AES\_STATE\_REG} becomes 2. For details on the working status of AES Accelerator, please refer to Table \ref{table:AESStateValue-dma}. At this step, no interrupt occurs.
    \item Obtain the $H$ value from the \hyperref[memdesc:AESHMEM]{AES\_H\_MEM} memory.
    \item Generate $J_0$ and write it to the \hyperref[memdesc:AESJ0MEM]{AES\_J0\_MEM} memory.
    \item Continue operating by writing 1 to the \hyperref[regdesc:AESCONTINUEOPREG]{AES\_CONTINUE\_OP\_REG} register.
    \item \label{wait GCM done}
            Wait for the completion of computation, which happens when the content of \hyperref[regdesc:AESSTATEREG]{AES\_STATE\_REG} becomes 2 or the AES interrupt occurs. For details on the working status of AES Accelerator, please refer to Table  \ref{table:AESStateValue-dma}.
    \item Obtain $T_0$ by reading \hyperref[memdesc:AEST0MEM]{AES\_T0\_MEM}, which is already ready at this step.
    \item Check if DMA completes data transmission from AES to memory. At this time, DMA had already written the result data in memory, which can be accessed directly. For details on DMA, please refer to Chapter \ref{mod:dma} \textit{\nameref{mod:dma}}.
    \item Clear interrupt by writing 1 to the \hyperref[regdesc:AESINTCLRREG]{AES\_INT\_CLR\_REG} register, if any AES interrupt occurred during the computation.
    \item Exit DMA by writing 1 to the \hyperref[regdesc:AESDMAEXITREG]{AES\_DMA\_EXIT\_REG} register. After this, the content of the \hyperref[regdesc:AESSTATEREG]{AES\_STATE\_REG} becomes 0. Note that, you can exit DMA earlier, but only after Step \ref{wait GCM done} is completed.
\end{enumerate}


\section{GCM Algorithm}\label{Label GCM algorithm}

ESP32-S2's AES accelerator fully supports GCM Algorithm. In reality, the \begin{math}AAD\end{math}, \begin{math}C\end{math} and \begin{math}P\end{math} that are longer than \begin{math}2^{32}\end{math}-1 bits are seldom used. Therefore, we specify that the length of \begin{math}AAD\end{math}, \begin{math}C\end{math} and \begin{math}P\end{math} should be no longer than \begin{math}2^{32}\end{math}-1 here.
Figure \ref{figure:GCMEncryptionOperation} below demonstrates how GCM encryption is implemented in the AES Accelerator of ESP32-S2.

\begin{figure}[H]
    \centering
    \includegraphics[width=0.8\textwidth]{\modulefiles/figures/GCM_encryption.eps}
    \caption{GCM Encryption Process}
    \label{figure:GCMEncryptionOperation}
\end{figure}

GCM encryption is implemented as follows:

\begin{enumerate}
    \item \label{GCM operation : Hardware Calculate H}
          Hardware executes the ECB Algorithm to obtain the Hash subkey \hyperref[Label Hash subkey]{$H$}, which is needed in the Hash computation.
    \item \label{GCM operation : Hardware Calculate AAD hash}
          Hardware executes the GHASH Algorithm to perform Hash computation with the \hyperref[api:testpadding]{padded} $AAD$.
    \item \label{GCM operation : Hardware Calculate J0 inc32}
          Hardware gets ready for CTR encryption by obtaining the result of applying Standard Incrementing Function INC$_{32}$ to \hyperref[Label J0]{$J_{0}$}.
    \item \label{GCM operation : Hardware Calculate C hash}
          Hardware executes the GCTR Algorithm to encrypt the padded plaintext $P$, then executes the GHASH Algorithm to perform Hash computation on the padded ciphertext $C$.
    \item \label{GCM operation : Hardware Calculate padding block hash}
          Hardware executes the GHASH Algorithm to perform Hash computation on AAD Blocks, obtaining a 128-bit Hash result.
    \item \label{GCM operation : Hardware Calculate T0}
          Hardware executes the GCTR Algorithm to encrypt $J_{0}$, obtaining $T_{0}$.
    \item \label{GCM operation : Software Calculate T}
          Software obtains the result $T_{0}$ from hardware, and execute MSB$_{t}$ Algorithm to obtain the final result Authenticated Tag $T$.
\end{enumerate}

The only difference between GCM decryption and GCM encryption lies in Step \ref{GCM operation : Hardware Calculate C hash} in Figure \ref{figure:GCMEncryptionOperation}. To be more specific, instead of executing GCTR Algorithm to encrypt the padded plaintext, the AES Accelerator executes the same Algorithm to decrypt the padded ciphertext in GCM decryption. For details, please see \href{https://nvlpubs.nist.gov/nistpubs/Legacy/SP/nistspecialpublication800-38d.pdf}{NIST SP 800-38D}.

\subsection{Hash Subkey}\label{Label Hash subkey}

\label{memdesc:AESHMEM}During GCM operation, the Hash subkey $H$ is a 128-bit value computed by hardware, which is demonstrated in Step \ref{GCM operation : Hardware Calculate H} in Figure \ref{figure:GCMEncryptionOperation}. Also you can find more information about Hash subkey at ``Step 1. Let $H$ = CIPH$_K$({$0^{128}$})" in Chapter 7 GCM Specification of \href{https://nvlpubs.nist.gov/nistpubs/Legacy/SP/nistspecialpublication800-38d.pdf}{NIST SP 800-38D}.\\
Just like all other Endianness, the Hash subkey $H$ is stored in the \hyperref[memdesc:AESHMEM]{AES\_H\_MEM} memory with its most significant (i.e., left-most) byte Byte0 stored at the lowest address and least significant (i.e., right-most) byte Byte15 at the highest address. For details, see Table \ref{table:DMAAESPlaintextCiphertextEndian}.

\subsection{\texorpdfstring{$J_0$}{J0}} \label{Label J0}
 The $J_0$ is a 128-bit value computed by hardware, which is required during Step \ref{GCM operation : Hardware Calculate J0 inc32} and Step \ref{GCM operation : Hardware Calculate T0} of the GCM process in Figure \ref{figure:GCMEncryptionOperation}. For details on the generation of $J_0$, please see Chapter 7 GCM Specification in \href{https://nvlpubs.nist.gov/nistpubs/Legacy/SP/nistspecialpublication800-38d.pdf}{NIST SP 800-38D} Specification.\\

 The $J_0$ is stored in the \hyperref[memdesc:AESJ0MEM]{AES\_J0\_MEM} memory as Endianness\label{memdesc:AESJ0MEM}. Just like all other Endianness, its most significant (i.e., left-most) byte Byte0 is stored at the lowest address in the memory while least significant (i.e., right-most) byte Byte15 at the highest address. For details, see Table \ref{table:DMAAESPlaintextCiphertextEndian}.

\subsection{Authenticated Tag}\label{Label Authenticated Tag}

Authenticated Tag (Tag for short) is one of the key results of GCM computation, which is demonstrated in Step \ref{GCM operation : Software Calculate T} of Figure \ref{figure:GCMEncryptionOperation}. The value of Tag is determined by the length of Authenticated Tag $t$ ($1 <= t <= 128$):

\begin{itemize}
    \item When $t = 128$, the value of Tag equals to $T_0$, a 128-bit string that is stored in the AES\_T0\_MEM as Endianness\label{memdesc:AEST0MEM}. Just like all other Endianness, its most significant (i.e., left-most) byte Byte0 is stored at the lowest address in the memory while least significant (i.e., right-most) byte Byte15 at the highest address. For details, see Table \ref{table:DMAAESPlaintextCiphertextEndian}.

    \item When $1 <= t < 128$, the value of Tag equals to the $t$ most significant (i.e., left-most) bits of $T_0$. In this case, Tag is represented as MSB$_t$($T_0$), which returns the $t$ most significant bits of $T_0$. For example, MSB$_4$(111011010) = 1110 and MSB$_5$(11010011010) = 11010. For details on the MSB$_t$(\ ) function, please refer to Chapter 6 Mathematical Components of GCM in the \href{https://nvlpubs.nist.gov/nistpubs/Legacy/SP/nistspecialpublication800-38d.pdf}{NIST SP 800-38D} specification.
\end{itemize}

\subsection{ AAD Block Number }\label{Label AAD Block Number}

Register \hyperref[regdesc:AESAADBLOCKNUMREG]{AES\_AAD\_BLOCK\_NUM\_REG} stores the Block Number of Additional Authenticated Data (AAD). The length of this register equals to length({\bfseries TEXT-PADDING}($AAD$))/128. AES Accelerator only uses this register when working in the DMA-AES mode.

\subsection{ Remainder Bit Number}\label{Label Remainder Bit Number}

Register \hyperref[regdesc:AESREMAINDERBITNUMREG]{AES\_REMAINDER\_BIT\_NUM\_REG} stores the Remainder Bit Number, which indicates the number of effective bits of incomplete blocks in plaintext/ciphertext. The value stored in this register equals to length($P$)\%128 or length($C$)\%128. AES Accelerator only uses this register when working in the DMA-AES mode.

Register \hyperref[regdesc:AESREMAINDERBITNUMREG]{AES\_REMAINDER\_BIT\_NUM\_REG} does not affect the results of plaintext or ciphertext, but does impact the value of $T_{0}$, therefore the Tag value too.

The GCM Algorithm can be viewed as the combination of GCTR operation and GHASH operation, among which, the GCTR performs the encryption and decryption, while the GHASH sloves the Tag.

Note that the \hyperref[regdesc:AESREMAINDERBITNUMREG]{AES\_REMAINDER\_BIT\_NUM\_REG} register is only effective for GCM encryption. To be more specific:

\begin{itemize}
    \item For GCM encryption, the Hardware firstly computes $C$, then passes it in the form of {\bfseries TEXT-PADDING}($C$) as the input of GHASH operation. In this case, hardware determines how many trailing ``0" should be added based on the content of \hyperref[regdesc:AESREMAINDERBITNUMREG]{AES\_REMAINDER\_BIT\_NUM\_REG}.
    \item For GCM decrpytion, the padding is completed with the {\bfseries TEXT-PADDING}($C$) function. In this case, the \hyperref[regdesc:AESREMAINDERBITNUMREG]{AES\_REMAINDER\_BIT\_NUM\_REG} register is not effective.
\end{itemize}
