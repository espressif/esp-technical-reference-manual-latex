\chapter{SPI Controller (SPI)}
\label{mod:spi}

\section{Overview}

\begin{figure}[H]
    \centering
    \fbox{\includegraphics[width=0.8\textwidth]{./20-SPI/figures/SPI_ARCH}}
    \caption{SPI Architecture}
    \label{Figure:SPI_ARCH}
\end{figure}

As Figure \ref{Figure:SPI_ARCH} shows, ESP32 integrates four SPI controllers which can be used to communicate with external devices that use the SPI protocol. Controller SPI0 is used as a buffer for accessing external memory. Controller SPI1 can be used as a master. Controllers SPI2 and SPI3 can be configured as either a master or a slave. When used as a master, each SPI controller can drive multiple CS signals (CS0\verb+~+CS2) to activate multiple slaves. Controllers SPI1\verb+~+SPI3 share two DMA channels.

The SPI signal buses consist of D, Q, CS0-CS2, CLK, WP, and HD signals, as Table \ref{table:SPI_signals} shows. Controllers SPI0 and SPI1 share one signal bus through an arbiter; the signals of the shared bus start with ``SPI". Controllers SPI2 and SPI3 use signal buses starting with ``HSPI" and ``VSPI" respectively. The I/O lines included in the above-mentioned signal buses can be mapped to pins via either the IO\_MUX module or the GPIO matrix. (Please refer to Chapter \hyperref[mod:iomuxgpio]{IO\_MUX} for details.)

The SPI controller supports four-line full-duplex/half-duplex communication (MOSI, MISO, CS, and CLK lines) and three-line half-duplex-only communication (DATA, CS, and CLK lines) in GP-SPI mode. In QSPI mode, an SPI controller accesses the flash or SRAM by using signal buses D, Q, CS0\verb+~+CS2, CLK, WP, and HD as a four-bit parallel SPI bus. The mapping between SPI bus signals and pin function signals under different communication modes is shown in Table \ref{table:SPI_signals}.

\begin{longtable}[c]{|p{3cm}|p{3cm}|p{2.5cm}|p{2cm}|p{2cm}|p{2cm}|}
    \caption{Mapping Between SPI Bus Signals and Pin Function Signals}
    \label{table:SPI_signals}\\
    \hline
    \rowcolor{lightgray}
    Four-line GP-SPI & Three-line GP-SPI & QSPI & \multicolumn{3}{c|}{Pin function signals}\\
    \cline{4-6}

\section{SPI Features}
{\bfseries General Purpose SPI (GP-SPI)}
\begin{itemize}
    \item Programmable data transfer length, in multiples of 1 byte
    \item Four-line full-duplex/half-duplex communication and three-line half-duplex communication support
     \item Master mode and slave mode
    \item Programmable CPOL and CPHA
    \item Programmable clock
\end{itemize}
{\bfseries Parallel QSPI}
\begin{itemize}
    \item Communication format support for specific slave devices such as flash
    \item Programmable communication format
    \item Six variations of flash-read operations available
    \item Automatic shift between flash and SRAM access
    \item Automatic wait states for flash access
\end{itemize}
{\bfseries SPI DMA Support}
\begin{itemize}
    \item Support for sending and receiving data using linked lists
\end{itemize}
{\bfseries SPI Interrupt Hardware}
\begin{itemize}
    \item SPI interrupts
    \item SPI DMA interrupts
\end{itemize}

\section{GP-SPI}\label{GP-SPI}

The SPI master mode supports four-line full-duplex/half-duplex communication and three-line half-duplex communication. Figure \ref{Figure:ESP32_SPI_full_duplex} outlines the connections needed for four-line full-duplex/half-duplex communications.

\begin{figure}[H]
    \centering
    \includegraphics[width=0.4\textwidth]{./20-SPI/figures/ESP32_SPI_full_duplex}
    \caption{SPI Master and Slave Full-duplex/Half-duplex Communication}
    \label{Figure:ESP32_SPI_full_duplex}
\end{figure}

The SPI1\verb+~+SPI3 controllers can communicate with other slaves as a standard SPI master. SPI2 and SPI3 can be configured as either a master or a slave. Every SPI master can be connected to three slaves at most by default. When not using DMA, the maximum length of data received/sent in one burst is 64 bytes. The data length is in multiples of one byte.

\subsection{GP-SPI Four-line Full-duplex Communication}

When configured to four-line full-duplex mode, the ESP32 SPI can act as either a master or a slave. The length of received and sent data needs to be set by configuring the SPI\_MISO\_DLEN\_REG, SPI\_MOSI\_DLEN\_REG registers for master mode as well as SPI\_SLV\_RDBUF\_DLEN\_REG, SPI\_SLV\_WRBUF\_DLEN\_REG registers for slave mode. The SPI\_DOUTDIN bit and SPI\_USR\_MOSI bit in register SPI\_USER\_REG should be configured to enable this communication mode. The SPI\_USR bit in register SPI\_CMD\_REG needs to be configured to initialize a data transfer.

\subsection{GP-SPI Four-line Half-duplex Communication}
When configured to four-line half-duplex mode, the ESP32 SPI can act as either a master or a slave. In this mode, the SPI communication supports flexible communication format as: command + address + dummy phase + received and/or sent data. The format is specified as follows:

\begin{enumerate}
    \item command: length of 0\verb+~+16 bits; Master Out Slave In (MOSI).
    \item address: length of 0\verb+~+32/64 bits; Master Out Slave In (MOSI).
    \item dummy phase: length of 0\verb+~+256 SPI clocks.
    \item received and/or sent data: length of 0\verb+~+512 bits (64 bytes); Master Out Slave In (MOSI) or Master In Slave Out (MISO).
\end{enumerate}

The address length is up to 32 bits in GP-SPI master mode and 64 bits in QSPI master mode. The command phase, address phase, dummy phase and received/sent data phase are controlled by bits SPI\_USR\_COMMAND, SPI\_USR\_ADDR, SPI\_USR\_DUMMY, and SPI\_USR\_MISO/SPI\_USR\_MOSI respectively in register SPI\_USER\_REG. A certain phase is enabled only when its corresponding control bit is set to 1. Details can be found in \hyperref[regdesc:SPIUSERREG]{register description}. When SPI works as a master, the register can be configured by software as required to determine whether or not to enable a certain phase.

When SPI works as a slave, the communication format must contain command, address, received and/or sent data, among which the command has several options listed in Table \ref{table:slave_cmd_description}. During data transmission or reception, the CS signal should keep logic level low. If the CS signal is pulled up during transmission, the internal state of the slave will be reset.

\begin{table}
    \centering
    \caption{Command Definitions Supported by GP-SPI Slave in Half-duplex Mode}
    \label{table:slave_cmd_description}
    \begin{tabular}[l]{|m{2cm}|m{14cm}|}
    \hline
    \rowcolor{lightgray}
    Command & Description\\
    \hline
    0x1 & Received by slave; writes data sent by the master into the slave status register via MOSI. \\
    \hline
    0x2 & Received by slave; writes data sent by the master into the slave data buffer via MOSI.\\
    \hline
    0x3 & Sent by slave; sends data in the slave buffer to master via MISO.\\
    \hline
    0x4 & Sent by slave; sends data in the slave status register to master via MISO. \\
    \hline
    0x6 & Writes master data on MOSI into data buffer and then sends the date in the slave data buffer to MISO.\\
    \hline
    \end{tabular}
\end{table}

The master can write the slave status register SPI\_SLV\_WR\_STATUS\_REG, and decide whether to read data from register SPI\_SLV\_WR\_STATUS\_REG or register SPI\_RD\_STATUS\_REG via the SPI\_SLV\_STATUS\_READBACK bit in register SPI\_SLAVE1\_REG. The SPI master can maintain communication with the slave by reading and writing slave status register, thus realizing complex communication with ease.

The length of received and sent data is controlled by SPI\_MISO\_DLEN\_REG and SPI\_MOSI\_DLEN\_REG in master mode, as well as SPI\_SLV\_RDBUF\_DLEN\_REG and SPI\_SLV\_WRBUF\_DLEN\_REG in slave mode. A reception or transmission of data is controlled by bit SPI\_USR\_MOSI or SPI\_USR\_MISO in SPI\_USER\_REG. The SPI\_USR bit in register SPI\_CMD\_REG needs to be configured to initialize a data transfer.

\subsection{GP-SPI Three-line Half-duplex Communication}

The three-line half-duplex communication differs from four-line half-duplex communication in that the reception and transmission shares one signal bus and that the communication format must contain command, address, received and/or sent data. Software can enable three-line half-duplex communication by configuring SPI\_SIO bit in SPI\_USER\_REG register.

\begin{tiplisting}
\vspace{-2em}
\begin{itemize}
\item In half-duplex communication, the order of command, address, received and/or sent data in the communication format should be followed strictly.
\item In half-duplex communication, communication formats "command + address + received data + sent data" and "received data + sent data" are not applicable to DMA.
\item When ESP32 SPI acts as a slave, the master CS should be active at least one SPI clock period before a read/write process is initiated, and should be inactive at least one SPI clock period after the read/write process is completed.
\end{itemize}
\end{tiplisting}


\subsection{GP-SPI Data Buffer}
\begin{figure}[H]
    \centering
    \includegraphics[width=0.5\textwidth]{./20-SPI/figures/SPI_data_buffer_e}
    \caption{SPI Data Buffer}
    \label{Figure:SPI_data_buffer_e}
\end{figure}

ESP32 SPI has 16 × 32 bits of data buffer to buffer data-send and data-receive operations. As is shown in Figure \ref{Figure:SPI_data_buffer_e}, received data is written from the low byte of SPI\_W0\_REG by default and the writing ends with SPI\_W15\_REG. If the data length is over 64 bytes, the extra part will be written from SPI\_W0\_REG.

Data buffer blocks SPI\_W0\_REG\verb+ ~ +SPI\_W7\_REG and SPI\_W8\_REG\verb+ ~ +SPI\_W15\_REG data correspond to the lower part and the higher part respectively. They can be used separately, and are controlled by the SPI\_USR\_MOSI\\\_HIGHPART bit and the SPI\_USR\_MISO\_HIGHPART bit in register SPI\_USER\_REG. For example, if SPI is configured as a master, when SPI\_USR\_MOSI\_HIGHPART = 1, SPI\_W8\_REG\verb+ ~ +SPI\_W15\_REG are used as buffer for sending data; when SPI\_USR\_MISO\_HIGHPART = 1, SPI\_W8\_REG\verb+ ~ +SPI\_W15\_REG are used as buffer for receiving data. If SPI acts as a slave, when SPI\_USR\_MOSI\_HIGHPART = 1, SPI\_W8\_REG\verb+ ~ +SPI\_W15\_REG are used as buffer for receiving data; when SPI\_USR\_MISO\_HIGHPART = 1, SPI\_W8\_REG\verb+ ~ +SPI\_W15\_REG are used as buffer for sending data.

\section{GP-SPI Clock Control}

The maximum output clock frequency of ESP32 GP-SPI master is $f_{\textrm{apb}}/2$, and the maximum input clock frequency of the ESP32 GP-SPI slave is $f_{\textrm{apb}}/8$. The master can derive other clock frequencies via frequency division.

 \[
    f_{\textrm{spi}}=\frac{f_{\textrm{apb}}} {\textrm{(SPI\_CLKCNT\_N+1)(SPI\_CLKDIV\_PRE+1)}}
    \]

SPI\_CLKCNT\_N and SPI\_CLKDIV\_PRE are two bits of register SPI\_CLOCK\_REG (Please refer to \ref{regdesc:SPICLOCKREG} Register Description for details). $\textrm{SPI\_CLKCNT\_H} =\lfloor \frac{\textrm{SPI\_CLKCNT\_N+1}}{2} – 1\rfloor$, SPI\_CLKCNT\_N=SPI\_CLKCNT\_L. When the SPI\_CLK\_EQU\_SYSCLK bit in register SPI\_CLOCK\_REG is set to 1, and the other bits are set to 0, SPI output clock frequency is $f_{\textrm{apb}}$. For other clock frequencies, SPI\_CLK\_EQU\_SYSCLK needs to be 0. In slave mode, SPI\_CLKCNT\_N, SPI\_CLKCNT\_L, SPI\_CLKCNT\_H and SPI\_CLKDIV\_PRE should all be 0.

\subsection{GP-SPI Clock Polarity (CPOL) and Clock Phase (CPHA)}\label{GP-SPI Clock Polarity (CPOL) and Clock Phase (CPHA)}

The clock polarity and clock phase of ESP32 SPI are controlled by SPI\_CK\_IDLE\_EDGE bit in register SPI\_PIN\_REG, SPI\_CK\_OUT\_EDGE bit and SPI\_CK\_I\_EDGE bit in register SPI\_USER\_REG, as well as SPI\_MISO\_DELAY\_MODE[1:0] bit, SPI\_MISO\_DELAY\_NUM[2:0] bit, SPI\_MOSI\_DELAY\_MODE[1:0] bit, SPI\_MOSI\_DELAY\_MUM[2:0] bit in register SPI\_CTRL2\_REG. Table \ref{table:clockpol_master} and Table \ref{table:clockpol_slave} show the clock polarity and phase as well as the corresponding register values for ESP32 SPI master and slave, respectively. Note that for mode0 and mode2 in Table \ref{table:clockpol_slave}, the registers are configured differently in non-DMA mode and DMA mode, and that the SPI slave data is output in advance in DMA mode.

\end{enumerate}

\subsection{GP-SPI Timing}\label{GP-SPI Timing}

The data signals of ESP32 GP-SPI can be mapped to physical pins either via IO\_MUX or via IO\_MUX and GPIO matrix. Input signals will be delayed by two $clk_{\textrm{apb}}$ clock cycles when they pass through the matrix. Output signals will not be delayed.

When GP-SPI is used as master and the data signals are not received by the SPI controller via GPIO matrix, if GP-SPI output clock frequency is $clk_{\textrm{apb}}/2$, register SPI\_MISO\_DELAY\_MODE should be set to 0 when configuring the clock polarity. If GP-SPI output clock frequency is not higher than $clk_{\textrm{apb}}/4$, register SPI\_MISO\_DELAY\_MODE can be set to the corresponding value in Table \ref{table:clockpol_master} when configuring the clock polarity.

When GP-SPI is used in master mode and the data signals enter the SPI controller via the GPIO matrix:

\begin{enumerate}
    \item If GP-SPI output clock frequency is $clk_{\textrm{apb}}/2$, register SPI\_MISO\_DELAY\_MODE should be set to 0 and the dummy phase should be enabled (SPI\_USR\_DUMMY = 1) for one $clk_{\textrm{spi}}$ clock cycle (SPI\_USR\_DUMMY\_CYC\\LELEN = 0) when configuring the clock polarity;
    \item If GP-SPI output clock frequency is $clk_{\textrm{apb}}/4$, register SPI\_MISO\_DELAY\_MODE should be set to 0 when configuring the clock polarity;
    \item If GP-SPI output clock frequency is not higher than $clk_{\textrm{apb}}/8$, register SPI\_MISO\_DELAY\_MODE can be set to the corresponding value in Table \ref{table:clockpol_master} when configuring the clock polarity.
\end{enumerate}

When GP-SPI is used in slave mode, the clock signal and the data signals should be routed to the SPI controller via the same path, i.e., neither the clock signal nor the data signals passes through GPIO matrix, or both of them pass through GPIO matrix. This is important in ensuring that the signals are not delayed by different time periods before they reach the SPI hardware.

Assume that $t_{\textrm{spi}}$, $t_{\textrm{pre}}$ and $t_{\textrm{v}}$ in Figure \ref{Figure:SPI_slv_o} denote SPI clock period, how far ahead data output is, and data output delay time, respectively. Assume the SPI slave's main clock period is $t_{\textrm{apb}}$. For non-DMA mode0, SPI slave data output is delayed by $t_{\textrm{v}}$:
\begin{itemize}
\item $t_{\textrm{v}} < 3.5*t_{\textrm{apb}}$, if CLK does not pass through GPIO matrix;
\item $t_{\textrm{v}} < 5.5*t_{\textrm{apb}}$, if CLK passes through GPIO matrix.
\end{itemize}

In DMA mode1 and mode3, SPI slave data output is delayed by the same period of time as in non-DMA mode. However, for mode0 and mode2, SPI slave data is output earlier by $t_{\textrm{pre}}$:
\begin{itemize}
\item $t_{\textrm{pre}} < ( t_{\textrm{spi}}/2 - 5.5*t_{\textrm{apb}} )$, if CLK does not pass through GPIO matrix;
\item $t_{\textrm{pre}} < ( t_{\textrm{spi}}/2 - 7.5*t_{\textrm{apb}} )$, if CLK passes through GPIO matrix.
\end{itemize}

\begin{figure}[H]
    \centering
    \fbox{\includegraphics[width=0.4\textwidth]{./20-SPI/figures/SPI_slv_o}}
    \caption{GP-SPI Slave Data Output}
    \label{Figure:SPI_slv_o}
\end{figure}

To conclude, if signals do not pass through GPIO matrix, the SPI slave clock frequency is up to $f_{\textrm{apb}}/8$; if signals pass through GPIO matrix, the SPI slave clock frequency is up to $f_{\textrm{apb}}/12$. Note that ($t_{\textrm{spi}}/2 – t_{\textrm{pre}}$) represents data output hold time for SPI slave in mode0 and mode2.


\section{Parallel QSPI}

ESP32 SPI controllers support SPI bus memory devices (such as flash and SRAM). The hardware connection between the SPI pins and the memories is shown by Figure \ref{Figure:ESP32_SPI_memory_inf}.
\begin{figure}[H]
    \centering
    \includegraphics[width=0.4\textwidth]{./20-SPI/figures/ESP32_SPI_memory_inf}
    \caption{Parallel QSPI}
    \label{Figure:ESP32_SPI_memory_inf}
\end{figure}

SPI1, SPI2 and SPI3 controllers can also be configured as QSPI master to connect to external memory. The maximum output clock frequency of the SPI memory interface is $f_{\textrm{apb}}$, with the same clock configuration as that of the GP-SPI master.

\subsection{Communication Format of Parallel QSPI}\label{Communication Format of Parallel QSPI}

To support communication with special slave devices, ESP32 QSPI implements a specifically designed communication protocol. The communication format of ESP32 QSPI master is the same as that of GP-SPI four-line half-duplex communication, except that in address phase and data phase, software can configure registers to enable two-line or four-line transmission. Figure  \ref{Figure:communication_format_c} shows a QSPI communication mode with four-line transmission in address phase and data phase.

\begin{figure}[H]
    \centering
    \fbox{\includegraphics[width=0.9\textwidth]{./20-SPI/figures/communication_format_e}}
    \caption{Communication Format of Parallel QSPI}
    \label{Figure:communication_format_c}
\end{figure}

ESP32 QSPI supports flash-read operation in one-line, two-line, and four-line modes. When working as a QSPI master, the command phase, address phase, dummy phase and data phase can be configured as needed, as flexible as in GP-SPI mode.

{\bfseries Note} that GPI-SPI full-duplex mode does not support dummy phase.

\section{GP-SPI Interrupt Hardware}

ESP32 SPI generates two types of interrupts. One is the SPI interrupt and the other is the SPI DMA interrupt.

ESP32 SPI reckons the completion of send- and/or receive-operations as the completion of one operation from the controller and generates one interrupt. When ESP32 SPI is configured to slave mode, the slave will generate read/write status registers and read/write buffer data interrupts according to different operations.

\subsection{SPI Interrupts}
The SPI\_*\_INTEN bits in the SPI\_SLAVE\_REG register can be set to enable SPI interrupts. When an SPI interrupt happens, the interrupt flag in the corresponding SPI\_*\_DONE register will get set. This flag is writable, and an interrupt can be cleared by setting the bit to zero.

\begin{itemize}
\item \label{int:SPITRANSDONEINT}SPI\_TRANS\_DONE\_INT: Triggered when an SPI operation is done.
\item \label{int:SPISLVWRSTAINT}SPI\_SLV\_WR\_STA\_INT: Triggered when an SPI slave status write is done.
\item \label{int:SPISLVRDSTAINT}SPI\_SLV\_RD\_STA\_INT: Triggered when an SPI slave status read is done.
\item \label{int:SPISLVWRBUFINT}SPI\_SLV\_WR\_BUF\_INT: Triggered when an SPI slave buffer write is done.
\item \label{int:SPISLVRDBUFINT}SPI\_SLV\_RD\_BUD\_INT: Triggered when an SPI slave buffer read is done.
\end{itemize}

\subsection{DMA Interrupts}
\begin{itemize}
\item \label{int:SPIOUTTOTALEOFINT}SPI\_OUT\_TOTAL\_EOF\_INT: Triggered when all linked lists are sent.
\item \label{int:SPIOUTEOFINT}SPI\_OUT\_EOF\_INT: Triggered when one linked list is sent.
\item \label{int:SPIOUTDONEINT}SPI\_OUT\_DONE\_INT: Triggered when the last linked list item has zero length.
\item \label{int:SPIINSUCEOFINT}SPI\_IN\_SUC\_EOF\_INT: Triggered when all linked lists are received.
\item \label{int:SPIINERREOFINT}SPI\_IN\_ERR\_EOF\_INT: Triggered when there is an error receiving linked lists.
\item \label{int:SPIINDONEINT}SPI\_IN\_DONE\_INT: Triggered when the last received linked list had a length of 0.
\item \label{int:SPIINLINKDSCRERRORINT}SPI\_INLINK\_DSCR\_ERROR\_INT: Triggered when the received linked list is invalid.
\item \label{int:SPIOUTLINKDSCRERRORINT}SPI\_OUTLINK\_DSCR\_ERROR\_INT: Triggered when the linked list to be sent is invalid.
\item \label{int:SPIINLINKDSCREMPTYINT}SPI\_INLINK\_DSCR\_EMPTY\_INT: Triggered when no valid linked list is available.
\end{itemize}
