\chapter{Low-Power Management (RTC\_CNTL)}
\label{mod:lowpowm}
\section{Introduction}

ESP32 offers efficient and flexible power-management technology to achieve the best balance between power consumption, wakeup latency and available wakeup sources. Users can select out of five predefined power modes of the main processors to suit specific needs of the application. In addition, to save power in power-sensitive applications, control may be executed by the Ultra-Low-Power coprocessor (ULP coprocessor), while the main processors are in Deep-sleep mode.

\section{Features}

\begin{itemize}
    \item Five predefined power modes to support various applications
    \item Up to 16 KB of retention memory
    \item 8 x 32 bits of retention registers
    \item ULP coprocessor enabled in all low-power modes
    \item RTC boot supported to shorten the wakeup latency
\end{itemize}

\section{Functional Description}

\subsection{Overview}

The low-power management unit includes voltage regulators, a power controller, power switch cells, power domain isolation cells, etc. Figure \ref{figure:ESP32 power control} shows the high-level architecture of ESP32's low-power management.


\subsection{Digital Core Voltage Regulator}

The built-in voltage regulator can convert the external power supply (typically 3.3 V) to 1.1 V to support the internal digital core. It receives a wide range of external power supplies from 1.8 V to 3.6 V, and provides an output voltage from 0.90 V to 1.25 V.

\begin{enumerate}
    \item The digital core voltage regulator can be controlled by software via \hyperref[regdesc:RTCCNTLVREGREG]{RTC\_CNTL\_VREG\_FORCE\_PU} and \hyperref[regdesc:RTCCNTLVREGREG]{RTC\_CNTL\_VREG\_FORCE\_PD}.
    \item \hyperref[regdesc:RTCCNTLVREGREG]{RTC\_CNTL\_DIG\_VREG\_DBIAS\_WAK} and \hyperref[regdesc:RTCCNTLVREGREG]{RTC\_CNTL\_DIG\_VREG\_DBIAS\_SLP} tune the supply voltage of the digital core: \\
    $$\mathrm{VDD\_DIG = 0.90 + DBIAS \cdot 0.05 V}$$
    \item The current to the digital core comes from pin VDD3P3\_CPU and pin VDD3P3\_RTC.
\end{enumerate}

Figure \ref{figure:Digital regulator} shows the structure of a digital core's voltage regulator.

\subsection{Low-Power Voltage Regulator}

The built-in low-power voltage regulator can convert the external power supply (typically 3.3 V) to 1.1 V to support the internal RTC core. To save power, it receives a wide range of external power supply from 1.8 V to 3.6 V, and supports an adjustable output voltage of 0.90 V to 1.25 V in normal work mode, a fixed output voltage of about 0.75 V both in Deep-sleep mode and Hibernation mode.

\begin{enumerate}

    \item When the pin CHIP\_PU is at a high level, the low-power voltage regulator cannot be turned off. It should be switched only between normal-work mode and Deep-sleep mode.

    \item In normal work mode, \hyperref[fielddesc:RTCCNTLDBIASWAK]{RTC\_CNTL\_DBIAS\_WAK} can be used to tune the output voltage: \\
    $$\mathrm{VDD\_RTC = 0.90 + DBIAS \cdot 0.05 V}$$

    \item In Deep-sleep mode, the regulator's output voltage is fixed at about 0.75 V.

    \item The current to the RTC core comes from pin VDD3P3\_RTC.

\end{enumerate}

\subsection{Flash Voltage Regulator}

The built-in flash voltage regulator can supply a voltage of 3.3 V or 1.8 V to other devices (flash, for example) in the system, with a maximum output current of 40 mA.

\begin{enumerate}
    \item When \hyperref[regdesc:RTCCNTLSDIOCONFREG]{XPD\_SDIO\_VREG} == 1, the regulator outputs a voltage of 3.3 V or 1.8 V; when \hyperref[regdesc:RTCCNTLSDIOCONFREG]{XPD\_SDIO\_VREG} == 0, the output is high-impedance and, in this case, the voltage is provided by the external power supply.

    \item When \hyperref[regdesc:RTCCNTLSDIOCONFREG]{SDIO\_TIEH} == 1, the regulator shorts pin VDD\_SDIO to pin VDD3P3\_RTC. The regulator then outputs a voltage of 3.3 V which is the voltage of pin VDD3P3\_RTC. When \hyperref[regdesc:RTCCNTLSDIOCONFREG]{SDIO\_TIEH} == 0, the inner loop ties the regulator output to the voltage of VREF, which is typically 1.8 V.

    \item \hyperref[fielddesc:RTCCNTLDREFHSDIO]{DREFH\_SDIO}, \hyperref[fielddesc:RTCCNTLDREFMSDIO]{DREFM\_SDIO} and \hyperref[fielddesc:RTCCNTLDREFLSDIO]{DREFL\_SDIO} could be used to tune the reference voltage VREF slightly. However, it is recommended that users do not change the value of these registers, since it may affect the stability of the inner loop.

    \item When the regulator output is 3.3 V or 1.8 V, the output current comes from the pin VDD3P3\_RTC.
\end{enumerate}

\subsection{Brownout Detector}

The brownout detector checks the voltage of pin VDD3P3\_RTC. If the voltage drops rapidly and becomes too low, the detector would trigger a signal to shut down some power-consuming blocks (such as LNA, PA, etc.) to allow extra time for the digital block to save and transfer important data. The power consumption of the detector is ultra low. It remains enabled whenever the chip is powered on, with an adjustable trigger level calibrated around 2.5V.

\begin{enumerate}
    \item As the output of the brownout detector, \hyperref[regdesc:RTCCNTLBROWNOUTREG]{RTC\_CNTL\_BROWN\_OUT\_DET} goes high when the voltage of pin VDD3P3\_RTC is lower than the threshold value.
    \item \hyperref[regdesc:RTCCNTLBROWNOUTREG]{RTC\_CNTL\_DBROWN\_OUT\_THRES[2:0]} is used to tune the threshold voltage, which is usually calibrated around 2.5V.
\end{enumerate}

Figure \ref{figure:Brownout detector} shows the structure of a brownout detector.

\subsection{RTC Module}

The RTC module is designed to handle the entry into, and exit from, the low-power mode, and control the clock sources, PLL, power switch and isolation cells to generate power-gating, clock-gating, and reset signals. As for the low-power management, RTC is composed of the following modules (see Figure \ref{figure:RTC structure}):

\begin{itemize}
    \item RTC main state machine: records the power state.
    \item Digital \& analog power controller: generates actual power-gating/clock-gating signals for digital parts and analog parts.
    \item Sleep \& wakeup controller: handles the entry into \& exit from the low-power mode.
    \item Timers: include RTC main timer, ULP coprocessor timer and touch timer.
    \item Low-Power processor and sensor controllers: include ULP coprocessor, touch controller, SAR ADC controller, etc.
    \item Retention memory:
    \begin{itemize}
        \item RTC slow memory: an 8 KB SRAM, mostly used as retention memory or instruction \& data memory for the ULP coprocessor. The CPU accesses it through the APB, starting from address 0x50000000.
        \item RTC fast memory: an 8 KB SRAM, mostly used as retention memory. The CPU accesses it through IRAM0/DRAM0. Fast RTC memory is about 10 times faster than the RTC slow memory.
    \end{itemize}
    \item Retention registers: always-on registers of 8 x 32 bits, serving as data storage.
    \item RTC IO pads: 18 always-on analog pads, usually functioning as wake-up sources.
\end{itemize}

\subsection{Low-Power Clocks}

In the low-power mode, the 40 MHz crystal and PLL are usually powered down to save power. But clocks are needed for the chip to remain active in the low-power mode.

For the RTC core, there are five possible clock sources:

\begin{itemize}
    \item external low-speed (32.768 kHz) crystal clock XTL32K\_CLK,
    \item external high-speed (2 MHz \verb+~+ 40 MHz) crystal clock XTAL\_DIV\_CLK,
    \item internal RC oscillator RC\_SLOW\_CLK (typically about 150 kHz and adjustable),
    \item internal 8-MHz oscillator RC\_FAST\_CLK, and
    \item internal 31.25-kHz clock RC\_FAST\_DIV\_CLK (derived from the internal 8-MHz oscillator divided by 256).
\end{itemize}

With these clocks, RTC\_FAST\_CLK and RTC\_SLOW\_CLK are derived. Configure \hyperref[fielddesc:RTCCNTLFASTCLKSEL]{RTC\_CNTL\_RTC\_FAST\_CLK\_SEL} to select the clock source of RTC\_FAST\_CLK and \hyperref[fielddesc:RTCCNTLANACLKRTCSEL]{RTC\_CNTL\_ANA\_CLK\_RTC\_SEL} to select the clock source of RTC\_SLOW\_CLK. Normally, choose RC\_FAST\_CLK as the clock source of RTC\_FAST\_CLK, and RC\_SLOW\_CLK as the clock source of RTC\_SLOW\_CLK. For details, please see Figure \ref{figure:RTC low-power clocks}.

\begin{figure}[H]
    \centering
    \includegraphics[width=0.6\textwidth]{\modulefiles/figures/rtc_clock.pdf}

For the digital core, LOW\_POWER\_CLK is switched among four sources (XTL32K\_CLK, RTC\_SLOW\_CLK, RC\_FAST\_CLK, and XTL\_CLK). Configure \hyperref[regdesc:BTLPCKDIVFRACREG]{DPORT\_BT\_LPCK\_DIV\_FRAC\_REG} to select the clock source of LOW\_POWER\_CLK.

\subsection{Predefined Power Modes}

In ESP32, we recommend that you always use the predefined power modes first, before trying to tune each power control signal. The predefined power modes should cover most scenarios:

\begin{itemize}
    \item Active mode
    \begin{itemize}
        \item The CPU is clocked at XTAL\_DIV\_N (40 MHz/26 MHz) or PLL (80 MHz/160 MHz/240 MHz).
        \item The chip can receive, transmit, or listen.
    \end{itemize}

    \item Modem-sleep mode
    \begin{itemize}
        \item The CPU is operational and the clock is configurable.
        \item The Wi-Fi/Bluetooth baseband is clock-gated or powered down. The radio is turned off.
        \item Current consumption: $\sim$30 mA with 80 MHz PLL.
        \item Current consumption: $\sim$3 mA with 2 MHz XTAL.
        \item Immediate wake-up.
    \end{itemize}

    \item Light-sleep mode
    \begin{itemize}
        \item The internal 8 MHz oscillator, 40 MHz high-speed crystal, PLL, and radio are disabled.
        \item The clock in the digital core is gated. The CPUs are stalled.
        \item The ULP coprocessor and touch controller can be periodically triggered by monitor sensors.
        \item Current consumption: $\sim$ 800 $\mu$A.
        \item Wake-up latency: less than 1 ms.
    \end{itemize}

    \item Deep-sleep mode
    \begin{itemize}
        \item The internal 8 MHz oscillator, 40 MHz high-speed crystal, PLL and radio are disabled.
        \item The digital core is powered down. The CPU context is lost.
        \item The supply voltage to the RTC core drops to 0.7V.
        \item 8 x 32 bits of data are kept in general-purpose retention registers.
        \item The RTC memory and fast RTC memory can be retained.
        \item Current consumption: $\sim$ 6.5 $\mu$A.
        \item Wake-up latency: less than 1 ms.
        \item Recommended for ultra-low-power infrequently-connected Wi-Fi/Bluetooth applications.
    \end{itemize}

    \item Hibernatation mode
    \begin{itemize}
        \item The internal 8 MHz oscillator, 40 MHz high-speed crystal, PLL, and radio are disabled.
        \item The digital core is powered down. The CPU context is lost.
        \item The RTC peripheral domain is powered down.
        \item The supply voltage to the RTC core drops to 0.7V.
        \item 8 x 32 bits of data are kept in general-purpose retention registers.
        \item The RTC memory and fast RTC memory are powered down.
        \item Current consumption: $\sim$ 4.5 $\mu$A.
        \item Wake-up source: RTC timer only.
        \item Wake-up latency: less than 1 ms.
        \item Recommended for ultra-low-power infrequently-connected Wi-Fi/Bluetooth applications.
    \end{itemize}
\end{itemize}

    \centering

By default, ESP32 first enters the Modem-sleep mode after a system reset and can be configured to Active mode when transmitting or receiving packets. After the CPU stalls for a while, the chip can enter several low-power modes. It is up to the user to select the mode that best balances power consumption, wake-up latency and available wake-up sources. For details, please see Figure \ref{figure:Power modes}.\\
Please note that the predefined power mode could be further optimized and adapted to any application.

\subsection{Reject Sleep}\label{sec:lpm-reject-source}

ESP32 implements a hardware mechanism that equips the chip with the ability to reject to sleep, which prevents the chip from going to sleep unexpectedly when some peripherals are still working but not detected by the CPU, thus guaranteeing the proper functioning of the peripherals.

Among the wakeup sources listed in Table \ref{tab:lpm-wakeup-source}, GPIO and SDIO can be configured as the causes to reject sleep.

Users can configure the reject to sleep option via the following registers.
\begin{itemize}
    \item Configure the \hyperref[regdesc:RTCCNTLSTATE0REG]{RTC\_CNTL\_SLP\_REJECT} field to enable or disable the option to reject to sleep:
        \begin{itemize}
            \item Set \hyperref[regdesc:RTCCNTLSLPREJECTCONFREG]{RTC\_CNTL\_LIGHT\_SLP\_REJECT\_EN} to enable reject-to-light-sleep.
            \item Set \hyperref[regdesc:RTCCNTLSLPREJECTCONFREG]{RTC\_CNTL\_DEEP\_SLP\_REJECT\_EN} to enable reject-to-deep-sleep.
        \end{itemize}
    \item Read \hyperref[regdesc:RTCCNTLSLPREJECTCONFREG]{RTC\_CNTL\_REJECT\_CAUSE} to check the reason for rejecting to sleep.
\end{itemize}

\subsection{RTC Timer}
The RTC timer is a 48-bit counter that can be read. The clock is RTC\_SLOW\_CLK. Any reset/sleep mode, except for the power-up reset, will not stop or reset the RTC timer.

The RTC timer can be used to wake up the CPU at a designated time, and to wake up TOUCH or the ULP coprocessor periodically.

\subsection{RTC Boot}

Since the CPU, ROM and RAM are powered down during Deep-sleep and Hibernation mode, the wake-up time is much longer than that in Light sleep/Modem sleep, because of the ROM unpacking and data-copying from the flash (SPI booting). There are two types of SRAM in the RTC, named slow RTC memory and fast RTC memory, which remain powered-on in Deep-sleep mode. For small-scale codes (less than 8 KB), there are two methods of speeding up the wake-up time, i.e. avoiding ROM unpacking and SPI booting.

The first method is to use the RTC slow memory:

\begin{enumerate}
    \item Set register \hyperref[regdesc:RTCCNTLRESETSTATEREG]{RTC\_CNTL\_PROCPU\_STAT\_VECTOR\_SEL} for PRO\_CPU (or register \hyperref[regdesc:RTCCNTLRESETSTATEREG]{RTC\_CNTL\_APPCPU\_STAT\_VECTOR\_SEL} for APP\_CPU) to 0.
    \item Put the chip into sleep.
    \item When the CPU is powered up, the reset vector starts from 0x50000000, instead of 0x40000400. ROM unpacking \& SPI boot are not needed. The code in RTC memory has to do itself some initialization for the C program environment.
\end{enumerate}

The second method is to use the fast RTC memory:

\begin{enumerate}
    \item Set register \hyperref[regdesc:RTCCNTLRESETSTATEREG]{RTC\_CNTL\_PROCPU\_STAT\_VECTOR\_SEL} for PRO\_CPU (or register \hyperref[regdesc:RTCCNTLRESETSTATEREG]{RTC\_CNTL\_APPCPU\_STAT\_VECTOR\_SEL} for APP\_CPU) to 1.
    \item Calculate CRC for the fast RTC memory, and save the result in register \hyperref[regdesc:RTCCNTLSTORE1nREG]{RTC\_CNTL\_RTC\_STORE6\_REG[31:0]}.
    \item Input register \hyperref[regdesc:RTCCNTLSTORE1nREG]{RTC\_CNTL\_RTC\_STORE7\_REG[31:0]} with the entry address in the fast RTC memory.
    \item Put the chip into sleep.
    \item When the CPU is powered up, after ROM unpacking and some necessary initialization, the CRC is calculated again. If the result matches with register \hyperref[regdesc:RTCCNTLSTORE1nREG]{RTC\_CNTL\_RTC\_STORE6\_REG[31:0]}, the CPU will jump to the entry address.
\end{enumerate}
