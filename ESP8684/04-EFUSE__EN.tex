\chapter{eFuse Controller (EFUSE)}
\label{mod:efuse}

\section{Overview}

\chipname{} contains a 1024-bit eFuse memory to store parameters and user data. The parameters include control parameters for some hardware modules, system data parameters and keys used for the decryption module. Once an eFuse bit is programmed to 1, it can never be reverted to 0. The eFuse controller programs individual bits of parameters in eFuse according to user configurations. From outside the chip, eFuse data can only be read via the eFuse Controller. If read-protection for some data is not enabled, that data is readable from outside the chip. If read-protection is enabled, that data can not be read from outside the chip. In all cases, however, some keys stored in eFuse can still be used internally by hardware cryptography modules such as Digital Signature, HMAC, etc., without exposing this data to the outside world.

\section{Features}

The eFuse controller has the following features:

\begin{itemize}
    \item 1024-bit one-time programmable storage, in which 256-bit is reserved for users
    \item Configurable write protection
    \item Configurable read protection
    \item Various hardware encoding schemes against data corruption in the eFuse memory
\end{itemize}


\section{Functional Description}

\subsection{Structure}

eFuse data is organized in 4 blocks (BLOCK0 \verb+~+ BLOCK3).

BLOCK0 holds control parameters for most hardware modules.

Table \ref{tab:efuse-block0-para} lists all the parameters in BLOCK0 that can be accessed (read and used) by users and their offsets, bit widths, as well as information on whether they can be used by hardware, which bits are write-protected, and corresponding descriptions.

The {\bfseries \hyperref[fielddesc:EFUSEWRDIS]{EFUSE\_WR\_DIS}} parameter is used to control the writing of other parameters, while {\bfseries \hyperref[fielddesc:EFUSERDDIS]{EFUSE\_RD\_DIS}} is used to disable users from reading BLOCK3. For more information on these two parameters, please see Section \ref{sec:efuse-wr-dis} and Section \ref{sec:efuse-rd-dis}.

\subsubsection{\hyperref[fielddesc:EFUSEWRDIS]{EFUSE\_WR\_DIS}}
\label{sec:efuse-wr-dis}

Parameter \hyperref[fielddesc:EFUSEWRDIS]{EFUSE\_WR\_DIS} determines whether individual eFuse parameters are write-protected. After \hyperref[fielddesc:EFUSEWRDIS]{EFUSE\_WR\_DIS} has been programmed, execute an eFuse read operation so that the write-protection status would take effect.

Column "Write Protection by \hyperref[fielddesc:EFUSEWRDIS]{EFUSE\_WR\_DIS} Bit Number" in Table \ref{tab:efuse-block0-para} and Table \ref{tab:efuse-block-1-3-para} lists the specific bits in \hyperref[fielddesc:EFUSEWRDIS]{EFUSE\_WR\_DIS} that disable writing.

When the write-protect bit of a parameter is set to 0, it means that this parameter is not write-protected and can be programmed, unless it has been programmed before.

When the write-protect bit of a parameter is set to 1, it means that this parameter is write-protected and none of its bits can be modified, with non-programmed bits always remaining 0 while programmed bits always remaining 1. That is to say, if a parameter is write-protected, it will always remain in this state and cannot be changed.

\subsubsection{\hyperref[fielddesc:EFUSERDDIS]{EFUSE\_RD\_DIS}}
\label{sec:efuse-rd-dis}

Only parameters in BLOCK3 can be read-protected to prevent any access from outside of the chip as shown in column "Read Protection by \hyperref[fielddesc:EFUSERDDIS]{EFUSE\_RD\_DIS} Bit Number" of Table \ref{tab:efuse-block-1-3-para}. After \hyperref[fielddesc:EFUSEWRDIS]{EFUSE\_RD\_DIS} has been programmed, execute an eFuse read operation so that the read-protection status would take effect.

If the corresponding \hyperref[fielddesc:EFUSERDDIS]{EFUSE\_RD\_DIS} bit is 0, then the eFuse block can be read by users; if the corresponding \hyperref[fielddesc:EFUSERDDIS]{EFUSE\_RD\_DIS} bit is 1, then the parameter controlled by this bit is user protected.

Other parameters that are not in BLOCK3 can always be read by users.


\subsubsection{Data Storage}
\label{sec:efuse-data-storage}

Internally, eFuse controller uses hardware encoding schemes to protect data from corruption, which are invisible for users.

All BLOCK0 parameters except for \hyperref[fielddesc:EFUSEWRDIS]{EFUSE\_WR\_DIS} are stored with four backups, meaning each bit is stored four times. This backup scheme is not visible to users.

Except for \hyperref[fielddesc:EFUSEWRDIS]{EFUSE\_WR\_DIS} which is 8-bit, all other parameters in BLOCK0 are 32-bit. Therefore, BLOCK0 occupies (8 + 32 * 4 = 136) bits of storage totally in eFuse memory.

BLOCK1 \verb+~+ BLOCK3 use RS (44, 32) coding scheme that supports up to 6 bytes of automatic error correction. The primitive polynomial of RS (44, 32) is \(p(x)=x^8+x^4+x^3+x^2+1\).

    \caption{Shift Register Circuit (former 32-byte)}
    \label{fig:efuse-lfsr-circuit-1}
\end{figure}

\begin{figure}[ht!]
    \centering
    \fbox{\includegraphics[width=1.0\textwidth]{\modulefiles/figures/04-eFuse-shift-reg-cir-2}}
    \caption{Shift Register Circuit (latter 12-byte)}
    \label{fig:efuse-lfsr-circuit-2}
\end{figure}

The shift register circuit shown in Figure \ref{fig:efuse-lfsr-circuit-1} and \ref{fig:efuse-lfsr-circuit-2} processes 32-byte data using RS (44, 32). This coding scheme encodes 32 bytes of data into 44 bytes:

\begin{itemize}
    \item Byte 0 \verb+~+ 31 is the data itself
    \item Byte 32 \verb+~+ 43 is the parity byte stored in the 8-bit trigger DFF1, DFF2,..., DFF12 (where gf\_mul\_n (n is an integer) is the result of multiplying a byte of data in the \(GF(2^8)\) field with the element \(\alpha^n\))
\end{itemize}

Then, hardware will write the 44-byte data to eFuse memory. The eFuse controller will automatically decode the data and correct errors when reading the eFuse block.

Because the RS check codes are generated on the entire 256-bit eFuse block, each block can only be written once.

Since the size of BLOCK1 is less than 256-bit, the unused bits will be treated as 0 by hardware during the RS (44, 32) decoding. Thus the final coding result will not be affected.

Among blocks using the RS (44, 32) coding scheme, the parameter in BLOCK1 is 88-bit, and the RS check code is 96-bit, so BLOCK1 occupies (88 + 96 = 184) bits in eFuse memory. The parameter in BLOCK2 and BLOCK3 is 256-bit respectively, and the RS check code is 96-bit, so these two blocks occupy ((256 + 96) * 2 = 704) bits in eFuse memory.


\subsection{Programming of Parameters}
\label{sec:efuse-software-programming-para}

The eFuse controller can only program eFuse parameters in one block at a time. BLOCK0 \verb+~+ BLOCK3 share the same address range to store the parameters to be programmed. Configure parameter \hyperref[fielddesc:EFUSEBLKNUM]{EFUSE\_BLK\_NUM} to indicate which block should be programmed.

{\bfseries Programming BLOCK0}

Set the \hyperref[fielddesc:EFUSEBLKNUM]{EFUSE\_BLK\_NUM} field to 0. The parameters to be programmed in BLOCK0 are stored in \hyperref[regdesc:EFUSEPGMDATA0REG]{EFUSE\_PGM\_DATA0\_REG} \verb+~+ \hyperref[regdesc:EFUSEPGMDATA1REG]{EFUSE\_PGM\_DATA1\_REG}. The data in \hyperref[regdesc:EFUSEPGMDATA6REG]{EFUSE\_PGM\_DATA2\_REG} \verb+~+ \hyperref[regdesc:EFUSEPGMDATA7REG]{EFUSE\_PGM\_DATA7\_REG} and \hyperref[regdesc:EFUSEPGMCHECKVALUE0REG]{EFUSE\_PGM\_CHECK\_VALUE0\_REG} \verb+~+ \hyperref[regdesc:EFUSEPGMCHECKVALUE2REG]{EFUSE\_PGM\_CHECK\_VALUE2\_REG} registers does not affect the programming of BLOCK0.

{\bfseries Programming BLOCK1}

Set the \hyperref[fielddesc:EFUSEBLKNUM]{EFUSE\_BLK\_NUM} field to 1. The parameters to be programmed in BLOCK1 are stored in \hyperref[regdesc:EFUSEPGMDATA0REG]{EFUSE\_PGM\_DATA0\_REG} \verb+~+ \hyperref[regdesc:EFUSEPGMDATA5REG]{EFUSE\_PGM\_DATA2\_REG}, while the corresponding RS check codes are stored in \hyperref[regdesc:EFUSEPGMCHECKVALUE0REG]{EFUSE\_PGM\_CHECK\_VALUE0\_REG} \verb+~+ \hyperref[regdesc:EFUSEPGMCHECKVALUE2REG]{EFUSE\_PGM\_CHECK\_VALUE2\_REG}. The data in \hyperref[regdesc:EFUSEPGMDATA6REG]{EFUSE\_PGM\_DATA3\_REG} \verb+~+ \hyperref[regdesc:EFUSEPGMDATA7REG]{EFUSE\_PGM\_DATA7\_REG} registers does not affect the programming of BLOCK1.

{\bfseries Programming BLOCK2 \verb+~+ 3}

Set the \hyperref[fielddesc:EFUSEBLKNUM]{EFUSE\_BLK\_NUM} field to 2 or 3, respectively. The parameters to be programmed are stored in \hyperref[regdesc:EFUSEPGMDATA0REG]{EFUSE\_PGM\_DATA0\_REG} \verb+~+ \hyperref[regdesc:EFUSEPGMDATA7REG]{EFUSE\_PGM\_DATA7\_REG}, while the corresponding RS check code is stored in \hyperref[regdesc:EFUSEPGMCHECKVALUE0REG]{EFUSE\_PGM\_CHECK\_VALUE0\_REG}  \verb+~+ \hyperref[regdesc:EFUSEPGMCHECKVALUE2REG]{EFUSE\_PGM\_CHECK\_VALUE2\_REG}.

{\bfseries Programming process}

The process of programming parameters is as follows:

\begin{enumerate}
\item Configure the value of parameter \hyperref[fielddesc:EFUSEBLKNUM]{EFUSE\_BLK\_NUM} to determine the block to be programmed.

\item Write the parameters to be programmed to registers \hyperref[regdesc:EFUSEPGMDATA0REG]{EFUSE\_PGM\_DATA0\_REG} \verb+~+ \hyperref[regdesc:EFUSEPGMDATA7REG]{EFUSE\_PGM\_DATA7\_REG} and \hyperref[regdesc:EFUSEPGMCHECKVALUE0REG]{EFUSE\_PGM\_CHECK\_VALUE0\_REG}  \verb+~+  \hyperref[regdesc:EFUSEPGMCHECKVALUE2REG]{EFUSE\_PGM\_CHECK\_VALUE2\_REG}.


\item Make sure the eFuse programming voltage VDDQ is configured correctly as described in Section \ref{sec:efuse-vddq-timing}.

\item Set the field \hyperref[fielddesc:EFUSEOPCODE]{EFUSE\_OP\_CODE} of register \hyperref[regdesc:EFUSECONFREG]{EFUSE\_CONF\_REG} to 0x5A5A.

\item Set the field \hyperref[fielddesc:EFUSEPGMCMD]{EFUSE\_PGM\_CMD} of register \hyperref[regdesc:EFUSECMDREG]{EFUSE\_CMD\_REG} to 1.

\item Poll register \hyperref[regdesc:EFUSECMDREG]{EFUSE\_CMD\_REG} until it is 0x0, or wait for a PGM\_DONE interrupt. For more information on how to identify a PGM/READ\_DONE interrupt, please see the end of Section \ref{sec:efuse-software-read-para}.

\item Clear the parameters in \hyperref[regdesc:EFUSEPGMDATA0REG]{EFUSE\_PGM\_DATA0\_REG} \verb+~+ \hyperref[regdesc:EFUSEPGMDATA7REG]{EFUSE\_PGM\_DATA7\_REG} and \hyperref[regdesc:EFUSEPGMCHECKVALUE0REG]{EFUSE\_PGM\_CHECK\_VALUE0\_REG}  \verb+~+  \hyperref[regdesc:EFUSEPGMCHECKVALUE2REG]{EFUSE\_PGM\_CHECK\_VALUE2\_REG}.

\item Trigger an eFuse read operation (see Section \ref{sec:efuse-software-read-para}) to update eFuse registers with the new values.
\item Check corresponding error registers. If the value read is not 0, above 1 \verb+~+ 7 steps should be executed again to prevent programming insufficiency. For different eFuse blocks, the corresponding error registers that need to be checked are listed as follows:
\begin{itemize}
    \item BLOCK0: \hyperref[regdesc:EFUSERDREPEATERRREG]{EFUSE\_RD\_REPEAT\_ERR\_REG}
    \item BLOCK1: \hyperref[regdesc:EFUSERDRSERRREG]{EFUSE\_RD\_RS\_ERR\_REG}[3:0]
    \item BLOCK2: \hyperref[regdesc:EFUSERDRSERRREG]{EFUSE\_RD\_RS\_ERR\_REG}[7:4]
    \item BLOCK3: \hyperref[regdesc:EFUSERDRSERRREG]{EFUSE\_RD\_RS\_ERR\_REG}[11:8]
\end{itemize}
\end{enumerate}


{\bfseries Restrictions}

In BLOCK0, each bit can be programmed separately. However, we recommend to minimize programming cycles and program all the bits of a parameter in one programming action. In addition, after all parameters controlled by a certain bit of \hyperref[fielddesc:EFUSEWRDIS]{EFUSE\_WR\_DIS} are programmed, that bit should be immediately programmed, or you could even program these parameters and the controlling bit at the same time. By doing so, programming can be effectively protected by preventing messing the programming of \hyperref[fielddesc:EFUSEWRDIS]{EFUSE\_WR\_DIS} with the bits controlled by it. Repeated programming of already programmed bits is strictly forbidden. Otherwise, programming errors will occur.

BLOCK2 cannot be programmed by users as it has been programmed at manufacturing.

Both BLOCK1 and BLOCK3 can only be programmed once. Repeated programming is not allowed.


\subsection{User Read of Parameters}
\label{sec:efuse-software-read-para}

Users cannot read data programmed in the eFuse controller directly. The eFuse controller reads all programmed data and stores the results to their corresponding programming registers in its memory space. Then, users can read eFuse bits by reading the registers that start with EFUSE\_RD\_. Details are provided in Table \ref{tab:efuse-softw-read-para}.

{\bfseries Updating eFuse controller read registers}
\label{sec:efuse-update-reg}

The eFuse controller reads eFuse memory to update corresponding registers. This read operation happens on system reset and can also be triggered manually by users as needed (e.g., if new eFuse values have been programmed). The process of triggering a read operation by users is as follows:

\begin{enumerate}
    \item Set the field \hyperref[fielddesc:EFUSEOPCODE]{EFUSE\_OP\_CODE} of register \hyperref[regdesc:EFUSECONFREG]{EFUSE\_CONF\_REG} to 0x5AA5.
    \item Set the field \hyperref[fielddesc:EFUSEREADCMD]{EFUSE\_READ\_CMD} of register \hyperref[regdesc:EFUSECMDREG]{EFUSE\_CMD\_REG} to 1.
    \item Poll register \hyperref[regdesc:EFUSECMDREG]{EFUSE\_CMD\_REG} until it is 0x0, or wait for a READ\_DONE interrupt. Information on how to identify a PGM/READ\_DONE interrupt is provided below in this section.
    \item Read the values of each parameter from eFuse memory.
\end{enumerate}

The eFuse controller read registers will hold all values until the next read operation.

{\bfseries Error detection}

Error record registers allow users to detect if there are any inconsistencies between the parameters stored in the eFuse memory and the parameters read by the eFuse controller.

The \hyperref[regdesc:EFUSERDREPEATERRREG]{EFUSE\_RD\_REPEAT\_ERR\_REG} register indicates if there are any errors of programmed parameters (except for \hyperref[fielddesc:EFUSEWRDIS]{EFUSE\_WR\_DIS}) in BLOCK0 (value 1 indicates an error is detected, and the bit becomes invalid; value 0 indicates no error).

The \hyperref[regdesc:EFUSERDRSERRREG]{EFUSE\_RD\_RS\_ERR\_REG} register stores the number of corrected bytes as well as the result of RS decoding during eFuse reading BLOCK1 \verb+~+ BLOCK3.

The values of above registers will be updated every time after the eFuse controller read registers have been updated.


{\bfseries Identifying programming/read operation}

The methods to identify the completion of a programming/read operation are described below. Please note that bit 1 corresponds to a programming operation, and bit 0 corresponds to a read operation.
\begin{itemize}

    \item Method 1:
        \begin{enumerate}
            \item Poll bit 1/0 in register \hyperref[regdesc:EFUSEINTRAWREG]{EFUSE\_INT\_RAW\_REG} until it becomes 1, which represents the completion of a program/read operation.
        \end{enumerate}
    \item Method 2:
        \begin{enumerate}
            \item Set bit 1/0 in register \hyperref[regdesc:EFUSEINTENAREG]{EFUSE\_INT\_ENA\_REG} to 1 to enable the eFuse controller to post a PGM/READ\_DONE interrupt.
            \item Configure the Interrupt Matrix to enable the CPU to respond to eFuse controller interrupt signals, see Chapter \ref{mod:intmtrx} \textit{\nameref{mod:intmtrx}}.
            \item Wait for the PGM/READ\_DONE interrupt.
            \item Set bit 1/0 in register \hyperref[regdesc:EFUSEINTCLRREG]{EFUSE\_INT\_CLR\_REG} to 1 to clear the PGM/READ\_DONE interrupt.
    \end{enumerate}
\end{itemize}

{\bfseries Attention}

When the eFuse controller updating registers, the \hyperref[regdesc:EFUSEPGMDATA0REG]{EFUSE\_PGM\_DATA\textcolor{red}{\it{n}}\_REG} (n=0,1,.., 7) register will be reused. Therefore, please do not write meaningful data to such register before the eFuse controller starts to update registers.

During the chip boot time, the eFuse controller will update eFuse data automatically and write it to registers that users can access. Users can get eFuse memory data by reading such registers. That is to say, it is no need to drive the eFuse controller manually again to update read registers.

\subsection{eFuse VDDQ Timing}
\label{sec:efuse-vddq-timing}

The eFuse controller operates with 20 MHz of clock frequency, and its programming voltage VDDQ should be configured as follows:

\begin{itemize}
    \item \hyperref[fielddesc:EFUSEDACNUM]{EFUSE\_DAC\_NUM} (the rising period of VDDQ): The default value of VDDQ is 2.5 V and the voltage increases by 0.01 V in each clock cycle. Thus, the default value of this parameter is 255.
    \item \hyperref[fielddesc:EFUSEDACCLKDIV]{EFUSE\_DAC\_CLK\_DIV} (the clock divisor of VDDQ): The clock period to program VDDQ should be larger than 1 $\mu$s.
    \item \hyperref[fielddesc:EFUSEPWRONNUM]{EFUSE\_PWR\_ON\_NUM} (the power-up time for VDDQ): The programming voltage should be stabilized after this time, which means the value of this parameter should be configured to exceed the result of \hyperref[fielddesc:EFUSEDACCLKDIV]{EFUSE\_DAC\_CLK\_DIV} times \hyperref[fielddesc:EFUSEDACNUM]{EFUSE\_DAC\_NUM}.
    \item \hyperref[fielddesc:EFUSEPWROFFNUM]{EFUSE\_PWR\_OFF\_NUM} (the power-down time for VDDQ): The value of this parameter should be larger than 10 $\mu$s.
\end{itemize}

\subsection{Parameters Used by Hardware Modules}

Some hardware modules are directly connected to the eFuse peripheral in order to use the parameters listed in Table \ref{tab:efuse-block0-para} and Table \ref{tab:efuse-block-1-3-para}, specifically those marked with "Y" in columns "Accessible by Hardware". Users cannot intervene in this process.

\subsection{Interrupts}
\begin{itemize}
\item PGM\_DONE interrupt: Triggered when eFuse programming has finished. To enable this interrupt, set the \hyperref[fielddesc:EFUSEPGMDONEINTENA]{EFUSE\_PGM\_DONE\_INT\_ENA} field of register \hyperref[regdesc:EFUSEINTENAREG]{EFUSE\_INT\_ENA\_REG} to 1.
\item READ\_DONE interrupt: Triggered when eFuse read has finished. To enable this interrupt, set the \\\hyperref[fielddesc:EFUSEREADDONEINTENA]{EFUSE\_READ\_DONE\_INT\_ENA} field of register \hyperref[regdesc:EFUSEINTENAREG]{EFUSE\_INT\_ENA\_REG} to 1.
\end{itemize}
