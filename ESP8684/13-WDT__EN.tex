\chapter{Watchdog Timers (WDT)}
\label{mod:wdt}

\section{Overview}

Watchdog timers are hardware timers used to detect and recover from malfunctions. They must be periodically fed (reset) to prevent a timeout. A system/software that is behaving unexpectedly (e.g. is stuck in a software loop or in overdue events) will fail to feed the watchdog thus trigger a watchdog timeout. Therefore, watchdog timers are useful for detecting and handling erroneous system/software behavior.

As shown in Figure \ref{fig:wdt-timers-overview}, \chipname{} contains two digital watchdog timers: one in the timer group in Chapter \ref{mod:timg} \textit{\nameref{mod:timg}} (called Main System Watchdog Timer, or MWDT) and one in the RTC Module (called the RTC Watchdog Timer, or RWDT).  Each digital watchdog timer allows for four separately configurable stages and each stage can be programmed to take one action upon timeout, unless the watchdog is fed or disabled. MWDT supports three timeout actions: interrupt, CPU reset, and core reset, while RWDT supports four timeout actions: interrupt, CPU reset, core reset, and system reset (see details in Section \ref{sec:wdt-timeout} \textit{\nameref{sec:wdt-timeout}}). A timeout value can be set for each stage individually.

During the flash boot process, RWDT and the MWDT in timergroup 0 are enabled automatically in order to detect and recover from booting errors.

\chipname{} also has one analog watchdog timer: Super watchdog (SWD). It is an ultra-low-power circuit in analog domain that helps to prevent the system from operating in a sub-optimal state and resets the system if required.

\section{Digital Watchdog Timers}
\subsection{Features}
Watchdog timers have the following features:
\begin{itemize}
    \item Four stages, each with a separately programmable timeout value and timeout action
    \item Timeout actions:
        \begin{itemize}
            \item MWDT: interrupt, CPU reset, core reset
            \item RWDT: interrupt, CPU reset, core reset, system reset
        \end{itemize}
    \item Flash boot protection at stage 0:
        \begin{itemize}
            \item MWDT: core reset upon timeout
            \item RWDT: system reset upon timeout
        \end{itemize}
    \item Write protection that makes WDT register read only unless unlocked
    \item 32-bit timeout counter
    \item Clock source:
        \begin{itemize}
            \item MWDT: 40 MHz PLL\_F40M\_CLK or XTAL\_CLK
            \item RWDT: RTC\_SLOW\_CLK
        \end{itemize}
\end{itemize}

\subsubsection{Clock Source and 32-Bit Counter}
At the core of each watchdog timer is a 32-bit counter.

MWDT can select between the PLL\_F40M\_CLK clock or external clock (XTAL\_CLK) as its clock source by setting the \hyperref[fielddesc:TIMGWDTUSEXTAL]{TIMG\_WDT\_USE\_XTAL} field of the \hyperref[regdesc:TIMGWDTCONFIG0REG]{TIMG\_WDTCONFIG0\_REG} register. Note that when the chip is in low-power mode and the clock source of CPU\_CLK is not PLL\_CLK (i.e. when \hyperref[fielddesc:SYSTEMSOCCLKSEL]{SYSTEM\_SOC\_CLK\_SEL} is not 1, see details in Table \ref{tab:resclk-cpu-clk-src} of Chapter \ref{mod:resclk} \textit{\nameref{mod:resclk}}), MWDT can only select XTAL\_CLK.
The selected clock is switched on by setting \hyperref[fielddesc:TIMGWDTCLKISACTIVE]{TIMG\_WDT\_CLK\_IS\_ACTIVE} field of the \hyperref[regdesc:TIMGREGCLKREG]{TIMG\_REGCLK\_REG} register to 1 and switched off by setting it to 0. Then the selected clock is
divided by a 16-bit configurable prescaler. The 16-bit prescaler for MWDT is configured via the \hyperref[fielddesc:TIMGWDTCLKPRESCALE]{TIMG\_WDT\_CLK\_PRESCALE} field of \hyperref[regdesc:TIMGWDTCONFIG1REG]{TIMG\_WDTCONFIG1\_REG}. When \hyperref[fielddesc:TIMGWDTDIVCNTRST]{TIMG\_WDT\_DIVCNT\_RST} field is set, the
prescaler is reset and it can be re-configured at once.

In contrast, the clock source of RWDT is derived directly from RTC\_SLOW\_CLK (see details in Chapter \ref{mod:resclk} \textit{\nameref{mod:resclk}}).


MWDT and RWDT are enabled by setting the \hyperref[fielddesc:TIMGWDTEN]{TIMG\_WDT\_EN} and \hyperref[fielddesc:RTCCNTLWDTEN]{RTC\_CNTL\_WDT\_EN} fields respectively. When enabled, the 32-bit counters of the watchdog will increment on each source clock cycle until the timeout value of the current stage is reached (i.e. timeout of the current stage). When this occurs, the current counter value is reset to zero and the next stage will become active.  If a watchdog timer is fed by software, the timer will return to stage 0 and reset its counter value to zero. Software can feed a watchdog timer by writing any value to \hyperref[regdesc:TIMGWDTFEEDREG]{TIMG\_WDTFEED\_REG} for MDWT and \hyperref[fielddesc:RTCCNTLWDTFEED]{RTC\_CNTL\_RTC\_WDT\_FEED} for RWDT.


\subsubsection{Stages and Timeout Actions}\label{sec:wdt-timeout}
Timer stages allow for a timer to have a series of different timeout values and corresponding timeout action. When one stage times out, the timeout action is triggered, the counter value is reset to zero, and the next stage becomes active. MWDT/ RWDT provide four stages (called stages 0 to 3). The watchdog timers will progress through each stage in a loop (i.e. from stage 0 to 3, then back to stage 0).

Timeout values of each stage for MWDT are configured in \hyperref[regdesc:TIMGWDTCONFIG2REG]{TIMG\_WDTCONFIG\textcolor{red}{\it{i}}\_REG} (where \textcolor{red}{\it{i}} ranges from 2 to 5), whilst timeout values for RWDT are configured using \hyperref[fielddesc:RTCCNTLWDTSTG3]{RTC\_CNTL\_WDT\_STG\textcolor{red}{\it{j}}\_HOLD} field (where \textcolor{red}{\it{j}} ranges from 0 to 3).

Please note that the timeout value of stage 0 for RWDT (T\textsubscript{hold0}) is determined by the combination of the \\ \hyperref[fielddesc:EFUSEWDTDELAYSEL]{EFUSE\_WDT\_DELAY\_SEL} field of eFuse register \hyperref[regdesc:EFUSERDREPEATDATA0REG]{EFUSE\_RD\_REPEAT\_DATA0\_REG} and \hyperref[fielddesc:RTCCNTLWDTSTG0HOLD]{RTC\_CNTL\_WDT\_STG0\_HOLD}. The relationship is as follows:

\begin{displaymath}
T_{hold0} = \hyperref[fielddesc:RTCCNTLWDTSTG0HOLD]{RTC\_CNTL\_WDT\_STG0\_HOLD}   << (\hyperref[fielddesc:EFUSEWDTDELAYSEL]{EFUSE\_WDT\_DELAY\_SEL} + 1)
\end{displaymath}

where $<<$ is a left-shift operator.

Upon the timeout of each stage, one of the following timeout actions will be executed:

\begin{table}[H]
\centering
\caption{Timeout Actions}
\label{tab:wdt-timeout-action}

\begin{tabular}{|l|m{12cm}|}
\hline
\rowcolor{lightgray}
    \textbf{Timeout Action} & \textbf{Description} \\\hline

    Interrupt &  Trigger an interrupt \\ \hline
    CPU reset &  Reset the CPU core \\ \hline
    Core reset &  Reset the main system (which includes MWDT, CPU, and all peripherals). The power management unit and RTC peripherals will not be reset \\ \hline
    System reset &  Reset the main system, power management unit and RTC peripherals (see details in Chapter \ref{mod:lowpowm} \textit{\nameref{mod:lowpowm}}). This action is only available in RWDT  \\ \hline
    Disabled &  No effect on the system \\ \hline

\end{tabular}
\end{table}

For MWDT, the timeout action of all stages is configured in \hyperref[regdesc:TIMGWDTCONFIG0REG]{TIMG\_WDTCONFIG0\_REG}. Likewise for RWDT, the timeout action is configured in \hyperref[regdesc:RTCCNTLWDTCONFIG0REG]{RTC\_CNTL\_WDTCONFIG0\_REG}.

\subsubsection{Write Protection}
Watchdog timers are critical to detecting and handling erroneous system/software behavior, thus should not be disabled easily (e.g. due to a misplaced register write). Therefore, MWDT and RWDT incorporate a write protection mechanism that prevent the watchdogs from being disabled or tampered with due to an accidental write.
The write protection mechanism is implemented using a write-key field for each timer (\hyperref[fielddesc:TIMGWDTWKEY]{TIMG\_WDT\_WKEY} for  MWDT, \hyperref[fielddesc:RTCCNTLWDTWKEY]{RTC\_CNTL\_WDT\_WKEY} for RWDT). The value 0x50D83AA1 must be written to the watchdog timer's write-key field before any other register of the same watchdog timer can be changed. Any attempts to write to a watchdog timer's registers (other than the write-key field itself) whilst the write-key field's value is not 0x50D83AA1 will be ignored. The recommended procedure for accessing a watchdog timer is as follows:
\begin{enumerate}
\item Disable the write protection by writing the value 0x50D83AA1 to the timer's write-key field.
\item Make the required modification of the watchdog such as feeding or changing its configuration.
\item Re-enable write protection by writing any value other than 0x50D83AA1 to the timer's write-key field.
\end{enumerate}

\subsubsection{Flash Boot Protection}
During flash booting process, MWDT as well as RWDT, are automatically enabled. Stage 0 for the enabled MWDT is automatically configured to reset the system upon timeout, known as core reset. Likewise, stage 0 for RWDT is configured to system reset, which resets the main system and RTC when it times out. After booting, \hyperref[fielddesc:TIMGWDTFLASHBOOTMODEN]{TIMG\_WDT\_FLASHBOOT\_MOD\_EN} and \hyperref[fielddesc:RTCCNTLWDTFLASHBOOTMODEN]{RTC\_CNTL\_WDT\_FLASHBOOT\_MOD\_EN}  should be cleared to stop the flash boot protection procedure for both MWDT and RWDT respectively. After this, MWDT and RWDT can be configured by software.

\section{Super Watchdog}\label{sec:wdt-swdt}
Super watchdog (SWD) is an ultra-low-power circuit in analog domain that helps to prevent the system from operating in a sub-optimal state and resets the system if required. SWD contains a watchdog circuit that needs to be fed for at least once during its timeout period, which is slightly less than one second. About 100 ms before watchdog timeout, it will also send out a WD\_INTR signal as a request to remind the system to feed the watchdog.

If the system doesn't respond to SWD feed request and watchdog finally times out, SWD will generate a system level signal SWD\_RSTB to reset whole digital circuits on the chip.

The source of the clock for SWD is constant and can not be selected.

\subsection{Features}
SWD has the following features:
\begin{itemize}
    \item Ultra-low power
    \item Interrupt to indicate that the SWD is about to time out
    \item Various dedicated methods for software to feed SWD, which enables SWD to monitor the working state of the whole operating system
\end{itemize}

\subsubsection{Structure}

\subsubsection{Workflow}
In normal state:
\begin{itemize}
    \item SWD controller receives feed request from SWD.
    \item SWD controller can send an interrupt to main CPU.
    \item Main CPU can feed SWD directly by setting \hyperref[fielddesc:RTCCNTLSWDFEED]{RTC\_CNTL\_SWD\_FEED}.
    \item When trying to feed SWD, CPU needs to disable SWD controller's write protection by writing 0x8F1D312A to \hyperref[fielddesc:RTCCNTLSWDWKEY]{RTC\_CNTL\_SWD\_WKEY}. This prevents SWD from being fed by mistake when the system is operating in sub-optimal state.
    \item If setting \hyperref[fielddesc:RTCCNTLSWDAUTOFEEDEN]{RTC\_CNTL\_SWD\_AUTO\_FEED\_EN} to 1, SWD controller can also feed SWD itself without any interaction with CPU.
\end{itemize}

After reset:
\begin{itemize}
    \item Check \hyperref[fielddesc:RTCCNTLRESETCAUSEPROCPU]{RTC\_CNTL\_RESET\_CAUSE\_PROCPU}[5:0] for the cause of CPU reset.\\
	If \hyperref[fielddesc:RTCCNTLRESETCAUSEPROCPU]{RTC\_CNTL\_RESET\_CAUSE\_PROCPU}[5:0] == 0x12, it indicates that the cause is SWD reset.
	\item Set \hyperref[fielddesc:RTCCNTLSWDRSTFLAGCLR]{RTC\_CNTL\_SWD\_RST\_FLAG\_CLR} to clear the SWD reset flag.
\end{itemize}
