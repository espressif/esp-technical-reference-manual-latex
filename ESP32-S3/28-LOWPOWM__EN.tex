\chapter{Low-power Management (RTC\_CNTL)}\label{mod:lowpowm}

\section{Introduction}

\chipname{} has an advanced Power Management Unit (PMU), which can flexibly power up different power domains of the chip, to achieve the best balance among chip performance, power consumption, and wakeup latency. To simplify power management for typical scenarios, \chipname{} has predefined four power modes, which are preset configurations that power up different combinations of power domains. On top of that, the chip also allows the users to independently power up any particular power domain to meet more complex requirements. \chipname{} has integrated two Ultra-Low-Power coprocessors (ULP co-processors), which allow the chip to work when most of the power domains are powered down, thus achieving extremely low-power consumption.

\section{Features}
\chipname{}'s low-power management supports the following features:
\begin{itemize}
    \item Four predefined power modes to simplify power management for typical scenarios
    \item Up to 16 KB of retention memory (slow memory and fast memory)
    \item 8 x 32-bit retention registers
    \item RTC Boot supported for reduced wakeup latency
    \item ULP co-processors supported in all power modes

\end{itemize}

In this chapter, we first introduce the working process of \chipname{}'s low-power management, then introduce the predefined power modes of the chip, and at last, introduce the RTC boot of the chip.

\section{Functional Description}

\chipname{}'s low-power management involves the following components:

\begin{itemize}
    \item Power management unit: controls the power supply to three power domain categories
    \begin{itemize}
        \item Real Time Controller (RTC)
        \item Digital
        \item Analog
    \end{itemize}
    For a complete list of 10 power domains grouped in these three power domain categories, see Section \ref{sec:lpm-power-domain}.
    \item Power isolation unit: isolates different power domains, so powered up and powered down domains do not affect each other.
    \item Low-power clocks: provide clocks to power domains working in low-power modes.
    \item Timers:
    \begin{itemize}
        \item RTC timer: logs the status of the RTC main state machine in dedicated registers.
        \item ULP timer: wakes up the ULP co-processors at a predefined time. For details, please refer to Chapter \ref{mod:ulp} \textit{\nameref{mod:ulp}}.
        \item Touch sensor timer: wakes up the touch sensor at a predefined time. For details, please refer to Chapter \ref{mod:sensor} \textit{\nameref{mod:sensor}}.
    \end{itemize}
    \item 8 x 32-bit ``always-on" retention registers: These registers are always powered up and are not affected by any low-power modes, thus can be used for storing data that cannot be lost.
    \item 22 x ``always-on" pins: These pins are always powered up and not affected by any low-power modes. They can be used as wakeup sources when the chip is working in the low-power modes (for details, please refer to Section \ref{sec:lpm-wakeup-source}), or can be used as regular GPIOs (for details, please refer to Chapter \ref{mod:iomuxgpio} \textit{\nameref{mod:iomuxgpio}}).
    \item RTC slow memory: 8 KB SRAM that works under RTC fast clock (rtc\_fast\_clk), which can be used as extended memory or to store ULP co-processor directives and data.
    \item RTC fast memory: 8 KB SRAM that works under CPU clock (CPU\_CLK), which can be used as extended memory.
    \item Voltage regulators: regulate the power supply to different power domains.
\end{itemize}

\subsection{Power Management Unit}

\chipname{}'s power management unit controls the power supply to different power domains. The main components of the power management unit include:

\begin{itemize}
    \item RTC main state machine: generates power gating, clock gating, and reset signals.
    \item Power controllers: power up and power down different power domains, according to the power gating signals from the main state machine.
    \item Sleep/wakeup controllers: send sleep or wakeup requests to the RTC main state machine.
    \item Clock controller: selects and powers up/down clock sources.
    \item Protection Timer: controls the transition interval between main state machine states.
\end{itemize}

In \chipname{}'s power management unit, the sleep/wakeup controllers send sleep or wakeup requests to the RTC main state machine, which then generates power gating, clock gating, and reset signals. Then, the power controller and clock controller power up and power down different power domains and clock sources, according to the signals generated by the RTC main state machine, so that the chip enters or exits the low-power modes. The main workflow is shown in Figure \ref{fig:lpm-RTC-structure}.

\subsection{Timers}

\chipname{}'s low-power management uses 3 timers:
\begin{itemize}
    \item RTC timer
    \item ULP timer
    \item Touch timer
\end{itemize}

This section only introduces the RTC timer. For detailed description of ULP timer, please refer to Chapters \ref{mod:ulp} \textit{\nameref{mod:ulp}}. For detailed description of touch timer, please refer to Chapters \ref{mod:sensor} \textit{\nameref{mod:sensor}}.

The RTC timer updates two groups of registers upon any new trigger. The first group logs the time of the current trigger, and the other logs the previous trigger. Detailed information about these two register groups is shown below:

\begin{itemize}
    \item Register group 0: logs the status of RTC timer at the current trigger.
    \begin{itemize}
        \item \hyperref[regdesc:RTCCNTLRTCTIMEHIGH0REG]{RTC\_CNTL\_RTC\_TIME\_HIGH0\_REG}
        \item \hyperref[regdesc:RTCCNTLRTCTIMELOW0REG]{RTC\_CNTL\_RTC\_TIME\_LOW0\_REG}
    \end{itemize}
    \item Register group 1: logs the status of RTC timer at the previous trigger.
    \begin{itemize}
        \item \hyperref[regdesc:RTCCNTLRTCTIMEHIGH1REG]{RTC\_CNTL\_RTC\_TIME\_HIGH1\_REG}
        \item \hyperref[regdesc:RTCCNTLRTCTIMELOW1REG]{RTC\_CNTL\_RTC\_TIME\_LOW1\_REG}
    \end{itemize}
\end{itemize}

On a new trigger, information on previous trigger is moved from register group 0 to register group 1 (and the original trigger logged in register group 1 is overwritten), and this new trigger is logged in register group 0. Therefore, only the last two triggers can be logged at any time.

It should be noted that any reset/sleep other than power-up reset will not stop or reset the RTC timer.

Also, the RTC timer can be used as a wakeup source. For details, see Section \ref{sec:lpm-wakeup-source}.

\chipname{}'s built-in flash voltage regulator can supply a voltage of 3.3 V or 1.8 V to other components outside of digital and RTC, such as flash. For the architecture of the \chipname{} flash voltage regulator, see Figure \ref{fig:lpm-Flash-regulator}.

\begin{enumerate}
    \item Configure XPD\_SDIO\_REG to select power source to components outside of the digital and RTC:
    \begin{itemize}
        \item 1: the voltage regulator outputs a voltage of 3.3 V or 1.8 V;
        \item 0: the voltage regulator outputs high-impedance. In this case, the voltage is provided by the external power supply.
    \end{itemize}
    \item Configure SDIO\_TIEH to choose between 3.3 V or 1.8 V:

        \item 1: the voltage regulator shorts pin VDD\_SPI to pin VDD3P3\_RTC, and outputs a voltage of 3.3 V, which is the voltage of pin VDD3P3\_RTC.
        \item 0: the voltage regulator outputs the reference voltage VREF, which is typically 1.8 V.
    \end{itemize}
\end{enumerate}
The signals mentioned above can be configured as below:

\begin{itemize}

       \item When the chip is in active mode, \hyperref[fielddesc:RTCCNTLSDIOFORCE]{RTC\_CNTL\_SDIO\_FORCE} == 0 and  \hyperref[fielddesc:EFUSEVDDSPIFORCE]{EFUSE\_VDD\_SPI\_FORCE} == 1, the XPD\_SDIO\_REG voltage is defined by \hyperref[fielddesc:EFUSEVDDSPIXPD]{EFUSE\_VDD\_SPI\_XPD};

       \item When \hyperref[fielddesc:RTCCNTLSDIOFORCE]{RTC\_CNTL\_SDIO\_FORCE} == 1, XPD\_SDIO\_REG is defined by \hyperref[fielddesc:RTCCNTLXPDSDIOREG]{RTC\_CNTL\_XPD\_SDIO\_REG}.
    \end{itemize}
    \item The configuration of SDIO\_TIEH:
    \begin{itemize}
        \item When \hyperref[fielddesc:RTCCNTLSDIOFORCE]{RTC\_CNTL\_SDIO\_FORCE}== 0 and \hyperref[fielddesc:EFUSEVDDSPIFORCE]{EFUSE\_VDD\_SPI\_FORCE} == 1, SDIO\_TIEH = \hyperref[fielddesc:EFUSEVDDSPITIEH]{EFUSE\_VDD\_SPI\_TIEH}
        \item Otherwise, SDIO\_TIEH = \hyperref[fielddesc:RTCCNTLSDIOTIEH]{RTC\_CNTL\_SDIO\_TIEH}.
    \end{itemize}
\end{itemize}

\subsubsection{Brownout Detector}

The brownout detector checks the voltage of pins VDD3P3, VDD3P3\_RTC and VDD3P3\_CPU. If the voltage of these pins drops below the predefined threshold (2.7 V by default), the detector would trigger a signal to shut down some power-consuming blocks (such as LNA, PA, etc.) to allow extra time for the digital to save and transfer important data.

The brownout detector has ultra-low power consumption and remains enabled whenever the chip is powered up.
For the architecture of the \chipname{} brownout detector, see Figure \ref{fig:lpm-Brown-out-detector}.

\begin{figure}[H]
    \centering
    \includegraphics[width=0.7\textwidth]{\modulefiles/figures/28-LPM-BROWN_OUT_DET.pdf}
   \caption{Brown-out detector}
    \label{fig:lpm-Brown-out-detector}
\end{figure}
\hyperref[fielddesc:RTCCNTLRTCBROWNOUTDET]{RTC\_CNTL\_RTC\_BROWN\_OUT\_DET} indicates the output level of brown-out detector. This register is low level by default, and outputs high level when the voltage of the detected pin drops below the predefined threshold.

When a brown-out signal is detected, the brownout detector can handle it in one of the two methods described below:
\begin{itemize}
    \item mode0: triggers an interrupt when the counter counts to the thresholds pre-defined in int comparer and rst comparer, then resets the chip based on the rst\_sel configuration. This method can be enabled by setting the bod\_mode0\_en signal.
    \item mode1: resets the system directly.
    \end{itemize}

Workflow is illustrated in the diagram below:

\begin{figure}[H]
    \centering
    \includegraphics[width=1.0\textwidth]{\modulefiles/figures/10-LOWPOWM_brownout.pdf}
   \caption{Brown-out detector}
    \label{fig:lpm-Brown-out-detector-mode}
\end{figure}

Registers for controlling related signals are described below:
    \begin{itemize}
        \item bod\_mode0\_en: \hyperref[fielddesc:RTCCNTLBROWNOUTENA]{RTC\_CNTL\_BROWN\_OUT\_ENA}
        \item bod\_mode0\_rst\_en: \hyperref[fielddesc:RTCCNTLBROWNOUTRSTENA]{RTC\_CNTL\_BROWN\_OUT\_RST\_ENA}
        \item bod\_mode0\_rst\_sel: \hyperref[fielddesc:RTCCNTLBROWNOUTRSTSEL]{RTC\_CNTL\_BROWN\_OUT\_RST\_SEL} configures the reset type. For more information regarding Chip Reset and System Reset, please refer to \ref{mod:resclk} \textit{\nameref{mod:resclk}}.
            \begin{itemize}
                \item 0: resets the chip
                \item 1: resets the system
            \end{itemize}
        \item bod\_mode1\_sel: the first bit of \hyperref[fielddesc:RTCCNTLRTCFIBSEL]{RTC\_CNTL\_RTC\_FIB\_SEL}.
        \item bod\_mode1\_rst\_en: \hyperref[fielddesc:RTCCNTLBROWNOUTANARSTEN]{RTC\_CNTL\_BROWN\_OUT\_ANA\_RST\_EN}
    \end{itemize}
\section{Power Modes Management}

\subsection{Power Domain}\label{sec:lpm-power-domain}

\chipname{} has 10 power domains in three power domain categories:
\begin{itemize}
    \item RTC
        \begin{itemize}
            \item Power management unit
            \item RTC peripherals, including RTC GPIO, RTC I2C, Temperature Sensor, Touch Sensor, RTC ADC Controller, and ULP Coprocessor
        \end{itemize}
    \item Digital
        \begin{itemize}
            \item Digital core
            \item Wireless digital circuits
            \item CPU
            \item PD peripherals, including SPI2, GDMA, SHA, RSA, AES, HMAC, DS, and Secure Boot
        \end{itemize}
    \item Analog
    \begin{itemize}
        \item Fast RC Oscillator (RC\_FAST\_CLK)
        \item External Main Clock (XTAL\_CLK)
        \item Phase Lock Loop (PLL)
        \item RF circuits
    \end{itemize}
\end{itemize}

\subsection{Reject Sleep}\label{sec:lpm-reject-source}

\chipname{} implements a hardware mechanism that equips the chip with the ability to reject to sleep, which prevents the chip from going to sleep unexpectedly when some peripherals are still working but not detected by the CPU, thus guaranteeing the proper functioning of the peripherals.

All the wakeup sources specified in Table \ref{tab:lpm-wakeup-source} (except UART) can also be configured as the causes to reject sleep.

Users can configure the reject to sleep option via the following registers.
\begin{itemize}
    \item Configure the \hyperref[fielddesc:RTCCNTLRTCSLEEPREJECTENA]{RTC\_CNTL\_RTC\_SLEEP\_REJECT\_ENA} field to enable or disable the option to reject to sleep:
        \begin{itemize}
            \item Set \hyperref[fielddesc:RTCCNTLLIGHTSLPREJECTEN]{RTC\_CNTL\_LIGHT\_SLP\_REJECT\_EN} to enable reject-to-light-sleep.
            \item Set \hyperref[fielddesc:RTCCNTLDEEPSLPREJECTEN]{RTC\_CNTL\_DEEP\_SLP\_REJECT\_EN} to enable reject-to-deep-sleep.
        \end{itemize}
    \item Read \hyperref[fielddesc:RTCCNTLREJECTCAUSE]{RTC\_CNTL\_SLP\_REJECT\_CAUSE\_REG} to check the reason for rejecting to sleep.
\end{itemize}

\section{Retention DMA}

\chipname{} can power off the CPU in the Light-sleep mode to further reduce the power consumption. To facilitate the CPU to wake up from Light-sleep and resume execution from the previous breakpoint, \chipname{} introduced a retention DMA.

\chipname{}'s retention DMA stores CPU information to the Internal SRAM Block2 to Block8 before CPU enters into sleep, and restore such information from Internal SRAM to CPU after CPU wakes up from sleep, thus enabling the CPU to resume execution from the previous breakpoint.

\chipname{}'s Retention DMA:

\begin{itemize}
    \item Retention DMA operates on date of 128 bits, and only supports address alignment of four words.
    \item Retention DMA's link list is specifically designed that it can be used to execute both write and read transactions. The configuration of Retention DMA is similar to that of GDMA:
    \begin{enumerate}
        \item First allocate enough memory in SRAM before CPU enters sleep to store 432 words\hyperref[dma-note1]{*}: CPU registers (428 words) and configuration information (4 words).
        \item Then configure the link list according to the memory allocated in the first step. See details in Chapter \ref{mod:dma} \textit{\nameref{mod:dma}}.
    \end{enumerate}


\begin{tiplisting}\label{dma-note1}
\textbf{*} Note that if the memory allocated is smaller than 432 words, then chip can only enter the Light-sleep mode and cannot further power down CPU.
\end{tiplisting}
\end{itemize}

After configuration, users can enable the Retention function by configuring the  \hyperref[fielddesc:RTCCNTLRETENTIONEN]{RTC\_CNTL\_RETENTION\_EN} field in Register \hyperref[regdesc:RTCCNTLRETENTIONCTRLREG]{RTC\_CNTL\_RETENTION\_CTRL\_REG} to:
\begin{itemize}
    \item Use Retention DMA to store CPU information before the chip enters sleep
    \item Restore information from Retention DMA to CPU after CPU wakes up.
\end{itemize}

\section{RTC Boot}
The wakeup time from Deep-sleep mode is much longer, compared to Light-sleep and Modem-sleep modes, because the ROMs and RAMs are both powered down in this case, and the CPU needs more time for SPI booting (data-copying from the flash). However, it's worth noting that both RTC fast memory and RTC slow memory remain powered up in the Deep-sleep mode. Therefore, users can store codes (so called ``deep sleep wake stub" of up to 8 KB) either in RTC fast memory or RTC slow memory, which doesn't require the above-mentioned  SPI booting, thus speeding up the wakeup process.

\textbf{Method one: Boot using RTC slow memory}
\begin{enumerate}
    \item Set \hyperref[fielddesc:RTCCNTLPROCPUSTATVECTORSEL]{RTC\_CNTL\_PROCPU\_STAT\_VECTOR\_SEL} to 0.
    \item Send the chip into sleep.
    \item After the CPU is powered up, the reset vector starts resetting from 0x{}50000000 instead of 0x{}40000400, which does not involve any SPI booting. The codes stored in RTC slow memory starts running immediately after the CPU reset. The code stored in the RTC slow memory only needs to be partially initialized in a C environment.
\end{enumerate}


\textbf{Method two: Boot using RTC fast memory}
\begin{enumerate}
    \item Set \hyperref[fielddesc:RTCCNTLPROCPUSTATVECTORSEL]{RTC\_CNTL\_PROCPU\_STAT\_VECTOR\_SEL} to 1.
    \item Calculate CRC for the RTC fast memory, and save the result in \hyperref[regdesc:RTCCNTLRTCSTORE7REG]{RTC\_CNTL\_RTC\_STORE7\_REG[31:0]}.
    \item Set \hyperref[regdesc:RTCCNTLRTCSTORE6REG]{RTC\_CNTL\_RTC\_STORE6\_REG[31:0]} to the entry address of RTC fast memory.
    \item Send the chip into sleep.
    \item ROM unpacking and some of the initialization starts after the CPU is powered up. After that, the CRC for the RTC fast memory will be calculated again. If the result matches with register \hyperref[regdesc:RTCCNTLRTCSTORE7REG]{RTC\_CNTL\_RTC\_STORE7\_REG[31:0]}, the CPU jumps to the entry address.
\end{enumerate}
