\chapter{ULP Coprocessor (ULP-FSM, ULP-RISC-V)}
\label{mod:ulp}


\section{Overview}
The ULP coprocessor is an ultra-low-power processor that remains powered on when the chip is in Deep-sleep (see Chapter \ref{mod:lowpowm} \textit{\nameref{mod:lowpowm}}). Hence, users can store in RTC memory a program for the ULP coprocessor to access RTC peripherals, internal sensors, and RTC registers during Deep-sleep.

In power-sensitive scenarios, the main CPU goes to sleep mode to lower power consumption. Meanwhile, the coprocessor is woken up by ULP timer, and then monitors the external environment or interacts with the external circuit by controlling peripherals such as RTC GPIO, RTC I2C, SAR ADC, or temperature sensor (TSENS). The coprocessor wakes the main CPU up once a wakeup condition is reached.

\begin{figure}[!ht]
    \centering
    \includegraphics[width=0.8\textwidth]{26-ULP/figures/26-ULP-ulp-overview.png}
    \caption{ULP Coprocessor Overview}
    \label{figure:coprocessor-overview}
\end{figure}

\chipname{} has two ULP coprocessors, with one based on RISC-V instruction set architecture (ULP-RISC-V) and the other on finite state machine (ULP-FSM). Users can choose between the two coprocessors depending on their needs.

\section{Features}

\begin{itemize}
    \item Access up to 8 KB of SRAM RTC slow memory for instructions and data
    \item Clocked with 17.5 MHz RTC\_FAST\_CLK
    \item Support working in normal mode and in monitor mode
    \item Wake up the CPU or send an interrupt to the CPU
    \item Access peripherals, internal sensors and RTC registers
\end{itemize}

ULP-FSM and ULP-RISC-V can not be used simultaneously. Users can only choose one of them as the ULP coprocessor of \chipname{}. The differences between the two coprocessors are shown in the table below.

\section{Programming Workflow}

The ULP-RISC-V is intended for programming using C language. The program in C is then compiled to\\ \href{https://riscv.org/specifications/}{RV32IMC} standard instruction code. \section{ULP Coprocessor Sleep and Wake-Up Workflow}

ULP coprocessor is designed to operate independently of the CPU, while the CPU is either in sleep or running.

In a typical power-saving scenario, the chip goes to Deep-sleep mode to lower power consumption. Before setting the chip to sleep mode, users should complete the following operations.

\begin{enumerate}
    \item Flash the program to be executed by ULP coprocessor into RTC slow memory.
    \item Select the working ULP coprocessor by configuring \hyperref[fielddesc:RTCCNTLCOCPUSEL]{RTC\_CNTL\_COCPU\_SEL}.
    \begin{itemize}
        \item 0: select ULP-RISC-V
        \item 1: select ULP-FSM
    \end{itemize}
    \item If ULP-RISC-V is selected as working ULP coprocessor, please

    \begin{itemize}
        \item set and reset \hyperref[fielddesc:RTCCNTLCOCPUCLKFO]{RTC\_CNTL\_COCPU\_CLK\_FO};
        \item set \hyperref[fielddesc:RTCCNTLCOCPUCLKGATEEN]{RTC\_CNTL\_COCPU\_CLKGATE\_EN}.
    \end{itemize}
    \item Set sleep cycles for the timer by configuring \hyperref[regdesc:RTCCNTLULPCPTIMER1REG]{RTC\_CNTL\_ULP\_CP\_TIMER\_1\_REG}.
    \item Enable the timer by software or by RTC GPIO;
        \begin{itemize}
            \item By software: set \hyperref[fielddesc:RTCCNTLULPCPSLPTIMEREN]{RTC\_CNTL\_ULP\_CP\_SLP\_TIMER\_EN}.
        \item By RTC GPIO: set \hyperref[fielddesc:RTCCNTLULPCPGPIOWAKEUPENA]{RTC\_CNTL\_ULP\_CP\_GPIO\_WAKEUP\_ENA}. For more information, see Chapter \ref{mod:lowpowm} \textit{\nameref{mod:lowpowm}}. 
        \end{itemize}
    \item Set the system into sleep mode.
\end{enumerate}

When the system is in Deep-sleep mode:

\begin{enumerate}
   \item The timer periodically sets the low-power controller (see Chapter \ref{mod:lowpowm} \textit{\nameref{mod:lowpowm}}) to Monitor mode and then wakes up the coprocessor.
   \item Coprocessor executes some necessary operations, such as monitoring external environment via low-power sensors.
   \item After the operations are finished, the system goes back to Deep-sleep mode.
   \item ULP coprocessor goes back to halt mode and waits for next wakeup.
\end{enumerate}

In monitor mode, ULP coprocessor is woken up and goes to halt as shown in Figure \ref{figure:ULP-program-control-flow}.

\begin{figure}[!ht ]
    \centering
    \includegraphics[width=0.8\textwidth]{./26-ULP/figures/26-ULP-ulp-exec-control-flow.eps}
    \caption{ULP Sleep and Wakeup Sequence}
    \label{figure:ULP-program-control-flow}
\end{figure}

\begin{enumerate}
    \item Enable the timer and the timer starts counting.
    \item The timer expires and wakes up the ULP coprocessor. ULP coprocessor starts running and executes the program flashed in RTC slow memory.
    \item ULP coprocessor goes to halt and the timer starts counting again.
    \begin{itemize}
        \item Put ULP-RISC-V into HALT: set \hyperref[fielddesc:RTCCNTLCOCPUDONE]{RTC\_CNTL\_COCPU\_DONE}.
        \item Put ULP-FSM into HALT: execute HALT instruction.
    \end{itemize}
    \item Disable the timer by ULP program or by software. ULP coprocessor exits from monitor mode.
    \begin{itemize}
        \item Disabled by software: clear \hyperref[fielddesc:RTCCNTLULPCPSLPTIMEREN]{RTC\_CNTL\_ULP\_CP\_SLP\_TIMER\_EN}.
        \item Disabled by RTC GPIO: clear \hyperref[fielddesc:RTCCNTLULPCPGPIOWAKEUPENA]{RTC\_CNTL\_ULP\_CP\_GPIO\_WAKEUP\_ENA} and set \hyperref[fielddesc:RTCCNTLULPCPGPIOWAKEUPCLR]{RTC\_CNTL\_ULP\_CP\_GPIO\_WAKEUP\_CLR}.
    \end{itemize}
\end{enumerate}

\textbf{Note:}
\begin{itemize}
    \item If the timer is enabled by software (RTC GPIO), it should be disabled by software (RTC GPIO).
    \item Before setting ULP-RISC-V to HALT, users should configure \hyperref[fielddesc:RTCCNTLCOCPUDONE]{RTC\_CNTL\_COCPU\_DONE} first, therefore, it is recommended to end the flashed program with the following pattern:
    \begin{itemize}
        \item Set \hyperref[fielddesc:RTCCNTLCOCPUDONE]{RTC\_CNTL\_COCPU\_DONE} to end the operation of ULP-RISC-V and put it into halt;
        \item Set \hyperref[fielddesc:RTCCNTLCOCPUSHUTRESETEN]{RTC\_CNTL\_COCPU\_SHUT\_RESET\_EN} to reset ULP-RISC-V.
    \end{itemize}
    Enough time is reserved for the ULP-RISC-V to complete the operations above before it goes to halt.
\end{itemize}

\section{ULP-FSM} \label{ulp-fsm}
\subsection{Features}
ULP-FSM is a programmable finite state machine that can work while the main CPU is in Deep-sleep. ULP-FSM supports instructions for complex logic and arithmetic operations, and also provides dedicated instructions for RTC controllers or peripherals. ULP-FSM can access up to 8 KB of SRAM RTC slow memory (accessible by the CPU) for instructions and data. Hence, such memory is usually used to store instructions and share data between the ULP coprocessor and the CPU. ULP-FSM can be stopped by running HALT instruction.

ULP-FSM has the following features.

\begin{itemize}
    \item Provides four 16-bit general-purpose registers (R0, R1, R2, and R3) for manipulating data and accessing memory.
    \item Provides one 8-bit stage count register (Stage\_cnt) which can be manipulated by ALU and used in JUMP instructions.
    \item Supports built-in instructions specially for direct control of low-power peripherals, such as SAR ADC and temperature sensor.
\end{itemize}

\subsection{Instruction Set} \label{section:instruction-set}
ULP-FSM supports the following instructions.

\begin{itemize}
    \item ALU: perform arithmetic and logic operations
    \item LD, ST, REG\_RD and REG\_WR: load and store data
    \item JUMP: jump to a certain address
    \item WAIT/HALT: manage program execution
    \item WAKE: wake up CPU or communicate with CPU
    \item TSENS and ADC: take measurements
\end{itemize}

Figure \ref{figure:ULP-instruction} shows the format of ULP-FSM instructions.

An instruction, which has one OpCode, can perform various operations, depending on the setting of Operands bits. A good example is the ALU instruction, which is able to perform 10 arithmetic and logic operations; or the JUMP instruction, which may be conditional or unconditional, absolute or relative.

Each instruction has a fixed width of 32 bits. A series of instructions can make a program be executed by the coprocessor. The execution flow inside the program uses 32-bit addressing. The program is stored in a dedicated region called Slow Memory, which is visible to the main CPU under an address range of 0x{}5000\_0000 to 0x{}5000\_1FFF (8 KB).

\subsubsection{ALU - Perform Arithmetic and Logic Operations}

ALU (Arithmetic and Logic Unit) performs arithmetic and logic operations on values stored in ULP coprocessor registers, and on immediate values stored in the instruction itself. The following operations are supported.
\begin{itemize}
    \item Arithmetic: ADD and SUB
    \item Logic: bitwise logical AND and bitwise logical OR
    \item Bit shifting: LSH and RSH
    \item Moving data to register: MOVE
    \item PC register operations - STAGE\_RST, STAGE\_INC, and STAGE\_DEC
\end{itemize}

The ALU instruction, which has one OpCode (7), can perform various arithmetic and logic operations, depending on the setting of the instruction bits [27:21].

    \item All ISR entries are located at 0x{}10, and the reset entry is at 0x{}0.

\subsubsection{Interrupt Handling}

When an interrupt occurs, ULP-RISC-V performs the following operations:

\begin{enumerate}
    \item Saves the current PC to Q0.
    \item Saves the interrupt being responded to in Q1.
    \item Jumps to the interrupt entry point (0x{}10).
\end{enumerate}

After ULP-RISC-V enters the interrupt entry point, please save the context and use Q2 and Q3 as temporary storage registers for backup. Before it exits the interrupt, restore the context. Then, exit the interrupt by executing the retirq instruction and jump to Q0. ULP does not support interrupt nesting and is not interrupted by any other interrupt before retirq is executed.

\section{RTC I2C Controller} \label{subsection:RTC_I2C-Controller}

ULP coprocessor reads from or writes to external I2C slave devices via RTC I2C controller.

\subsection{Connecting RTC I2C Signals}

SDA and SCL signals can be mapped onto two out of the four GPIO pins, which are identified in Table RTC\_MUX Pad List in Chapter \ref{mod:iomuxgpio} \textit{\nameref{mod:iomuxgpio}}, using the register \hyperref[regdesc:RTCIOSARI2CIOREG]{RTCIO\_SAR\_I2C\_IO\_REG}.

\subsection{Configuring RTC I2C}

Before ULP coprocessor can communicate using I2C instruction, RTC I2C need to be configured. Configuration is performed by writing certain timing parameters into the RTC I2C registers. This can be done by the program running on the main CPU, or by the ULP coprocessor itself.
\begin{tiplisting}
The timing parameters are configured in cycles of RTC\_FAST\_CLK running at 17.5 MHz.
\end{tiplisting}
\begin{enumerate}
    \item Set the low and high SCL half-periods by configuring \hyperref[fielddesc:RTCI2CSCLLOWPERIODREG]{RTC\_I2C\_SCL\_LOW\_PERIOD\_REG} and \hyperref[fielddesc:RTCI2CSCLHIGHPERIODREG]{RTC\_I2C\_SCL\\\_HIGH\_PERIOD\_REG} in RTC\_FAST\_CLK cycles (e.g., \hyperref[fielddesc:RTCI2CSCLLOWPERIODREG]{RTC\_I2C\_SCL\_LOW\_PERIOD\_REG} = 40, \hyperref[fielddesc:RTCI2CSCLHIGHPERIODREG]{RTC\_I2C\_\\SCL\_HIGH\_PERIOD\_REG} = 40 for 100 kHz frequency).
    \item Set the number of cycles between the SDA switch and the falling edge of SCL by using \hyperref[regdesc:RTCI2CSDADUTYREG]{RTC\_I2C\_SDA\_DUTY\\\_REG} in RTC\_FAST\_CLK (e.g., \hyperref[regdesc:RTCI2CSDADUTYREG]{RTC\_I2C\_SDA\_DUTY\_REG} = 16).
    \item Set the waiting time after the START signal by using \hyperref[regdesc:RTCI2CSCLSTARTPERIODREG]{RTC\_I2C\_SCL\_START\_PERIOD\_REG} (e.g., \hyperref[fielddesc:RTCI2CSCLSTARTPERIOD]{RTC\_I2C\\\_SCL\_START\_PERIOD} = 30).
    \item Set the waiting time before the END signal by using \hyperref[regdesc:RTCI2CSCLSTOPPERIODREG]{RTC\_I2C\_SCL\_STOP\_PERIOD\_REG} (e.g., \hyperref[fielddesc:RTCI2CSCLSTOPPERIOD]{RTC\_I2C\_\\SCL\_STOP\_PERIOD} = 44).
    \item Set the transaction timeout by using  \hyperref[fielddesc:RTCI2CTIMEOUTREG]{RTC\_I2C\_TIME\_OUT\_REG} (e.g., \hyperref[fielddesc:RTCI2CTIMEOUTREG]{RTC\_I2C\_TIME\_OUT\_REG} = 200).
    \item Configure the RTC I2C controller into master mode by setting the bit \hyperref[fielddesc:RTCI2CMSMODE]{RTC\_I2C\_MS\_MODE} in \hyperref[regdesc:RTCI2CCTRLREG]{RTC\_I2C\_CTRL\\\_REG}.
    \item Configure the address(es) of external slave(s):
    \begin{itemize}
        \item If ULP-RISC-V or main CPU is used, then write the slave address to \hyperref[fielddesc:SENSSARI2CSLAVEADDR]{SENS\_SAR\_I2C\_SLAVE\_ADDR}.
        \item If ULP-FSM is used, then write the slave address to SENS\_I2C\_SLAVE\_ADDR\regindex{{\it{n}}} (\regindex{{\it{n}}}: 0-7)
    \end{itemize} 
    Up to eight slave addresses can be pre-programmed. One of these addresses can then be selected for each transaction as part of the RTC I2C instruction.
\end{enumerate}

Once RTC I2C is configured, the main CPU or the ULP coprocessor can communicate with the external I2C devices.

\subsection{Using RTC I2C}
\subsubsection{Instruction Format}

The format of RTC I2C instruction is basically consistent with that of I2C0/I2C1, see Section I2C CMD Controller 
in Chapter \ref{mod:i2c} \textit{\nameref{mod:i2c}} except the following:
\begin{itemize}
    \item RTC I2C has different op\_code mapping:
    \begin{itemize}
        \item RSTART: op\_code = 0
        \item WRITE: op\_code = 1
        \item READ: op\_code = 2
        \item STOP: op\_code = 3
        \item END: op\_code = 4
    \end{itemize}
    \item RTC I2C provides fixed instructions for different operations, as follows:
    \begin{itemize}
        \item Command 0 \verb+~+ Command 1: specifically for I2C write operation
        \item Command 2 \verb+~+ Command 6: specifically for I2C read operation
    \end{itemize}
\end{itemize}

\textbf{Note:}
All slave addresses are expressed in 7 bits.


\subsubsection{I2C\_RD - I2C Read Workflow}

Preparation for RTC I2C read:

\begin{itemize}
    \item Configure the instruction list of RTC I2C (see Section CMD\_Controller in Chapter \ref{mod:i2c} \textit{\nameref{mod:i2c}}), including instruction order, instruction code, read data number (byte\_num), and other information.
    \item Configure the slave register address by setting the register \hyperref[fielddesc:SENSSARI2CREGADDR]{SENS\_SAR\_I2C\_REG\_ADDR}.
    \item Start RTC I2C transmission by setting \hyperref[fielddesc:SENSSARI2CSTARTFORCE]{SENS\_SAR\_I2C\_START\_FORCE} and \hyperref[fielddesc:SENSSARI2CSTART]{SENS\_SAR\_I2C\_START}.
    \item When an RTC\_I2C\_RX\_DATA\_INT interrupt is received, transfer the read data stored in \hyperref[fielddesc:RTCI2CI2CRDATA]{RTC\_I2C\_RDATA} to SRAM RTC slow memory, or use the data directly.
\end{itemize}


The I2C\_RD instruction performs the following operations (see Figure \ref{figure:I2C-read-operation}):

\begin{enumerate}
    \item Master generates a START signal.
    \item Master sends slave address, with r/w bit set to 0 (``write"). Slave address is obtained from  SENS\_I2C\_SLAVE\_ADDR\regindex{n}.
    \item Slave generates ACK.
    \item Master sends slave register address.
    \item Slave generates ACK.
    \item Master generates a repeated START (RSTART) signal.
    \item Master sends slave address, with r/w bit set to 1 (``read").
    \item Slave sends one byte of data.
    \item Master checks whether the number of transmitted bytes reaches the number set by the current instruction (byte\_num). If yes, master jumps out of the read instruction and sends an NACK signal. Otherwise master repeats Step 8 and waits for the slave to send the next byte.
    \item Master generates a STOP signal and stops reading.
\end{enumerate}

\subsubsection{I2C\_WR - I2C Write Workflow}

Preparation for RTC I2C write:

\begin{itemize}
    \item Configure RTC I2C instruction list, including instruction order, instruction code, and the data to be written in byte (byte\_num). See the configuration of I2C0/I2C1 in Section CMD\_Controller in Chapter \ref{mod:i2c} \textit{\nameref{mod:i2c}}.
    \item Configure the slave register address by setting the register \hyperref[fielddesc:SENSSARI2CREGADDR]{SENS\_SAR\_I2C\_REG\_ADDR}, and the data to be transmitted in \hyperref[fielddesc:SENSSARI2CWDATA]{SENS\_SAR\_I2C\_WDATA}.
    \item Set \hyperref[fielddesc:SENSSARI2CSTARTFORCE]{SENS\_SAR\_I2C\_START\_FORCE} and \hyperref[fielddesc:SENSSARI2CSTART]{SENS\_SAR\_I2C\_START} to start the transmission.
    \item Update the next data to be transmitted in \hyperref[fielddesc:SENSSARI2CWDATA]{SENS\_SAR\_I2C\_WDATA}, each time when an \hyperref[interrupt:RTCI2CTXDATAINT]{RTC\_I2C\_TX\_DATA\_INT} interrupt is received.
\end{itemize}

The I2C\_WR instruction performs the following operations, see Figure \ref{figure:I2C-write-operation}.

\begin{enumerate}
    \item Master generates a START signal.
    \item Master sends slave address, with r/w bit set to 0 (``write"). Slave address is obtained from  SENS\_I2C\_SLAVE\_ADDR\regindex{n}.
    \item Slave generates ACK.
    \item Master sends slave register address.
    \item Slave generates ACK.
    \item Master generates a repeated START (RSTART) signal.
    \item Master sends slave address, with r/w bit set to 0 (``write").
    \item Master sends one byte of data.
    \item Slave generates ACK. Master checks whether the number of transmitted bytes reaches the number set by the current instruction (byte\_num). If yes, master jumps out of the write instruction and starts the next instruction. Otherwise the master repeats Step 8 and sends the next byte.
    \item Master generates a STOP signal and stops the transmission.
\end{enumerate}

\subsubsection{Detecting Error Conditions} \label{paragraph:ULP-detecting-errors}
Applications can query specific bits in the \hyperref[regdesc:RTCI2CINTSTREG]{RTC\_I2C\_INT\_ST\_REG} register to check if the transaction is successful. To enable checking for specific communication events, their corresponding bits should be set in register \hyperref[regdesc:RTCI2CINTENAREG]{RTC\_I2C\_INT\_ENA\_REG}. \textcolor{red}{Note that the bit map is shifted by 1.} If a specific communication event is detected and its corresponding bit in register  \hyperref[regdesc:RTCI2CINTSTREG]{RTC\_I2C\_INT\_ST\_REG} is set, the event can then be cleared using register \hyperref[regdesc:RTCI2CINTCLRREG]{RTC\_I2C\_INT\_CLR\_REG}.

    \item ULP (ALWAYS\_ON) registers: not reset due to power down of RTC\_PERI domain. See Chapter \ref{mod:lowpowm} \textit{\nameref{mod:lowpowm}}.
    \item ULP (RTC\_PERI) registers: reset due to power down of RTC\_PERI domain. See Chapter \ref{mod:lowpowm} \textit{\nameref{mod:lowpowm}}.
    \item RTC I2C registers: I2C related registers, including RTC I2C (RTC\_PERI) and RTC I2C (I2C) registers.
\end{itemize}
