\hypertarget{uart}{}
\chapter{UART Controller (UART)}
\label{mod:uart}


\section{Overview}

In embedded system applications, data are required to be transferred in a simple way with minimal system resources. This can be achieved by a Universal Asynchronous Receiver/Transmitter (UART), which flexibly exchanges data with other peripheral devices in full-duplex mode. \chipname{} has three UART controllers compatible with various UART devices. They support Infrared Data Association (IrDA) and RS485 transmission.

Each of the three UART controllers has a group of registers that function identically. In this chapter, the three UART controllers are referred to as UART\regindex{n}, in which \regindex{n} denotes 0, 1, or 2.

A UART is a character-oriented data link for asynchronous communication between devices. Such communication does not provide any clock signal to send data. Therefore, in order to communicate successfully, the transmitter and the receiver must operate at the same baud rate with the same stop bit(s) and parity bit.

A UART data frame usually begins with one start bit, followed by data bits, one parity bit (optional) and one or more stop bits. UART controllers on \chipname{}  support various lengths of data bits and stop bits. These controllers also support software and hardware flow control as well as GDMA for seamless high-speed data transfer. This allows developers to use multiple UART ports at minimal software cost.

\section{Features}

Each UART controller has the following features:

\begin{itemize}
    \item Three clock sources that can be divided
    \item Programmable baud rate
    \item 1024 x 8-bit RAM shared by TX FIFOs and RX FIFOs of the three UART controllers
    \item Full-duplex asynchronous communication
    \item Automatic baud rate detection of input signals
    \item Data bits ranging from 5 to 8
    \item Stop bits of 1, 1.5, or 2  bits
    \item Parity bit
    \item Special character AT\_CMD detection
    \item RS485 protocol
    \item IrDA protocol
    \item High-speed data communication using GDMA
    \item UART as wake-up source
    \item Software and hardware flow control
\end{itemize}

\section{UART Structure}

\begin{figure}[H]
\includegraphics[width=0.6\textwidth]{\modulefiles/figures/UART-architecture-overview-20211123}
\centering
\caption{UART Architecture Overview}
\label{fig:uart-arch-overview}
\end{figure}

\begin{figure}[H]
\includegraphics[width=1.0\textwidth]{\modulefiles/figures/UART-uart-architecture-c3}
\centering
\caption{UART Structure}
\label{fig:one-uart-structure}
\end{figure}

Figure \ref{fig:one-uart-structure} shows the basic structure of a UART controller. A UART controller works in two clock domains, namely APB\_CLK domain and Core Clock domain (the UART Core's clock domain). The UART Core has three clock sources: 80 MHz APB\_CLK, RC\_FAST\_CLK and external crystal clock XTAL\_CLK (for details, please refer to Chapter \ref{mod:resclk} \textit{\nameref{mod:resclk}}), which are selected by configuring \hyperref[fielddesc:UARTSCLKSEL]{UART\_SCLK\_SEL}. The selected clock source is divided by a divider to generate clock signals that drive the UART
Core.

A UART controller is broken down into two parts: a transmitter and a receiver.

The transmitter contains a FIFO, called TX FIFO (or Tx\_FIFO), which buffers data to be sent. Software can write data to the Tx\_FIFO either via the APB bus, or using GDMA. Tx\_FIFO\_Ctrl controls writing and reading the Tx\_FIFO. When Tx\_FIFO is not empty, Tx\_FSM reads data bits in the data frame via Tx\_FIFO\_Ctrl, and converts them into a bitstream. The levels of output signal txd\_out can be inverted by configuring the \hyperref[fielddesc:UARTTXDINV]{UART\_TXD\_INV} field.

The receiver also contains a FIFO, called RX FIFO (or Rx\_FIFO), which buffers received data. The levels of input signal rxd\_in can be inverted by configuring \hyperref[fielddesc:UARTRXDINV]{UART\_RXD\_INV} field. Baudrate\_Detect measures the baud rate of input signal rxd\_in by detecting its minimum pulse width. Start\_Detect detects the start bit in a data frame. If the start bit is detected, Rx\_FSM stores data bits in the data frame into Rx\_FIFO by Rx\_FIFO\_Ctrl. Software can read data from Rx\_FIFO via the APB bus, or receive data using GDMA.

HW\_Flow\_Ctrl controls rxd\_in and txd\_out data flows by standard UART RTS and CTS flow control signals (rtsn\_out and ctsn\_in). SW\_Flow\_Ctrl controls data flows by automatically adding special characters to outgoing data and detecting special characters in incoming data.

When a UART controller is in Light-sleep mode (see Chapter \ref{mod:lowpowm} \textit{\nameref{mod:lowpowm}} for more details), Wakeup\_Ctrl counts up rising edges of rxd\_in. When the number is equal to or greater than \hyperref[fielddesc:UARTACTIVETHRESHOLD]{UART\_ACTIVE\_THRESHOLD} + 3, a wake\_up signal is generated and sent to RTC, which then wakes up the \chipname{}  chip.

\section{Functional Description}

\subsection{Clock and Reset}

UART controllers are asynchronous. their configuration registers, TX FIFOs, and RX FIFOs are in APB\_CLK domain, while the module controlling transmission and reception (i.e., UART Core) is in Core Clock domain. The latter can be sourced out of three clocks, namely APB\_CLK, RC\_FAST\_CLK and external crystal clock XTAL\_CLK, which can be selected by configuring \hyperref[fielddesc:UARTSCLKSEL]{UART\_SCLK\_SEL}. The selected clock source can be divided. This divider supports fractional division, and the divisor is equal to:
\[
\hyperref[fielddesc:UARTSCLKDIVNUM]{UART\_SCLK\_DIV\_NUM} + \frac{\hyperref[fielddesc:UARTSCLKDIVB]{UART\_SCLK\_DIV\_B}}{\hyperref[fielddesc:UARTSCLKDIVA]{UART\_SCLK\_DIV\_A}}
\]
The divisor ranges from 1 \verb+~+ 256.

When the frequency of the UART Core's clock is higher than the frequency needed to generate the baud rate, the UART Core can be clocked at a lower frequency by the divider, in order to reduce power consumption. Usually, the UART Core's clock frequency is lower than the APB\_CLK's frequency, and can be divided by the largest divisor value when higher than the frequency needed to generate the baud rate. The frequency of the UART Core's clock can also be at most twice higher than the APB\_CLK. The clock for the UART transmitter and the UART receiver can be controlled independently. To enable the clock for the UART transmitter, \hyperref[fielddesc:UARTTXSCLKEN]{UART\_TX\_SCLK\_EN} shall be set; to enable the clock for the UART receiver, \hyperref[fielddesc:UARTRXSCLKEN]{UART\_RX\_SCLK\_EN} shall be set.

Section \ref{sec:uart-program} explains the procedure to ensure that the configured register values are synchronized between APB\_CLK domain and Core Clock domain.

Section \ref{sec:uart-initial} explains the procedure to reset the whole UART controller. Note that it is not recommended to only reset the APB clock domain module or UART Core.

\subsection{UART RAM}

\begin{figure}[H]
\includegraphics[width=0.4\textwidth]{\modulefiles/figures/uart_mem}
\centering
\caption{UART Controllers Sharing RAM}
\label{fig:uart-mem}
\end{figure}

All three UART controllers on \chipname{}  share 1024 × 8 bits of RAM. As Figure \ref{fig:uart-mem} illustrates, the RAM is divided into 8 blocks, each having 128 × 8 bits. Figure \ref{fig:uart-mem} shows how many RAM blocks are allocated by default to TX and RX FIFOs for each of the three UART controllers. UART\regindex{n} Tx\_FIFO can be expanded by configuring \hyperref[fielddesc:UARTTXSIZE]{UART\_TX\_SIZE}, while UART\regindex{n} Rx\_FIFO can be expanded by configuring \hyperref[fielddesc:UARTRXSIZE]{UART\_RX\_SIZE}.
Some limits are imposed:
\begin{itemize}
    \item UART0 Tx\_FIFO can be increased up to 8 blocks (the whole RAM);
    \item UART1 Tx\_FIFO can be increased up to 7 blocks (from offset 128 to the end address);
    \item UART2 Tx\_FIFO can be increased up to 6 blocks (from offset 256 to the end address);
    \item UART0 Rx\_FIFO can be increased up to 4 blocks (from offset 512 to the end address);
    \item UART1 Rx\_FIFO can be increased up to 3 blocks (from offset 640 to the end address);
    \item UART2 Rx\_FIFO can be increased up to 2 blocks (from offset 768 to the end address).
\end{itemize}
Please note that starting addresses of all FIFOs are fixed, so expanding one FIFO may take up the default space of other FIFOs. For example, by setting \hyperref[fielddesc:UARTTXSIZE]{UART\_TX\_SIZE} of UART0 to 2, the size of UART0 Tx\_FIFO is increased by 128 bytes (from offset 0 to offset 255). In this case, UART0 Tx\_FIFO takes up the default space for UART1 Tx\_FIFO, and UART1's transmitting function cannot be used as a result.

When neither of the three UART controllers is active, RAM can enter low-power mode by setting \hyperref[fielddesc:UARTMEMFORCEPD]{UART\_MEM\_FORCE\_PD}.

UART\regindex{n} Tx\_FIFO is reset by setting \hyperref[fielddesc:UARTTXFIFORST]{UART\_TXFIFO\_RST}. UART\regindex{n} Rx\_FIFO is reset by setting \hyperref[fielddesc:UARTRXFIFORST]{UART\_RXFIFO\_RST}.



The "empty" signal threshold for Tx\_FIFO is configured by setting \hyperref[fielddesc:UARTTXFIFOEMPTYTHRHD]{UART\_TXFIFO\_EMPTY\_THRHD}. When data stored in Tx\_FIFO is less than \hyperref[fielddesc:UARTTXFIFOEMPTYTHRHD]{UART\_TXFIFO\_EMPTY\_THRHD}, a UART\_TXFIFO\_EMPTY\_INT interrupt is generated. The "full" signal threshold for Rx\_FIFO is configured by setting \hyperref[fielddesc:UARTRXFIFOFULLTHRHD]{UART\_RXFIFO\_FULL\_THRHD}. When data stored in Rx\_FIFO is equal to or greater than \hyperref[fielddesc:UARTRXFIFOFULLTHRHD]{UART\_RXFIFO\_FULL\_THRHD}, a UART\_RXFIFO\_FULL\_INT interrupt is generated. In addition, when Rx\_FIFO receives more data than its capacity, a UART\_RXFIFO\_OVF\_INT interrupt is generated.

TX FIFO and RX FIFO can be accessed via the APB bus or GDMA. Access via the APB bus is performed through register \hyperref[regdesc:UARTFIFOREG]{UART\_FIFO\_REG}. You can put data into TX FIFO by writing \hyperref[fielddesc:UARTRXFIFORDBYTE]{UART\_RXFIFO\_RD\_BYTE}, and get data in RX FIFO by reading this exact same field. For access via GDMA, please refer to Section \ref{sec:uart-gdma-control}.

\subsection{Baud Rate Generation and Detection}

\subsubsection{Baud Rate Generation}

Before a UART controller sends or receives data, the baud rate should be configured by setting corresponding registers. The baud rate generator of a UART controller functions by dividing the input clock source. It can divide the clock source by a fractional amount. The divisor is configured by \hyperref[regdesc:UARTCLKDIVREG]{UART\_CLKDIV\_REG}: \hyperref[fielddesc:UARTCLKDIV]{UART\_CLKDIV} for the integer part, and \hyperref[fielddesc:UARTCLKDIVFRAG]{UART\_CLKDIV\_FRAG} for the fractional part. When using the 80 MHz input clock, the UART controller supports a maximum baud rate of 5 Mbaud.

The divisor of the baud rate is equal to
\[
    \hyperref[fielddesc:UARTCLKDIV]{UART\_CLKDIV} + \frac{\hyperref[fielddesc:UARTCLKDIVFRAG]{UART\_CLKDIV\_FRAG}}{16}
\]
meaning that the final baud rate is equal to
\[
    \frac{INPUT\_FREQ}{\hyperref[fielddesc:UARTCLKDIV]{UART\_CLKDIV} + \frac{\hyperref[fielddesc:UARTCLKDIVFRAG]{UART\_CLKDIV\_FRAG}}{16}}
\]
where INPUT\_FREQ is the frequency of UART Core's source clock. For example, if \hyperref[fielddesc:UARTCLKDIV]{UART\_CLKDIV} = 694 and \hyperref[fielddesc:UARTCLKDIVFRAG]{UART\_CLKDIV\_FRAG} = 7
then the divisor value is
\[
    694 + \frac{7}{16} = 694.4375
\]

When \hyperref[fielddesc:UARTCLKDIVFRAG]{UART\_CLKDIV\_FRAG} is 0, the baud rate generator is an integer clock divider where an output pulse is generated every
\hyperref[fielddesc:UARTCLKDIV]{UART\_CLKDIV} input pulses.

When \hyperref[fielddesc:UARTCLKDIVFRAG]{UART\_CLKDIV\_FRAG} is not 0, the divider is fractional and the output baud rate clock pulses are not strictly uniform.
As shown in Figure \ref{fig:uart-frag-div}, for every 16 output pulses, the frequency of some pulses is INPUT\_FREQ/(\hyperref[fielddesc:UARTCLKDIV]{UART\_CLKDIV} + 1), and the frequency of the other pulses is INPUT\_FREQ/\hyperref[fielddesc:UARTCLKDIV]{UART\_CLKDIV}. A total of \hyperref[fielddesc:UARTCLKDIVFRAG]{UART\_CLKDIV\_FRAG} output pulses are generated by dividing (\hyperref[fielddesc:UARTCLKDIV]{UART\_CLKDIV} + 1) input pulses, and the remaining
(16 - \hyperref[fielddesc:UARTCLKDIVFRAG]{UART\_CLKDIV\_FRAG}) output pulses are generated by dividing \hyperref[fielddesc:UARTCLKDIV]{UART\_CLKDIV} input pulses.

The output pulses are interleaved as shown in Figure \ref{fig:uart-frag-div} below, to make the output timing more uniform:

\begin{figure}[H]
\includegraphics[width=1.0\textwidth]{\modulefiles/figures/uart_frag_div_en.eps}
\centering
\caption{UART Controllers Division}
\label{fig:uart-frag-div}
\end{figure}

To support IrDA (see Section \ref{sub:uart-func-desc-irda} for details), the fractional clock divider for IrDA data transmission generates clock signals divided by 16 × \hyperref[regdesc:UARTCLKDIVREG]{UART\_CLKDIV\_REG}. This divider works similarly as the one elaborated above: it takes \hyperref[fielddesc:UARTCLKDIV]{UART\_CLKDIV}/16 as the integer value and the lowest four bits of \hyperref[fielddesc:UARTCLKDIV]{UART\_CLKDIV} as the fractional value.

\subsubsection{Baud Rate Detection}

Automatic baud rate detection (Autobaud) on UARTs is enabled by setting \hyperref[fielddesc:UARTAUTOBAUDEN]{UART\_AUTOBAUD\_EN}. The Baudrate\_Detect module shown in Figure \ref{fig:one-uart-structure} filters any noise whose pulse width is shorter than \hyperref[fielddesc:UARTGLITCHFILT]{UART\_GLITCH\_FILT}.

Before communication starts, the transmitter can send random data to the receiver for baud rate detection. \hyperref[fielddesc:UARTLOWPULSEMINCNT]{UART\_LOWPULSE\_MIN\_CNT} stores the minimum low pulse width, \hyperref[fielddesc:UARTHIGHPULSEMINCNT]{UART\_HIGHPULSE\_MIN\_CNT} stores the minimum high pulse width, \hyperref[fielddesc:UARTPOSEDGEMINCNT]{UART\_POSEDGE\_MIN\_CNT} stores the minimum pulse width between two rising edges, and \hyperref[fielddesc:UARTNEGEDGEMINCNT]{UART\_NEGEDGE\_MIN\_CNT} stores the minimum pulse width between two falling edges. These four fields are read by software to determine the transmitter's baud rate.

\begin{figure}[H]
\includegraphics[width=0.6\textwidth]{\modulefiles/figures/baud_det}
\centering
\caption{The Timing Diagram of Weak UART Signals Along Falling Edges}
\label{fig:baud-det}
\end{figure}

The baud rate can be determined in the following three ways:

\begin{enumerate}
\item Normally, to avoid sampling erroneous data along rising or falling edges in a metastable state, which results in the inaccuracy of \hyperref[fielddesc:UARTLOWPULSEMINCNT]{UART\_LOWPULSE\_MIN\_CNT} or \hyperref[fielddesc:UARTHIGHPULSEMINCNT]{UART\_HIGHPULSE\_MIN\_CNT}, use a weighted average of these two values to eliminate errors. In this case, the baud rate is calculated as follows:
    \[
    B_{\textrm{uart}}=\frac{f_{\textrm{clk}}} {\textrm{(\hyperref[fielddesc:UARTLOWPULSEMINCNT]{UART\_LOWPULSE\_MIN\_CNT} + \hyperref[fielddesc:UARTHIGHPULSEMINCNT]{UART\_HIGHPULSE\_MIN\_CNT} + 2)/2}}
    \]
\item If UART signals are weak along falling edges as shown in Figure \ref{fig:baud-det}, which leads to an inaccurate average of \hyperref[fielddesc:UARTLOWPULSEMINCNT]{UART\_LOWPULSE\_MIN\_CNT} and \hyperref[fielddesc:UARTHIGHPULSEMINCNT]{UART\_HIGHPULSE\_MIN\_CNT}, use \hyperref[fielddesc:UARTPOSEDGEMINCNT]{UART\_POSEDGE\_MIN\_CNT} to determine the transmitter's baud rate as follows:
    \[
    B_{\textrm{uart}}=\frac{f_{\textrm{clk}}} {\textrm{(\hyperref[fielddesc:UARTPOSEDGEMINCNT]{UART\_POSEDGE\_MIN\_CNT} + 1)/2}}
    \]
\item If UART signals are weak along rising edges, use \hyperref[fielddesc:UARTNEGEDGEMINCNT]{UART\_NEGEDGE\_MIN\_CNT} to determine the transmitter's baud rate as follows:
    \[
    B_{\textrm{uart}}=\frac{f_{\textrm{clk}}} {\textrm{(\hyperref[fielddesc:UARTNEGEDGEMINCNT]{UART\_NEGEDGE\_MIN\_CNT} + 1)/2}}
    \]
\end{enumerate}

\subsection{UART Data Frame}

\begin{figure}[H]
\includegraphics[width=0.9\textwidth]{\modulefiles/figures/uart_data_frame-2}
\centering
\caption{Structure of UART Data Frame}
\label{fig:uart-data-frame}
\end{figure}

Figure \ref{fig:uart-data-frame} shows the basic structure of a data frame. A frame starts with one START bit, and ends with STOP bits which can be 1, 1.5, or 2 bits long, configured by \hyperref[fielddesc:UARTSTOPBITNUM]{UART\_STOP\_BIT\_NUM} (in RS485 mode turnaround delay may be added. See details in Section \ref{sec:uart-turnaround-delay}). The START bit is logical low, whereas STOP bits are logical high.

The actual data length can be anywhere between 5 \verb+~+ 8 bits, configured by \hyperref[fielddesc:UARTBITNUM]{UART\_BIT\_NUM}. When \hyperref[fielddesc:UARTPARITYEN]{UART\_PARITY\_EN} is set, a parity bit is added after data bits. \hyperref[fielddesc:UARTPARITY]{UART\_PARITY} is used to choose even parity or odd parity. When the receiver detects a parity bit error in the data received, a UART\_PARITY\_ERR\_INT interrupt is generated, and the erroneous data are still stored into the RX FIFO. When the receiver detects a data frame error, a UART\_FRM\_ERR\_INT interrupt is generated, and the erroneous data by default is stored into the RX FIFO.

If all data in Tx\_FIFO have been sent, a UART\_TX\_DONE\_INT interrupt is generated. After this, if the \hyperref[fielddesc:UARTTXDBRK]{UART\_TXD\_BRK} bit is set then the transmitter will enter the Break condition and send several NULL characters in which the TX data line is logical low. The number of NULL characters is configured by \hyperref[fielddesc:UARTTXBRKNUM]{UART\_TX\_BRK\_NUM}. Once the transmitter has sent all NULL characters, a UART\_TX\_BRK\_DONE\_INT interrupt is generated. The minimum interval between data frames can be configured using \hyperref[fielddesc:UARTTXIDLENUM]{UART\_TX\_IDLE\_NUM}. If the transmitter stays idle for \hyperref[fielddesc:UARTTXIDLENUM]{UART\_TX\_IDLE\_NUM} or more time (in the unit of bit time, i.e., the time it takes to transfer one bit), a UART\_TX\_BRK\_IDLE\_DONE\_INT interrupt is generated.

The receiver can also detect the Break conditions when the RX data line remains logical low for one NULL character transmission, and a UART\_BRK\_DET\_INT interrupt will be triggered to detect that a Break condition has been completed.

The receiver can detect the current bus state through the timeout interrupt UART\_RXFIFO\_TOUT\_INT. The UART\_RXFIFO\_TOUT\_INT interrupt will be triggered when the bus is in the idle state for more than \hyperref[fielddesc:UARTRXTOUTTHRHD]{UART\_RX\_TOUT\_THRHD} bit time on current baud rate after the receiver has received at least one byte. You can use this interrupt to detect whether all the data from the transmitter has been sent.

\subsection{AT\_CMD Character Structure}

\begin{figure}[H]
\includegraphics[width=0.8\textwidth]{\modulefiles/figures/uart_at_cmd}
\centering
\caption{AT\_CMD Character Structure}
\label{fig:uart-at-cmd}
\end{figure}

Figure \ref{fig:uart-at-cmd} is the structure of a special character AT\_CMD. If the receiver constantly receives AT\_CMD\_CHAR and the following conditions are met, a UART\_AT\_CMD\_CHAR\_DET\_INT interrupt is generated. The specific value of AT\_CMD\_CHAR can be read from UART\regindex{n}\_AT\_CMD\_CHAR.

\begin{itemize}
    \item The interval between the first AT\_CMD\_CHAR and the last non-AT\_CMD\_CHAR character is at least \hyperref[fielddesc:UARTPREIDLENUM]{UART\_PRE\_IDLE\_NUM} cycles.
    \item The interval between two AT\_CMD\_CHAR characters is less than \hyperref[fielddesc:UARTRXGAPTOUT]{UART\_RX\_GAP\_TOUT} cycles.
    \item The number of AT\_CMD\_CHAR characters is equal to or greater than \hyperref[fielddesc:UARTCHARNUM]{UART\_CHAR\_NUM}.
    \item The interval between the last AT\_CMD\_CHAR character and next non-AT\_CMD\_CHAR character is at least \hyperref[fielddesc:UARTPOSTIDLENUM]{UART\_POST\_IDLE\_NUM} cycles.
\end{itemize}

\subsection{RS485}

All three UART controllers support RS485 protocol. This protocol uses differential signals to transmit data, so it can communicate over longer distances at higher bit rates than RS232. RS485 has two-wire half-duplex mode and four-wire full-duplex modes. UART controllers support two-wire half-duplex transmission and bus snooping. In a two-wire RS485 multidrop network, there can be 32 slaves at most.

\subsubsection{Driver Control}

As shown in Figure \ref{fig:rs485}, in a two-wire multidrop network, an external RS485 transceiver is needed for differential to single-ended conversion. An RS485 transceiver contains a driver and a receiver. When a UART controller is not in transmitter mode, the connection to the differential line can be broken by disabling the driver. When the DE (Driver Enable) signal is 1, the driver is enabled; when DE is 0, the driver is disabled.

The UART receiver converts differential signals to single-ended signals via an external receiver. RE is the enable control signal for the receiver. When RE is 0, the receiver is enabled; when RE is 1, the receiver is disabled. If RE is configured as 0, the UART controller is allowed to snoop data on the bus, including the data sent by itself.

DE can be controlled by either software or hardware. To reduce the cost of software, DE is controlled by hardware in our design. As shown in Figure \ref{fig:rs485}, DE is connected to dtrn\_out of UART (please refer to Section \ref{sec:uart-hd-flow-control} for more details).

\begin{figure}[H]
\includegraphics[width=0.5\textwidth]{\modulefiles/figures/rs485}
\centering
\caption{Driver Control Diagram in RS485 Mode}
\label{fig:rs485}
\end{figure}

\subsubsection{Turnaround Delay}\label{sec:uart-turnaround-delay}

By default, all three UART controllers work in receiver mode. When a UART controller is switched from transmitter mode to receiver mode, the RS485 protocol requires a turnaround delay of one cycle after the stop bit. The UART transmitter supports adding a turnaround delay of one cycle not only before the start bit but also after the stop bit. When \hyperref[fielddesc:UARTDL0EN]{UART\_DL0\_EN} is set, a turnaround delay of one cycle is added before the start bit; when \hyperref[fielddesc:UARTDL1EN]{UART\_DL1\_EN} is set, a turnaround delay of one cycle is added after the stop bit.

\subsubsection{Bus Snooping}

In a two-wire multidrop network,  UART controllers support bus snooping if RE of the external RS485 transceiver is 0. By default, a UART controller is not allowed to transmit and receive data simultaneously. If \hyperref[fielddesc:UARTRS485TXRXEN]{UART\_RS485TX\_RX\_EN} is set and the external RS485 transceiver is configured as in Figure \ref{fig:rs485}, a UART controller may receive data in transmitter mode and snoop the bus. If \hyperref[fielddesc:UARTRS485RXBYTXEN]{UART\_RS485RXBY\_TX\_EN} is set, a UART controller may transmit data in receiver mode.

All three UART controllers can snoop the data sent by themselves. In transmitter mode, when a UART controller monitors a collision between the data sent and the data received, a UART\_RS485\_CLASH\_INT interrupt is generated; when it monitors a data frame error, a UART\_RS485\_FRM\_ERR\_INT interrupt is generated; when it monitors a polarity error, a UART\_RS485\_PARITY\_ERR\_INT is generated.

\subsection{IrDA} \label{sub:uart-func-desc-irda}

IrDA protocol consists of three layers, namely the physical layer, the link access protocol, and the link management protocol. The three UART controllers implement IrDA's physical layer. In IrDA encoding, a UART controller supports data rates up to 115.2 kbit/s (SIR, or serial infrared mode). As shown in Figure \ref{fig:sir-endec}, the IrDA encoder converts a NRZ (non-return to zero code) signal to a RZI (return to zero inverted code) signal and sends it to the external driver and infrared LED. This encoder uses modulated signals whose pulse width is 3/16 bits to indicate logic ``0", and low levels to indicate logic ``1". The IrDA decoder receives signals from the infrared receiver and converts them to NRZ signals. In most cases, the receiver is high when it is idle, and the encoder output polarity is the opposite of the decoder input polarity. If a low pulse is detected, it indicates that a start bit has been received.

When IrDA function is enabled, one bit is divided into 16 clock cycles. If the bit to be sent is zero, then the 9th, 10th and 11th clock cycle are high.

\begin{figure}[H]
\includegraphics[width=0.8\textwidth]{\modulefiles/figures/sir_endec}
\centering
\caption{The Timing Diagram of Encoding and Decoding in SIR mode}
\label{fig:sir-endec}
\end{figure}


The IrDA transceiver is half-duplex, meaning that it cannot send and receive data simultaneously. As shown in Figure \ref{fig:irda_arc}, IrDA function is enabled by setting \hyperref[fielddesc:UARTIRDAEN]{UART\_IRDA\_EN}. When \hyperref[fielddesc:UARTIRDATXEN]{UART\_IRDA\_TX\_EN} is set (high), the IrDA transceiver is enabled to send data and not allowed to receive data; when \hyperref[fielddesc:UARTIRDATXEN]{UART\_IRDA\_TX\_EN} is reset (low), the IrDA transceiver is enabled to receive data and not allowed to send data.

\begin{figure}[H]
\includegraphics[width=0.6\textwidth]{\modulefiles/figures/IrDA_arc}
\centering
\caption{IrDA Encoding and Decoding Diagram}
\label{fig:irda_arc}
\end{figure}

\subsection{Wake-up}

UART0 and UART1 can be set as a wake-up source for Light-sleep mode. To be specific Wakeup\_Ctrl counts up the rising edges of rxd\_in, and when this count is equal to or greater than \hyperref[fielddesc:UARTACTIVETHRESHOLD]{UART\_ACTIVE\_THRESHOLD} + 3, a wake\_up signal is generated and sent to RTC, which then wakes \chipname{} up.

After the chip is woken up by UART, it is necessary to clear the wake\_up signal by transmitting data to UART in Active mode or resetting the whole UART, otherwise the number of rising edges required for the next wakeup will be reduced.

\subsection{Loopback Test}

UART\regindex{n} supports loopback testing, which can be enabled by setting \hyperref[fielddesc:UARTLOOPBACK]{UART\_LOOPBACK}. When loopback testing is enabled, UART output signal txd\_out is connected to its input signal rxd\_in, rtsn\_out is connected to ctsn\_in, and dtrn\_out is connected to dsrn\_out. Data are then sent out through txd\_out. If the data received match the data sent, it indicates that UART\regindex{n} controller is working properly.

\subsection{Flow Control}

UART controllers have two ways to control data flow, namely hardware flow control and software flow control. Hardware flow control is achieved using output signal rtsn\_out and input signal dsrn\_in. Software flow control is achieved by inserting special characters (XON or XOFF) in the data flow sent and detecting special characters in the data flow received.

\subsubsection{Hardware Flow Control}\label{sec:uart-hd-flow-control}

\includegraphics[width=0.7\textwidth]{\modulefiles/figures/UART-hardware-flow-control-20211123}
\centering
\caption{Hardware Flow Control Diagram}
\label{fig:uart-flow-control}
\end{figure}

Figure \ref{fig:uart-flow-control} shows the hardware flow control of a UART controller. Hardware flow control uses output signal rtsn\_out and input signal dsrn\_in. Figure \ref{fig:hw-flow-control-connect} illustrates how these signals are connected between UART on \chipname{}  (hereinafter referred to as IU0) and the external UART (hereinafter referred to as EU0).

When rtsn\_out of IU0 is low, EU0 is allowed to send data; when rtsn\_out of IU0 is high, EU0 is notified to stop sending data until rtsn\_out of IU0 returns to low. The output signal rtsn\_out can be controlled in two ways.

\begin{itemize}
    \item Software control: Enter this mode by clearing \hyperref[fielddesc:UARTRXFLOWEN]{UART\_RX\_FLOW\_EN} to 0. In this mode, the level of rtsn\_out is changed by configuring \hyperref[fielddesc:UARTSWRTS]{UART\_SW\_RTS}.
    \item Hardware control: Enter this mode by setting \hyperref[fielddesc:UARTRXFLOWEN]{UART\_RX\_FLOW\_EN} to 1. In this mode, rtsn\_out is pulled high when data in Rx\_FIFO exceeds \hyperref[fielddesc:UARTRXFLOWTHRHD]{UART\_RX\_FLOW\_THRHD}.
\end{itemize}

\begin{figure}[H]
\includegraphics[width=0.55\textwidth]{\modulefiles/figures/hardware_flow_control_connect_figure}
\centering
\caption{Connection between Hardware Flow Control Signals}
\label{fig:hw-flow-control-connect}
\end{figure}

When ctsn\_in of IU0 is low, IU0 is allowed to send data; when ctsn\_in is high, IU0 is not allowed to send data. When IU0 detects an edge change on ctsn\_in, a UART\_CTS\_CHG\_INT interrupt is generated.

If dtrn\_out of IU0 is high, it indicates that IU0 is ready to transmit data. dtrn\_out is generated by configuring the \hyperref[fielddesc:UARTSWDTR]{UART\_SW\_DTR} field. When the IU0 transmitter detects a edge change on dsrn\_in, a UART\_DSR\_CHG\_INT interrupt is generated. After this interrupt is detected, software can obtain the level of input signal dsrn\_in by reading \hyperref[fielddesc:UARTDSRN]{UART\_DSRN}. If dsrn\_in is high, it indicates that EU0 is ready to transmit data.

In a two-wire RS485 multidrop network enabled by setting \hyperref[fielddesc:UARTRS485EN]{UART\_RS485\_EN}, dtrn\_out is generated by hardware and used for transmit/receive turnaround. When data transmission starts, dtrn\_out is pulled high and the external driver is enabled; when data transmission completes, dtrn\_out is pulled low and the external driver is disabled. Please note that when there is a turnaround delay of one cycle added after the stop bit, dtrn\_out is pulled low after the delay.



\subsubsection{Software Flow Control}

Instead of CTS/RTS lines, software flow control uses XON/XOFF characters to start or stop data transmission. Such flow control can be enabled by setting \hyperref[fielddesc:UARTSWFLOWCONEN]{UART\_SW\_FLOW\_CON\_EN} to 1.

When choosing software flow control, the hardware automatically detects if XON and XOFF characters are used in the data flow, and generates a UART\_SW\_XOFF\_INT or a UART\_SW\_XON\_INT interrupt accordingly. When XOFF character is detected, the transmitter stops data transmission once the current byte has been transmitted; when XON character is detected, the transmitter starts data transmission. In addition, software can force the transmitter to stop sending data or to start sending data by setting respectively \hyperref[fielddesc:UARTFORCEXOFF]{UART\_FORCE\_XOFF} or \hyperref[fielddesc:UARTFORCEXON]{UART\_FORCE\_XON}.

Software determines whether to insert flow control characters according to the remaining room in the RX FIFO. When \hyperref[fielddesc:UARTSENDXOFF]{UART\_SEND\_XOFF} is set, the transmitter sends an XOFF character configured by \hyperref[fielddesc:UARTXOFFCHAR]{UART\_XOFF\_CHAR} after the current byte in transmission; when \hyperref[fielddesc:UARTSENDXON]{UART\_SEND\_XON} is set, the transmitter sends an XON character configured by \hyperref[fielddesc:UARTXONCHAR]{UART\_XON\_CHAR} after the current byte in transmission. If the RX FIFO of a UART controller stores more data than \hyperref[fielddesc:UARTXOFFTHRESHOLD]{UART\_XOFF\_THRESHOLD}, \hyperref[fielddesc:UARTSENDXOFF]{UART\_SEND\_XOFF} is set by hardware. As a result, the transmitter sends an XOFF character configured by \hyperref[fielddesc:UARTXOFFCHAR]{UART\_XOFF\_CHAR} after the current byte in transmission. If the RX FIFO of a UART controller stores less data than \hyperref[fielddesc:UARTXONTHRESHOLD]{UART\_XON\_THRESHOLD}, \hyperref[fielddesc:UARTSENDXON]{UART\_SEND\_XON} is set by hardware. As a result, the transmitter sends an XON character configured by \hyperref[fielddesc:UARTXONCHAR]{UART\_XON\_CHAR} after the current byte in transmission.

\subsection{GDMA Mode}\label{sec:uart-gdma-control}

All three UART controllers on \chipname{}  share one TX/RX GDMA (general direct memory access) channel via UHCI. In GDMA mode, UART controllers support the decoding and encoding of HCI data packets. The UHCI\_UART\regindex{n}\_CE field determines which UART controller occupies the GDMA TX/RX channel.

\begin{figure}[H]
\includegraphics[width=0.6\textwidth]{\modulefiles/figures/UDMA}
\centering
\caption{Data Transfer in GDMA Mode}
\label{fig:uart-gdma}
\end{figure}

Figure \ref{fig:uart-gdma} shows how data are transferred using GDMA. Before GDMA receives data, software prepares an inlink (i.e., a linked list of receive descriptors. For details, see Chapter \ref{mod:dma} \nameref{mod:dma}). \hyperref[fielddesc:DMAINLINKADDRCHN]{GDMA\_INLINK\_ADDR\_CH\regindex{n}} points to the first receive descriptor in the inlink. After \hyperref[fielddesc:DMAINLINKSTARTCHN]{GDMA\_INLINK\_START\_CH\regindex{n}} is set, UHCI passes data that UART has received to the decoder. The decoded data are then stored into the RAM pointed by the inlink under the control of GDMA.

Before GDMA sends data, software prepares an outlink and data to be sent. \hyperref[fielddesc:DMAOUTLINKADDRCHN]{GDMA\_OUTLINK\_ADDR\_CH\regindex{n}} points to the first transmit descriptor in the outlink. After \hyperref[fielddesc:DMAOUTLINKSTARTCHN]{GDMA\_OUTLINK\_START\_CH\regindex{n}} is set, GDMA reads data from the RAM pointed by outlink. The data are then encoded by the encoder, and sent sequentially by the UART transmitter.

HCI data packets have separators at the beginning and the end, with data bits in the middle (separators + data bits + separators). The encoder inserts separators in front of and after data bits, and replaces data bits identical to separators with special characters (i.e., escape characters). The decoder removes separators in front of and after data bits, and replaces escape characters with separators. There can be more than one continuous separator at the beginning and the end of a data packet. The separator is configured by \hyperref[fielddesc:UHCISEPERCHAR]{UHCI\_SEPER\_CHAR}, 0x{}C0 by default. The escape characters are configured by \hyperref[fielddesc:UHCIESCSEQ0CHAR0]{UHCI\_ESC\_SEQ0\_CHAR0} (0x{}DB by default) and \hyperref[fielddesc:UHCIESCSEQ0CHAR1]{UHCI\_ESC\_SEQ0\_CHAR1} (0x{}DD by default). When all data have been sent, a GDMA\_OUT\_TOTAL\_EOF\_CH\regindex{n}\_INT interrupt is generated. When all data have been received, a GDMA\_IN\_SUC\_EOF\_CH\regindex{n}\_INT is generated.

\section{Programming Procedures}\label{sec:uart-program}

\subsection{Register Type}

All UART registers are in APB\_CLK domain. According to whether clock domain crossing and synchronization are required, UART registers that can be configured by software are classified into three types, namely synchronous registers, static registers, and immediate registers. Synchronous registers are read in Core Clock domain, and take effect after synchronization. Static registers are also read in Core Clock domain, but would not change dynamically. Therefore, for static registers, clock domain crossing is not required, and software can turn on and off the clock for the UART transmitter or receiver to ensure that the configuration sampled in Core Clock domain is correct. Immediate registers are read in APB\_CLK domain, and take effect after being configured via the APB bus.

\subsubsection{Synchronous Registers}

Since synchronous registers are read in core clock domain, but written in APB\_CLK domain, they implement the clock domain crossing design to ensure that their values sampled in Core Clock domain are correct. These registers as listed in Table \ref{tab:uart-sync-reg} are configured as follows:

\begin{itemize}
    \item Enable register synchronization by clearing \hyperref[fielddesc:UARTUPDATECTRL]{UART\_UPDATE\_CTRL} to 0;
    \item Wait for \hyperref[fielddesc:UARTREGUPDATE]{UART\_REG\_UPDATE} to become 0, which indicates the completion of last synchronization;
    \item Configure synchronous registers;
    \item Synchronize the configured values to Core Clock domain by writing 1 to \hyperref[fielddesc:UARTREGUPDATE]{UART\_REG\_UPDATE}.
\end{itemize}

\subsubsection{Initializing UART\regindex{n}}\label{sec:uart-initial}

Initializing UART\regindex{n} requires two steps: resetting UART\regindex{n} and enabling register synchronization.

To reset UART\regindex{n}:

\begin{itemize}
    \item enable the clock for UART RAM by setting \hyperref[fielddesc:SYSTEMUARTMEMCLKEN]{SYSTEM\_UART\_MEM\_CLK\_EN} to 1;
    \item enable APB\_CLK for UART\regindex{n} by setting \hyperref[fielddesc:SYSTEMUARTCLKEN]{SYSTEM\_UART\regindex{n}\_CLK\_EN} to 1;
    \item clear \hyperref[fielddesc:SYSTEMUARTRST]{SYSTEM\_UART\regindex{n}\_RST};
    \item write 1 to \hyperref[fielddesc:UARTRSTCORE]{UART\_RST\_CORE};
    \item write 1 to \hyperref[fielddesc:SYSTEMUARTRST]{SYSTEM\_UART\regindex{n}\_RST};
    \item clear \hyperref[fielddesc:SYSTEMUARTRST]{SYSTEM\_UART\regindex{n}\_RST};
    \item clear \hyperref[fielddesc:UARTRSTCORE]{UART\_RST\_CORE}.
\end{itemize}

To enable register synchronization, clear \hyperref[fielddesc:UARTUPDATECTRL]{UART\_UPDATE\_CTRL}.


\subsubsection{Configuring UART\regindex{n} Communication}

To configure UART\regindex{n} communication:

\begin{itemize}
    \item wait for \hyperref[fielddesc:UARTREGUPDATE]{UART\_REG\_UPDATE} to become 0, which indicates the completion of the last synchronization;
    \item configure static registers (if any) following Section \ref{sec:uart-static-reg};
    \item select the clock source via \hyperref[fielddesc:UARTSCLKSEL]{UART\_SCLK\_SEL};
    \item configure divisor of the divider via \hyperref[fielddesc:UARTSCLKDIVNUM]{UART\_SCLK\_DIV\_NUM}, \hyperref[fielddesc:UARTSCLKDIVA]{UART\_SCLK\_DIV\_A}, and \hyperref[fielddesc:UARTSCLKDIVB]{UART\_SCLK\_DIV\_B};
    \item configure the baud rate for transmission via \hyperref[fielddesc:UARTCLKDIV]{UART\_CLKDIV} and \hyperref[fielddesc:UARTCLKDIVFRAG]{UART\_CLKDIV\_FRAG};
    \item configure data length via \hyperref[fielddesc:UARTBITNUM]{UART\_BIT\_NUM};
    \item configure odd or even parity check via \hyperref[fielddesc:UARTPARITYEN]{UART\_PARITY\_EN} and \hyperref[fielddesc:UARTPARITY]{UART\_PARITY};
    \item optional steps depending on application ...
    \item synchronize the configured values to the Core Clock domain by writing 1 to \hyperref[fielddesc:UARTREGUPDATE]{UART\_REG\_UPDATE}.
\end{itemize}

\subsubsection{Enabling UART\regindex{n}}

To enable UART\regindex{n} transmitter:

\begin{itemize}
    \item configure the TX FIFO's empty threshold via \hyperref[fielddesc:UARTTXFIFOEMPTYTHRHD]{UART\_TXFIFO\_EMPTY\_THRHD};
    \item disable UART\_TXFIFO\_EMPTY\_INT interrupt by clearing \hyperref[fielddesc:UARTTXFIFOEMPTYINTENA]{UART\_TXFIFO\_EMPTY\_INT\_ENA};
    \item write data to be sent to \hyperref[fielddesc:UARTRXFIFORDBYTE]{UART\_RXFIFO\_RD\_BYTE};
    \item clear UART\_TXFIFO\_EMPTY\_INT interrupt by setting  \hyperref[fielddesc:UARTTXFIFOEMPTYINTCLR]{UART\_TXFIFO\_EMPTY\_INT\_CLR};
    \item enable UART\_TXFIFO\_EMPTY\_INT interrupt by setting \hyperref[fielddesc:UARTTXFIFOEMPTYINTENA]{UART\_TXFIFO\_EMPTY\_INT\_ENA};
    \item detect UART\_TXFIFO\_EMPTY\_INT and wait for the completion of data transmission.
\end{itemize}

To enable UART\regindex{n} receiver:
\begin{itemize}
    \item configure RXFIFO's full threshold via \hyperref[fielddesc:UARTRXFIFOFULLTHRHD]{UART\_RXFIFO\_FULL\_THRHD};
    \item enable UART\_RXFIFO\_FULL\_INT interrupt by setting \hyperref[fielddesc:UARTRXFIFOFULLINTENA]{UART\_RXFIFO\_FULL\_INT\_ENA};
    \item detect UART\_TXFIFO\_FULL\_INT and wait until the RXFIFO is full;
    \item read data from RXFIFO via  \hyperref[fielddesc:UARTRXFIFORDBYTE]{UART\_RXFIFO\_RD\_BYTE}, and obtain the number of bytes received in RXFIFO via \hyperref[fielddesc:UARTRXFIFOCNT]{UART\_RXFIFO\_CNT}.
\end{itemize}
